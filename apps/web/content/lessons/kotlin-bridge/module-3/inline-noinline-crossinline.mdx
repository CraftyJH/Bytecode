You do not need to author **`inline`** APIs day one â€” you need to **recognize** them in stdlib signatures and know why stack traces sometimes show duplicated lines.

## Mental model

- **`inline`**: compiler copies the function body at call sites; lambdas become plain bytecode without an extra `Function` object (often).
- **`noinline`**: opt a lambda out when you must store it or pass it as a value.
- **`crossinline`**: lambda cannot use non-local `return` because it might execute in another execution context.

## Why it matters for Java devs

Library authors use **inline** to keep higher-order helpers (`with`, `use`, `measureTimeMillis`) cheap. When debugging, read source knowing calls may not appear as separate frames.

## Recap

- **`inline`** trades bytecode size for speed and enables non-local control flow rules with lambdas.
- **`noinline` / `crossinline`** fine-tune lambda parameters.
