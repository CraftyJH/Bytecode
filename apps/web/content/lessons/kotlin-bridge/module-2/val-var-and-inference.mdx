`val` and `var` split **reassignment**, not deep immutability. A `val` list can still be a `mutableListOf` under the hood (the **reference** is fixed, the list content may change). Prefer **read-only interfaces** in public APIs: expose `List<T>` and keep `MutableList` private.

<Note>Track baseline: Kotlin 2.x on the **JVM** with **Java 21**-aligned targets where we mention versions.</Note>

## Compared to Java

- Java: `final String name = "Ada";` ‚Üí Kotlin: `val name = "Ada"`.
- Java: `String name = "Ada"; name = "Bob";` ‚Üí Kotlin: `var name = "Ada"; name = "Bob"`.

## Type inference

Kotlin infers from the right-hand side:

```kotlin
val count = 10        // Int
val title = "Kotlin"  // String
```

Add types when it helps readers or the compiler:

```kotlin
val ids: List<Long> = emptyList()
```

## Common mistakes

- Using `var` everywhere out of Java habit ‚Äî default to `val` until you need mutation.
- Confusing **read-only view** (`List`) with **immutable data** ‚Äî Kotlin‚Äôs `listOf` returns a read-only list, not a mathematically immutable structure.

## Recap

- **`val`**: immutable reference; **`var`**: reassignable.
- **Inference** reduces noise; annotate public contracts and ambiguous sites.
