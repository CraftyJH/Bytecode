Kotlin collapses Java’s constructor boilerplate:

```kotlin
class User(val name: String, val id: Long)
```

This declares **properties** and assigns them from parameters automatically.

## `init` blocks

Use **`init { ... }`** for validation or derived setup:

```kotlin
class Email(raw: String) {
    val normalized: String

    init {
        require(raw.contains("@")) { "not an email" }
        normalized = raw.trim().lowercase()
    }
}
```

## Secondary constructors

Rare when defaults + factory helpers cover cases:

```kotlin
class Person(val name: String) {
    constructor(full: String, age: Int) : this("$full ($age)") { ... }
}
```

Prefer **named factory functions** (`fun Person.adult(...)`) when readability wins.

## Design notes for Java developers

A Kotlin primary constructor often replaces three Java artifacts at once:

- field declaration,
- constructor assignment,
- and boilerplate getter methods.

That is a readability gain, but the underlying JVM model still matters. You should still think in terms of:

- construction invariants (what must always be true),
- visibility (`private` constructor for factory-only creation),
- and side effects during initialization (`init` should stay lean).

If validation can fail, prefer failing immediately in `init` with clear messages (`require(...)`). This keeps invalid state from leaking into later logic.

## Java ↔ Kotlin mapping

```java
// Java
public final class User {
  private final String name;
  private final int id;
  public User(String name, int id) {
    this.name = name;
    this.id = id;
  }
}
```

```kotlin
// Kotlin
class User(val name: String, val id: Int)
```

Same domain model, significantly less ceremony.
