You do not need to become a Gradle expert to learn Kotlin, but you should know **where** the language plugs in. In the JVM world, most teams use **Gradle** (Groovy or **Kotlin DSL**) or **Maven** with the Kotlin plugin. The important idea: the **Kotlin compiler** runs as part of the build, emitting JVM bytecode like `javac`.

<Note>
  In this course, the **playground** compiles a single `main` for you. Locally, you would use your IDE (IntelliJ IDEA, Android Studio, or VS Code with extensions) to run the same `main` and see stack traces. Keep that mental model: playground ≈ one-off script, project ≈ full module path and tests.
</Note>

## Gradle mental model (high level)

- **Plugin**: `org.jetbrains.kotlin.jvm` (or a preset) turns on Kotlin compilation.
- **Source sets**: `src/main/kotlin` is conventional next to `src/main/java` when you mix languages in one module.
- **Toolchain / jvmTarget**: align Kotlin’s JVM target with the Java version you run in production (this track assumes **21** where it matters).

## REPL and scratch files

- **REPL**: Useful for trying expressions (` kotli-jvm` / IDEA scratch files). Great for learning APIs — not where you ship production code.
- **Scratch files / worksheets**: IntelliJ lets you execute snippets against module classpath — handy when exploring Java libraries from Kotlin.

## Compared to `javac` only

In Java-only projects you compile `.java` → `.class`. With Kotlin you compile `.kt` → `.class` as well, possibly alongside `.java` in the same module. The classpath is shared; Kotlin sees Java declarations (often as **platform types** when nullability is unknown).

## Common mistakes

- Forgetting to align **JVM target** between Kotlin and Java in mixed modules — subtle bytecode mismatches at runtime.
- Assuming Gradle Kotlin DSL is “another language” — it **is** Kotlin, just with Gradle APIs in scope.

## Recap

- The Kotlin **Gradle plugin** compiles `.kt` sources on the JVM alongside Java when configured.
- **`fun main()`** is the idiomatic entry point you will use in exercises.
- Local tooling adds debugging and classpath fidelity beyond the playground.
