If you already ship Java, you are not starting from zero. **Kotlin** is another language for the same runtime: the JVM, the same class files, the same libraries, and the same deployment story. The difference is ergonomic — fewer ceremonies for common tasks — and **null safety** woven into the type system instead of bolted on with annotations alone.

This track (**Kotlin Bridge**) assumes you are comfortable with Java basics through collections and objects. We compare idioms side by side so you can translate mental models, not memorize keywords in isolation.

<Note>
  **Technical baseline for this track:** Kotlin **2.x** language features where noted, **JVM target** aligned with **Java 21**, and the **Kotlin standard library** before frameworks. Execution in the playground uses Judge0’s Kotlin/JVM runtime — minor version differences may apply; lesson exercises stick to stable syntax.
</Note>

## What Kotlin adds on top of Java interop

- **Concision**: Data-heavy code often shrinks because getters, `equals`/`hashCode`, and builders collapse into language features like data classes and default arguments.
- **Nullability in types**: `String` vs `String?` is tracked by the compiler so many null-pointer failures become compile-time errors.
- **Expression-oriented control flow**: `if` and `when` can be expressions, which pairs well with `val` and single-assignment style.
- **Extension functions**: You can add namespaced helpers without subclassing (with different dispatch rules than Java instance methods — we cover that later).

## Migration expectations (realistic)

Kotlin is not magic and does not remove architectural complexity by itself. You still need:

- clear module boundaries,
- explicit domain models,
- and disciplined tests.

What Kotlin does improve is the signal-to-noise ratio. Teams usually see faster reviews because intent is clearer:

- nullability is explicit,
- mutability is explicit,
- and boilerplate does not hide business rules.

A practical migration pattern is **strangler-style adoption**:

1. Keep existing Java modules stable.
2. Add new features in Kotlin where possible.
3. Convert high-churn Java classes opportunistically.
4. Leave low-risk legacy code until there is business value to touch it.

What we **defer** to **Kotlin Advanced**: coroutines, multiplatform, compiler plugins, and deep reflection. You will see pointers so you know where to go next.

## How this track is organized

Each module maps a Java habit to a Kotlin idiom, then gives you a small exercise in the **playground** (Kotlin on the right, like Java lessons). The editor runs on **Kotlin/JVM** — the same conceptual output you would get from a local `main` function.

## Common migration mistakes (preview)

- Treating Kotlin **extension functions** like overridden Java methods (they are not dynamically dispatched the same way).
- Using `!!` to silence nullability instead of modeling `?` and `let` / early returns.
- Exposing **mutable** collection types from public APIs when Kotlin’s read-only views would be safer.

## Recap

- Kotlin targets the **JVM** you already use; interop with Java is a first-class design goal.
- This track is **translation-focused** for experienced Java developers, not a second beginner course.
- **Coroutines and advanced topics** are out of scope here but pointed to explicitly.
