In Java, `public static void main(String[] args)` lives **inside a named class** — conventionally `Main`. Kotlin lets you declare **`fun main()` at file scope**. The compiler still emits class files the JVM can execute; you simply skip writing the hosting class by hand in most learning and service code.

## Packages

Kotlin uses the `package` keyword like Java. Files in `src/main/kotlin/com/example/app/Foo.kt` should declare `package com.example.app` (or a shorter segment matching directory layout, depending on your convention). **Mixed Java/Kotlin modules** share package namespaces — a Kotlin `UserRepository` and a Java `UserService` can sit side by side under `com.example`.

## Entry point shapes

**Default (what we use in exercises):**

```kotlin
fun main() {
    println("Hello, Kotlin")
}
```

**With command-line args:**

```kotlin
fun main(args: Array<String>) {
    println(args.firstOrNull() ?: "no args")
}
```

**Java-facing static entry (preview):**

```kotlin
class Launcher {
    companion object {
        @JvmStatic
        fun main(args: Array<String>) {
            println("Called from Java-style tooling")
        }
    }
}
```

## Common mistakes

- Copy-pasting Java `public class Main` into `.kt` files when learning — valid Java interop exists, but it hides Kotlin’s default ergonomics.
- Misaligning **directory layout** and `package` declarations, which breaks tooling (IDE red code) even if a toy snippet compiles in the playground.

## Recap

- **`fun main()`** at top level is idiomatic Kotlin on the JVM.
- **Packages** mirror Java’s namespacing; mixed modules share packages naturally.
- **Annotations** adjust JVM names when Java callers or scripts need a specific shape — covered again in the interop module.
