Modern software is event-driven. A button click, a stock price change, a sensor reading, a new message in a queue — in each case, one object produces a change and an unknown number of other objects need to react. Hard-coding these relationships produces tightly coupled, brittle code. The **Observer pattern** formalises the publish-subscribe relationship: a *subject* maintains a list of *observers* and notifies them automatically when its state changes, without knowing who those observers are.

## The Observer interfaces

```java
// Observer — the subscriber interface
public interface Observer {
    void update(String event, Object data);
}

// Subject — the event source
public interface Subject {
    void subscribe(Observer o);
    void unsubscribe(Observer o);
    void notifyObservers(String event, Object data);
}
```

## EventSource — maintaining the observer list

```java
import java.util.*;

public class EventSource implements Subject {
    private final List<Observer> observers = new ArrayList<>();

    @Override
    public void subscribe(Observer o) {
        observers.add(o);
    }

    @Override
    public void unsubscribe(Observer o) {
        observers.remove(o);
    }

    @Override
    public void notifyObservers(String event, Object data) {
        // Iterate over a snapshot to allow safe unsubscription during notification
        for (Observer o : new ArrayList<>(observers)) {
            o.update(event, data);
        }
    }
}
```

## Concrete subject and observers

```java
public class StockMarket extends EventSource {
    private String ticker;
    private double price;

    public StockMarket(String ticker, double price) {
        this.ticker = ticker;
        this.price  = price;
    }

    public void updatePrice(double newPrice) {
        double oldPrice = this.price;
        this.price = newPrice;
        notifyObservers("PRICE_CHANGE", ticker + " " + oldPrice + " -> " + newPrice);
    }
}

// Observer implementations
public class AlertService implements Observer {
    private final double threshold;

    public AlertService(double threshold) { this.threshold = threshold; }

    @Override
    public void update(String event, Object data) {
        System.out.println("AlertService [" + event + "]: " + data);
    }
}

public class AuditLogger implements Observer {
    @Override
    public void update(String event, Object data) {
        System.out.println("AUDIT | " + event + " | " + data);
    }
}
```

```java
public class Main {
    public static void main(String[] args) {
        StockMarket aapl = new StockMarket("AAPL", 180.00);

        aapl.subscribe(new AlertService(200.00));
        aapl.subscribe(new AuditLogger());

        aapl.updatePrice(185.50);
        aapl.updatePrice(210.00);
    }
}
// AlertService [PRICE_CHANGE]: AAPL 180.0 -> 185.5
// AUDIT | PRICE_CHANGE | AAPL 180.0 -> 185.5
// AlertService [PRICE_CHANGE]: AAPL 185.5 -> 210.0
// AUDIT | PRICE_CHANGE | AAPL 185.5 -> 210.0
```

<Note>Java's `java.util.Observer` and `java.util.Observable` were **deprecated in Java 9**. They are poorly designed (Observable is a class, not an interface; the update method is weakly typed). Always define your own Observer interface as shown above.</Note>

## Lambda observers

Because `Observer` is a functional interface (one abstract method), observers can be expressed as lambdas:

```java
StockMarket tsla = new StockMarket("TSLA", 250.00);

// Lambda observer — no boilerplate class needed
tsla.subscribe((event, data) -> System.out.println("Dashboard: " + data));
tsla.subscribe((event, data) -> {
    if (event.equals("PRICE_CHANGE")) {
        System.out.println("SMS alert sent for: " + data);
    }
});

tsla.updatePrice(265.00);
// Dashboard: TSLA 250.0 -> 265.0
// SMS alert sent for: TSLA 250.0 -> 265.0
```

## Comparison to reactive streams

The basic Observer pattern has three gaps that reactive libraries address:

| | Basic Observer | Reactive Streams (Reactor/RxJava) |
|---|---|---|
| Back-pressure | None — slow observers get overwhelmed | `Subscription.request(n)` lets subscribers control rate |
| Error handling | Out-of-band | `onError(Throwable)` built into the protocol |
| Composition | Manual loops | `map`, `filter`, `flatMap`, `zip`, ... |
| Threading | Synchronous | Configurable schedulers |

```java
// Reactor Flux — reactive equivalent of our Observer
Flux.just(180.0, 185.5, 210.0)
    .filter(p -> p > 200)
    .map(p -> "HIGH PRICE: " + p)
    .subscribe(System.out::println);
// HIGH PRICE: 210.0
```

## Common mistakes

- **Forgetting to unsubscribe** — long-lived subjects holding references to short-lived observers prevent garbage collection. Always provide `unsubscribe` and call it when the observer is no longer needed.
- **Modifying the observer list during notification** — iterating the list while a callback calls `subscribe`/`unsubscribe` throws `ConcurrentModificationException`. Iterate over a copy.
- **Using `java.util.Observable`** — it's deprecated and class-based. Define your own interface.
- **Ordering assumptions** — observers are notified in subscription order, but do not design logic that depends on which observer runs first.

## Recap

- Observer defines a **one-to-many** relationship: one subject, many observers, decoupled through an interface
- The subject maintains a **subscriber list** and iterates it on each `notifyObservers` call
- **Lambdas** work as observers when the interface has a single abstract method
- `java.util.Observer` is **deprecated** — use a custom typed interface
- Reactive streams (Reactor, RxJava) extend Observer with **back-pressure**, **error channels**, and **composable operators**
