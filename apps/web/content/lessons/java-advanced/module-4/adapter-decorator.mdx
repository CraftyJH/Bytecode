Legacy code never goes away, and requirements never stop growing. The **Adapter** pattern bridges the gap between an interface you have and an interface you need ‚Äî without touching either side. The **Decorator** pattern lets you stack new behaviours on top of an object at runtime, one thin wrapper at a time. Together they are the workhorse structural patterns for real-world Java development.

## Adapter ‚Äî two flavours

**Object Adapter** uses composition: the adapter holds a reference to the adaptee. This is the preferred Java form because Java supports only single inheritance.

```java
// Client's expected interface
public interface ModernPrinter {
    void print(String text);
}

// Existing class with a different interface (cannot modify)
public class LegacyPrinter {
    public void printDocument(String document) {
        System.out.println("Legacy output: " + document);
    }
}

// Adapter ‚Äî implements the target, delegates to the adaptee
public class LegacyPrinterAdapter implements ModernPrinter {
    private final LegacyPrinter legacy;  // composition

    public LegacyPrinterAdapter(LegacyPrinter legacy) {
        this.legacy = legacy;
    }

    @Override
    public void print(String text) {
        legacy.printDocument(text);  // translation
    }
}

// Client sees only ModernPrinter ‚Äî unaware of the legacy class
ModernPrinter printer = new LegacyPrinterAdapter(new LegacyPrinter());
printer.print("Design Patterns in Java");
// Legacy output: Design Patterns in Java
```

**Class Adapter** uses multiple inheritance (via an interface + a class) and is less common in Java:

```java
// Class adapter ‚Äî inherits from LegacyPrinter, implements ModernPrinter
public class LegacyPrinterClassAdapter extends LegacyPrinter implements ModernPrinter {
    @Override
    public void print(String text) {
        printDocument(text);  // inherited from LegacyPrinter
    }
}
```

<Note>Prefer **object adapter** in Java. Class adapter locks you into a specific adaptee class. Object adapter works with any subclass of the adaptee and can be easier to mock in tests.</Note>

## Decorator ‚Äî building on `java.io`

The `java.io` package is the most-used Decorator in the JDK. Every stream class shares the `InputStream`/`OutputStream` interface, and each wrapper adds one feature:

```java
import java.io.*;

// Layer by layer ‚Äî each wrap adds one responsibility
InputStream raw        = new FileInputStream("data.bin");        // reads bytes from file
InputStream buffered   = new BufferedInputStream(raw);           // adds an 8KB buffer
DataInputStream typed  = new DataInputStream(buffered);          // adds readInt(), readDouble()

int value = typed.readInt();  // uses all three layers
```

The chain `FileInputStream ‚Üí BufferedInputStream ‚Üí DataInputStream` stacks three behaviours without subclassing. You can reorder or remove layers with no changes to the inner classes.

## Building your own Decorator

```java
// Component interface
public interface TextProcessor {
    String process(String input);
}

// Concrete component
public class PlainTextProcessor implements TextProcessor {
    public String process(String input) { return input; }
}

// Base decorator ‚Äî holds the wrapped component
public abstract class TextDecorator implements TextProcessor {
    protected final TextProcessor wrapped;
    protected TextDecorator(TextProcessor tp) { this.wrapped = tp; }
}

// Concrete decorators
public class TrimDecorator extends TextDecorator {
    public TrimDecorator(TextProcessor tp) { super(tp); }
    public String process(String input) {
        return wrapped.process(input).trim();
    }
}

public class UpperCaseDecorator extends TextDecorator {
    public UpperCaseDecorator(TextProcessor tp) { super(tp); }
    public String process(String input) {
        return wrapped.process(input).toUpperCase();
    }
}

public class StarWrapDecorator extends TextDecorator {
    public StarWrapDecorator(TextProcessor tp) { super(tp); }
    public String process(String input) {
        return "*** " + wrapped.process(input) + " ***";
    }
}
```

Stack them in any order:

```java
TextProcessor pipeline = new StarWrapDecorator(
        new UpperCaseDecorator(
        new TrimDecorator(
        new PlainTextProcessor())));

System.out.println(pipeline.process("  hello world  "));
// *** HELLO WORLD ***
```

## When Decorator beats subclassing

Suppose you need `BufferedLoggingPrinter`, `TimedPrinter`, `BufferedTimedPrinter`, and `LoggingTimedPrinter`. With subclassing that is 4 classes plus a base class. With Decorator it is 2 decorator classes (`BufferingDecorator`, `TimingDecorator`) composed as needed ‚Äî any combination without new subclasses.

```java
// Without decorator: combinatorial explosion of subclasses
class BufferedLoggingPrinter extends Printer { ... }
class TimedLoggingPrinter    extends Printer { ... }
class BufferedTimedPrinter   extends Printer { ... }
// ...

// With decorator: compose on demand
Printer p1 = new TimingDecorator(new BufferingDecorator(new ConsolePrinter()));
Printer p2 = new LoggingDecorator(new TimingDecorator(new ConsolePrinter()));
```

## Common mistakes

- **Confusing Adapter with Decorator** ‚Äî Adapter changes the interface; Decorator keeps the same interface and adds behaviour. If the interface changes, it's an Adapter.
- **Storing the adaptee as the concrete type in the Decorator** ‚Äî always type the wrapped field as the component interface, not a concrete class. This enables stacking.
- **Deep inheritance chains for combinations** ‚Äî whenever you find yourself creating `BufferedLoggingTimedPrinter`, reach for Decorator instead.
- **Forgetting to delegate in the Decorator** ‚Äî if a Decorator overrides a method and forgets to call `wrapped.method()`, it silently drops the wrapped behaviour.

## Recap

- **Adapter** converts one interface into another ‚Äî use it when you can't modify existing code but need it to satisfy a different contract
- **Object Adapter** uses composition (preferred in Java); **Class Adapter** uses multiple inheritance
- **Decorator** wraps a component implementing the same interface, adding one responsibility per layer
- `java.io` streams are the canonical JDK Decorator: `FileInputStream ‚Üí BufferedInputStream ‚Üí DataInputStream`
- Decorator avoids the **combinatorial subclass explosion** that results from using inheritance to combine features
