Not every class should be accessed directly, and not every subsystem should be exposed in full. The **Proxy** pattern sits in front of a real object, controlling who can use it and when — enabling lazy loading, caching, and access control. The **Facade** pattern provides a single simplified entry point to a complex subsystem, hiding its moving parts from the caller. Both reduce coupling; Proxy does it at the object level, Facade does it at the subsystem level.

## Virtual Proxy — lazy initialisation

A virtual proxy defers creation of an expensive object until it is first needed:

```java
public interface Image {
    void display();
}

// Expensive — reads from disk on construction
public class HighResImage implements Image {
    private final String filename;
    private final byte[] data;

    public HighResImage(String filename) {
        this.filename = filename;
        System.out.println("Loading image from disk: " + filename);
        // Simulate expensive I/O
        this.data = new byte[1024 * 1024]; // 1 MB
    }

    public void display() {
        System.out.println("Displaying: " + filename + " (" + data.length + " bytes)");
    }
}

// Proxy — creates the real object only on first display()
public class LazyImageProxy implements Image {
    private final String filename;
    private HighResImage realImage;  // null until first use

    public LazyImageProxy(String filename) {
        this.filename = filename;
        System.out.println("Proxy created for: " + filename);
    }

    public void display() {
        if (realImage == null) {
            realImage = new HighResImage(filename);
        }
        realImage.display();
    }
}
```

```java
Image img = new LazyImageProxy("banner.png"); // no I/O yet
System.out.println("--- later ---");
img.display(); // I/O happens here, first access
img.display(); // cached — no second load
// Proxy created for: banner.png
// --- later ---
// Loading image from disk: banner.png
// Displaying: banner.png (1048576 bytes)
// Displaying: banner.png (1048576 bytes)
```

## Protection Proxy — access control

A protection proxy checks permissions before delegating:

```java
public interface AdminService {
    void deleteUser(String userId);
    List<String> listUsers();
}

public class AdminServiceImpl implements AdminService {
    public void deleteUser(String userId) { System.out.println("Deleted: " + userId); }
    public List<String> listUsers()       { return List.of("alice", "bob"); }
}

public class AuthProxy implements AdminService {
    private final AdminService real;
    private final String currentRole;

    public AuthProxy(AdminService real, String currentRole) {
        this.real = real;
        this.currentRole = currentRole;
    }

    public void deleteUser(String userId) {
        if (!"ADMIN".equals(currentRole)) {
            throw new SecurityException("Access denied: deleteUser requires ADMIN role");
        }
        real.deleteUser(userId);
    }

    public List<String> listUsers() {
        return real.listUsers(); // all roles may list users
    }
}
```

<Note>The protection proxy is a lightweight alternative to full-blown AOP frameworks for simple access control rules. For complex cross-cutting concerns, consider Spring Security or a proper AOP library.</Note>

## Dynamic Proxy with `java.lang.reflect.Proxy`

Java's `java.lang.reflect.Proxy` generates a proxy class at runtime for any interface:

```java
import java.lang.reflect.*;

public class LoggingProxyFactory {
    @SuppressWarnings("unchecked")
    public static <T> T wrap(T target, Class<T> iface) {
        return (T) Proxy.newProxyInstance(
            iface.getClassLoader(),
            new Class<?>[]{ iface },
            (proxy, method, args) -> {
                System.out.println("Calling: " + method.getName());
                Object result = method.invoke(target, args);
                System.out.println("Done:    " + method.getName());
                return result;
            }
        );
    }
}

// Usage — no subclassing, works for any interface
AdminService logged = LoggingProxyFactory.wrap(new AdminServiceImpl(), AdminService.class);
logged.listUsers();
// Calling: listUsers
// Done:    listUsers
```

Dynamic proxies are the foundation of Spring AOP, JPA lazy-loading, and mocking frameworks like Mockito.

## Facade — hiding subsystem complexity

A Facade provides a single method that orchestrates multiple subsystem classes. The caller never needs to know the subsystem exists:

```java
// Subsystem classes — complex on their own
public class Amplifier {
    public void on()            { System.out.println("Amp on"); }
    public void setVolume(int v){ System.out.println("Volume: " + v); }
    public void off()           { System.out.println("Amp off"); }
}

public class Projector {
    public void on()  { System.out.println("Projector on"); }
    public void off() { System.out.println("Projector off"); }
}

public class Lights {
    public void dim(int level)  { System.out.println("Lights dimmed to " + level + "%"); }
    public void full()          { System.out.println("Lights full"); }
}

// Facade — a single orchestrating class
public class HomeTheaterFacade {
    private final Amplifier amp;
    private final Projector proj;
    private final Lights lights;

    public HomeTheaterFacade(Amplifier amp, Projector proj, Lights lights) {
        this.amp = amp; this.proj = proj; this.lights = lights;
    }

    public void watchMovie(String title) {
        System.out.println("Get ready to watch: " + title);
        lights.dim(20);
        proj.on();
        amp.on();
        amp.setVolume(8);
    }

    public void endMovie() {
        System.out.println("Shutting down...");
        amp.off();
        proj.off();
        lights.full();
    }
}
```

```java
HomeTheaterFacade theater = new HomeTheaterFacade(
    new Amplifier(), new Projector(), new Lights()
);
theater.watchMovie("Inception");
// Get ready to watch: Inception
// Lights dimmed to 20%
// Projector on
// Amp on
// Volume: 8
```

}

## Common mistakes

- **Confusing Proxy with Decorator** — same structure, different intent. If the wrapper controls access, it's a Proxy; if it adds transparent features, it's a Decorator.
- **Tight coupling in Facade constructors** — never `new` subsystem objects inside the Facade constructor. Inject them so they can be tested and swapped.
- **Dynamic proxies only work with interfaces** — `java.lang.reflect.Proxy` cannot proxy concrete classes. For class proxying you need a library like CGLIB (used internally by Spring).
- **Forgetting to synchronise lazy proxy init** — `LazyImageProxy` as written is not thread-safe. In multithreaded code, protect the null check with `synchronized` or a `volatile` field.

## Recap

- **Virtual proxy** defers expensive construction until first use (lazy init)
- **Protection proxy** enforces access control before delegating to the real subject
- **`java.lang.reflect.Proxy`** generates dynamic proxies for any interface at runtime — the foundation of Spring AOP and Mockito
- **Facade** provides a simple, unified interface to an otherwise complex subsystem
- Both Proxy and Facade reduce coupling; Proxy works at the object level, Facade at the subsystem level
