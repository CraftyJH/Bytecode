The most durable way to cement your knowledge of design patterns is to spot them in code you already use every day. The Java standard library is built almost entirely from the patterns covered in this module. This lesson maps each pattern to its JDK incarnation so you can see the theory in production-grade code — and recognise patterns instantly when you encounter them in any large codebase.

## Pattern map — JDK at a glance

| Pattern | JDK Example | Key signal |
|---|---|---|
| **Decorator** | `BufferedInputStream`, `InputStreamReader` | Wraps same interface, adds behaviour |
| **Observer** | `PropertyChangeListener`, `EventListener` | `addXxxListener` / `removeXxxListener` |
| **Strategy** | `Comparator`, `Predicate`, `Function` | Single-method interface, passed as argument |
| **Builder** | `StringBuilder`, `Stream`, `ProcessBuilder` | Fluent chain ending in a terminal method |
| **Proxy** | `java.lang.reflect.Proxy`, `Collections.unmodifiableList` | Wraps same interface, controls access |
| **Singleton** | `Runtime.getRuntime()`, `System.getSecurityManager()` | Private constructor + static accessor |
| **Iterator** | `java.util.Iterator`, enhanced for-each | `hasNext()` / `next()` protocol |
| **Template Method** | `AbstractList`, `AbstractMap`, `HttpServlet` | `final` method calling abstract hooks |
| **Factory Method** | `Calendar.getInstance()`, `NumberFormat.getInstance()` | Static method returning interface/abstract type |

## Decorator — java.io streams

```java
import java.io.*;

// FileInputStream → BufferedInputStream → DataInputStream
// Each layer adds one responsibility, same InputStream interface throughout
try (DataInputStream in = new DataInputStream(
        new BufferedInputStream(
        new FileInputStream("scores.bin")))) {

    int score = in.readInt();    // DataInputStream adds typed reads
    System.out.println(score);
}
// FileInputStream  — raw bytes from disk
// BufferedInputStream — adds an 8 KB buffer (fewer OS calls)
// DataInputStream  — adds readInt(), readDouble(), readUTF()
```

Adding a checksum layer requires only wrapping with `CheckedInputStream` — zero changes to the inner classes.

## Observer — PropertyChangeListener

```java
import java.beans.*;

public class Temperature {
    private final PropertyChangeSupport pcs = new PropertyChangeSupport(this);
    private double value;

    public void addChangeListener(PropertyChangeListener l) { pcs.addPropertyChangeListener(l); }
    public void removeChangeListener(PropertyChangeListener l) { pcs.removePropertyChangeListener(l); }

    public void setValue(double newVal) {
        double old = this.value;
        this.value = newVal;
        pcs.firePropertyChange("value", old, newVal);
    }
}

Temperature temp = new Temperature();
temp.addChangeListener(e ->
    System.out.println("Changed: " + e.getOldValue() + " -> " + e.getNewValue())
);
temp.setValue(22.5);
// Changed: 0.0 -> 22.5
```

<Note>`PropertyChangeSupport` is the JDK's ready-made observer list. It is the standard convention in JavaBeans and Swing. In modern code, custom typed Observer interfaces (as taught in Lesson 6) are often cleaner.</Note>

## Strategy — Comparator and Stream

```java
import java.util.*;
import java.util.stream.*;

List<String> words = Arrays.asList("banana", "fig", "cherry", "kiwi", "apple");

// Comparator IS a strategy: plug in any ordering algorithm
words.sort(Comparator.comparingInt(String::length)
                     .thenComparing(Comparator.naturalOrder()));
System.out.println(words);  // [fig, kiwi, apple, banana, cherry]

// Stream pipeline: each lambda is a pluggable strategy
long count = words.stream()
    .filter(w -> w.length() > 4)       // Predicate strategy
    .map(String::toUpperCase)          // Function strategy
    .peek(System.out::println)         // Consumer strategy
    .count();
System.out.println("Count: " + count);
```

## Builder — StringBuilder and ProcessBuilder

```java
// StringBuilder is a classic Builder (mutable, fluent chain)
String result = new StringBuilder()
    .append("Hello")
    .append(", ")
    .append("World")
    .append("!")
    .toString();  // final "build" step
System.out.println(result);  // Hello, World!

// ProcessBuilder shows Builder for configuration
ProcessBuilder pb = new ProcessBuilder("ls", "-la")
    .directory(new java.io.File("/tmp"))
    .redirectErrorStream(true);
// pb.start() would launch the process
```

## Singleton — Runtime

```java
// Runtime.getRuntime() — the classic eager Singleton
Runtime rt = Runtime.getRuntime();
System.out.println("Processors: " + rt.availableProcessors());
System.out.println("Free memory: " + rt.freeMemory() / 1024 / 1024 + " MB");

// Same instance every time — reference equality is true
System.out.println(Runtime.getRuntime() == Runtime.getRuntime()); // true
```

## Try it

Use three JDK patterns in one program: `Comparator` strategy, `BufferedReader` decorator chain, and a builder-style `Stream` pipeline:

```java
import java.util.*;
import java.util.stream.*;

public class Main {
    public static void main(String[] args) {
        // 1. Strategy: custom Comparator
        List<String> fruits = Arrays.asList("banana", "apple", "cherry", "fig");
        fruits.sort(Comparator.comparingInt(String::length)
                              .thenComparing(Comparator.naturalOrder()));
        System.out.println("Sorted: " + fruits);
        // Sorted: [fig, apple, banana, cherry]

        // 2. Decorator: BufferedReader wraps a StringReader (no file I/O needed)
        String data = "one\ntwo\nthree";
        try (var reader = new java.io.BufferedReader(new java.io.StringReader(data))) {
            reader.lines()
                  .map(String::toUpperCase)
                  .forEach(System.out::println);
        } catch (Exception e) { e.printStackTrace(); }
        // ONE  TWO  THREE

        // 3. Builder-style Stream pipeline
        long longWords = Stream.of("design", "patterns", "in", "Java", "are", "powerful")
            .filter(w -> w.length() > 4)
            .peek(w -> System.out.println("Kept: " + w))
            .count();
        System.out.println("Long words: " + longWords);
    }
}
```

## Common mistakes

- **Confusing `StringBuilder` with GoF Builder** — `StringBuilder` is a fluent accumulator, not the immutable-object-from-named-parameters pattern. Recognise it as a Builder-style API, not a strict GoF Builder.
- **Calling `Collections.unmodifiableList` a Decorator** — it controls access (throws on mutation), making it a Protection Proxy, not a Decorator. The interface is the same but the intent is access control.
- **Assuming every static factory method is a Singleton** — `Calendar.getInstance()` creates a new instance each call. It's Factory Method, not Singleton. Only methods that always return the same instance are Singleton.
- **Missing the Template Method in `AbstractList`** — `AbstractList.indexOf()` calls `listIterator()`, which calls `get()` — an abstract method you override. That's Template Method calling back into your subclass.

## Recap

- `BufferedInputStream`, `InputStreamReader` → **Decorator**
- `PropertyChangeListener`, `EventListener` → **Observer**
- `Comparator`, `Predicate`, `Function` → **Strategy**
- `StringBuilder`, `Stream`, `ProcessBuilder` → **Builder-style API**
- `java.lang.reflect.Proxy`, `Collections.unmodifiableList` → **Proxy**
- `Runtime.getRuntime()` → **Singleton**
- `AbstractList.get()` / `AbstractMap.entrySet()` → **Template Method**
