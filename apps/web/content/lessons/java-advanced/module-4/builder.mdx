Constructors with many parameters are one of the most pervasive sources of bugs in Java codebases. Have you ever seen a call like `new Order(true, false, true, null, 3, "USD", false)`? Without looking up the class definition you cannot know which boolean is which. The **Builder pattern** solves the telescoping constructor problem by replacing positional constructor arguments with named, chainable setter methods, and produces an immutable object only when you call `build()`.

## The telescoping constructor problem

```java
// v1 — one constructor
public class Pizza {
    public Pizza(String size) { ... }
}

// v2 — add optional cheese
public class Pizza {
    public Pizza(String size) { ... }
    public Pizza(String size, boolean cheese) { ... }
}

// v3 — add pepperoni, mushrooms, sauce...
// This is where it falls apart:
Pizza p = new Pizza("large", true, false, true, "tomato", false, true);
//                            ^      ^      ^       ^         ^     ^
//                           cheese pepperoni mushrooms sauce olives bacon
// Which boolean is which?!
```

A class with 6 optional boolean parameters needs up to 64 constructor overloads to cover every combination. The Builder pattern eliminates all of them.

## Builder with inner static class

```java
public class Pizza {
    // All fields are final — the object is immutable after build()
    private final String size;
    private final String crust;
    private final boolean cheese;
    private final boolean pepperoni;
    private final boolean mushrooms;

    // Private constructor — only Builder can call it
    private Pizza(Builder builder) {
        this.size      = builder.size;
        this.crust     = builder.crust;
        this.cheese    = builder.cheese;
        this.pepperoni = builder.pepperoni;
        this.mushrooms = builder.mushrooms;
    }

    public static class Builder {
        // Required field — set in Builder constructor
        private final String size;

        // Optional fields with defaults
        private String  crust      = "regular";
        private boolean cheese     = false;
        private boolean pepperoni  = false;
        private boolean mushrooms  = false;

        public Builder(String size) {
            this.size = size;
        }

        // Each setter returns 'this' for chaining (fluent API)
        public Builder crust(String crust)         { this.crust = crust;           return this; }
        public Builder cheese(boolean cheese)       { this.cheese = cheese;         return this; }
        public Builder pepperoni(boolean pepperoni) { this.pepperoni = pepperoni;   return this; }
        public Builder mushrooms(boolean mushrooms) { this.mushrooms = mushrooms;   return this; }

        public Pizza build() {
            if (size == null || size.isBlank()) {
                throw new IllegalStateException("Pizza size is required");
            }
            return new Pizza(this);
        }
    }

    @Override public String toString() {
        return String.format("Pizza[%s, crust=%s, cheese=%b, pepperoni=%b, mushrooms=%b]",
                size, crust, cheese, pepperoni, mushrooms);
    }
}
```

Now the call site is self-documenting:

```java
Pizza p = new Pizza.Builder("large")
        .crust("thin")
        .cheese(true)
        .pepperoni(true)
        .build();

System.out.println(p);
// Pizza[large, crust=thin, cheese=true, pepperoni=true, mushrooms=false]
```

<Note>Mandatory fields belong in the **Builder's constructor**, not in setter methods. This makes it a compile-time error to omit them, rather than a runtime exception. Optional fields live in setter methods with sensible defaults.</Note>

## Immutability from a Builder

The target object (`Pizza`) has all `final` fields and no setters. Once `build()` returns the object, its state cannot change. This makes it safe to share across threads and cache without defensive copying.

```java
// This is fine — Pizza is immutable
Pizza p = new Pizza.Builder("medium").cheese(true).build();
cache.put("standard", p);           // safe to share
list.add(p);                        // safe to put in collections
```

## Lombok @Builder

In real projects, the boilerplate above is often generated by **Lombok's `@Builder` annotation**:

```java
import lombok.Builder;
import lombok.ToString;

@Builder
@ToString
public class User {
    private final String username;
    private final String email;
    @Builder.Default private final boolean active = true;
    @Builder.Default private final int loginCount = 0;
}

// Lombok generates the Builder class automatically
User u = User.builder()
             .username("alice")
             .email("alice@example.com")
             .build();
System.out.println(u);
// User(username=alice, email=alice@example.com, active=true, loginCount=0)
```

`@Builder.Default` tells Lombok to use a default value for fields not set by the caller. Without it, unset fields default to Java's zero values (`null`, `0`, `false`).

## Try it

Build `Pizza.Builder` from scratch and verify the output:

```java
public class Main {
    public static void main(String[] args) {
        Pizza p1 = new Pizza.Builder("small")
                .build();

        Pizza p2 = new Pizza.Builder("large")
                .crust("stuffed")
                .cheese(true)
                .pepperoni(true)
                .mushrooms(true)
                .build();

        System.out.println(p1);
        System.out.println(p2);

        // Demonstrate immutability — p1 and p2 can be stored safely
        java.util.List<Pizza> menu = java.util.List.of(p1, p2);
        System.out.println("Menu size: " + menu.size());
    }
}
// Pizza[small, crust=regular, cheese=false, pepperoni=false, mushrooms=false]
// Pizza[large, crust=stuffed, cheese=true, pepperoni=true, mushrooms=true]
// Menu size: 2
```

## Common mistakes

- **Making the target class's constructor public** — callers bypass the builder and can create objects in inconsistent states. The constructor must be `private`.
- **Forgetting to return `this` in setter methods** — without `return this`, the fluent chain breaks and every setter must be on a separate line with the builder variable.
- **Putting mandatory fields in setter methods** — mandatory fields in the Builder constructor give compile-time safety; as setter methods they silently default to `null`/`0`.
- **Sharing a Builder across threads** — the builder itself is mutable. Build the final object, then share the immutable result.

## Recap

- The **telescoping constructor problem** arises when optional fields multiply constructor overloads beyond readability
- Builder uses a **static inner class** that accumulates configuration through chainable setter methods
- Calling `build()` triggers validation and returns an **immutable** target object
- The **fluent API** (each setter returns `this`) makes call sites self-documenting
- **Lombok `@Builder`** eliminates boilerplate in production code while keeping the same pattern
