Some problems are about choosing the right algorithm. Others are about encapsulating an action so it can be queued, logged, or undone. **Strategy** makes algorithms interchangeable at runtime by hiding them behind an interface. **Command** turns a request into an object, enabling deferred execution, undo/redo queues, and request logging. Both patterns convert what was previously hard-coded control flow into flexible, composable first-class objects.

## Strategy pattern

The Strategy pattern defines a family of algorithms, encapsulates each one in a class, and makes them interchangeable. The *context* delegates the algorithm to the current strategy:

```java
// Strategy interface
public interface EncryptionStrategy {
    String encrypt(String plaintext);
    String decrypt(String ciphertext);
}

// Concrete strategies
public class AesStrategy implements EncryptionStrategy {
    public String encrypt(String p) {
        return "AES[" + p + "]"; // simplified
    }
    public String decrypt(String c) {
        return c.replace("AES[", "").replace("]", "");
    }
}

public class Base64Strategy implements EncryptionStrategy {
    public String encrypt(String p) {
        return java.util.Base64.getEncoder().encodeToString(p.getBytes());
    }
    public String decrypt(String c) {
        return new String(java.util.Base64.getDecoder().decode(c));
    }
}

// Context — delegates to whichever strategy is plugged in
public class MessageService {
    private EncryptionStrategy strategy;

    public MessageService(EncryptionStrategy strategy) {
        this.strategy = strategy;
    }

    public void setStrategy(EncryptionStrategy strategy) {
        this.strategy = strategy;
    }

    public String send(String message) {
        String encrypted = strategy.encrypt(message);
        System.out.println("Sending: " + encrypted);
        return encrypted;
    }
}
```

```java
MessageService svc = new MessageService(new AesStrategy());
svc.send("Hello");
// Sending: AES[Hello]

svc.setStrategy(new Base64Strategy());
svc.send("Hello");
// Sending: SGVsbG8=
```

## Lambda as Strategy

When the strategy interface has a single abstract method, a lambda is all you need:

```java
// Comparator<T> IS a strategy interface
List<String> names = Arrays.asList("Charlie", "Alice", "Bob");

// Swap strategies at runtime with lambdas — no class definitions needed
names.sort((a, b) -> a.compareTo(b));           // alphabetical
System.out.println(names);  // [Alice, Bob, Charlie]

names.sort((a, b) -> a.length() - b.length());  // by length
System.out.println(names);  // [Bob, Alice, Charlie]

names.sort(Comparator.reverseOrder());           // reverse
System.out.println(names);  // [Charlie, Bob, Alice]
```

<Note>The `Comparator`, `Predicate`, `Function`, and `Runnable` interfaces in `java.util.function` are all Strategy interfaces. Java's Stream API is essentially Strategy applied pervasively — every lambda you pass to `map`, `filter`, or `sorted` is a pluggable algorithm.</Note>

## Command pattern

The Command pattern encapsulates a request as an object. Each command knows how to execute one operation, and optionally how to reverse it:

```java
// Command interface
public interface Command {
    void execute();
    void undo();
}

// Receiver — the object the command actually operates on
public class TextBuffer {
    private StringBuilder text = new StringBuilder();

    public void append(String s) { text.append(s); }
    public void delete(int count) {
        int start = Math.max(0, text.length() - count);
        text.delete(start, text.length());
    }
    public String getText() { return text.toString(); }
}

// Concrete command
public class InsertCommand implements Command {
    private final TextBuffer buffer;
    private final String textToInsert;

    public InsertCommand(TextBuffer buffer, String text) {
        this.buffer = buffer;
        this.textToInsert = text;
    }

    @Override
    public void execute() { buffer.append(textToInsert); }

    @Override
    public void undo() { buffer.delete(textToInsert.length()); }
}
```

## Invoker — the undo/redo queue

```java
import java.util.*;

public class TextEditor {
    private final TextBuffer buffer = new TextBuffer();
    private final Deque<Command> history = new ArrayDeque<>();

    public void executeCommand(Command cmd) {
        cmd.execute();
        history.push(cmd);
    }

    public void undo() {
        if (!history.isEmpty()) {
            history.pop().undo();
        }
    }

    public String getText() { return buffer.getText(); }
    public TextBuffer getBuffer() { return buffer; }
}
```

```java
public class Main {
    public static void main(String[] args) {
        TextEditor editor = new TextEditor();

        editor.executeCommand(new InsertCommand(editor.getBuffer(), "Hello"));
        editor.executeCommand(new InsertCommand(editor.getBuffer(), " World"));
        System.out.println(editor.getText()); // Hello World

        editor.undo();
        System.out.println(editor.getText()); // Hello

        editor.undo();
        System.out.println(editor.getText()); // (empty)
    }
}
```

## Try it

Build a `SortContext` with pluggable strategy and a `TextEditor` with full undo:

```java
public class Main {
    public static void main(String[] args) {
        // --- Strategy demo ---
        SortContext ctx = new SortContext(arr -> {
            java.util.Arrays.sort(arr);
            System.out.println("Sorted: " + java.util.Arrays.toString(arr));
        });

        int[] nums = {5, 3, 9, 1};
        ctx.sort(nums);   // Sorted: [1, 3, 5, 9]

        // Swap to reverse sort strategy
        ctx.setStrategy(arr -> {
            java.util.Arrays.sort(arr);
            // reverse
            for (int i = 0, j = arr.length - 1; i < j; i++, j--) {
                int tmp = arr[i]; arr[i] = arr[j]; arr[j] = tmp;
            }
            System.out.println("Reverse sorted: " + java.util.Arrays.toString(arr));
        });
        ctx.sort(nums);   // Reverse sorted: [9, 5, 3, 1]

        // --- Command / undo demo ---
        TextEditor editor = new TextEditor();
        editor.executeCommand(new InsertCommand(editor.getBuffer(), "Design"));
        editor.executeCommand(new InsertCommand(editor.getBuffer(), " Patterns"));
        System.out.println(editor.getText()); // Design Patterns
        editor.undo();
        System.out.println(editor.getText()); // Design
    }
}
```

## Common mistakes

- **Not separating the Invoker from the Command** — the invoker (queue, button, scheduler) should only call `execute()`/`undo()`. Business logic belongs inside the Command or its Receiver.
- **Making Commands depend on the Invoker** — Commands should be self-contained. A Command that calls back into the invoker creates a circular dependency.
- **Using Strategy where a simple method parameter suffices** — if you only ever have one algorithm and never need to swap it, a plain method parameter or lambda suffices. Don't extract a class just to have one.
- **Forgetting undo state in Commands** — if `InsertCommand` doesn't store the inserted text, it cannot compute what to delete on `undo()`. Each Command must capture all state needed for reversal at construction time.

## Recap

- **Strategy** encapsulates interchangeable algorithms behind an interface; the context swaps them at runtime
- Strategy eliminates `if/else` algorithm forks and makes behaviour composable
- **Lambdas are anonymous strategy instances** — `Comparator`, `Predicate`, `Function` are all strategy interfaces
- **Command** encapsulates a request as an object, enabling deferred execution and undo queues
- The **Invoker** (queue/history stack) drives commands; the **Receiver** holds domain state; Commands bridge the two
