Before you can apply design patterns well, you need to recognise what you're refactoring away from. **Anti-patterns** are recurring poor solutions — patterns of failure that appear again and again in real codebases. This capstone lesson names the most damaging ones, shows how to identify them in code, and maps each to the pattern that fixes it.

## God Class

A God Class does too much. It accumulates responsibilities over years of "easiest place to add it" decisions until it becomes the centre of everything:

```java
// Anti-pattern: OrderManager with 8 responsibilities
public class OrderManager {
    public void validateOrder(Order o)       { /* validation logic */ }
    public double calculateTotal(Order o)    { /* pricing, tax, discounts */ }
    public void chargeCustomer(Order o)      { /* payment */ }
    public void sendConfirmation(Order o)    { /* email / SMS */ }
    public void updateInventory(Order o)     { /* stock levels */ }
    public void generatePdf(Order o)         { /* PDF export */ }
    public void logAuditTrail(Order o)       { /* audit */ }
    public void scheduleDelivery(Order o)    { /* logistics */ }
    // ... 300 more lines ...
}
```

Every change to any concern touches this one class. Merges conflict constantly. Nothing is testable in isolation.

**Fix:** apply the Single Responsibility Principle — split each responsibility into its own class. Use **Strategy** for pluggable concerns (pricing rules, discount policies) and **Observer** to decouple side-effects (email, audit, PDF) from the core order workflow:

```java
// After refactoring: each class has one reason to change
public class OrderValidator      { public void validate(Order o)    { ... } }
public class PricingService      { public Money calculate(Order o)  { ... } }
public class PaymentService      { public void charge(Order o, Money total) { ... } }

// Side effects driven by Observer — OrderService doesn't know about email or PDF
public class OrderService {
    private final OrderValidator validator;
    private final PricingService pricer;
    private final PaymentService payment;
    private final List<OrderObserver> observers = new ArrayList<>();

    public void processOrder(Order order) {
        validator.validate(order);
        Money total = pricer.calculate(order);
        payment.charge(order, total);
        observers.forEach(o -> o.onOrderProcessed(order, total));
    }
}
// EmailNotificationService, AuditService, PdfService each implement OrderObserver
```

## Primitive Obsession

Using raw primitives (`double`, `String`, `int`) where a value object should exist:

```java
// Anti-pattern
public void transfer(String fromAccount, String toAccount, double amount, String currency) {
    // What if currency is null? What if amount is negative?
    // Nothing stops: transfer("ACC1", "ACC2", -500.0, "RUBISH");
}

// Fix: value objects make invalid states unrepresentable
public record Money(BigDecimal amount, Currency currency) {
    public Money {
        if (amount.compareTo(BigDecimal.ZERO) < 0)
            throw new IllegalArgumentException("Amount cannot be negative");
    }
}

public record AccountId(String value) {
    public AccountId { if (value == null || value.isBlank()) throw new IllegalArgumentException("Invalid account id"); }
}

public void transfer(AccountId from, AccountId to, Money amount) { ... }
```

<Note>Java **records** (Java 16+) are the fastest way to create value objects. They are immutable, provide `equals`, `hashCode`, and `toString` automatically, and compact constructors let you add validation concisely.</Note>

## Feature Envy

A method is more interested in another class's data than its own. It calls getter after getter on a foreign object to do work that belongs there:

```java
// Anti-pattern: OrderManager envies Customer
public class OrderManager {
    public String buildGreeting(Order order) {
        String name    = order.getCustomer().getName();
        String tier    = order.getCustomer().getLoyaltyTier();
        String address = order.getCustomer().getShippingAddress().getCity();
        // This logic belongs in Customer or a dedicated service
        return "Dear " + tier + " member " + name + " in " + address;
    }
}

// Fix: move behavior to where the data lives
public class Customer {
    public String buildGreeting() {
        return "Dear " + loyaltyTier + " member " + name
               + " in " + shippingAddress.getCity();
    }
}
```

## Anemic Domain Model

Domain objects are pure data bags — all getters/setters, zero behavior. All logic lives in service classes:

```java
// Anti-pattern: Order is just a DTO
public class Order {
    private List<Item> items;
    private double total;
    private String status;
    // only getters and setters — no behavior
}

// All logic scattered in service
public class OrderService {
    public double getTotal(Order o) { /* calculates here */ }
    public boolean canCancel(Order o) { /* decides here */ }
}

// Fix: move domain logic into Order
public class Order {
    private final List<Item> items;
    private OrderStatus status = OrderStatus.PENDING;

    public Money calculateTotal(DiscountPolicy policy) {
        // Logic belongs here — Order owns its items
        return items.stream()
            .map(Item::subtotal)
            .reduce(Money.ZERO, Money::add)
            .apply(policy);
    }

    public void cancel() {
        if (status != OrderStatus.PENDING)
            throw new IllegalStateException("Cannot cancel a " + status + " order");
        this.status = OrderStatus.CANCELLED;
    }
}
```

## Spaghetti inheritance

Modelling every behavior combination through subclasses:

```java
// Anti-pattern: class explosion
class Dog                 extends Animal { }
class FlyingDog           extends Dog    { }   // hypothetical
class SwimmingDog         extends Dog    { }
class FlyingSwimmingDog   extends Dog    { }   // ...

// Fix: Strategy (composition over inheritance)
public interface MovementBehaviour { void move(); }
public class Walk  implements MovementBehaviour { public void move() { System.out.println("walks"); } }
public class Swim  implements MovementBehaviour { public void move() { System.out.println("swims"); } }
public class Fly   implements MovementBehaviour { public void move() { System.out.println("flies"); } }

public class Animal {
    private final MovementBehaviour movement;
    public Animal(MovementBehaviour movement) { this.movement = movement; }
    public void move() { movement.move(); }
}

Animal duck = new Animal(new Fly());
Animal dog  = new Animal(new Swim());
```

## Try it — Refactor the God Class

Refactor the provided `OrderManager` by splitting it into `OrderValidator`, `PricingService`, `PaymentService`, and an Observer-driven notification pipeline:

```java
// Step 1: Observer interface for post-processing side effects
public interface OrderObserver {
    void onOrderProcessed(Order order, Money total);
}

// Step 2: Concrete observers
public class EmailNotifier implements OrderObserver {
    public void onOrderProcessed(Order order, Money total) {
        System.out.println("Email sent to " + order.getCustomer().getEmail()
                + " for " + total);
    }
}

public class AuditLogger implements OrderObserver {
    public void onOrderProcessed(Order order, Money total) {
        System.out.println("AUDIT: order " + order.getId() + " processed, " + total);
    }
}

// Step 3: Strategy for pricing
public interface DiscountStrategy {
    Money apply(Money subtotal, Order order);
}

public class PremiumDiscount implements DiscountStrategy {
    public Money apply(Money subtotal, Order order) {
        return order.getCustomer().isPremium()
            ? subtotal.multiply(0.9)
            : subtotal;
    }
}

// Step 4: Lean OrderService — orchestrates, does not implement
public class OrderService {
    private final DiscountStrategy discount;
    private final List<OrderObserver> observers;

    public OrderService(DiscountStrategy discount, List<OrderObserver> observers) {
        this.discount  = discount;
        this.observers = observers;
    }

    public void process(Order order) {
        if (order.getItems().isEmpty()) throw new IllegalArgumentException("Empty order");
        Money total = order.calculateTotal(discount);
        observers.forEach(o -> o.onOrderProcessed(order, total));
    }
}
```

## Common mistakes

- **Refactoring too aggressively at once** — break one responsibility out at a time. Large-scale rewrites introduce regressions and make code review impossible.
- **Replacing one God Class with several Feature Envy services** — distributing behavior into services that all reach back into domain objects is not a real improvement. Move logic to where the data lives.
- **Over-applying patterns as the fix** — not every God Class becomes a Strategy + Observer + Builder. Use the simplest fix that eliminates the problem.
- **Forgetting to test the seams** — before refactoring, write characterization tests that capture current behavior. After, verify they still pass.

## Recap

- **God Class** — split into focused classes using SRP; use Strategy for algorithms and Observer for side effects
- **Primitive Obsession** — introduce value objects (`Money`, `EmailAddress`, `AccountId`) — Java records make this cheap
- **Feature Envy** — move methods to the class that owns the data they consume
- **Anemic Domain Model** — put domain logic back into domain objects; services should orchestrate, not implement
- **Spaghetti Inheritance** — replace class hierarchies of behavior combinations with **composition over inheritance** (Strategy)
