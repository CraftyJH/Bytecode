Software development is full of recurring problems. How do you create an object without hard-coding its exact class? How do you add behavior to an object at runtime without altering its class? How do you notify many objects when something changes? Experienced engineers solved these problems, recognised the solutions were reusable, and wrote them down. The result — the 1994 book by Gamma, Helm, Johnson, and Vlissides — became one of the most influential software engineering texts ever written.

## What is a design pattern?

A **design pattern** is a named, reusable solution to a commonly occurring problem in a given context. It is not finished code you paste in. It is a template describing a structure, the roles of the participants, and the trade-offs.

Patterns give teams a shared vocabulary. Instead of a five-minute explanation, an engineer can say "use a Factory Method here" and convey a precise, well-understood structure.

```
Pattern anatomy (GoF style)
──────────────────────────
Name        — the shared vocabulary word (e.g., "Strategy")
Intent      — what problem it solves
Motivation  — a concrete scenario
Structure   — a UML class diagram
Participants — the roles and their responsibilities
Consequences — trade-offs, benefits, liabilities
```

## The three GoF categories

The 23 GoF patterns are grouped by their primary purpose:

| Category | Concern | Examples |
|---|---|---|
| **Creational** | How objects are created | Singleton, Factory Method, Abstract Factory, Builder, Prototype |
| **Structural** | How classes/objects are composed | Adapter, Decorator, Proxy, Facade, Composite, Bridge, Flyweight |
| **Behavioral** | How objects communicate | Observer, Strategy, Command, Iterator, Template Method, Chain of Responsibility |

```java
// Creational: you say WHAT you need, not HOW to build it
Shape s = ShapeFactory.create("circle");   // Factory Method

// Structural: you compose behavior without changing the original class
InputStream in = new BufferedInputStream(new FileInputStream("data.bin")); // Decorator

// Behavioral: you define how objects interact
button.addActionListener(e -> handleClick()); // Observer (simplified)
```

<Note>These categories overlap at the edges. A Proxy, for instance, has elements of both structural composition and behavioral delegation. When classifying a pattern, focus on its primary intent.</Note>

## UML notation basics for Java developers

You will encounter UML class diagrams whenever patterns are documented. The essentials:

```
─────────────────────────────────────────────────────
Notation            Meaning
─────────────────────────────────────────────────────
A ──────► B         Association: A holds a reference to B
A ------► B         Dependency: A uses B (dashed, open arrow)
A ═══════► B        Realisation: A implements interface B (dashed, filled triangle)
A ────────► B       Inheritance: A extends B (solid, filled triangle)
◆──────── B         Composition: A owns B (filled diamond on A's side)
◇──────── B         Aggregation: A has B (open diamond on A's side)
─────────────────────────────────────────────────────
```

In Java terms:
- **Inheritance** → `class Dog extends Animal`
- **Realisation** → `class Dog implements Runnable`
- **Composition** → `Engine` exists only inside `Car`; destroyed with it
- **Aggregation** → `Team` holds `Player` references, but players outlive the team

## When patterns help — and when they hurt

Patterns are useful when:
- The same structural problem recurs across the codebase
- You need to isolate a variation point (e.g., pluggable algorithms)
- You want to enforce constraints (e.g., only one instance)
- You need to decouple producers from consumers

Patterns hurt when:
- The problem they solve does not exist in your code
- The added indirection exceeds the complexity it removes
- A simpler construct (a lambda, an interface, a record) suffices
- They are applied for resume-padding rather than necessity

> **Rule of thumb:** if you cannot name the concrete problem the pattern solves in your specific code, do not introduce it.

```

```
// Which category?

_Answer: Structural (Adapter pattern wraps the legacy interface inside the expected one.)_

**Scenario C:** A spreadsheet cell should update all dependent formula cells and chart widgets whenever its value changes, without the cell knowing what depends on it.

## Common mistakes

- **Treating patterns as requirements** — patterns are optional tools. Do not force every class to use one.
- **Confusing category with intent** — Decorator is Structural, not Behavioral, even though it changes behavior. Always check the intent column.
- **Memorising UML instead of understanding it** — focus on what each arrow communicates about dependency and ownership.
- **Equating patterns with best practices** — a pattern is a known trade-off, not an uncritically good idea. Understand the liabilities column of each pattern.

## Recap

- The Gang of Four published 23 patterns in 1994, grouped into **Creational**, **Structural**, and **Behavioral** families
- A pattern is a **named, reusable solution template**, not copy-paste code
- UML class diagrams use arrows to express inheritance, implementation, association, and composition
- Patterns help when a concrete, recurring problem exists in your code; they hurt when applied speculatively
- The shared vocabulary patterns provide is often their greatest day-to-day value
