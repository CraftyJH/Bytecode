Shared mutable collections are one of the most common sources of bugs in concurrent Java code. A plain `ArrayList` or `HashMap` is emphatically not thread-safe: concurrent modifications corrupt their internal state in unpredictable ways — `ConcurrentModificationException` is the benign outcome; silent data corruption or infinite loops are the dangerous ones. Java's `java.util.concurrent` package provides thread-safe collection alternatives that are both correct *and* high-performance, designed specifically for concurrent access patterns.

## ConcurrentHashMap

`ConcurrentHashMap` is the go-to replacement for `HashMap` in multi-threaded code. Unlike the old `Hashtable`, it does not lock the entire map on every operation:

```java
import java.util.concurrent.ConcurrentHashMap;

ConcurrentHashMap<String, Integer> wordCount = new ConcurrentHashMap<>();

// Thread-safe put, get, putIfAbsent
wordCount.put("java", 1);
wordCount.putIfAbsent("java", 99);          // no-op — "java" already exists
wordCount.merge("java", 1, Integer::sum);   // atomic increment

System.out.println(wordCount.get("java")); // 2
```

Key atomic operations:
- `putIfAbsent(key, value)` — inserts only if absent.
- `computeIfAbsent(key, fn)` — computes and inserts if absent.
- `merge(key, value, fn)` — atomically combines existing and new values.
- `compute(key, fn)` — atomically updates a mapping.

<Note>`ConcurrentHashMap` does not allow `null` keys or `null` values. A `null` value is used internally to signal "key absent", so allowing nulls would be ambiguous. If you need null support, use a sentinel object instead.</Note>

## CopyOnWriteArrayList

`CopyOnWriteArrayList` is ideal for collections that are read far more often than written:

```java
import java.util.concurrent.CopyOnWriteArrayList;

CopyOnWriteArrayList<String> listeners = new CopyOnWriteArrayList<>();
listeners.add("listener-1");
listeners.add("listener-2");

// Safe to iterate while another thread is adding — uses a snapshot
for (String l : listeners) {
    System.out.println(l);
}
```

Every write (add, remove, set) creates a **new copy** of the underlying array. Iterators use the array snapshot taken at their creation time, so they never throw `ConcurrentModificationException` and are not affected by concurrent modifications.

The trade-off: writes are O(n) due to copying. Use `CopyOnWriteArrayList` when writes are rare and the list is small-to-medium (e.g. an event listener registry).

## BlockingQueue — producer-consumer made easy

`BlockingQueue` is the standard interface for producer-consumer patterns:

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;

BlockingQueue<String> queue = new ArrayBlockingQueue<>(10);  // bounded

// Producer — blocks if queue is full
queue.put("task-1");

// Consumer — blocks if queue is empty
String task = queue.take();
```

| Method | Behaviour when full/empty |
|---|---|
| `put(e)` | Blocks until space is available |
| `take()` | Blocks until an element is available |
| `offer(e, t, u)` | Times out |
| `poll(t, u)` | Times out |
| `add(e)` / `remove()` | Throws exception |

`LinkedBlockingQueue` is unbounded (or optionally bounded); `ArrayBlockingQueue` is always bounded, which adds back-pressure that prevents unbounded queue growth.

<Exercise>
Implement a producer-consumer pair. The producer puts 5 strings (`"item-1"` through `"item-5"`) into an `ArrayBlockingQueue<String>` with capacity 2. The consumer takes from the queue and prints each item. The small capacity should cause the producer to block naturally when the queue is full.

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;

public class Main {
    public static void main(String[] args) throws InterruptedException {
        BlockingQueue<String> queue = new ArrayBlockingQueue<>(2);

        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    String item = "item-" + i;
                    queue.put(item);   // blocks when queue is full
                    System.out.println("Produced: " + item);
                }
                queue.put("DONE");   // poison pill
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    String item = queue.take();   // blocks when queue is empty
                    if ("DONE".equals(item)) break;
                    System.out.println("Consumed: " + item);
                    Thread.sleep(80);   // consumer is slower than producer
                }
                System.out.println("Consumer done");
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        producer.start();
        consumer.start();
        producer.join();
        consumer.join();
    }
}
```

**What to verify:**
- All 5 items are produced and consumed
- `"Consumer done"` is printed at the end
- The producer occasionally prints ahead of the consumer, but the lines interleave — output order may vary, demonstrating back-pressure

<Note>Hint: with capacity 2 and a slow consumer, the producer blocks on `put()` after item 2 or 3, waiting for the consumer to `take()` before it can add more. This is back-pressure in action.</Note>
</Exercise>

## ConcurrentLinkedQueue

`ConcurrentLinkedQueue` is a lock-free, unbounded FIFO queue built on CAS operations. It is appropriate when you need high-throughput, non-blocking polling and can tolerate an unbounded queue:

```java
import java.util.concurrent.ConcurrentLinkedQueue;

ConcurrentLinkedQueue<String> queue = new ConcurrentLinkedQueue<>();
queue.offer("a");
queue.offer("b");
System.out.println(queue.poll());  // "a"
System.out.println(queue.size());  // 1
```

Use `ConcurrentLinkedQueue` over `BlockingQueue` when producers should never block (tasks can pile up) and consumers poll opportunistically.

## When to use each

| Collection | Use when |
|---|---|
| `ConcurrentHashMap` | Shared map with concurrent reads and writes |
| `CopyOnWriteArrayList` | Mostly-read list (e.g. listener registries) |
| `ArrayBlockingQueue` | Bounded producer-consumer with back-pressure |
| `LinkedBlockingQueue` | Unbounded (or large) producer-consumer |
| `ConcurrentLinkedQueue` | Non-blocking, high-throughput queue |

## Try it

Implement a producer-consumer pair using `ArrayBlockingQueue`:

```java
import java.util.concurrent.*;

public class Main {
    public static void main(String[] args) throws InterruptedException {
        BlockingQueue<String> queue = new ArrayBlockingQueue<>(5);

        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 8; i++) {
                    String item = "item-" + i;
                    queue.put(item);
                    System.out.println("Produced: " + item);
                    Thread.sleep(50);
                }
                queue.put("DONE");   // poison pill to signal completion
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    String item = queue.take();
                    if ("DONE".equals(item)) break;
                    System.out.println("Consumed: " + item);
                    Thread.sleep(100);
                }
                System.out.println("Consumer finished");
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        producer.start();
        consumer.start();
        producer.join();
        consumer.join();
    }
}
```

The queue's capacity of 5 creates natural back-pressure: the producer blocks when the queue is full, waiting for the consumer to catch up.

## Common mistakes

- **Using `Collections.synchronizedList` and iterating without external synchronisation** — the wrapper synchronises individual calls, but iteration requires manually locking the list for the entire loop. `CopyOnWriteArrayList` handles this transparently.
- **Calling `size()` then `take()` as a pair** — `size() > 0` then `take()` is not atomic; another thread can drain the queue between the two calls. Use `poll()` and check for null, or just `take()` with a timeout.
- **Using `CopyOnWriteArrayList` for write-heavy workloads** — every write copies the entire array; on a large, frequently modified list this is a severe performance problem.
- **Not using `putIfAbsent` or `computeIfAbsent`** — a separate `containsKey` + `put` has a race window. Always use the atomic compound methods on `ConcurrentHashMap`.

## Recap

- `ConcurrentHashMap` allows high-concurrency reads and writes via fine-grained locking; prefer it over `Hashtable` or `synchronizedMap`. No null keys/values.
- `CopyOnWriteArrayList` is safe for concurrent iteration; writes copy the array, so it's best for rarely-modified lists.
- `BlockingQueue` implementations (`ArrayBlockingQueue`, `LinkedBlockingQueue`) provide blocking `put`/`take` for producer-consumer patterns.
- `ConcurrentLinkedQueue` is a non-blocking, unbounded FIFO queue for high-throughput polling.
