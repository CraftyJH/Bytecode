`synchronized` is powerful but coarse: it serialises all access to a block, which can create contention bottlenecks in high-throughput code. Java provides two lighter-weight alternatives for specific scenarios: the `volatile` keyword for simple visibility guarantees, and the `java.util.concurrent.atomic` package for lock-free compound operations. Knowing when to reach for each — and when `synchronized` is still the right choice — is a key concurrency skill.

## The visibility problem

Modern CPUs do not read and write directly to main memory for every operation. They maintain per-core caches, and the JVM compiler can keep a value in a register across multiple iterations of a loop. This is a valid optimization for single-threaded code, but it means that a write by Thread A may not be visible to Thread B without a memory barrier.

```java
// Without volatile — the reader thread may loop forever
public class StopFlag {
    private boolean running = true;

    public void stop() { running = false; }

    public void loop() {
        while (running) { /* work */ }   // may never see running=false
    }
}
```

## volatile — visibility without locking

Declaring a field `volatile` inserts memory barriers around every read and write:

- Every **read** of a volatile field fetches from main memory (never from a cache).
- Every **write** to a volatile field is immediately flushed to main memory.

```java
public class StopFlag {
    private volatile boolean running = true;

    public void stop() { running = false; }

    public void loop() {
        while (running) { /* work */ }   // guaranteed to see the write
        System.out.println("stopped cleanly");
    }
}
```

`volatile` is ideal for **single-writer, single-reader flags** — boolean stop signals, published references to immutable objects. It is not enough when multiple threads perform compound operations on the field.

<Note>`volatile` does NOT make `counter++` atomic. That operation is three steps: read current value, add 1, write back. Two threads can interleave those steps and lose updates, even on a volatile field. Use `AtomicInteger` for counter patterns.</Note>

## AtomicInteger and AtomicLong

The `java.util.concurrent.atomic` package provides classes that perform compound operations atomically using CPU-level compare-and-swap (CAS) instructions — no lock required:

```java
import java.util.concurrent.atomic.AtomicInteger;

AtomicInteger counter = new AtomicInteger(0);

counter.incrementAndGet();   // ++counter, returns new value
counter.getAndIncrement();   // counter++, returns old value
counter.addAndGet(5);        // counter += 5, returns new value
counter.get();               // plain read
```

Running the same 5-thread × 1 000 increments test with `AtomicInteger` always yields exactly 5 000, without any `synchronized`:

```java
import java.util.concurrent.atomic.AtomicInteger;

public class Main {
    static AtomicInteger counter = new AtomicInteger(0);

    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[5];
        for (int i = 0; i < 5; i++) {
            threads[i] = new Thread(() -> {
                for (int j = 0; j < 1000; j++) counter.incrementAndGet();
            });
        }
        for (Thread t : threads) t.start();
        for (Thread t : threads) t.join();
        System.out.println(counter.get());   // always 5000
    }
}
```

## AtomicReference and compareAndSet

`AtomicReference<T>` lets you atomically swap object references. The core operation is **compareAndSet** (CAS):

```java
import java.util.concurrent.atomic.AtomicReference;

AtomicReference<String> config = new AtomicReference<>("v1");

// Only updates if current value is still "v1"
boolean swapped = config.compareAndSet("v1", "v2");
System.out.println(swapped);         // true
System.out.println(config.get());    // v2

// Second attempt fails — value is now "v2"
boolean swapped2 = config.compareAndSet("v1", "v3");
System.out.println(swapped2);        // false
System.out.println(config.get());    // still v2
```

CAS is the building block of all lock-free algorithms. The pattern is: read a value, compute a new value, attempt CAS; if the CAS fails (someone else changed it), retry.

## When to use what

| Scenario | Best tool |
|---|---|
| Simple boolean stop flag | `volatile boolean` |
| Counter incremented by multiple threads | `AtomicInteger` |
| Swapping an object reference atomically | `AtomicReference` |
| Multiple variables must change together | `synchronized` |
| Complex conditional logic on shared state | `synchronized` or `ReentrantLock` |

## Common mistakes

- **Using `volatile` for compound operations** — `volatile int counter; counter++` is still not thread-safe. Use `AtomicInteger`.
- **Assuming `volatile` replaces `synchronized` for complex logic** — if you need to check a value and then update it as one logical unit, you need mutual exclusion (`synchronized` or a lock), not just visibility.
- **Forgetting to read volatile/atomic values into a local variable** — calling `if (counter.get() > 0) doSomethingWith(counter.get())` can race between the two reads. Read once into a local variable when the reads must be consistent.
- **Ignoring the return value of `compareAndSet`** — a false return means your update was lost; in retry-based algorithms you must loop and retry, not silently drop the update.

## Recap

- `volatile` guarantees visibility (no caching) and write-ordering but does NOT make compound operations atomic.
- `AtomicInteger`, `AtomicLong`, and `AtomicReference` provide lock-free atomic operations using hardware CAS instructions.
- `compareAndSet(expected, update)` succeeds only if the current value equals `expected`, returning a boolean result.
- Use atomics for single-variable operations; use `synchronized` when multiple variables must be updated together.
