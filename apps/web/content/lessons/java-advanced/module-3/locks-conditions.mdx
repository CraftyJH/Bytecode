The `synchronized` keyword covers the majority of concurrency needs, but it has real limitations: you cannot attempt to acquire a lock with a timeout, you cannot interrupt a thread that is waiting for a lock, and a single object can only have one associated wait-set. The `java.util.concurrent.locks` package provides explicit lock implementations that overcome these limitations. `ReentrantLock` is the most commonly used; `ReadWriteLock` optimises read-heavy workloads; and `Condition` replaces `wait`/`notify` with a cleaner, more flexible API.

## ReentrantLock — explicit locking

`ReentrantLock` implements `Lock` and works like `synchronized` but with explicit `lock()` and `unlock()` calls:

```java
import java.util.concurrent.locks.ReentrantLock;

public class Counter {
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();   // ALWAYS in finally
        }
    }

    public int getCount() {
        lock.lock();
        try {
            return count;
        } finally {
            lock.unlock();
        }
    }
}
```

<Note>Always release a `ReentrantLock` in a `finally` block. If an exception escapes the `try` and there is no `finally`, the lock is never released and every other thread waiting for it will block forever.</Note>

## tryLock — non-blocking and timed acquisition

```java
// Non-blocking attempt
if (lock.tryLock()) {
    try {
        // got the lock
    } finally {
        lock.unlock();
    }
} else {
    // could not acquire — do something else
}

// Timed attempt
if (lock.tryLock(200, TimeUnit.MILLISECONDS)) {
    try { /* got it within 200ms */ }
    finally { lock.unlock(); }
} else {
    System.out.println("Lock not available — giving up");
}
```

This directly addresses the "no preemption" Coffman condition from lesson 4.

## lockInterruptibly — respond to cancellation

```java
try {
    lock.lockInterruptibly();   // throws InterruptedException if interrupted while waiting
    try {
        doWork();
    } finally {
        lock.unlock();
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    System.out.println("Cancelled while waiting for lock");
}
```

With `synchronized`, there is no way to interrupt a thread that is blocked waiting for a monitor — it will wait forever. `lockInterruptibly()` makes locks responsive to cancellation.

## ReadWriteLock — maximising read throughput

For data structures that are read far more often than written, `ReadWriteLock` provides two separate locks: a shared *read* lock (multiple holders) and an exclusive *write* lock (one holder, no readers):

```java
import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class Cache {
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Map<String, String> map = new HashMap<>();

    public String get(String key) {
        rwLock.readLock().lock();     // shared — many threads can read simultaneously
        try {
            return map.get(key);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    public void put(String key, String value) {
        rwLock.writeLock().lock();    // exclusive — all readers and writers blocked
        try {
            map.put(key, value);
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

## Condition — structured wait/signal

A `Condition` provides `await()` (releases the lock and waits) and `signal()` / `signalAll()` (wakes waiters). Unlike `Object.wait()`/`notify()`, you can create multiple `Condition` objects for the same lock:

```java
import java.util.concurrent.locks.*;

public class BoundedBuffer<T> {
    private final ReentrantLock lock  = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();
    private final Object[]  buffer;
    private int head, tail, count;

    public BoundedBuffer(int capacity) { buffer = new Object[capacity]; }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == buffer.length) notFull.await();  // wait for space
            buffer[tail] = item;
            tail = (tail + 1) % buffer.length;
            count++;
            notEmpty.signal();   // wake a waiting consumer
        } finally { lock.unlock(); }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) notEmpty.await();  // wait for an item
            T item = (T) buffer[head];
            head = (head + 1) % buffer.length;
            count--;
            notFull.signal();   // wake a waiting producer
            return item;
        } finally { lock.unlock(); }
    }
}
```

The two `Condition` objects mean producers only wake consumers and consumers only wake producers — no spurious wake-ups of the wrong party.

        private final ReentrantLock lock = new ReentrantLock();
        private final Condition notFull  = lock.newCondition();

                head = (head + 1) % buf.length;

                notFull.signal();    // wake a waiting producer
                return item;
            } finally { lock.unlock(); }
        }
    }

    public static void main(String[] args) throws InterruptedException {
        BoundedBuffer buffer = new BoundedBuffer();

        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 6; i++) {

                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }

            try {

                    String item = buffer.take();
                    System.out.println("Took: " + item);
                    Thread.sleep(60);   // consumer is slower than producer
                }

        producer.join();
        consumer.join();

## Common mistakes

- **Forgetting `unlock()` in a finally block** — an exception in the critical section that isn't caught will leave the lock acquired forever.
- **Calling `await()` without a loop** — spurious wake-ups can occur; always re-check the condition in a `while` loop, not an `if`.
- **Acquiring the read lock inside a write-locked section without upgrading** — `ReentrantReadWriteLock` does not support lock upgrading (read → write). Attempting it deadlocks. Always upgrade by releasing the read lock first.
- **Using `ReentrantLock` when `synchronized` suffices** — `ReentrantLock` adds code complexity. Use it only when you need `tryLock`, `lockInterruptibly`, or multiple `Condition` objects.

## Recap

- `ReentrantLock.lock()`/`unlock()` gives manual control; always release in `finally`. It adds `tryLock()` and `lockInterruptibly()` over `synchronized`.
- `ReadWriteLock` allows concurrent reads (shared read lock) and exclusive writes, ideal for read-heavy caches.
- `Condition` (from `lock.newCondition()`) replaces `wait`/`notify` with named, per-lock wait-sets, enabling multiple distinct waiting conditions.
- Prefer `synchronized` for simple use cases; escalate to `ReentrantLock` only when you need its extra capabilities.
