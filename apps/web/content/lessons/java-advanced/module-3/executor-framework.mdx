Creating a raw `Thread` for every unit of work is expensive: each thread allocates a stack (typically 512 KB–1 MB), and creating thousands of them exhausts memory and overwhelms the OS scheduler. The **Executor framework**, introduced in Java 5 as part of `java.util.concurrent`, decouples *what* you want to run from *how* it is run. You submit tasks; the framework manages a pool of reusable threads. This is how every production Java application handles concurrent work — from web servers processing HTTP requests to batch processors crunching data.

## The ExecutorService interface

`ExecutorService` extends `Executor` and is the main entry point:

```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

ExecutorService pool = Executors.newFixedThreadPool(4);

pool.submit(() -> System.out.println("Task A"));
pool.submit(() -> System.out.println("Task B"));

pool.shutdown();
```

The factory class `Executors` provides several ready-made implementations.

## Thread pool flavours

### Fixed thread pool

Keeps exactly *n* threads alive. Extra tasks queue up until a thread becomes free:

```java
ExecutorService fixed = Executors.newFixedThreadPool(4);
```

Best for CPU-bound work where you want to cap parallelism at the number of cores.

### Cached thread pool

Spins up new threads on demand and reuses idle ones. Idle threads are recycled after 60 seconds:

```java
ExecutorService cached = Executors.newCachedThreadPool();
```

Good for short-lived, bursty I/O tasks. Avoid for CPU-bound work — it will happily create thousands of threads.

### Single-thread executor

Exactly one thread, tasks run sequentially in submission order:

```java
ExecutorService single = Executors.newSingleThreadExecutor();
```

Useful for serialising access to a non-thread-safe resource without writing `synchronized` everywhere.

<Note>Avoid `Executors.newFixedThreadPool` with an unbounded queue for work that can grow without limit — if tasks arrive faster than they are processed the queue fills heap memory indefinitely. In production, construct a `ThreadPoolExecutor` directly with a bounded queue and an explicit rejection policy.</Note>

## submit() vs execute()

```java
// execute — fire and forget, void return
pool.execute(() -> processOrder(orderId));

// submit — returns a Future
Future<?> future = pool.submit(() -> processOrder(orderId));
future.get();  // blocks until done; rethrows exceptions as ExecutionException
```

`execute()` accepts only `Runnable`. `submit()` accepts `Runnable` or `Callable<T>` and returns a `Future`. Exceptions thrown by a Runnable passed to `execute()` are handed to the thread's uncaught-exception handler and are otherwise lost. With `submit()`, exceptions are captured inside the `Future` and thrown when you call `get()`.

## Lifecycle: shutdown() and shutdownNow()

An `ExecutorService` must be shut down or its threads will keep the JVM alive:

```java
ExecutorService pool = Executors.newFixedThreadPool(3);
try {
    for (int i = 0; i < 10; i++) {
        pool.submit(task);
    }
} finally {
    pool.shutdown();                            // stop accepting new tasks
    pool.awaitTermination(10, TimeUnit.SECONDS); // wait up to 10s for current tasks
}
```

- `shutdown()` — orderly: no new tasks accepted, existing queued tasks complete.
- `shutdownNow()` — aggressive: attempts to interrupt running tasks, returns the list of queued-but-not-started tasks.

## Try it

Submit 10 tasks to a 3-thread pool and observe which thread handles each:

```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.TimeUnit;

public class Main {
    public static void main(String[] args) throws InterruptedException {
        ExecutorService pool = Executors.newFixedThreadPool(3);

        for (int i = 1; i <= 10; i++) {
            final int taskId = i;
            pool.submit(() -> {
                System.out.printf("Task %2d handled by %s%n",
                    taskId, Thread.currentThread().getName());
                try { Thread.sleep(100); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
            });
        }

        pool.shutdown();
        pool.awaitTermination(5, TimeUnit.SECONDS);
        System.out.println("All done");
    }
}
```

You will see only three distinct thread names cycling through the ten tasks — the pool is reusing its three threads.

## Common mistakes

- **Forgetting to call `shutdown()`** — pool threads are non-daemon; the JVM will hang after `main()` returns if the pool is never shut down.
- **Using `newCachedThreadPool()` for CPU-heavy tasks** — it creates a new thread for every task if the pool is saturated, quickly exhausting memory and thrashing the scheduler.
- **Calling `shutdownNow()` and assuming tasks stopped** — `shutdownNow()` sends interrupt signals; tasks that ignore `InterruptedException` will keep running.
- **Not handling `RejectedExecutionException`** — if you submit tasks after `shutdown()`, or if a bounded-queue pool is full, submissions are rejected. Always catch or prevent this.

## Recap

- `ExecutorService` decouples task submission from thread management.
- `newFixedThreadPool(n)` — fixed threads, bounded parallelism. `newCachedThreadPool()` — dynamic threads. `newSingleThreadExecutor()` — serial execution.
- `submit()` returns a `Future`; `execute()` is void. Use `submit()` whenever you need to track completion or handle exceptions.
- Always call `shutdown()` (and optionally `awaitTermination()`) to release pool threads and allow JVM exit.
