Every pattern and primitive covered in this module — threads, locks, atomics, executors, futures, concurrent collections — exists to solve real architectural problems. This capstone lesson ties them together by examining four foundational concurrency patterns that appear repeatedly in production Java: **thread-per-request**, **worker pool**, **pipeline**, and **publish-subscribe**. Understanding these patterns lets you design concurrent systems correctly from the first sketch, rather than bolting on thread-safety as an afterthought. The lesson closes with graceful shutdown — often overlooked but essential for production correctness.

## Pattern 1: Thread-per-request

The simplest model: dedicate one thread to each incoming request for its entire lifecycle. Code is easy to read and debug because each request's logic executes sequentially top-to-bottom.

```java
// Classic servlet-container style (Tomcat does this)
ExecutorService requestPool = Executors.newFixedThreadPool(200);

server.onRequest(request -> {
    requestPool.submit(() -> {
        User user     = db.findUser(request.userId());        // blocks
        List<Order> o = db.findOrders(request.userId());      // blocks
        respond(formatResponse(user, o));
    });
});
```

With **virtual threads** (lesson 11), replace the fixed pool with `newVirtualThreadPerTaskExecutor()` and scale to thousands of concurrent requests at minimal memory cost.

**When to use:** Simple CRUD services, internal tools, any workload where each task is independent and the number of concurrent tasks is bounded.

## Pattern 2: Worker pool

A fixed number of worker threads draw tasks from a shared queue. Producers submit tasks without knowing which worker will handle them:

```java
import java.util.concurrent.*;

public class WorkerPool {
    private final ExecutorService pool;
    private final BlockingQueue<Runnable> queue;

    public WorkerPool(int workers) {
        // Custom ThreadPoolExecutor with a bounded queue and CallerRunsPolicy back-pressure
        queue = new ArrayBlockingQueue<>(100);
        pool  = new ThreadPoolExecutor(workers, workers, 0L, TimeUnit.MILLISECONDS,
                    queue, new ThreadPoolExecutor.CallerRunsPolicy());
    }

    public void submit(Runnable task) { pool.execute(task); }

    public void shutdown() throws InterruptedException {
        pool.shutdown();
        pool.awaitTermination(30, TimeUnit.SECONDS);
    }
}
```

`CallerRunsPolicy` makes the submitting thread run the task itself when the queue is full — natural back-pressure without dropping work.

<Note>Avoid `Executors.newFixedThreadPool` when you need back-pressure: its internal `LinkedBlockingQueue` is unbounded and will happily queue millions of tasks until you hit `OutOfMemoryError`. Build `ThreadPoolExecutor` directly with a bounded `ArrayBlockingQueue` and an explicit rejection policy.</Note>

## Pattern 3: Pipeline

Work flows through independent stages connected by bounded queues. Each stage runs on its own thread (or pool), transforming items and passing them forward:

```java
import java.util.concurrent.*;

public class Pipeline {
    private static final String DONE = "__DONE__";

    public static void main(String[] args) throws InterruptedException {
        BlockingQueue<String> raw       = new ArrayBlockingQueue<>(10);
        BlockingQueue<String> processed = new ArrayBlockingQueue<>(10);
        BlockingQueue<String> output    = new ArrayBlockingQueue<>(10);

        // Stage 0: Producer
        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    raw.put("msg-" + i);
                    System.out.println("[Producer] sent msg-" + i);
                }
                raw.put(DONE);
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        // Stage 1: Transform (add prefix)
        Thread stage1 = new Thread(() -> {
            try {
                while (true) {
                    String item = raw.take();
                    if (DONE.equals(item)) { processed.put(DONE); break; }
                    String transformed = "STAGE1[" + item.toUpperCase() + "]";
                    processed.put(transformed);
                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        // Stage 2: Enrich (add timestamp)
        Thread stage2 = new Thread(() -> {
            try {
                while (true) {
                    String item = processed.take();
                    if (DONE.equals(item)) { output.put(DONE); break; }
                    output.put(item + "@" + System.currentTimeMillis());
                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        // Consumer
        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    String item = output.take();
                    if (DONE.equals(item)) break;
                    System.out.println("[Consumer] " + item);
                }
                System.out.println("[Consumer] finished");
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        producer.start(); stage1.start(); stage2.start(); consumer.start();
        producer.join();  stage1.join();  stage2.join();  consumer.join();
    }
}
```

The bounded queues provide back-pressure: if a stage is slow, the queue upstream fills and pauses the previous stage — the whole pipeline throttles naturally.

Implement a 3-stage pipeline using `BlockingQueue`. A producer puts integers 1–5 into `queue1`. Stage 1 reads from `queue1`, doubles each value, and puts into `queue2`. import java.util.concurrent.*;

```java
public class Main {
    static final int POISON = -1;

    public static void main(String[] args) throws InterruptedException {
        BlockingQueue<Integer> queue1 = new ArrayBlockingQueue<>(5);
        BlockingQueue<Integer> queue2 = new ArrayBlockingQueue<>(5);

        // Producer: puts 1..5 then poison pill
        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    queue1.put(i);
                    System.out.println("Produced: " + i);
                }
                queue1.put(POISON);

        Thread stage1 = new Thread(() -> {
            try {

                    int val = queue1.take();
                    if (val == POISON) { queue2.put(POISON); break; }

                    queue2.put(val * 2);
                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }

        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    int val = queue2.take();
                    if (val == POISON) break;
                    System.out.println("Consumed: " + val);
                }
                System.out.println("Consumer done");

        });

        producer.start(); stage1.start(); consumer.start();
        producer.join();  stage1.join();  consumer.join();
    }
}
```

- Consumed values are `2, 4, 6, 8, 10` (each original doubled)
- `"Consumer done"` is printed last
- Output order may vary between produced/consumed lines

## Pattern 4: Publish-subscribe with ConcurrentHashMap

Decouple event producers from consumers by maintaining a registry of listeners per event type:

```java
import java.util.List;
import java.util.concurrent.*;

public class EventBus {
    private final ConcurrentHashMap<String, CopyOnWriteArrayList<Runnable>> listeners =
        new ConcurrentHashMap<>();

    public void subscribe(String event, Runnable handler) {
        listeners.computeIfAbsent(event, k -> new CopyOnWriteArrayList<>()).add(handler);
    }

    public void unsubscribe(String event, Runnable handler) {
        listeners.computeIfPresent(event, (k, list) -> { list.remove(handler); return list; });
    }

    public void publish(String event) {
        List<Runnable> handlers = listeners.get(event);
        if (handlers != null) handlers.forEach(Runnable::run);
    }
}
```

`CopyOnWriteArrayList` ensures that iterating the handler list in `publish` is safe even if another thread is adding a subscriber at the same time.

Implement a thread-safe publish-subscribe system. Subscribers register a `Runnable` handler for a topic string. When the publisher calls `publish(topic)`, all registered handlers for that topic are invoked. Use `ConcurrentHashMap` with `CopyOnWriteArrayList` for the registry.

```java
import java.util.concurrent.*;

public class Main {
    static class EventBus {
        private final ConcurrentHashMap<String, CopyOnWriteArrayList<Runnable>> registry =
            new ConcurrentHashMap<>();

        public void subscribe(String topic, Runnable handler) {
            // TODO: add handler to the list for this topic (create list if absent)
            registry.computeIfAbsent(topic, k -> new CopyOnWriteArrayList<>()).add(handler);
        }

        public void publish(String topic) {
            // TODO: retrieve the handler list and invoke each handler
            CopyOnWriteArrayList<Runnable> handlers = registry.get(topic);
            if (handlers != null) handlers.forEach(Runnable::run);
        }
    }

    public static void main(String[] args) {
        EventBus bus = new EventBus();

        bus.subscribe("login",  () -> System.out.println("Handler A: user logged in"));
        bus.subscribe("login",  () -> System.out.println("Handler B: send welcome email"));
        bus.subscribe("logout", () -> System.out.println("Handler C: user logged out"));

        System.out.println("-- Publishing login --");
        bus.publish("login");

        System.out.println("-- Publishing logout --");
        bus.publish("logout");

        System.out.println("-- Publishing unknown --");
        bus.publish("unknown");   // no handlers — should not throw
    }
}
```

**What to verify:**
- `"-- Publishing login --"` is followed by both `"Handler A: user logged in"` and `"Handler B: send welcome email"`
- `"-- Publishing logout --"` is followed by `"Handler C: user logged out"`
- `"-- Publishing unknown --"` produces no output and no exception

## Graceful shutdown

A well-behaved concurrent application shuts down cleanly — finishing in-flight work, releasing resources, and not leaving background threads to die with errors:

```java
public class Application {
    private final ExecutorService pool = Executors.newFixedThreadPool(4);
    private volatile boolean running   = true;

    public void start() {
        Runtime.getRuntime().addShutdownHook(new Thread(this::shutdown));
        while (running) {
            pool.submit(this::processNextItem);
        }
    }

    private void shutdown() {
        System.out.println("Shutting down…");
        running = false;
        pool.shutdown();
        try {
            if (!pool.awaitTermination(10, TimeUnit.SECONDS)) {
                List<Runnable> dropped = pool.shutdownNow();
                System.err.println("Forced shutdown; dropped " + dropped.size() + " tasks");
            }
        } catch (InterruptedException e) {
            pool.shutdownNow();
            Thread.currentThread().interrupt();
        }
        System.out.println("Shutdown complete");
    }
}
```

The pattern: `shutdown()` → `awaitTermination(timeout)` → `shutdownNow()` as a last resort.

<Exercise>
**Challenge:** Extend the 3-stage pipeline from the earlier exercise with a `shutdown()` method that combines a volatile stop flag and a poison pill. The pipeline should drain any items already in the queues before stopping, rather than dropping them abruptly.

```java
import java.util.concurrent.*;

public class Main {
    static final int POISON = -1;
    static volatile boolean running = true;

    static BlockingQueue<Integer> queue1 = new ArrayBlockingQueue<>(10);
    static BlockingQueue<Integer> queue2 = new ArrayBlockingQueue<>(10);

    static void shutdown() throws InterruptedException {
        // Set stop flag then inject poison pill to drain remaining items
        running = false;
        queue1.put(POISON);   // signals stage1 to stop after draining
    }

    public static void main(String[] args) throws InterruptedException {
        // Producer: generates items while running
        Thread producer = new Thread(() -> {
            try {
                int i = 1;
                while (running) {
                    queue1.put(i++);
                    Thread.sleep(30);
                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        // Stage 1: doubles items, propagates poison pill
        Thread stage1 = new Thread(() -> {
            try {
                while (true) {
                    int val = queue1.take();
                    if (val == POISON) { queue2.put(POISON); break; }
                    queue2.put(val * 2);
                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        // Consumer: prints items, stops on poison pill
        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    int val = queue2.take();
                    if (val == POISON) break;
                    System.out.println("Consumed: " + val);
                }
                System.out.println("Pipeline drained and stopped");
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        producer.start(); stage1.start(); consumer.start();

        // Let the pipeline run briefly, then shut it down gracefully
        Thread.sleep(150);
        shutdown();

        producer.join(); stage1.join(); consumer.join();
        System.out.println("Shutdown complete");
    }
}
```

**What to verify:**
- Several items are consumed (roughly 4–5 depending on timing)
- `"Pipeline drained and stopped"` is printed — showing remaining queued items were processed before stop
- `"Shutdown complete"` is printed last
- No items already in the queue are dropped

<Note>Hint: setting `running = false` stops the producer from adding new items. Inserting the poison pill into `queue1` then signals stage1 to stop — but only after processing any items ahead of it in the queue, so nothing is lost.</Note>
</Exercise>

## Common mistakes

- **Using unbounded queues in pipelines** — `LinkedBlockingQueue` without a capacity grows without limit. Under sustained load it causes `OutOfMemoryError`. Always choose a capacity and handle back-pressure explicitly.
- **Skipping graceful shutdown** — threads left running after the application "stops" can corrupt shared state, exhaust resources, or delay JVM exit. Always provide a `shutdown()` path.
- **Using ArrayList in concurrent publish-subscribe** — a plain `ArrayList` in the listener registry throws `ConcurrentModificationException` if a subscriber is added while an event is being published. Use `CopyOnWriteArrayList`.
- **Blocking threads in short-lived tasks** — submitting a task to a pool and immediately calling `.get()` on its `Future` negates the pool's benefit. Chain with `thenAccept` / `CompletableFuture` instead.

## Module recap

You have now completed Module 3: Concurrency. Here is what you have learned:

- **Threads & Runnable** — the fundamental unit of concurrency; `start()` vs `run()`, daemon threads, lifecycle.
- **Synchronisation** — `synchronized` provides mutual exclusion and visibility; both reads and writes need protection.
- **volatile & Atomics** — `volatile` solves visibility; `AtomicInteger`/`AtomicReference` add lock-free compound operations.
- **Deadlock** — four Coffman conditions; prevent with consistent lock ordering or `tryLock` with timeout.
- **Executor framework** — `ExecutorService` decouples task submission from thread management; always shut down cleanly.
- **Callable & Future** — return typed results from background tasks; `get()` surfaces exceptions via `ExecutionException`.
- **CompletableFuture (Part 1)** — non-blocking async pipelines with `supplyAsync`, `thenApply`, `thenCompose`, `exceptionally`.
- **CompletableFuture (Part 2)** — combining futures with `thenCombine`, `allOf`, `anyOf`; unified error handling with `handle`.
- **Thread-safe collections** — `ConcurrentHashMap`, `CopyOnWriteArrayList`, `BlockingQueue` for correct concurrent data structures.
- **ReentrantLock & Conditions** — explicit locking with `tryLock`, `ReadWriteLock` for read-heavy workloads, `Condition` for fine-grained signalling.
- **Virtual threads** — JVM-managed lightweight threads (Java 21) for scalable I/O-bound concurrency without reactive complexity.
- **Concurrency patterns** — thread-per-request, worker pool, pipeline with back-pressure, publish-subscribe, graceful shutdown.

Concurrency is hard. The tools are mature, the primitives are well-documented, but the bugs are subtle and timing-dependent. The most important habit is to minimise shared mutable state: the less data crosses thread boundaries, the less there is to get wrong. When sharing is necessary, choose the highest-level safe abstraction available — `BlockingQueue` over hand-rolled producer-consumer, `ConcurrentHashMap` over a synchronised `HashMap`, `CompletableFuture` over manually joined threads. Write tests that exercise code under concurrent load using tools like `jcstress` or simple multi-thread stress tests. And always read thread dumps when production concurrency issues arise — the JVM tells you exactly which thread holds which lock.
