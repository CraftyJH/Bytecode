For decades, the Java threading model worked like this: one platform thread per concurrent task, each backed by an OS thread. This scaled to thousands of threads but hit a wall at higher concurrency — OS threads consume significant memory (typically 512 KB–1 MB of stack) and context-switching them is expensive. Frameworks responded by introducing callback-based and reactive programming, which are correct but notoriously hard to read and debug. Project Loom, delivered as a preview in Java 19 and finalised in Java 21, solves this at the JVM level with **virtual threads** — making the simple thread-per-task model scalable to millions of concurrent tasks.

## Platform threads vs virtual threads

| Property | Platform thread | Virtual thread |
|---|---|---|
| Managed by | OS | JVM |
| Typical stack | 512 KB–1 MB | ~1 KB (grows on demand) |
| Practical limit | ~10 000 | ~1 000 000+ |
| Blocking behavior | Holds OS thread | Unmounts from carrier thread |
| Best for | CPU-bound work | I/O-bound work |

A virtual thread is *mounted* on a carrier (platform) thread from a small pool. When it blocks (I/O, `sleep`, lock waiting), it *unmounts* — the carrier thread is freed to run another virtual thread. When the blocking operation completes, the virtual thread is rescheduled on a (possibly different) carrier.

## Creating virtual threads

### Thread.ofVirtual()

```java
Thread vt = Thread.ofVirtual()
    .name("my-virtual-thread")
    .start(() -> {
        System.out.println("Running: " + Thread.currentThread().isVirtual());
        // isVirtual() returns true
    });
vt.join();
```

### Executors.newVirtualThreadPerTaskExecutor()

The idiomatic way for server-style concurrency: one virtual thread per submitted task:

```java
try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        exec.submit(() -> {
            Thread.sleep(100);   // simulate I/O wait — unmounts the virtual thread
            return "done";
        });
    }
}  // auto-closes, waits for all tasks
System.out.println("All 10 000 tasks done");
```

This starts 10 000 virtual threads concurrently. With platform threads this would likely crash with `OutOfMemoryError`; with virtual threads it completes in ~100 ms.

<Note>Virtual threads are pinned to their carrier platform thread while executing `synchronized` blocks or native methods. If your virtual thread blocks inside a `synchronized` block, it holds the carrier thread — effectively behaving like a platform thread for that duration. Replace `synchronized` with `ReentrantLock` in code that must not pin.</Note>

## When virtual threads shine

Virtual threads are transformative for **I/O-bound, thread-per-task** patterns:

```java
// HTTP server handler — one virtual thread per request
try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) {
    server.setExecutor(exec);
    server.start();
}
```

Each request blocks on database queries, HTTP calls, file reads — all of which unmount the virtual thread. The same server that previously handled 500 concurrent requests with a fixed thread pool can now handle 50 000 with virtual threads.

**Virtual threads do NOT help with:**
- CPU-bound computation (image processing, encryption) — use `ForkJoinPool` or a fixed platform-thread pool sized to CPU cores.
- Code with heavy `synchronized` usage — pinning negates the benefit.

## Structured concurrency (preview)

Java 21 also previews **Structured Concurrency** (`StructuredTaskScope`) which enforces that subtasks started within a scope are all completed or cancelled before the scope exits:

```java
import java.util.concurrent.StructuredTaskScope;

try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<String> user  = scope.fork(() -> fetchUser(id));
    Subtask<Integer> score = scope.fork(() -> fetchScore(id));

    scope.join();           // wait for both
    scope.throwIfFailed();  // propagate any exception

    System.out.println(user.get() + " scored " + score.get());
}
```

Structured concurrency makes it impossible for subtask lifetimes to leak beyond their enclosing scope — a significant safety improvement over raw `CompletableFuture` chains.

## Common mistakes

- **Using virtual threads for CPU-bound work** — the benefit is for I/O waiting; CPU-heavy code needs real parallelism, not more threads on the same cores.
- **Using `synchronized` in hot paths with virtual threads** — pinning holds carrier threads and reintroduces the scalability problem. Profile and replace with `ReentrantLock`.
- **Using `ThreadLocal` in virtual thread code** — with millions of virtual threads, one `ThreadLocal` per thread means millions of map entries. Consider passing data explicitly or using scoped values (JEP 446).
- **Expecting virtual threads to improve latency** — a single task's latency is unchanged. Virtual threads improve *throughput* (more concurrent tasks) not the speed of individual operations.

## Recap

- Virtual threads (Java 21) are JVM-managed, lightweight threads that unmount from OS threads while blocked, enabling millions of concurrent tasks.
- Create with `Thread.ofVirtual().start(...)` or `Executors.newVirtualThreadPerTaskExecutor()`.
- They excel for I/O-bound, thread-per-task patterns; they do not help CPU-bound workloads.
- Avoid `synchronized` in virtual threads that block — use `ReentrantLock` to prevent carrier thread pinning.
- Structured concurrency (`StructuredTaskScope`) enforces task-lifetime scoping and simplifies cancellation.
