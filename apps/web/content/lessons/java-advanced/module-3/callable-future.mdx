`Runnable` is fine when you want to fire off work and don't need a result. But most real tasks *produce* something: a database query returns rows, an HTTP call returns a response body, a computation returns a number. Java's `Callable<T>` interface and `Future<T>` handle exactly this scenario — submit work that returns a typed value, retrieve it later (blocking only when the result is actually needed), and propagate any exceptions that occur on the background thread back to the caller.

## Callable vs Runnable

`Runnable`:
```java
void run();          // no return value, cannot throw checked exceptions
```

`Callable<T>`:
```java
T call() throws Exception;   // returns T, can throw any exception
```

`Callable` is also a functional interface, so lambdas work perfectly:

```java
Callable<Integer> computePi = () -> {
    Thread.sleep(200);   // simulate work
    return 314159;
};
```

## Submitting a Callable and getting a Future

```java
ExecutorService exec = Executors.newFixedThreadPool(2);

Future<String> future = exec.submit(() -> {
    Thread.sleep(100);
    return "result";
});

// Do other work here — the Callable runs concurrently
System.out.println("Working on something else…");

// Block until the result is ready
String result = future.get();   // throws ExecutionException or InterruptedException
System.out.println(result);

exec.shutdown();
```

`get()` blocks the calling thread until the task completes and returns the value. If the task threw an exception, `get()` wraps it in `ExecutionException`.

## Future's full API

```java
future.get();                       // block indefinitely
future.get(2, TimeUnit.SECONDS);    // block at most 2 seconds (throws TimeoutException)
future.isDone();                    // true if finished (normally, exceptionally, or cancelled)
future.isCancelled();               // true if cancel() succeeded
future.cancel(true);               // attempt to cancel; true = interrupt if running
```

<Note>`future.get()` with no timeout will block forever if the task never completes (e.g. it's stuck in an infinite loop or waiting for I/O that never arrives). Always prefer `get(timeout, unit)` in production code and handle `TimeoutException` by cancelling the future or logging a warning.</Note>

## Handling ExecutionException

```java
try {
    String value = future.get();
} catch (ExecutionException e) {
    Throwable cause = e.getCause();   // the original exception from the Callable
    if (cause instanceof IOException) {
        System.err.println("I/O failed: " + cause.getMessage());
    } else {
        throw new RuntimeException("Unexpected failure", cause);
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();   // restore interrupt flag
    System.err.println("Interrupted while waiting");
}
```

## Collecting multiple results with invokeAll

When you have a batch of homogeneous tasks, `invokeAll` submits them all and returns a `List<Future<T>>` only after *all* tasks complete:

```java
List<Callable<Integer>> tasks = List.of(
    () -> 1 + 1,
    () -> 2 * 3,
    () -> 10 - 4
);

List<Future<Integer>> results = exec.invokeAll(tasks);

for (Future<Integer> f : results) {
    System.out.println(f.get());   // 2, 6, 6
}
```

## Try it

Submit 5 Callable tasks that compute squares, collect the `Future` results, and print them:

```java
import java.util.*;
import java.util.concurrent.*;

public class Main {
    public static void main(String[] args) throws Exception {
        ExecutorService exec = Executors.newFixedThreadPool(3);
        List<Future<Integer>> futures = new ArrayList<>();

        for (int i = 1; i <= 5; i++) {
            final int n = i;
            futures.add(exec.submit(() -> {
                Thread.sleep(50);   // simulate work
                return n * n;
            }));
        }

        System.out.println("All tasks submitted, now collecting results:");

        for (int i = 0; i < futures.size(); i++) {
            try {
                System.out.printf("%d² = %d%n", i + 1, futures.get(i).get());
            } catch (ExecutionException e) {
                System.err.println("Task " + (i+1) + " failed: " + e.getCause());
            }
        }

        exec.shutdown();
    }
}
```

Expected output (order may vary in submission but results collected in order):
```
All tasks submitted, now collecting results:
1² = 1
2² = 4
3² = 9
4² = 16
5² = 25
```

## Common mistakes

- **Calling `get()` immediately after `submit()`** — this defeats concurrency entirely; you block immediately and the task runs sequentially. Submit all tasks first, then collect results.
- **Not unwrapping `ExecutionException`** — `e.getMessage()` on an `ExecutionException` is often unhelpful; always call `e.getCause()` to inspect the real failure.
- **Ignoring `InterruptedException`** — catching and swallowing it silences the JVM's cooperative cancellation mechanism. Always restore the interrupt flag.
- **Using `future.get()` without a timeout in production** — a stuck background task will block the calling thread forever. Use `get(timeout, unit)` and handle `TimeoutException`.

## Recap

- `Callable<T>` is like `Runnable` but returns a typed value and can throw checked exceptions.
- `ExecutorService.submit(callable)` returns a `Future<T>` representing the pending result.
- `future.get()` blocks until the result is ready; it throws `ExecutionException` if the task failed and `InterruptedException` if the waiting thread was interrupted.
- `invokeAll(callables)` submits a batch and returns a `List<Future<T>>` once all tasks complete.
