JUnit 5 is the de-facto standard test framework for Java. It ships in three modules — JUnit Platform (the launcher), JUnit Jupiter (the annotations and assertions you use daily), and JUnit Vintage (for running JUnit 4 tests). This lesson focuses on Jupiter, the part you interact with directly when writing tests.

## Core lifecycle annotations

JUnit 5 controls test execution through a set of lifecycle annotations:

```java
class BankAccountTest {

    BankAccount account;

    @BeforeAll
    static void initDatabase() {
        // Runs once before any test in this class.
        // Must be static (unless @TestInstance(PER_CLASS) is used).
        System.out.println("Setting up shared resources");
    }

    @BeforeEach
    void setUp() {
        // Runs before EACH test method — gives each test a fresh state.
        account = new BankAccount(0.0);
    }

    @Test
    void depositAddsToBalance() {
        account.deposit(100.0);
        assertEquals(100.0, account.getBalance());
    }

    @AfterEach
    void tearDown() {
        // Runs after each test — close connections, reset state, etc.
    }

    @AfterAll
    static void cleanUpDatabase() {
        // Runs once after all tests in this class complete.
    }
}
```

The most important lifecycle rule: **JUnit 5 creates a new instance of the test class for every @Test method**. That is why `setUp()` can assign `account = new BankAccount(...)` and every test gets a clean, independent account without any shared-state bugs.

## @DisplayName

Test method names have to be valid Java identifiers — no spaces, no punctuation. `@DisplayName` lets you write a human-readable label shown in the IDE and CI output:

```java
@Test
@DisplayName("Deposit of negative amount throws IllegalArgumentException")
void negativeDepositThrows() {
    assertThrows(IllegalArgumentException.class,
        () -> account.deposit(-10.0));
}
```

<Note>Some teams skip @DisplayName and instead follow a strict naming convention such as `given_when_then` or `should_when_given`. Both approaches are valid — pick one and apply it consistently across the codebase.</Note>

## Assertions

JUnit 5's `Assertions` class (statically imported in most test files) covers the full set:

```java
import static org.junit.jupiter.api.Assertions.*;

// Basic equality
assertEquals(150.0, account.getBalance(), "balance after deposit");

// Boolean conditions
assertTrue(account.isOpen());
assertFalse(account.isClosed());

// Null checks
assertNotNull(account.getId());

// Exception assertion — returns the exception for further inspection
IllegalArgumentException ex = assertThrows(
    IllegalArgumentException.class,
    () -> account.withdraw(9999.0)
);
assertEquals("Insufficient funds", ex.getMessage());

// All-or-nothing block — reports every failure, not just the first
assertAll("account state",
    () -> assertEquals(100.0, account.getBalance()),
    () -> assertTrue(account.isOpen()),
    () -> assertNotNull(account.getId())
);
```

`assertAll` is particularly valuable when checking multiple properties of a result: without it, a failure on the first assertion hides whether the remaining ones pass.

## @Disabled

Sometimes a test cannot run in the current environment — it depends on a service that isn't available in CI, or it's a known failing case being tracked in an issue. `@Disabled` skips it without deleting it:

```java
@Test
@Disabled("Flaky in CI until DB migration is complete — see issue #417")
void withdrawCreatesAuditLog() {
    // ...
}
```

Skipped tests show up in the test report so they stay visible and don't silently rot.

## Nested tests

`@Nested` inner classes let you group related tests and share nested `@BeforeEach` setup without polluting the outer class:

```java
class BankAccountTest {

    BankAccount account;

    @BeforeEach
    void createAccount() { account = new BankAccount(100.0); }

    @Nested
    @DisplayName("Deposit")
    class DepositTests {
        @Test void positiveAmountIncreasesBalance() { /* ... */ }
        @Test void negativeAmountThrows() { /* ... */ }
    }

    @Nested
    @DisplayName("Withdraw")
    class WithdrawTests {
        @Test void sufficientFundsDecreasesBalance() { /* ... */ }
        @Test void insufficientFundsFails() { /* ... */ }
    }
}
```

The IDE renders nested classes as a tree, making large test files easy to navigate.

## Try it

Write a complete test class for `BankAccount` with the following behaviour:

```java
public class BankAccount {
    private double balance;
    public BankAccount(double initialBalance) { this.balance = initialBalance; }
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Amount must be positive");
        balance += amount;
    }
    public void withdraw(double amount) {
        if (amount > balance) throw new IllegalStateException("Insufficient funds");
        balance -= amount;
    }
    public double getBalance() { return balance; }
}
```

Your test class should cover:
1. Deposit increases balance correctly
2. Deposit with zero or negative amount throws `IllegalArgumentException`
3. Withdraw reduces balance correctly
4. Withdraw more than the balance throws `IllegalStateException`
5. Check both the exception type *and* the message for the overdraft case using `assertThrows`

Use `@BeforeEach` to create a fresh account for every test and `@DisplayName` on at least two tests.

## Common mistakes

- **Sharing mutable state across tests without @BeforeEach** — if two tests modify the same object and test execution order changes, one test will corrupt the other's state. Use `@BeforeEach` to create a fresh instance per test.
- **Using @BeforeAll on a non-static method** — JUnit 5 creates a new test instance per method, so a non-static @BeforeAll has no single instance to call. Add `static` or switch to `@TestInstance(Lifecycle.PER_CLASS)` when you intentionally share state.
- **Catching exceptions manually instead of assertThrows** — wrapping code in a try/catch and calling `fail()` in the catch is more verbose and error-prone. `assertThrows` is cleaner and returns the exception for inspection.
- **Putting multiple unrelated scenarios in one @Test** — when that test fails you don't know which scenario broke. One clear scenario per test method makes failures immediately actionable.

## Recap

- **@BeforeAll / @AfterAll** run once per class (must be static by default)
- **@BeforeEach / @AfterEach** run before/after every individual test method
- JUnit creates a **new test instance per @Test** to prevent shared-state bugs
- **assertThrows** returns the caught exception — assert its message too
- **assertAll** reports every failure in a block, not just the first
- **@Nested** classes group related tests and can have their own lifecycle hooks
- **@Disabled** skips a test visibly rather than deleting it
