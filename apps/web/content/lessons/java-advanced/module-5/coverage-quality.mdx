A project's test coverage percentage is one of the first numbers a new team member sees on the CI dashboard — and one of the most misunderstood. Coverage tells you which code was *executed* during the test run. It says nothing about whether the tests were *correct*. Understanding what coverage measures, and equally what it does not, is essential for building a test suite you can actually trust.

## Line coverage vs branch coverage

**Line coverage** marks each executable line as covered or uncovered. A line is covered if any test causes its execution at least once.

**Branch coverage** tracks every possible outcome of every decision point. An `if (condition)` has two branches: true and false. A `switch` with four cases has four branches. Line coverage can be 100% while a significant branch is never exercised.

```java
public String classify(int n) {
    if (n > 0) {           // line 2
        return "positive"; // line 3
    }
    return "non-positive"; // line 5
}
```

If a test only calls `classify(5)`:
- **Line coverage:** lines 2, 3 are covered (line 5 is not — 75% line coverage)
- **Branch coverage:** true branch covered, false branch (n ≤ 0) not covered

If a second test calls `classify(-1)`:
- **Line coverage:** all four lines covered (100%)
- **Branch coverage:** both branches covered (100%)

Now consider a more subtle case:

```java
public void process(String value) {
    if (value != null) {        // branch: null and non-null
        save(value.trim());
    }
    // no else — null case falls through silently
}
```

100% line coverage is achievable by only passing a non-null value. Branch coverage exposes the untested null case.

## JaCoCo basics

JaCoCo (Java Code Coverage) is the standard coverage tool for Maven and Gradle projects. Add it to `pom.xml`:

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.12</version>
    <executions>
        <execution>
            <goals><goal>prepare-agent</goal></goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals><goal>report</goal></goals>
        </execution>
    </executions>
</plugin>
```

Run `mvn test` and open `target/site/jacoco/index.html`. JaCoCo colour-codes each line green (fully covered), yellow (partially covered — some branches missed), or red (not covered at all).

<Note>JaCoCo's yellow highlighting is your most actionable signal. A yellow `if` line means some branches are tested and others are not — exactly the kind of subtle gap that hides real bugs. Prioritise yellows over hunting for more green lines.</Note>

## Why 100% coverage is not the goal

Consider this test:

```java
@Test
void coverageGame() {
    String result = classify(5);
    // No assertion — the method ran, so the line is covered
}
```

This test contributes to coverage metrics but verifies nothing. A method that returns the wrong value, throws an unexpected exception, or corrupts state will not be caught by it. **Coverage measures execution, not correctness.**

The real goal is *meaningful assertions* on covered code. A 70% coverage suite where every test has clear assertions is far more valuable than a 100% suite padded with assertion-free calls.

## Mutation testing — evaluating test quality

Mutation testing answers the question coverage cannot: "If I introduced a bug right here, would any test catch it?"

A mutation testing tool (PIT/Pitest is the Java standard) automatically:
1. Copies the production code
2. Applies one small change (*mutation*): negates a boolean, replaces `+` with `-`, removes a return statement, changes `>` to `>=`, etc.
3. Runs the full test suite against the mutated code
4. Records whether any test failed

A **killed mutant** — the test suite detected the change — is good. A **surviving mutant** — all tests passed despite the bug — reveals a gap.

```java
// Original
if (age >= 18) return "adult";

// Mutant: > instead of >=
if (age > 18) return "adult";
```

If no test checks the boundary value `age == 18`, this mutant survives. The test suite has 100% line coverage but misses the boundary condition that matters most.

## Common mistakes

- **Treating coverage percentage as a quality target** — "we must reach 80%" leads teams to write meaningless tests just to hit the number. Coverage should be a *diagnostic* tool, not a goal.
- **Ignoring branch coverage in favour of line coverage** — most CI pipelines are configured for line coverage because it is simpler. Add branch coverage checks too; the extra configuration is minimal and the signal is significantly more valuable.
- **Not running mutation testing** — even with 100% branch coverage, a suite can miss boundary conditions. Running PIT/Pitest on CI at least periodically catches these gaps before production does.
- **Excluding generated or trivial code from coverage and forgetting about it** — excluding getter/setter boilerplate is reasonable. Excluding complex business logic from the coverage report because it was hard to test is a red flag that indicates untested risk.

## Recap

- **Line coverage** marks lines as executed; **branch coverage** requires every if/switch outcome to be tested
- JaCoCo's **yellow lines** show partially covered branches — the most actionable coverage signal
- 100% coverage does not mean bug-free: a test with no assertions contributes coverage but verifies nothing
- **Mutation testing** (PIT/Pitest) evaluates whether your tests would catch real bugs by introducing deliberate faults
- Use coverage as a diagnostic to find gaps, not as a target to maximise for its own sake
