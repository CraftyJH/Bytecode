Unit tests verify individual classes in isolation. Integration tests verify that those classes collaborate correctly when connected to real implementations — real repositories, real serialisers, real HTTP handlers. This capstone lesson covers the strategies teams use to test across component boundaries without sacrificing speed, reproducibility, or confidence.

## Unit vs integration vs system testing

The three layers of the test pyramid serve different purposes and make different trade-offs:

| Type | Tests | Uses real | Speed | Typical count |
|---|---|---|---|---|
| Unit | One class | Nothing | ~1ms | Hundreds |
| Integration | 2+ components | DB, serialiser, HTTP stub | ~100ms | Tens |
| System / E2E | Full stack | Everything | Seconds–minutes | Few |

A common misconception is that integration tests must be slow. An integration test that hits a real in-memory H2 database or an in-process HTTP server can run in under 100ms. Speed depends on the infrastructure, not the classification.

## Testing with a real in-memory repository

The simplest form of integration test replaces a mock with a lightweight real implementation. An in-memory repository backed by a `HashMap` or `ArrayList` is a *fake* — real logic, production-unsuitable infrastructure:

```java
class InMemoryProductRepository implements ProductRepository {
    private final Map<Long, Product> store = new HashMap<>();
    private long nextId = 1;

    @Override
    public Product save(Product product) {
        product.setId(nextId++);
        store.put(product.getId(), product);
        return product;
    }

    @Override
    public Optional<Product> findById(Long id) {
        return Optional.ofNullable(store.get(id));
    }

    @Override
    public List<Product> findAll() {
        return new ArrayList<>(store.values());
    }
}
```

An integration test wires the real service to this fake — no mocks, no framework, just business logic exercised against a realistic data layer:

```java
class ProductServiceIntegrationTest {
    ProductRepository repository;
    ProductService service;

    @BeforeEach
    void setUp() {
        repository = new InMemoryProductRepository();  // fresh per test
        service = new ProductService(repository);
    }

    @Test
    void savedProductCanBeRetrievedById() {
        Product saved = service.createProduct("Widget", 9.99);

        Optional<Product> found = service.findById(saved.getId());

        assertTrue(found.isPresent());
        assertEquals("Widget", found.get().getName());
        assertEquals(9.99, found.get().getPrice());
    }

    @Test
    void duplicateProductNameThrowsDuplicateProductException() {
        service.createProduct("Gadget", 19.99);

        assertThrows(DuplicateProductException.class,
            () -> service.createProduct("Gadget", 5.00));
    }
}
```

No Spring context, no database connection, no network. These tests run in milliseconds and give full confidence in the service layer's logic.

Implement an in-memory `TaskRepository`, a `TaskService` layer on top, and write 3 integration checks: create a task, find it by id, and mark it complete — using only real objects, no mocks.

```java
import java.util.*;

public class Main {
    static void check(boolean cond, String msg) {
        if (!cond) throw new AssertionError(msg);
        System.out.println("PASS: " + msg);
    }

    enum Status { PENDING, DONE }

    static class Task {
        Long id;
        String title;
        Status status;
        Task(String title) { this.title = title; this.status = Status.PENDING; }
    }

    interface TaskRepository {
        Task save(Task task);
        Optional<Task> findById(Long id);
    }

    static class InMemoryTaskRepository implements TaskRepository {
        private final Map<Long, Task> store = new HashMap<>();
        private long nextId = 1;

        @Override
        public Task save(Task task) {
            if (task.id == null) task.id = nextId++;
            store.put(task.id, task);
            return task;
        }

        @Override
        public Optional<Task> findById(Long id) {
            return Optional.ofNullable(store.get(id));
        }
    }

    static class TaskService {
        private final TaskRepository repo;
        TaskService(TaskRepository repo) { this.repo = repo; }

        Task create(String title) {
            if (title == null || title.isBlank())
                throw new IllegalArgumentException("Title required");
            return repo.save(new Task(title));
        }

        void complete(Long id) {
            Task t = repo.findById(id)
                .orElseThrow(() -> new NoSuchElementException("Task not found: " + id));
            t.status = Status.DONE;
            repo.save(t);
        }
    }

    public static void main(String[] args) {
        TaskRepository repo = new InMemoryTaskRepository();
        TaskService service  = new TaskService(repo);

        // Check 1: create task
        Task created = service.create("Write tests");
        check(created.id != null,               "created task has an id");
        check("Write tests".equals(created.title), "created task has correct title");

        // Check 2: find by id
        Optional<Task> found = repo.findById(created.id);
        check(found.isPresent(),                "task can be found by id");

        // Check 3: mark complete
        service.complete(created.id);
        Task updated = repo.findById(created.id).orElseThrow();
        check(updated.status == Status.DONE,    "completed task has DONE status");
    }
}
```

**What to verify:**
- All 4 checks print `PASS:`
- Removing `task.id = nextId++` from `save()` causes check 1 to fail

<Note>Hint: give each check a distinct, readable description so a failure message immediately tells you which integration point broke.</Note>
</Exercise>

## Test containers — real databases in CI

When you need exact production database behaviour (specific SQL dialect, constraint enforcement, stored procedures), Testcontainers spins up a real Docker container during the test run:

```java
@Testcontainers
class ProductRepositoryJpaTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16")
        .withDatabaseName("testdb")
        .withUsername("test")
        .withPassword("test");

    @DynamicPropertySource
    static void overrideProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired ProductRepository repository;

    @Test
    void uniqueConstraintIsEnforcedByDatabase() {
        repository.save(new Product("UniqueItem", 10.0));
        assertThrows(DataIntegrityViolationException.class,
            () -> repository.save(new Product("UniqueItem", 20.0)));
    }
}
```

The container starts once for the test class (thanks to `static @Container`), is shared across all tests in the class, and is stopped automatically when the JVM exits.

<Note>Testcontainers requires Docker to be available on the machine running the tests. Most CI platforms (GitHub Actions, GitLab CI, CircleCI) support Docker out of the box. Developers without Docker installed can fall back to an H2 profile with a separate Maven profile flag.</Note>

```java
import java.util.*;

public class Main {
    static void check(boolean cond, String msg) {
        if (!cond) throw new AssertionError(msg);
        System.out.println("PASS: " + msg);
    }

    static class Task {
        Long id; String title;
        Task(String title) { this.title = title; }
    }

    interface TaskRepository {
        Task save(Task task);
        Optional<Task> findById(Long id);
    }

    // Strict repository: rejects blank titles at the persistence layer
    static class StrictInMemoryRepo implements TaskRepository {
        private final Map<Long, Task> store = new HashMap<>();
        private long nextId = 1;

        @Override
        public Task save(Task task) {
            if (task.title == null || task.title.isBlank())
                throw new IllegalArgumentException("Repo: title must not be blank");
            task.id = nextId++;
            store.put(task.id, task);
            return task;
        }

        @Override
        public Optional<Task> findById(Long id) {
            return Optional.ofNullable(store.get(id));
        }
    }

    static class TaskService {
        private final TaskRepository repo;
        TaskService(TaskRepository repo) { this.repo = repo; }

        Task create(String title) {
            // Service validates first, then delegates to repo
            // TODO: throw IllegalArgumentException("Title required") for blank title
            if (title == null || title.isBlank())
                throw new IllegalArgumentException("Title required");
            return repo.save(new Task(title));
        }
    }

    public static void main(String[] args) {
        TaskRepository repo    = new StrictInMemoryRepo();
        TaskService    service = new TaskService(repo);

        // Check: blank title — service should throw IllegalArgumentException
        IllegalArgumentException caught = null;
        try {
            service.create("   ");
        } catch (IllegalArgumentException e) {
            caught = e;
        }
        check(caught != null, "service throws IllegalArgumentException for blank title");
        check(caught != null && caught.getMessage() != null
              && caught.getMessage().contains("Title"),
              "exception message mentions 'Title'");

        // Check: normal title still works after a failed attempt
        Task ok = service.create("Valid task");
        check(ok.id != null, "valid task saved successfully after failed attempt");
    }
}
```

**What to verify:**
- All 3 checks print `PASS:`
- Removing the blank-title guard from `TaskService.create()` causes the first check to fail (exception propagates from the repo with a different message, causing the second check to fail too)

<Note>Hint: the service layer should validate *before* calling the repo — this gives a consistent error message regardless of which persistence backend is used.</Note>

## @SpringBootTest overview

For testing the full Spring application context — controllers, services, repositories, and all their wiring — JUnit 5 + Spring Boot provides `@SpringBootTest`:

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class ProductApiIntegrationTest {

    @Autowired TestRestTemplate restTemplate;

    @Test
    void createProductReturns201() {
        ResponseEntity<Product> response = restTemplate.postForEntity(
            "/api/products",
            new CreateProductRequest("Thingamajig", 14.99),
            Product.class
        );

        assertEquals(HttpStatus.CREATED, response.getStatusCode());
        assertNotNull(response.getBody().getId());
    }
}
```

`@SpringBootTest` starts the full context and can boot an embedded HTTP server on a random port. It is the slowest option but the most realistic. Spring Boot's slice annotations — `@DataJpaTest`, `@WebMvcTest`, `@JsonTest` — load only the relevant part of the context and are significantly faster.

## WireMock — stubbing HTTP dependencies

When the system under test calls an external HTTP service, WireMock starts an in-process stub server that returns pre-programmed responses:

```java
@SpringBootTest
@AutoConfigureWireMock(port = 0)  // random port, configured automatically
class PaymentClientIntegrationTest {

    @Autowired PaymentClient paymentClient;

    @Test
    void successfulChargeReturnsTransactionId() {
        stubFor(post(urlEqualTo("/charge"))
            .willReturn(aResponse()
                .withStatus(200)
                .withHeader("Content-Type", "application/json")
                .withBody("{\"transactionId\": \"txn-123\"}")));

        String txnId = paymentClient.charge(50.0);

        assertEquals("txn-123", txnId);
    }
}
```

The real `PaymentClient` makes an actual HTTP call — WireMock intercepts it without any network egress, making the test fast and deterministic.

## Contract testing concept

In a microservices architecture, Service A (consumer) depends on Service B's (provider) API. Both evolve independently. Contract testing (using tools like Pact) lets the consumer record its expectations as a *contract file*, and the provider verifies it can satisfy the contract in its own CI pipeline — without either service running simultaneously.

This catches breaking API changes at the earliest possible point — in CI before deployment — rather than in an integration environment at 2am.

```java
public interface TaskRepository {
    Task save(Task task);
    Optional<Task> findById(Long id);
    List<Task> findAllByStatus(TaskStatus status);
    void delete(Long id);

public class TaskService {
    private final TaskRepository repository;

    public Task create(String title) {

    public void complete(Long id) {
        Task task = repository.findById(id)

        task.setStatus(TaskStatus.DONE);

    }
}
```

## Common mistakes

- **Writing integration tests for everything** — integration tests are slower and more complex than unit tests. Start with unit tests; add integration tests at the boundaries that matter (database queries, HTTP clients, event serialisation).
- **Using @SpringBootTest when a slice annotation would suffice** — loading the full application context to test one JPA repository adds seconds of startup time per test class. Use `@DataJpaTest` for repository tests; it loads only the persistence layer.
- **Not cleaning up between tests** — in-memory fakes are naturally reset by `@BeforeEach`. Real databases persist state. Always truncate or delete data in `@BeforeEach` (or rely on transaction rollback knowingly), or use Testcontainers with a fresh container per class.
- **Mocking the class under test in integration tests** — an integration test that mocks the service being tested is neither a unit test nor an integration test. It tests nothing meaningful. Integration tests exist precisely to test the wiring; do not undermine that with mocks of the SUT.

## Recap

- **Integration tests** verify component collaboration with real (or realistic) collaborators, stopping short of the full deployed system
- **In-memory fakes** (repositories backed by HashMap/ArrayList) enable fast, dependency-free integration tests for service logic
- **Testcontainers** provisions real Docker containers (PostgreSQL, Redis, Kafka) for tests that need exact production database semantics
- **@SpringBootTest** loads the full Spring context — use slice annotations (@DataJpaTest, @WebMvcTest) for faster, more focused tests
- **WireMock** stubs HTTP dependencies so HTTP clients can be tested against realistic responses without network access
- **Contract testing** (Pact) verifies consumer–provider API compatibility in CI without running both services simultaneously
