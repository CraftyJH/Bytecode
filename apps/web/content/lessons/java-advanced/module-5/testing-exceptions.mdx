Most developers test the happy path first — the input is valid, the service responds, everything works. Edge cases are where bugs actually live: null inputs that slip through, zero-divisors that crash arithmetic, string boundaries that exceed column widths, or negative values that make no physical sense. Testing exceptions and edges is not pessimism — it is completeness.

## assertThrows — the complete pattern

`assertThrows` verifies the exception type and returns the thrown instance so you can inspect it:

```java
@Test
void withdrawNegativeAmountThrowsIllegalArgument() {
    BankAccount account = new BankAccount(100.0);

    IllegalArgumentException ex = assertThrows(
        IllegalArgumentException.class,
        () -> account.withdraw(-50.0)
    );

    assertTrue(ex.getMessage().contains("negative"),
        "Expected message to mention 'negative' but was: " + ex.getMessage());
}
```

The third argument to `assertThrows` (not used above) is a message supplier — it prints a description when the assertion itself fails (i.e., when no exception is thrown at all), not when the exception has the wrong message.

## assertThrowsExactly

```java
// assertThrows passes if FileNotFoundException is thrown (it IS an IOException)
assertThrows(IOException.class, () -> readFile("missing.txt"));

// assertThrowsExactly fails if FileNotFoundException is thrown instead of plain IOException
assertThrowsExactly(IOException.class, () -> readFile("missing.txt"));
```

Use `assertThrowsExactly` when the precise type in the public contract matters — for example, when callers catch the specific exception class and behave differently for subtypes.

<Note>Testing the exception message is particularly important when a method can throw the same exception type for multiple reasons. "Amount must be positive" and "Amount exceeds limit" are both IllegalArgumentException but represent different bugs. Your test should verify which one actually fired.</Note>

## Testing null inputs

A well-designed public API should document and enforce null-safety. Test it explicitly:

```java
@Test
void nullNameThrowsNullPointerException() {
    assertThrows(NullPointerException.class,
        () -> new Customer(null, "alice@example.com"));
}

@Test
void nullEmailThrowsIllegalArgument() {
    IllegalArgumentException ex = assertThrows(
        IllegalArgumentException.class,
        () -> new Customer("Alice", null));
    assertEquals("Email must not be null", ex.getMessage());
}
```

If you use `Objects.requireNonNull` in the constructor, JUnit will produce a clear failure message if the null check ever gets accidentally removed.

## Boundary value analysis

For any range `[min, max]`, test:
- The minimum value
- Just below minimum (should fail/throw)
- Just above minimum (should succeed)
- The maximum value
- Just above maximum (should fail/throw)

```java
@ParameterizedTest
@CsvSource({
    "-1,  false",   // just below minimum (age cannot be negative)
    " 0,  true",    // minimum boundary
    " 1,  true",    // just above minimum
    "150, true",    // maximum reasonable boundary
    "151, false",   // just above maximum
})
void ageValidationBoundaries(int age, boolean valid) {
    assertEquals(valid, Person.isValidAge(age));
}
```

        if (value < lo) return lo;

        return value;

        check(clamp(5, 5, 10) == 5,  "clamp(5,  5, 10) == 5  (at lo)");

        check(clamp(7, 5, 10) == 7,  "clamp(7,  5, 10) == 7  (between)");

        // Boundary 4: at hi — should return hi exactly
        check(clamp(10, 5, 10) == 10, "clamp(10, 5, 10) == 10 (at hi)");

## Equivalence partitioning

Rather than testing every possible integer, group inputs into *equivalence classes* — sets where the system is expected to behave identically for every member:

| Class | Representative | Expected |
|---|---|---|
| Negative divisor | -3 | Result is negative |
| Zero divisor | 0 | ArithmeticException |
| Positive divisor | 5 | Result is positive |
| Integer overflow | MAX_INT / -1 | Overflow or exception |

Test one representative from each class. Testing `divide(10, 3)` and `divide(100, 7)` in the same class adds no new information.

Write tests covering these equivalence classes and boundaries:
1. Happy path: positive / positive
2. Happy path: negative numerator / positive denominator

4. Zero denominator — verify exception type AND message
5. `Integer.MAX_VALUE / 1` — verify no overflow occurs
6. `Integer.MIN_VALUE / -1` — this overflows silently in Java; document the behaviour with a test

## Common mistakes

- **Only asserting exception type, not message** — when a method throws the same exception for multiple reasons, a type-only assertion can pass even when the wrong code path fired. Always check the message when the contract distinguishes causes.
- **Testing implementation instead of contract** — if `withdraw(-1)` throws `IllegalArgumentException` today but you later refactor to throw a custom `InvalidAmountException` (a subtype), `assertThrows(IllegalArgumentException.class, ...)` still passes. That may be intentional with subtype matching, or it may hide a breaking contract change. Know which you want.
- **Forgetting the overflow case for integer arithmetic** — `Integer.MIN_VALUE / -1` and `Integer.MAX_VALUE + 1` are classic bugs. Test them explicitly; Java does not throw on integer overflow by default.
- **Empty catch blocks in test code** — writing `try { method(); } catch (Exception e) { }` and calling it a test means exceptions are silently swallowed. Always let unexpected exceptions propagate, or explicitly call `fail()` if the code should have thrown but did not.

## Recap

- **assertThrows** verifies exception type AND returns the exception for message/cause inspection
- **assertThrowsExactly** requires the precise class, rejecting subtypes
- Always test null inputs for any public method that does not accept null
- **Boundary value analysis**: test at min, just below, just above, at max, just above max
- **Equivalence partitioning**: group inputs into classes; test one representative per class
- Integer overflow is silent in Java — document and test its behaviour explicitly
