A unit test is supposed to test *one unit* in isolation. But most classes depend on other classes — services talk to repositories, services call external APIs, controllers delegate to services. Wiring up all those real dependencies in every test makes tests slow, flaky, and tightly coupled to the entire call graph. Mockito solves this by letting you replace any dependency with a programmable stand-in.

## mock() vs spy()

```java
// mock() — creates a complete substitute. All methods return safe defaults.
PaymentGateway gateway = mock(PaymentGateway.class);
// gateway.charge(100.0) returns false (default boolean)
// gateway.getTransactionId() returns null (default reference)

// spy() — wraps a real object. Real methods run unless explicitly stubbed.
List<String> realList = new ArrayList<>();
List<String> spyList = spy(realList);
spyList.add("hello");           // real add() is called
when(spyList.size()).thenReturn(99); // only size() is stubbed
```

Mocks are the everyday tool. Spies are useful when you have a large real object and only want to stub *one* troublesome method without replacing everything.

## Stubbing with when/thenReturn

```java
PaymentGateway gateway = mock(PaymentGateway.class);

// Stub a return value
when(gateway.charge(100.0)).thenReturn(true);

// Stub to throw an exception
when(gateway.charge(-1.0)).thenThrow(new IllegalArgumentException("Negative amount"));

// Stub with argument matchers
when(gateway.charge(anyDouble())).thenReturn(true);
```

`thenReturn`, `thenThrow`, and `thenAnswer` can be chained: the first call returns the first value, subsequent calls return the last one. `when(mock.method()).thenReturn(1).thenReturn(2)` returns 1 on the first call and 2 on every call after that.

<Note>Never stub `void` methods with `when(...).thenReturn(...)` — void methods have nothing to return. Use `doThrow(ex).when(mock).voidMethod()` or simply leave them unstubbed (void methods are no-ops on mocks by default).</Note>

## @Mock and @InjectMocks

Declaring mocks as fields and using `@InjectMocks` is cleaner than calling `mock()` manually, especially when a class has several dependencies:

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock PaymentGateway paymentGateway;
    @Mock EmailSender emailSender;
    @InjectMocks OrderService orderService;  // Mockito creates this and injects the mocks above

    @Test
    void successfulOrderChargesAndSendsEmail() {
        when(paymentGateway.charge(150.0)).thenReturn(true);

        orderService.submit(new Order(150.0, "alice@example.com"));

        verify(paymentGateway).charge(150.0);
        verify(emailSender).send(any(Email.class));
    }
}
```

`@InjectMocks` tries constructor injection first, then setter injection, then field injection. Constructor injection is the most explicit and the most testable — prefer it in production code.

## verify() and ArgumentCaptor

`verify()` checks that a method was called with certain arguments. When you need to inspect *what* was passed (e.g., the content of the email), use `ArgumentCaptor`:

```java
@Captor ArgumentCaptor<Email> emailCaptor;

@Test
void emailContainsOrderId() {
    when(paymentGateway.charge(anyDouble())).thenReturn(true);
    orderService.submit(new Order(75.0, "bob@example.com"));

    verify(emailSender).send(emailCaptor.capture());
    Email sent = emailCaptor.getValue();

    assertEquals("bob@example.com", sent.getTo());
    assertTrue(sent.getSubject().contains("Order confirmed"));
}
```

Without `ArgumentCaptor` you would have to stub a custom matcher, or worse, expose internal state just to test it. The captor lets you assert on the exact object passed to the collaborator.

## Strict stubs

`MockitoExtension` (the JUnit 5 integration) enables *strict stubbing* by default. This means:

- **Unused stubs fail the test** — if you stub a method that is never actually called, Mockito throws `UnnecessaryStubbingException`. This catches dead test code.
- **Argument mismatch detection** — if stubbed with `when(x.m(1))` but called with `x.m(2)`, the stub doesn't apply and Mockito reports the inconsistency clearly.

## When NOT to mock

Mockito is a tool, not a mandate. Do not mock:

- **Value objects** — `Money`, `Address`, `LocalDate`. Use the real thing; they have no side effects.
- **The class under test** — mocking the SUT defeats the purpose of the test.
- **Simple collaborators** — if a helper does a trivial calculation with no I/O, instantiate it for real.
- **Everything** — a test that mocks five objects and asserts only on mock interactions is testing your Mockito configuration, not your logic.

## Common mistakes

- **Calling real methods on a mock** — `spy()` delegates to the real method; `mock()` does not. Accidentally using `spy()` when you wanted `mock()` can cause unexpected side effects during test setup.
- **Mixing argument matchers and raw values** — `when(m.method(anyInt(), "hello"))` is a compile-time valid but runtime-invalid mix. Once you use a matcher for one argument, all arguments in that call must use matchers (`eq("hello")` for the fixed value).
- **Stubbing after execution** — placing `when(...)` after the method under test has already run creates unused stubs and never affects the already-completed call. Always stub before the action (Act) step.
- **Over-verifying** — asserting every method call on every mock is brittle. Verify only the interactions that matter for the behavior being tested; leave incidental internal calls unverified.

## Recap

- **mock()** creates a full substitute with no-op defaults; **spy()** wraps a real object
- **when(...).thenReturn(...)** stubs return values; use **doThrow/doAnswer** for void methods
- **@Mock + @InjectMocks** wire up the test class cleanly via `@ExtendWith(MockitoExtension.class)`
- **verify()** asserts invocation count and arguments; **ArgumentCaptor** captures what was passed
- Strict stubbing (default with MockitoExtension) fails on unused stubs — keeps test code clean
- Do not mock value objects, simple helpers, or the class under test
