Writing the same assertion five times with slightly different inputs is the kind of copy-paste noise that makes test suites hard to maintain. JUnit 5's parameterized test support lets you declare the logic once and supply the data separately — the framework runs the method once per data row, reports each invocation individually, and makes the intent explicit.

## @ParameterizedTest + @ValueSource

The simplest parameterized form: one value, one type, no ceremony.

```java
@ParameterizedTest
@ValueSource(strings = {"", " ", "\t", "\n"})
void isBlankReturnsTrueForWhitespaceAndEmpty(String input) {
    assertTrue(StringUtils.isBlank(input));
}
```

`@ValueSource` supports `ints`, `longs`, `doubles`, `strings`, `chars`, `booleans`, `bytes`, `shorts`, and `classes`. Each element triggers one test invocation with its own result line in the report.

## @CsvSource — multiple arguments per row

When the test needs both an input and an expected value, `@CsvSource` maps comma-separated columns to method parameters:

```java
@ParameterizedTest(name = "toUpperCase({0}) = {1}")
@CsvSource({
    "hello,  HELLO",
    "world,  WORLD",
    "Java,   JAVA",
    "'',     ''",          // empty string — quoted so CSV parser doesn't swallow it
})
void toUpperCase(String input, String expected) {
    assertEquals(expected, input.toUpperCase());
}
```

The optional `name` attribute uses `{0}`, `{1}` placeholders for the arguments, giving each run a readable display name in the IDE.

<Note>Use single quotes inside @CsvSource strings to represent empty strings or strings containing commas: `"'hello, world', 11"` — the outer double-quotes are Java string delimiters; the inner single-quotes are CSV quoting.</Note>

## @MethodSource — factory methods for complex inputs

When inputs are objects or the data needs construction logic, point to a static factory method:

```java
@ParameterizedTest
@MethodSource("triangleProvider")
void triangleAreaIsCorrect(double base, double height, double expectedArea) {
    assertEquals(expectedArea, Geometry.triangleArea(base, height), 0.001);
}

static Stream<Arguments> triangleProvider() {
    return Stream.of(
        Arguments.of(3.0, 4.0, 6.0),
        Arguments.of(5.0, 2.0, 5.0),
        Arguments.of(10.0, 1.0, 5.0)
    );
}
```

The factory must be **static** (unless `@TestInstance(PER_CLASS)` is active). It returns `Stream<Arguments>`, `List<Arguments>`, or any `Iterable<Arguments>`.

## @EnumSource

When the test should run for every (or a selected subset of) enum constants:

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

@ParameterizedTest
@EnumSource(value = Day.class, names = {"SAT", "SUN"})
void weekendDaysAreNotWorkDays(Day day) {
    assertFalse(Schedule.isWorkDay(day));
}
```

Omit the `names` attribute to run the test for every constant. Use `mode = EXCLUDE` to blacklist specific constants instead of whitelisting them.

## @DynamicTest factory

Sometimes test cases are not known at compile time — they come from a file, a database query, or a computation. `@TestFactory` returns a `Stream<DynamicTest>` where each test has a runtime-computed name and executable:

```java
@TestFactory
Stream<DynamicTest> dynamicPrimeTests() {
    List<Integer> primes = List.of(2, 3, 5, 7, 11, 13);
    return primes.stream()
        .map(n -> DynamicTest.dynamicTest(
            n + " should be prime",
            () -> assertTrue(MathUtils.isPrime(n))
        ));
}
```

Each returned `DynamicTest` appears as its own entry in the test report with its custom name.

```java
// Test 1 — @ValueSource for known primes
@ParameterizedTest
@ValueSource(ints = {2, 3, 5, 7, 11, 13, 17, 19})
void knownPrimesReturnTrue(int n) {
    assertTrue(MathUtils.isPrime(n));

@ParameterizedTest
@CsvSource({
    "1,  false",
    "4,  false",
    "9,  false",
    "2,  true",
    "7,  true",
    "25, false",
})
void isPrimeReturnsExpected(int input, boolean expected) {
    assertEquals(expected, MathUtils.isPrime(input));
}
```

Run both tests and observe how each CSV row appears as a separate result. Then add a third test using `@MethodSource` that also checks negative numbers and zero — all should return `false`.

## Common mistakes

- **Forgetting to annotate with both @ParameterizedTest and a source annotation** — @Test and @ParameterizedTest are mutually exclusive. Using @Test with @ValueSource silently runs only the first value; using @ParameterizedTest with no source is a configuration error.
- **Non-static @MethodSource factory** — JUnit invokes the factory before instantiating the test class. Make it `static` or add `@TestInstance(Lifecycle.PER_CLASS)` to the class.
- **Type mismatches in @CsvSource** — JUnit converts strings to the declared parameter types automatically, but the conversion has limits. Complex types need a registered `TypeConverter` or a `@MethodSource` instead.
- **Overusing @DynamicTest** — dynamic tests have weaker IDE support (no run-single-test button by name). Prefer @ParameterizedTest with a data source unless you genuinely need runtime test generation.

## Recap

- **@ParameterizedTest + @ValueSource** is the simplest form: one type, one array, one argument per invocation
- **@CsvSource** maps comma-separated rows to multiple method parameters
- **@MethodSource** delegates to a static factory returning `Stream<Arguments>` for complex inputs
- **@EnumSource** iterates over enum constants, with optional include/exclude filtering
- **@TestFactory + DynamicTest** generates tests at runtime with per-case display names
- JUnit reports each parameterized invocation as an independent result in the test output
