Good tests are not just correct — they are readable, independent, and maintainable. A test suite where tests fail randomly, require a specific execution order, or take ten minutes to understand is a liability rather than an asset. This lesson covers the conventions that turn a collection of @Test methods into a well-structured safety net.

## Test naming conventions

The method name is the first thing a developer reads when a test fails. Names like `test1`, `checkIt`, or `verifyLogic` provide no information. The two most popular alternatives:

**given_when_then:**
```java
@Test
void givenInsufficientFunds_whenWithdraw_thenThrowsInsufficientFundsException() { }
```

**should_when:**
```java
@Test
void shouldThrowInsufficientFunds_whenBalanceIsTooLow() { }
```

Both are self-documenting. The name serves as a failing-test error message: if `shouldThrowInsufficientFunds_whenBalanceIsTooLow` fails, you immediately know the precondition and the expected outcome without opening the method.

## The single-assertion guideline

The *one assertion per test* guideline is frequently misunderstood. It does not mean one `assertEquals` call — it means one *logical scenario*. `assertAll` groups related assertions on the same object into one conceptual assertion:

```java
// Good — one scenario, multiple related checks via assertAll
@Test
void depositUpdatesAllAccountFields() {
    account.deposit(50.0);
    assertAll("post-deposit state",
        () -> assertEquals(150.0, account.getBalance()),
        () -> assertEquals(1, account.getTransactionCount()),
        () -> assertNotNull(account.getLastTransactionTime())
    );
}

// Bad — two unrelated scenarios in one method
@Test
void depositAndWithdraw() {
    account.deposit(50.0);
    assertEquals(150.0, account.getBalance());
    account.withdraw(30.0);
    assertEquals(120.0, account.getBalance()); // if deposit assertion fails, this never runs
}
```

## Test isolation

Each test must start from a known, clean state independent of every other test. The three rules:

1. **Never share mutable state between tests** — static fields are fine for immutable constants; catastrophic for mutable objects.
2. **Use @BeforeEach for setup, not static initialisers** — each test gets a fresh instance.
3. **Do not depend on test execution order** — JUnit 5 does not guarantee order by default. A suite that only passes in alphabetical order is hiding an isolation bug.

```java
class CartServiceTest {
    Cart cart;
    CartService service;

    @BeforeEach
    void setUp() {
        cart = new Cart();                    // fresh per test
        service = new CartService(cart);      // fresh per test
    }

    @Test
    void addItemIncreasesItemCount() {
        service.add(new Item("book", 20.0));
        assertEquals(1, service.itemCount());
    }

    @Test
    void removeItemDecreasesItemCount() {
        service.add(new Item("book", 20.0));
        service.remove("book");
        assertEquals(0, service.itemCount());
    }
}
```

## Test doubles taxonomy

The word "mock" is often used loosely to mean any kind of substitute object. Gerard Meszaros defined five precise types:

| Type | Description | Example |
|---|---|---|
| **Dummy** | Passed but never used | `null` or empty object to satisfy a constructor parameter |
| **Stub** | Returns canned answers to calls | `when(repo.findById(1)).thenReturn(user)` |
| **Spy** | Records calls; real or partial logic | `Mockito.spy(realList)` |
| **Mock** | Pre-programmed with expectations, verified | `verify(gateway).charge(100.0)` |
| **Fake** | Simplified but working implementation | `InMemoryUserRepository` used in tests |

Knowing which type you are using makes the test's intent clear. A test that uses a **stub** is checking return-value logic. A test that uses a **mock** is verifying interaction behaviour. A test that uses a **fake** is running a slice of real logic cheaply.

<Note>Fakes are underused. An in-memory repository fake lets service-layer tests run against real repository logic without a database, without Mockito ceremony, and without the fragility of interaction verification.</Note>

## Avoiding test interdependence

Test interdependence has two failure modes:

1. **Test A pollutes state, causing Test B to fail** — share mutable state between tests.
2. **Test B depends on Test A having run first** — tests rely on execution order.

Both are fixed the same way: isolate setup in `@BeforeEach` and tear down in `@AfterEach`. If cleanup is genuinely expensive (e.g., database schema creation), use `@BeforeAll`/`@AfterAll` with care and ensure each test resets the *data* (not the schema) in its own setup.

## Common mistakes

- **Static mutable fields in test classes** — they persist across tests, causing subtle order-dependent failures that are hard to diagnose. Use instance fields initialised in `@BeforeEach`.
- **Asserting on private implementation details** — accessing private fields via reflection or testing exact internal call sequences makes every refactoring break the tests. Test the public contract.
- **Identical setup duplicated across test methods** — copy-pasted `new BankAccount(100.0)` in every test method means a single constructor signature change requires updating dozens of lines. Factor setup into `@BeforeEach`.
- **Using mocks when a fake would be clearer** — Mockito interaction verification for every call on a repository creates fragile tests that break when the implementation is optimised. An in-memory fake lets the test focus on outcome, not call sequence.

## Recap

- Use **given_when_then** or **should_when** naming so the test name is a self-contained failure description
- One **logical scenario** per test; use `assertAll` for multiple checks on the same scenario
- **@BeforeEach** ensures every test starts from a fresh, known state
- Know your test doubles: **Dummy**, **Stub**, **Spy**, **Mock**, **Fake** — use the right one for the right job
- Tests must pass in **any execution order** — order dependence always hides an isolation bug
