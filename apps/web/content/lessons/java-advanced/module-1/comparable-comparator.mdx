Almost every real-world Java application needs to sort objects: sort orders by date, employees by department, products by price and name. Java provides two complementary tools. `Comparable` bakes the natural order *into the class* — a class that implements `Comparable` tells the world "here is my default sort order". `Comparator` is an external strategy object — reusable, composable, and reversible without touching the original class. Production code almost always ends up using `Comparator.comparing().thenComparing().reversed()` chains, which are expressive enough to replace hundreds of lines of hand-written comparison code.

## `Comparable<T>` — the natural order

A class implements `Comparable<T>` to define how its instances compare to each other by default:

```java
public class Temperature implements Comparable<Temperature> {
    private final double celsius;

    public Temperature(double celsius) { this.celsius = celsius; }

    @Override
    public int compareTo(Temperature other) {
        // Negative → this is smaller; 0 → equal; positive → this is larger
        return Double.compare(this.celsius, other.celsius);
    }

    @Override
    public String toString() { return celsius + "°C"; }
}
```

Once a class implements `Comparable`, it works with all standard JDK sorting and searching:

```java
List<Temperature> temps = new ArrayList<>(List.of(
    new Temperature(100), new Temperature(-10), new Temperature(37)
));
Collections.sort(temps);
System.out.println(temps);   // [-10.0°C, 37.0°C, 100.0°C]
```

<Note>Always use `Double.compare(a, b)`, `Integer.compare(a, b)`, etc. instead of subtraction (`a - b`). Subtraction overflows for large integers and produces NaN for certain doubles, leading to incorrect sort orders.</Note>

## `Comparator<T>` — external, composable ordering

`Comparator` is a functional interface — you can create one with a lambda or method reference. The `Comparator.comparing` factory is the modern entry point:

```java
import java.util.*;

record Person(String name, int age) {}

List<Person> people = new ArrayList<>(List.of(
    new Person("Zara", 28),
    new Person("Alice", 35),
    new Person("Bob", 28)
));

// Sort by age
people.sort(Comparator.comparingInt(Person::age));
System.out.println(people);
// [Person[name=Zara, age=28], Person[name=Bob, age=28], Person[name=Alice, age=35]]
```

## Chaining with `thenComparing`

`thenComparing` adds a secondary sort key that breaks ties:

```java
// Sort by age ascending, then name alphabetically for equal ages
Comparator<Person> byAgeThenName =
    Comparator.comparingInt(Person::age)
              .thenComparing(Person::name);

people.sort(byAgeThenName);
System.out.println(people);
// [Person[name=Bob, age=28], Person[name=Zara, age=28], Person[name=Alice, age=35]]
```

You can chain as many `thenComparing` calls as needed.

## `reversed()` and `reverseOrder()`

```java
// Sort by age descending
Comparator<Person> byAgeDesc = Comparator.comparingInt(Person::age).reversed();

// Sort strings by length descending, then alphabetically
Comparator<String> c = Comparator.comparingInt(String::length)
                                  .reversed()
                                  .thenComparing(Comparator.naturalOrder());
```

`Comparator.reverseOrder()` gives the reverse of natural order for directly comparable types:

```java
List<Integer> nums = new ArrayList<>(List.of(3, 1, 4, 1, 5));
nums.sort(Comparator.reverseOrder());
System.out.println(nums);  // [5, 4, 3, 1, 1]
```

## Practical example: multi-key Employee sort

A classic production scenario — sort employees by department ascending, then salary descending, then name ascending:

```java
import java.util.*;

public class Main {
    record Employee(String name, String dept, int salary) {}

    public static void main(String[] args) {
        List<Employee> employees = new ArrayList<>(List.of(
            new Employee("Zara",  "Engineering", 90_000),
            new Employee("Alice", "Engineering", 75_000),
            new Employee("Bob",   "Marketing",   80_000),
            new Employee("Dave",  "Marketing",   80_000),
            new Employee("Eve",   "Engineering", 90_000)
        ));

        Comparator<Employee> order =
            Comparator.comparing(Employee::dept)               // dept A→Z
                      .thenComparing(
                          Comparator.comparingInt(Employee::salary).reversed()  // salary Z→A
                      )
                      .thenComparing(Employee::name);          // name A→Z (tie-break)

        employees.sort(order);
        employees.forEach(e ->
            System.out.printf("%-12s %-14s %,d%n", e.name(), e.dept(), e.salary())
        );
    }
}
```

## Common mistakes

- **Subtracting instead of using `Integer.compare`** — `return a.age - b.age` overflows for large negative ages and gives wrong results. Always use `Integer.compare(a.age, b.age)` or `Comparator.comparingInt(...)`.
- **Implementing `Comparable` inconsistently with `equals`** — the contract says if `a.compareTo(b) == 0`, then `a.equals(b)` should be `true`. Violating this breaks `TreeSet` and `TreeMap` (they use `compareTo` as equality).
- **Applying `reversed()` to the whole chain when you only want one key reversed** — `comparator.reversed()` flips every level. Wrap only the relevant level: `thenComparing(Comparator.comparingInt(Employee::salary).reversed())`.
- **Forgetting `List.sort` vs `Collections.sort`** — both work on `ArrayList`, but `List.sort(comparator)` is the modern API (Java 8+) and accepts `null` as a comparator (uses natural order). Prefer `list.sort(comparator)`.

## Recap

- `Comparable<T>` is implemented by a class to define its **natural ordering**; used by default in `sort`, `TreeSet`, `TreeMap`.
- `Comparator<T>` is an **external strategy** — create it with `Comparator.comparing()`, chain with `thenComparing()`, flip with `reversed()`.
- Use `Integer.compare`, `Double.compare`, etc. — never subtraction.
- Keep `compareTo` consistent with `equals` to avoid surprises in sorted collections.
