Type erasure is the reason Java generics have several surprising limitations: you can't write `new T()`, `T.class`, or `instanceof List<String>`. Understanding *why* these restrictions exist — and how the JVM actually executes generic code — turns frustrating compiler errors into clear design decisions. It also explains heap pollution, bridge methods, and why mixing generic and legacy (raw-type) code can produce `ClassCastException` far from the original bug.

## What type erasure does

When the Java compiler compiles a generic class, it replaces every type parameter with its **erasure**:

- An unbounded `T` becomes `Object`.
- A bounded `T extends Comparable<T>` becomes `Comparable`.
- All casts that preserve type safety at call sites are inserted automatically.

After compilation, `Box<String>` and `Box<Integer>` produce identical bytecode:

```java
// Source
public class Box<T> {
    private T value;
    public T get() { return value; }
}

// Bytecode equivalent (what the compiler emits)
public class Box {
    private Object value;
    public Object get() { return value; }
}

// Call site — compiler inserts the cast automatically:
Box<String> b = new Box<>("hi");
String s = b.get();           // source code

// Compiled as:
String s = (String) b.get();  // inserted by compiler
```

The JVM sees no generics — all safety is a compile-time guarantee.

## What you cannot do because of erasure

```java
// ILLEGAL — T is not available at runtime:
T obj = new T();             // cannot instantiate T
T[] arr = new T[10];        // cannot create generic array
if (x instanceof Box<String>) { ... }  // cannot check parameterised type
Class<T> c = T.class;       // no class literal for type parameters
```

Each of these requires a runtime representation of T, which erasure removes.

<Note>You *can* use `instanceof Box<?>` (unbounded wildcard) because it doesn't depend on knowing the type argument — it just checks that the object is a `Box`.</Note>

## Raw types and heap pollution

A **raw type** is a generic class used without type arguments. Raw types exist for backward compatibility with pre-generics code:

```java
// Raw type — avoid in new code
List raw = new ArrayList();
raw.add("hello");
raw.add(42);           // compiles with unchecked warning

List<String> strings = raw;   // unchecked assignment
String s = strings.get(1);   // ClassCastException at runtime!
```

This is **heap pollution**: a `List<String>` reference points to a list that actually contains an Integer. The cast fails far from where the bad element was inserted, making the bug hard to find.

## Bridge methods

When a generic class overrides a method from a non-generic parent, the compiler may insert a **bridge method** to maintain polymorphism correctly:

```java
public interface Comparable<T> {
    int compareTo(T other);
}

public class StringWrapper implements Comparable<StringWrapper> {
    private final String value;
    StringWrapper(String value) { this.value = value; }

    @Override
    public int compareTo(StringWrapper other) {
        return this.value.compareTo(other.value);
    }
    // Compiler also generates the bridge method:
    // public int compareTo(Object other) {
    //     return compareTo((StringWrapper) other);
    // }
}
```

The bridge method preserves the `compareTo(Object)` signature from the erased `Comparable` interface. You never write bridge methods; the compiler generates them silently. They appear in stack traces and can occasionally cause confusion.

## `Box<?>` and type tokens

Two common patterns for working with generic types at runtime:

**Pattern 1 — Unbounded wildcard for containers of unknown type:**

```java
static void printBox(Box<?> box) {
    System.out.println(box.get());   // Object — safe, prints anything
}
```

**Pattern 2 — `Class<T>` as a type token (preview for lesson 9):**

```java
public static <T> T cast(Object obj, Class<T> type) {
    return type.cast(obj);   // throws ClassCastException immediately if wrong
}

// Usage:
String s = cast(someObject, String.class);
```

`Class<T>` survives erasure because it's a regular object — not a type parameter. This pattern is used in Jackson, Spring, and most serialization frameworks.

## Try it

Observe erasure in action and use the type-token pattern:

```java
import java.util.*;

public class Main {
    static class Box<T> {
        private final T value;
        Box(T value) { this.value = value; }
        T get() { return value; }
    }

    static <T> T safeCast(Object obj, Class<T> type) {
        return type.cast(obj);  // immediate, clear ClassCastException if wrong
    }

    public static void main(String[] args) {
        Box<String> boxStr = new Box<>("hello");
        Box<Integer> boxInt = new Box<>(42);

        // Both boxes are the same raw class after erasure:
        System.out.println(boxStr.getClass() == boxInt.getClass()); // true

        // Safe cast with type token:
        System.out.println(safeCast("world", String.class));  // world
        try {
            safeCast("world", Integer.class);
        } catch (ClassCastException e) {
            System.out.println("caught: " + e.getMessage());
        }
    }
}
```

## Common mistakes

- **Mixing raw and generic types** — assigning a raw `List` to a `List<String>` reference compiles with warnings and causes heap pollution. Enable `-Xlint:unchecked` and treat all warnings as errors in new code.
- **Casting to a type parameter** — `(T) someObject` at runtime erases to `(Object)` — the cast does nothing and defers the error to wherever the result is used. Use a `Class<T>` type token and `type.cast()` for genuine runtime safety.
- **Expecting `getClass()` to return the parameterised type** — `new ArrayList<String>().getClass()` returns `class java.util.ArrayList`, not `ArrayList<String>`. The parameter is gone.
- **Trying `new T[n]`** — create `Object[]` and cast the array reference: `T[] arr = (T[]) new Object[n]`. This is an unchecked cast but is the standard workaround (see lesson 7).

## Recap

- **Type erasure** replaces type parameters with `Object` (or the bound) in bytecode; the compiler inserts casts at read sites.
- `Box<String>` and `Box<Integer>` are the **same class** at runtime.
- `instanceof Box<String>` is illegal; use `instanceof Box<?>`.
- **Raw types** bypass generic safety and can cause heap pollution.
- **Bridge methods** are compiler-generated to maintain polymorphism with erased signatures.
- `Class<T>` as a type token is the safe way to do generic operations at runtime.
