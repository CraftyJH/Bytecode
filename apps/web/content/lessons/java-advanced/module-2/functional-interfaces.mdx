The `java.util.function` package is the vocabulary of functional Java. Every lambda you write plugs into one of these interfaces — either directly or through a library method that accepts them. Knowing which interface fits which shape of function lets you read and write modern Java fluently, without guessing what the compiler expects.

## The core interfaces

Java's functional interfaces are named by *shape*: how many inputs, what output.

| Interface | Method | Shape |
|---|---|---|
| `Function<T, R>` | `R apply(T t)` | one in, one out |
| `BiFunction<T, U, R>` | `R apply(T t, U u)` | two in, one out |
| `Consumer<T>` | `void accept(T t)` | one in, no out |
| `Supplier<T>` | `T get()` | no in, one out |
| `Predicate<T>` | `boolean test(T t)` | one in, boolean out |
| `UnaryOperator<T>` | `T apply(T t)` | one in, same type out |
| `BinaryOperator<T>` | `T apply(T t1, T t2)` | two same-type in, same type out |

`UnaryOperator<T>` extends `Function<T, T>` and `BinaryOperator<T>` extends `BiFunction<T, T, T>`. They exist for the common case where the input and output type are the same, so you don't repeat the type parameter.

```java
import java.util.function.*;

Function<String, Integer>   toLength  = String::length;        // String -> Integer
BiFunction<Integer,Integer,Integer> mul = (a, b) -> a * b;     // two ints -> int
Consumer<String>            log       = System.out::println;    // String -> void
Supplier<Double>            rand      = Math::random;           // () -> Double
Predicate<String>           nonEmpty  = s -> !s.isEmpty();     // String -> boolean
UnaryOperator<String>       shout     = String::toUpperCase;   // String -> String
BinaryOperator<Integer>     sum       = Integer::sum;          // (int,int) -> int
```

## @FunctionalInterface

Any interface with exactly one abstract method is a functional interface, and a lambda can implement it. The `@FunctionalInterface` annotation is optional but recommended — it tells the compiler to error if you accidentally add a second abstract method:

```java
@FunctionalInterface
public interface Transformer<T> {
    T transform(T input);
    // adding another abstract method here would be a compile error
}

Transformer<String> reverse = s -> new StringBuilder(s).reverse().toString();
System.out.println(reverse.transform("Java")); // avaJ
```

The annotation documents intent and protects the interface's contract. Any interface in the JDK that carries `@FunctionalInterface` is guaranteed safe to use as a lambda target.

<Note>Default methods and static methods do **not** count toward the abstract-method count. An interface can have many default/static methods and still be a valid functional interface as long as it has exactly one abstract method.</Note>

## Composing with andThen, compose, and negate

The standard functional interfaces include composition helpers — methods that combine two functions into one.

**Function composition:**

```java
Function<String, String> trim   = String::trim;
Function<String, String> upper  = String::toUpperCase;
Function<String, Integer> count = String::length;

// andThen: left-to-right — trim first, then uppercase
Function<String, String> trimThenUpper = trim.andThen(upper);
System.out.println(trimThenUpper.apply("  hello  ")); // HELLO

// compose: right-to-left — upper runs first, then trim
// (rarely used, but mirrors mathematical function composition)
Function<String, String> upperThenTrim = trim.compose(upper);
System.out.println(upperThenTrim.apply("  hello  ")); // HELLO (same here, order matters when operations differ)
```

**Predicate composition:**

```java
Predicate<String> longEnough = s -> s.length() >= 5;
Predicate<String> noSpaces   = s -> !s.contains(" ");

Predicate<String> valid = longEnough.and(noSpaces);
System.out.println(valid.test("hello"));     // true
System.out.println(valid.test("hi"));        // false (too short)
System.out.println(valid.test("hello world")); // false (has space)

Predicate<String> either = longEnough.or(noSpaces);
Predicate<String> notLong = longEnough.negate();
```

## Try it

Write a method that accepts a `Function<String, String>` and a list of strings, applies the function to each, and returns the results:

```java
import java.util.*;
import java.util.function.*;
import java.util.stream.*;

public class Main {
    static List<String> transform(List<String> items, Function<String, String> fn) {
        return items.stream().map(fn).collect(Collectors.toList());
    }

    public static void main(String[] args) {
        List<String> words = List.of("hello", "world", "java");

        // Pass different functional interface implementations
        System.out.println(transform(words, String::toUpperCase));
        System.out.println(transform(words, s -> s + "!"));

        // Compose: uppercase then exclaim
        Function<String, String> shoutAndExclaim =
            ((Function<String,String>) String::toUpperCase).andThen(s -> s + "!");
        System.out.println(transform(words, shoutAndExclaim));
    }
}
```

## Common mistakes

- **Using `Function<T, Boolean>` instead of `Predicate<T>`** — `Predicate<T>` returns a primitive `boolean`, not a boxed `Boolean`, and comes with `and()`, `or()`, `negate()`. Prefer it for boolean tests.
- **Confusing `andThen` and `compose`** — `f.andThen(g)` means `g(f(x))` (f first). `f.compose(g)` means `f(g(x))` (g first). When in doubt, `andThen` reads left-to-right and is usually what you want.
- **Ignoring primitive specialisations** — `IntFunction<R>`, `ToIntFunction<T>`, `IntUnaryOperator` etc. avoid boxing overhead in tight loops. For high-throughput code, check whether a primitive variant exists.
- **Implementing `@FunctionalInterface` with more than one abstract method** — the interface stops being a functional interface and can no longer be used as a lambda target. The annotation turns this into a compile-time error, which is exactly why you should use it.

## Recap

- `Function<T,R>`, `Consumer<T>`, `Supplier<T>`, `Predicate<T>` are the four fundamental shapes
- `BiFunction`, `UnaryOperator`, `BinaryOperator` cover common two-argument and same-type variants
- `@FunctionalInterface` documents intent and guards the single-abstract-method constraint
- `andThen` / `compose` chain `Function`s; `and` / `or` / `negate` combine `Predicate`s
