Stream pipelines are how modern Java expresses data transformation. A pipeline is a sequence of operations applied to a source: each element flows through each stage and exits at the terminal operation. Understanding which operations are *intermediate* (lazy, return a new stream) versus *terminal* (eager, produce a result) is the key mental model for working with streams fluently.

## Lazy evaluation — why it matters

Intermediate operations do not execute immediately. They build a *description* of what to do. Execution only begins when a terminal operation runs — and it runs only enough to satisfy the terminal:

```java
List<Integer> nums = List.of(1, 2, 3, 4, 5);

// This does nothing — no terminal operation
nums.stream()
    .filter(n -> { System.out.println("filter: " + n); return n > 2; })
    .map(n -> { System.out.println("map: " + n); return n * 10; });

// Add a terminal — now it runs, and the output interleaves:
// filter: 1, filter: 2, filter: 3, map: 3, filter: 4, map: 4, ...
nums.stream()
    .filter(n -> n > 2)
    .map(n -> n * 10)
    .forEach(System.out::println); // triggers the whole pipeline
```

Laziness enables short-circuit operations like `findFirst()` and `limit()` to stop processing early, which would be impossible if every stage materialised a full intermediate list.

## filter

`filter(Predicate<T>)` keeps elements for which the predicate returns `true`.

```java
List<String> names = List.of("Alice", "Bob", "Anna", "Charlie");

List<String> aNames = names.stream()
    .filter(s -> s.startsWith("A"))
    .collect(Collectors.toList());
// [Alice, Anna]
```

The predicate is called for every element. Elements where it returns `false` are silently dropped from the downstream stages.

## map

`map(Function<T, R>)` transforms each element from type `T` to type `R`.

```java
List<String> words = List.of("hello", "world");

List<Integer> lengths = words.stream()
    .map(String::length)
    .collect(Collectors.toList());
// [5, 5]
```

The output stream has the same number of elements as the input; each is just a different type or value.

<Note>When mapping to a primitive, prefer `mapToInt`, `mapToLong`, or `mapToDouble`. These return a primitive stream (`IntStream`, `LongStream`, `DoubleStream`) that avoids boxing and adds arithmetic terminals like `sum()` and `average()`.</Note>

## flatMap

`flatMap(Function<T, Stream<R>>)` maps each element to a stream, then *flattens* all those streams into one:

```java
List<String> sentences = List.of("hello world", "java streams");

List<String> words = sentences.stream()
    .flatMap(sentence -> Arrays.stream(sentence.split(" ")))
    .collect(Collectors.toList());
// [hello, world, java, streams]
```

Without `flatMap`, using `map` would produce a `Stream<String[]>` or `Stream<Stream<String>>`, requiring an extra step to unwrap. `flatMap` is the idiomatic way to go from one-to-many.

```java
// Another common pattern: flatten nested lists
List<List<String>> groups = List.of(
    List.of("a", "b"),
    List.of("c"),
    List.of("d", "e", "f")
);
List<String> flat = groups.stream()
    .flatMap(Collection::stream)
    .collect(Collectors.toList());
// [a, b, c, d, e, f]
```

## Other useful intermediate operations

```java
.distinct()           // remove duplicates (uses equals/hashCode)
.sorted()             // natural order
.sorted(comparator)   // custom order
.limit(n)             // keep at most n elements
.peek(action)         // run a side-effect, pass element through unchanged
```

`peek` is a debugging aid: insert it between stages to see what is flowing through without disturbing the pipeline.

```java
List.of(1, 2, 3, 4, 5).stream()
    .filter(n -> n % 2 == 0)
    .peek(n -> System.out.println("after filter: " + n))
    .map(n -> n * n)
    .peek(n -> System.out.println("after map: " + n))
    .forEach(System.out::println);
```

<Note>Never rely on `peek` for production logic — its execution is not guaranteed when a terminal short-circuits. Use it only for debugging.</Note>

## Common mistakes

- **Forgetting `flatMap` when mapping to collections** — `words.stream().map(s -> Arrays.stream(s.split("")))` gives a `Stream<IntStream>`, not the flat stream you wanted. Use `flatMap`.
- **Using `peek` for production side effects** — `peek` is not guaranteed to run for every element when the terminal short-circuits (e.g., `findFirst`). Use `map` with a side effect inside if you truly need both, or separate the side effect and the transformation.
- **Sorting before filtering** — sorting is O(n log n); filtering is O(n). Filter first to reduce the set, then sort the smaller result.
- **Confusing `map` with `forEach`** — `map` transforms and returns a stream; `forEach` is a terminal that performs a side effect and returns `void`. You cannot chain after `forEach`.

## Recap

- Stream pipelines are **lazy**: intermediate operations run only when a terminal is reached
- `filter(pred)` keeps elements matching the predicate
- `map(fn)` transforms each element (same count, different value/type)
- `flatMap(fn)` maps each element to a stream, then flattens all streams into one
- `distinct()`, `sorted()`, `limit(n)`, `peek(action)` are handy intermediate helpers
