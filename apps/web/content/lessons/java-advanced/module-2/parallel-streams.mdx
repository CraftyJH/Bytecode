Modern CPUs have multiple cores, but a sequential stream uses only one. Parallel streams are Java's built-in way to split a computation across the common fork-join pool, potentially cutting wall-clock time for CPU-bound work. The catch: parallel streams are not always faster, and used incorrectly they produce wrong answers or silently corrupt data. This lesson gives you the mental model to make the right call.

## How parallel streams work

When you call `.parallel()` on a stream (or start with `.parallelStream()`), the JVM routes the pipeline through the common `ForkJoinPool`. The pool splits the data source into chunks using a *spliterator*, processes each chunk on a separate thread, and combines the results.

```java
List<Integer> nums = List.of(1, 2, 3, 4, 5, 6, 7, 8);

// Sequential
long seqSum = nums.stream().mapToLong(n -> n).sum();

// Parallel — same result, potentially using multiple cores
long parSum = nums.parallelStream().mapToLong(n -> n).sum();

// Or: convert mid-pipeline
long parSum2 = nums.stream().parallel().mapToLong(n -> n).sum();
```

For a tiny list of eight numbers, the parallel version is almost certainly *slower* due to thread coordination overhead. Parallelism pays off only when:

1. The work per element is significant (CPU-heavy, not just a field lookup)
2. The dataset is large (thousands to millions of elements)
3. The operation is *stateless* and *associative*

## Stateless vs stateful operations

**Stateless** — each element is processed independently, no shared mutable state:

```java
// Safe: map and filter are stateless
stream.parallel()
    .filter(n -> n > 0)        // no shared state
    .map(n -> n * n)           // no shared state
    .sum();                    // safe reduction
```

**Stateful** — results depend on other elements seen so far:

```java
// UNSAFE: ArrayList is not thread-safe
List<Integer> results = new ArrayList<>();
stream.parallel().forEach(results::add); // data corruption!

// Safe alternative: use collect
List<Integer> results = stream.parallel()
    .filter(n -> n > 0)
    .collect(Collectors.toList()); // thread-safe via Collector
```

<Note>Sorted and distinct operations have to collect some state to work correctly (what they've seen so far), so they perform worse in parallel than purely stateless operations. Avoid `.sorted()` and `.distinct()` immediately before a parallel terminal if performance is critical.</Note>

## reduce — identity requirement

For `reduce(identity, op)` to give correct results in parallel, the operation must be **associative** and the identity must be **genuinely neutral**:

```java
// Correct: 0 is neutral for sum, + is associative
int sum = IntStream.rangeClosed(1, 10).parallel().reduce(0, Integer::sum); // 55

// WRONG identity: using 1 instead of 0 for sum
// In parallel, the identity is applied once per *partition*, not once overall
int wrong = IntStream.rangeClosed(1, 10).parallel().reduce(1, Integer::sum);
// Result is unpredictable — adds extra 1s depending on partition count
```

The JVM applies the identity to *each partition* before combining. This is correct behavior when the identity is truly neutral (1+x = x for `+` with identity 0). A wrong identity causes extra contributions from each partition.

## Encounter order

Some sources have a guaranteed *encounter order* (arrays, `ArrayList`, ranges); some do not (`HashSet`). Parallel streams preserve encounter order only when the source and operations support it:

```java
// Ordered — result list order matches source order
List<Integer> sorted = IntStream.rangeClosed(1, 10)
    .parallel()
    .filter(n -> n % 2 == 0)
    .boxed()
    .collect(Collectors.toList());
// [2, 4, 6, 8, 10] — order preserved

// If you don't need order, hint with unordered() for better performance
long count = IntStream.rangeClosed(1, 1_000_000)
    .parallel()
    .unordered()
    .filter(n -> n % 7 == 0)
    .count();
```

## When parallel helps vs hurts

| Scenario | Parallel? |
|---|---|
| Large data (millions of elements), CPU-bound transform | Yes |
| Stateless map/filter/reduce | Yes |
| Small collections (< thousands) | No — overhead dominates |
| I/O operations inside the lambda | No — blocks pool threads |
| `.sorted()` / `.distinct()` | Usually no — coordination overhead |
| Shared mutable state | Never |

Measure the difference between sequential and parallel sum on a large range:

```java

        long seqTime = System.nanoTime() - start;

    }
}
```

The results vary by machine and JVM warmup. On a multi-core box with 10 million elements, the parallel version is often 2-4x faster for this pure arithmetic task.

## Common mistakes

- **Using parallel on small collections** — the fork-join overhead makes the parallel version slower. Always measure before optimising.
- **Mutating shared state inside a parallel stream** — adds to a non-thread-safe collection, increments a shared counter, etc. The results are silently wrong or throw `ConcurrentModificationException`.
- **Blocking inside a parallel lambda** — sleeping, waiting on I/O, or acquiring a lock inside a `.parallel()` pipeline starves the common fork-join pool, hurting every other parallel operation in the JVM.
- **Wrong identity in parallel reduce** — the identity is applied per partition. `reduce(1, Integer::sum)` gives wrong results in parallel (adds an extra 1 per partition). Use `reduce(0, Integer::sum)`.

## Recap

- `.parallel()` or `.parallelStream()` routes work through the common fork-join pool
- The default pool size is `availableProcessors - 1`
- Only stateless, side-effect-free operations are safe in parallel
- Use `collect` to gather results — never mutate a shared collection from `forEach`
- `reduce(identity, op)` requires a truly neutral identity and an associative operator
- Parallel helps for large, CPU-bound, stateless work; hurts for small data, I/O, or stateful ops
