The previous lessons introduced lambdas, functional interfaces, streams, Optional, and composition as individual tools. This capstone lesson shows how those tools combine into recognisable *design patterns*: approaches to recurring problems that functional style solves more elegantly than traditional object-oriented solutions. By the end you'll have a mini stream-like fluent API you wrote yourself, giving you insight into how the real Java streams library works internally.

## Pattern 1: Strategy with lambdas

The classic Strategy pattern replaces a family of interchangeable algorithms with a common interface. Before lambdas, this required a class per strategy. With lambdas, a strategy is just a functional interface value:

```java
import java.util.function.*;
import java.util.*;

// Sorting strategies as lambdas — no extra classes needed
Comparator<String> byLength = Comparator.comparingInt(String::length);
Comparator<String> alphabetical = Comparator.naturalOrder();
Comparator<String> byLengthThenAlpha = byLength.thenComparing(alphabetical);

List<String> words = new ArrayList<>(List.of("fig", "apple", "kiwi", "cherry", "banana"));
words.sort(byLengthThenAlpha);
System.out.println(words); // [fig, kiwi, apple, cherry, banana]
```

A pricing engine using a discount strategy:

```java
@FunctionalInterface
interface DiscountStrategy {
    double apply(double price);
}

DiscountStrategy noDiscount   = price -> price;
DiscountStrategy tenPercent   = price -> price * 0.9;
DiscountStrategy halfPrice    = price -> price * 0.5;

double checkout(double price, DiscountStrategy strategy) {
    return strategy.apply(price);
}

System.out.println(checkout(100.0, tenPercent)); // 90.0
```

## Pattern 2: Pipeline

A pipeline is a sequence of transformations applied to data. This is exactly what streams do, but you can build a lightweight version for specialised use cases:

```java
import java.util.*;
import java.util.function.*;
import java.util.stream.*;

public class Pipeline<T> {
    private final List<Function<List<T>, List<T>>> steps = new ArrayList<>();

    public static <T> Pipeline<T> of() { return new Pipeline<>(); }

    public Pipeline<T> filter(Predicate<T> pred) {
        steps.add(list -> list.stream().filter(pred).collect(Collectors.toList()));
        return this;
    }

    public Pipeline<T> map(UnaryOperator<T> fn) {
        steps.add(list -> list.stream().map(fn).collect(Collectors.toList()));
        return this;
    }

    public List<T> execute(List<T> input) {
        List<T> result = new ArrayList<>(input);
        for (Function<List<T>, List<T>> step : steps) {
            result = step.apply(result);
        }
        return result;
    }
}
```

Usage:

```java
Pipeline<String> pipeline = Pipeline.<String>of()
    .filter(s -> s.length() > 3)
    .map(String::toUpperCase);

List<String> output = pipeline.execute(
    List.of("hi", "hello", "java", "streams")
);
System.out.println(output); // [HELLO, JAVA, STREAMS]
```

## Pattern 3: Memoization

Pure functions always return the same output for the same input. When computation is expensive, cache results with `Map.computeIfAbsent`:

```java
import java.util.*;
import java.util.function.*;

public class Memoize {
    public static <A, R> Function<A, R> of(Function<A, R> fn) {
        Map<A, R> cache = new HashMap<>();
        return arg -> cache.computeIfAbsent(arg, fn);
    }
}

// Memoize an expensive computation
Function<Integer, Long> fib = Memoize.of(n -> {
    // naive recursive Fibonacci — slow without memoisation
    if (n <= 1) return (long) n;
    return fib.apply(n - 1) + fib.apply(n - 2); // fib captured below
});
// NOTE: for recursive memoisation, use a static Map (see Try it section)

Function<String, Integer> countVowels = Memoize.of(s ->
    (int) s.chars().filter("aeiou"::indexOf).count()
);

System.out.println(countVowels.apply("functional")); // 4
System.out.println(countVowels.apply("functional")); // 4 (cached)
```

<Note>`Map.computeIfAbsent` is not thread-safe for recursive lambdas in Java 8 (it can deadlock). Use `ConcurrentHashMap` only for non-recursive memoization in concurrent contexts. For recursive memoization, a plain `HashMap` is simplest and correct when accessed single-threaded.</Note>

## Pattern 4: Lazy evaluation with Supplier

`Supplier<T>` defers computation until the value is actually needed:

```java
import java.util.function.*;

// Without lazy: always computed, even if unused
String eagerLog = "Result: " + expensiveComputation();

// With lazy: only computed when logged
Supplier<String> lazyLog = () -> "Result: " + expensiveComputation();
if (debugEnabled) System.out.println(lazyLog.get()); // only calls expensiveComputation here
```

Optional uses the same idea with `orElseGet(() -> fallback())`. Libraries like SLF4J use suppliers for lazy log message formatting to avoid string concatenation when the log level is off.

Build the complete `Pipeline<T>` class with `filter`, `map`, and `toList`, then exercise it:

```java
import java.util.*;
import java.util.function.*;
import java.util.stream.*;

public class Main {
    static class Pipeline<T> {
        private final List<UnaryOperator<List<T>>> steps = new ArrayList<>();

        public static <T> Pipeline<T> start() { return new Pipeline<>(); }

        public Pipeline<T> filter(Predicate<T> pred) {
            steps.add(list -> list.stream()
                .filter(pred).collect(Collectors.toList()));
            return this;
        }

        @SuppressWarnings("unchecked")
        public <R> Pipeline<R> map(Function<T, R> fn) {
            Pipeline<R> next = new Pipeline<>();
            next.steps.addAll((List) steps);
            next.steps.add(list -> ((List<T>) list).stream()
                .map(fn).collect(Collectors.toList()));
            return next;
        }

        public List<T> toList(List<T> source) {
            List<T> result = new ArrayList<>(source);
            for (UnaryOperator<List<T>> step : steps) result = step.apply(result);
            return result;
        }
    }

    public static void main(String[] args) {
        List<String> words = List.of("hi", "hello", "world", "java", "fun", "streams");

        List<String> result = Pipeline.<String>start()
            .filter(s -> s.length() > 3)
            .map(String::toUpperCase)
            .toList(words);

        System.out.println(result); // [HELLO, WORLD, JAVA, STREAMS]

        // Memoised vowel counter
        Map<String, Long> vowelCache = new HashMap<>();
        Function<String, Long> countVowels = w ->
            vowelCache.computeIfAbsent(w,
                s -> s.chars().filter(c -> "aeiou".indexOf(c) >= 0).count());

        words.forEach(w -> System.out.println(w + " -> " + countVowels.apply(w)));
    }
}
```

## Common mistakes

- **Memoizing impure functions** — if the function has side effects or depends on external state, caching silently returns stale results. Memoize only pure, deterministic functions.
- **Building overly clever pipelines** — functional pipelines are great for transforming data, but deeply nested `andThen` / `compose` chains without named intermediate steps become unreadable. Extract named variables.
- **Using `HashMap` for concurrent memoization** — `computeIfAbsent` on `HashMap` is not thread-safe. Use `ConcurrentHashMap` for shared caches, being aware that the supplier may be called more than once for the same key under race conditions.
- **Forgetting that `Pipeline.execute` eagerly materialises each step** — unlike Java streams, this simple implementation builds an intermediate `List` at each step. Real streams avoid this with lazy spliterators. For high-throughput production work, prefer `Stream` over a hand-rolled pipeline.

The functional style is not a replacement for every Java idiom — loops, classes, and inheritance still have their place. But you now have the vocabulary to choose: when the data flows in a clear transformation chain, reach for streams and lambdas; when the logic is stateful and complex, reach for imperative control flow. Knowing both — and the trade-offs between them — is what makes you a well-rounded Java developer.
