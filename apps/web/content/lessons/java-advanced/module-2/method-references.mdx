Method references are method references are a shorthand notation for lambdas that do nothing but call a single existing method. Where a lambda would be `s -> s.toUpperCase()`, a method reference is `String::toUpperCase`. The JVM generates the same bytecode — the difference is purely in how much you type and how clearly the intent reads.

The four forms cover every scenario you'll encounter. Learning to recognise them by shape turns unfamiliar library code from mysterious to immediately readable.

## Form 1 — Static method reference

`ClassName::staticMethodName`

Used when the method is `static`. The lambda's arguments are passed directly to the static method.

```java
// Lambda                         Method reference
s -> Integer.parseInt(s)          Integer::parseInt
n -> Math.abs(n)                  Math::abs
(a, b) -> Integer.max(a, b)       Integer::max
```

```java
import java.util.function.*;

Function<String, Integer> parse = Integer::parseInt;
System.out.println(parse.apply("42")); // 42
```

## Form 2 — Bound instance method reference

`instance::instanceMethodName`

The receiver is fixed to a specific object that was captured when the reference was created. The lambda's arguments become the method's arguments.

```java
String prefix = "Hello, ";
// Lambda:                              Bound reference:
s -> prefix.concat(s)                  prefix::concat

PrintStream out = System.out;
s -> out.println(s)                    System.out::println
```

```java
Consumer<String> log = System.out::println;
log.accept("bound reference");  // bound reference
```

## Form 3 — Unbound instance method reference

`ClassName::instanceMethodName`

No receiver is captured. When the lambda is invoked, the *first* argument becomes `this` for the method call.

```java
// Lambda                              Unbound reference
s -> s.toUpperCase()                   String::toUpperCase
s -> s.length()                        String::length
(s, regex) -> s.split(regex)           String::split
```

This form is especially common in stream pipelines:

```java
List.of("alice", "bob", "carol")
    .stream()
    .map(String::toUpperCase)   // unbound: each element becomes `this`
    .sorted(String::compareTo)  // unbound Comparator: (a, b) -> a.compareTo(b)
    .forEach(System.out::println); // bound: always prints to System.out
```

<Note>The difference between bound and unbound is whether the receiver is fixed at reference creation time. `"prefix"::concat` captures the specific string `"prefix"`. `String::concat` expects the receiver to be supplied as the first argument at call time.</Note>

## Form 4 — Constructor reference

`ClassName::new`

Constructs a new instance. The lambda's arguments become constructor arguments.

```java
// Supplier — zero-argument constructor
Supplier<ArrayList<String>> maker = ArrayList::new;
ArrayList<String> list = maker.get();

// Function — single-argument constructor
Function<String, StringBuilder> builder = StringBuilder::new;
StringBuilder sb = builder.apply("start");

// BiFunction — two-argument constructor
BiFunction<String, Integer, String> repeated = String::new; // not useful here, just illustrative
```

Constructor references are most useful with `Collectors.toCollection()` and factory methods:

```java
// Collect into a specific List implementation
List<String> mutable = stream.collect(Collectors.toCollection(ArrayList::new));
```

## When to prefer method references over lambdas

A method reference is preferable when the lambda *only* calls a single method and passes the arguments through unchanged:

```java
// Prefer method reference — the lambda adds nothing
.map(s -> s.trim())          →  .map(String::trim)
.forEach(x -> list.add(x))  →  .forEach(list::add)

// Keep the lambda — the lambda adds real logic
.map(s -> s.trim().toLowerCase())   // two chained calls, no single method
.filter(n -> n > 0 && n < 100)      // compound expression
```

The rule of thumb: if reading the method reference name is *more* informative than reading the lambda body, use the reference.

## Common mistakes

- **Using `List::new` for a constructor reference** — `List` is an interface with no constructor. Use `ArrayList::new`, `LinkedList::new`, or another concrete class.
- **Confusing bound and unbound** — `System.out::println` is bound to the specific `out` object. `PrintStream::println` would be unbound and require a `PrintStream` as the first argument. Pick bound for "always use this object", unbound for "apply to each element".
- **Overusing method references** — when the lambda has more logic than a single forwarding call, a lambda is clearer. `s -> prefix + s.trim()` is better left as a lambda.
- **Forgetting that an unbound reference adds a parameter** — `String::compareTo` as a `Comparator<String>` receives two strings: the receiver and the argument. If the arity doesn't match the target functional interface, the compiler will tell you.

## Recap

- **Static:** `ClassName::staticMethod` — arguments forwarded directly
- **Bound:** `instance::method` — receiver is a captured object
- **Unbound:** `ClassName::instanceMethod` — first argument becomes the receiver
- **Constructor:** `ClassName::new` — constructs an instance
- Prefer method references when they name the operation more clearly than the lambda body does
