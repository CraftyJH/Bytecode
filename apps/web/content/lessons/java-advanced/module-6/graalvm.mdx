For the first 25 years of Java's life, every deployment came with the cost of a JVM startup, a warm-up period, and significant baseline memory. That was an acceptable tradeoff for long-running servers. Then serverless computing changed the equation: functions that start per-request, containers that scale to zero, CLI tools that users expect to launch instantly. GraalVM's native-image addresses all of these by compiling Java ahead-of-time into a standalone executable — no JVM required.

## GraalVM CE vs EE

GraalVM comes in two editions:

| Edition | License | Features |
|---------|---------|----------|
| Community Edition (CE) | GPLv2 + CE | native-image, polyglot, Graal JIT |
| Enterprise Edition (EE) | Commercial | All CE + G1/ZGC in native, PGO, compressed images |

For most teams, **GraalVM CE** is sufficient. It is available as a drop-in replacement for standard OpenJDK builds and powers the `native-image` tool.

```bash
# Install GraalVM CE via SDKMAN
sdk install java 21.0.3-graal

# Verify
java -version
# openjdk version "21.0.3" 2024-04-16
# OpenJDK Runtime Environment GraalVM CE 21.0.3+7.1 (build ...)

# Install native-image toolchain
gu install native-image
```

## Building a native image

```bash
# Compile Java source
javac Hello.java

# Build native executable (requires a C compiler in PATH)
native-image Hello -o hello

# Run — no JVM needed
./hello
# Startup: ~15 ms
# vs: java Hello  → Startup: ~150–300 ms
```

For Maven projects:

```bash
# Using the Native Build Tools plugin
./mvnw -Pnative native:compile

# Result: target/myapp  (self-contained executable)
./target/myapp
```

## Startup and memory tradeoffs

```
                   JVM (JIT)     Native Image
Startup time       200–1000 ms   10–80 ms
Memory at startup  100–300 MB    20–80 MB
Peak throughput    Very high     Moderate (no JIT)
Pause times        GC-dependent  Low (no JIT deopt)
```

<Note>Native images do still have garbage collection — GraalVM includes a serial GC, G1GC (EE), and experimental Epsilon/Shenandoah support. The difference is there is no JIT compiler overhead and no JVM startup cost.</Note>

## The closed-world assumption

Native-image performs **points-to analysis** at build time: it starts from `main()` and traces every reachable class, method, and field. Anything not reachable is excluded from the executable.

This breaks features that load code dynamically:

| Feature | Issue | Fix |
|---------|-------|-----|
| Reflection (`Class.forName`, `getDeclaredMethods`) | Classes not in analysis | `reflect-config.json` |
| JNI | Native bindings unknown | `jni-config.json` |
| Dynamic proxies | Proxy classes generated at runtime | `proxy-config.json` |
| Resources (`getResourceAsStream`) | Files not bundled | `resource-config.json` |
| Serialization | Classes serialized by name | `serialization-config.json` |

## Reflection configuration

```json
// reflect-config.json
[
  {
    "name": "com.example.MyService",
    "allDeclaredConstructors": true,
    "allDeclaredMethods": true,
    "allDeclaredFields": true
  },
  {
    "name": "com.example.dto.UserDTO",
    "allDeclaredFields": true
  }
]
```

```bash
native-image -H:ReflectionConfigurationFiles=reflect-config.json -jar myapp.jar
```

**Tracing agent** — GraalVM includes an agent that runs your tests and generates all configuration files automatically:

```bash
# Run with tracing agent to auto-generate config
java -agentlib:native-image-agent=config-output-dir=src/main/resources/META-INF/native-image \
     -jar myapp.jar

# Now run your test suite or exercise all code paths
# Agent writes reflect-config.json, proxy-config.json, etc.

# Build native image using generated configs
native-image -jar myapp.jar
```

## Spring Boot 3 native

Spring Boot 3 + Spring Framework 6 has first-class native support via an AOT engine that automatically generates native-image hints for all Spring-managed beans:

```xml
<!-- pom.xml -->
<plugin>
    <groupId>org.graalvm.buildtools</groupId>
    <artifactId>native-maven-plugin</artifactId>
</plugin>
```

```bash
# Build Spring Boot native image
./mvnw -Pnative native:compile

# Run the native Spring Boot app
./target/myapp
# Started MyApplication in 0.087 seconds (process running for 0.1)
```

Compare that to the JVM version which typically reports 1–3 seconds startup time.

## Use cases: when to choose native image

**Good fit:**
- Serverless functions (AWS Lambda, Google Cloud Functions)
- CLI tools and dev tooling
- Microservices with strict startup SLAs
- Short-lived containers that scale to zero

**Poor fit:**
- Long-running services with complex reflection/serialization libraries
- Applications requiring maximum sustained throughput (JIT wins here)
- Teams using dynamic languages features heavily (Groovy scripts, runtime code gen)

## Common mistakes

- **Not running the tracing agent** — manually writing reflection configuration is error-prone and incomplete. Always run the agent against a comprehensive test suite to generate config automatically.
- **Assuming all libraries are native-image compatible** — some libraries use aggressive reflection, class generation, or JVM internals that are hard to configure. Check GraalVM's compatibility list before adopting native image for a complex app.
- **Choosing native image for peak throughput** — a warm JIT consistently outperforms AOT for sustained high-load services. Native image wins on startup and memory, not on throughput.
- **Ignoring the build time cost** — native-image compilation can take 2–10 minutes depending on application size. Factor this into your CI pipeline planning.

## Recap

- GraalVM **native-image** compiles Java AOT into a standalone executable with no JVM required at runtime.
- Native images start in milliseconds and use less memory — ideal for serverless and CLI tools.
- The **closed-world assumption** requires all reflective, JNI, proxy, and resource access to be declared in configuration files.
- The **tracing agent** generates configuration automatically by observing a live run.
- **Spring Boot 3** AOT engine handles native-image hints for Spring-managed components automatically.
