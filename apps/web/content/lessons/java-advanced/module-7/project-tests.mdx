Four parts in, we have a domain model, a generic repository, a service layer with functional validation, and concurrent background workers. Before we add more features it is time to verify what we have. This lesson covers the full testing strategy for the Task Management System, from fast unit tests on the domain model all the way to multi-threaded stress tests on the repository.

Testing connects to every previous module: parameterized tests use streams to generate inputs, integration tests exercise the service + repository stack together, and concurrent tests expose races that single-threaded tests never catch. This is also where the discipline of the architecture pays off — because domain, repository, and service are cleanly separated, we can test each in isolation.

## Testing pyramid for this system

| Layer | What to test | Test type | Speed |
|---|---|---|---|
| Domain (`Task`, `TaskStatus`) | Record invariants, transition guards | Unit | < 1 ms each |
| Validation pipeline | All valid/invalid combos | Parameterised unit | < 1 ms each |
| `TaskService` | Business rules, Optional semantics | Integration (real repo) | < 10 ms |
| Concurrent code | Race conditions, queue draining | Concurrent / stress | 100–500 ms |
| Notification pipeline | Async completion, error handling | Async unit | < 50 ms |

Run the unit tests on every commit; run the concurrent tests in CI but not necessarily on every local build (they are slower and nondeterministic by nature).

## Domain unit tests — TaskStatus transitions

```java
package com.bytecode.tasks.domain;

import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.MethodSource;

import java.util.stream.Stream;

import static org.junit.jupiter.api.Assertions.*;

class TaskStatusTest {

    record TransitionCase(TaskStatus from, TaskStatus to, boolean expected) {}

    static Stream<TransitionCase> transitions() {
        return Stream.of(
            new TransitionCase(TaskStatus.TODO,        TaskStatus.IN_PROGRESS, true),
            new TransitionCase(TaskStatus.IN_PROGRESS, TaskStatus.DONE,        true),
            new TransitionCase(TaskStatus.TODO,        TaskStatus.DONE,        false),
            new TransitionCase(TaskStatus.DONE,        TaskStatus.TODO,        false),
            new TransitionCase(TaskStatus.DONE,        TaskStatus.IN_PROGRESS, false),
            new TransitionCase(TaskStatus.IN_PROGRESS, TaskStatus.TODO,        false)
        );
    }

    @ParameterizedTest(name = "{0} → {1} should be {2}")
    @MethodSource("transitions")
    void canTransitionTo(TransitionCase tc) {
        assertEquals(tc.expected(), tc.from().canTransitionTo(tc.to()),
            tc.from() + " → " + tc.to());
    }
}
```

Each row is a separate test invocation with a descriptive name. If the `DONE → TODO` case regresses, the CI report names it explicitly.

        assertEquals(TaskStatus.TODO, taskRepo.findById(t.id()).get().status());

## Concurrent stress test — repository consistency

```java
class RepositoryConcurrencyTest {

    @Test
    void tenThreadsCreateTasks_allPersisted() throws Exception {
        var taskRepo    = new InMemoryTaskRepository();
        var projectRepo = new InMemoryProjectRepository();
        var service     = new TaskService(taskRepo, projectRepo);

        int threadCount = 10;
        CountDownLatch ready  = new CountDownLatch(threadCount);
        CountDownLatch start  = new CountDownLatch(1);
        CountDownLatch done   = new CountDownLatch(threadCount);
        ExecutorService exec  = Executors.newFixedThreadPool(threadCount);

        for (int i = 0; i < threadCount; i++) {
            final String title = "Task-" + i;
            exec.submit(() -> {
                ready.countDown();
                try {
                    start.await();   // all threads start simultaneously
                    service.createTask(title, "desc", Priority.MEDIUM,
                                       Optional.empty(), Optional.empty());
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    done.countDown();
                }
            });
        }

        ready.await();    // wait until all threads are at the start line
        start.countDown(); // fire!
        done.await(5, TimeUnit.SECONDS);
        exec.shutdown();

        assertEquals(threadCount, taskRepo.count(),
            "Repository should contain exactly " + threadCount + " tasks");
    }
}
```

<Note>The `start` latch synchronises all threads to begin at the same instant, maximising contention. Without it, threads would start staggered and the test would be far less likely to expose races.</Note>

## Common mistakes

- **Mocking the repository in integration tests** — mocking `InMemoryTaskRepository` defeats the purpose of an integration test. Use the real in-memory implementation; it is fast (microseconds) and exercises the actual code path. Reserve mocks for external systems (email APIs, databases) that you genuinely cannot run in a test.
- **Non-deterministic async assertions** — asserting on a `CompletableFuture` result without calling `.join()` or `.get()` first creates a race between the test assertion and the async computation. Always block before asserting.
- **Missing `@BeforeEach` reset** — if the repository is a shared field, tasks created in one test pollute the next. Always create a fresh instance in `@BeforeEach`.
- **Testing only the happy path** — validation logic is most valuable when it is tested on the *edges*: blank strings, null values, yesterday's date. Every validation rule deserves at least one invalid-input test that confirms the rejection.

## Recap

- A **parameterized test** with `@MethodSource` gives each test case a separate named result in the CI report — critical for state-machine transition matrices.
- **Integration tests** with real in-memory implementations are preferred over mocks for testing service + repository interactions; they are fast enough and test real behavior.
- `CountDownLatch` with a shared `start` latch forces maximum thread contention in concurrency tests, surfacing races that staggered starts would miss.
- `CompletableFuture` tests use `.join()` or `.get(timeout, unit)` to synchronize the test thread before asserting.
- Test coverage goals: 100% on domain logic and validation, ≥ 90% on the service layer, at least one concurrent stress test per shared mutable resource.
