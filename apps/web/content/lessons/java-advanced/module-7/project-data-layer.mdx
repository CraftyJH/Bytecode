In Part 1 we built a domain model that compiles cleanly and enforces its own invariants. Now we need somewhere to store and retrieve domain objects. In a production system that would be a database; in our capstone it will be an **in-memory repository** backed by a `ConcurrentHashMap`. The design pattern is the same either way: the service layer talks to a `Repository<T, ID>` interface, and the concrete storage strategy can be swapped without touching the service code.

This lesson also applies everything from Module 1 (generic classes), Module 2 (streams and functional interfaces), and a taste of Module 4 (concurrency-safe collections). Watch how those pieces snap together.

## The Repository pattern

The Repository pattern, popularised by Domain-Driven Design, places a collection-like interface between domain logic and storage. The service layer treats the repository as if it were a smart, queryable in-memory collection — it does not need to know whether the underlying store is a `HashMap`, a PostgreSQL database, or a remote REST API.

The simplest possible repository interface:

```java
package com.bytecode.tasks.repository;

import java.util.List;
import java.util.Optional;
import java.util.function.Predicate;

public interface Repository<T, ID> {
    void         save(T entity);
    Optional<T>  findById(ID id);
    List<T>      findAll();
    boolean      delete(ID id);
    List<T>      findWhere(Predicate<T> predicate);
}
```

`T` is the entity type (`Task`, `Project`, `User`). `ID` is the identifier type (we use `UUID` for all three). `findWhere(Predicate<T>)` is the functional escape hatch: callers compose any filter they need without us adding dozens of `findByXxx` overloads to the interface.

<Note>`Repository` is an interface, not an abstract class. This keeps the contract pure and allows a concrete repository to implement multiple interfaces (e.g., `Repository<Task, UUID>` and `Closeable`) without conflicting state.</Note>

## The generic in-memory base class

Now we write `InMemoryRepository<T, ID>` once. Every concrete repository inherits CRUD for free:

```java
package com.bytecode.tasks.repository;

import java.util.*;
import java.util.concurrent.ConcurrentHashMap;
import java.util.function.Function;
import java.util.function.Predicate;
import java.util.stream.Collectors;

public class InMemoryRepository<T, ID> implements Repository<T, ID> {

    protected final Map<ID, T> store = new ConcurrentHashMap<>();
    private final Function<T, ID> idExtractor;

    public InMemoryRepository(Function<T, ID> idExtractor) {
        this.idExtractor = idExtractor;
    }

    @Override
    public void save(T entity) {
        store.put(idExtractor.apply(entity), entity);
    }

    @Override
    public Optional<T> findById(ID id) {
        return Optional.ofNullable(store.get(id));
    }

    @Override
    public List<T> findAll() {
        return List.copyOf(store.values());   // immutable snapshot
    }

    @Override
    public boolean delete(ID id) {
        return store.remove(id) != null;
    }

    @Override
    public List<T> findWhere(Predicate<T> predicate) {
        return store.values().stream()
                    .filter(predicate)
                    .collect(Collectors.toUnmodifiableList());
    }

    public long count() {
        return store.size();
    }
}
```

The `idExtractor` `Function<T, ID>` is injected via the constructor — this is the only thing that varies between entity types and it keeps the base class fully generic.

## Pagination — a practical addition

Real systems never load every entity in one shot. A lightweight pagination helper keeps responses bounded:

```java
package com.bytecode.tasks.repository;

import java.util.List;

public record Page<T>(List<T> content, int pageNumber, int pageSize, long totalElements) {

    public int totalPages() {
        return (int) Math.ceil((double) totalElements / pageSize);
    }

    public boolean hasNext() {
        return pageNumber < totalPages() - 1;
    }
}
```

And we add a `findPage` method to `InMemoryRepository`:

```java
public Page<T> findPage(int pageNumber, int pageSize) {
    List<T> all = List.copyOf(store.values());
    long total  = all.size();
    List<T> content = all.stream()
            .skip((long) pageNumber * pageSize)
            .limit(pageSize)
            .collect(Collectors.toUnmodifiableList());
    return new Page<>(content, pageNumber, pageSize, total);
}
```

<Note>In-memory pagination is deterministic only if the iteration order of the underlying collection is stable. `ConcurrentHashMap` does not guarantee insertion order. In production you would sort before paginating; here it is fine for the capstone.</Note>

## The concrete task repository

`InMemoryTaskRepository` extends the base class and adds domain-specific convenience queries:

```java
package com.bytecode.tasks.repository;

import com.bytecode.tasks.domain.*;

import java.util.List;
import java.util.UUID;

public class InMemoryTaskRepository extends InMemoryRepository<Task, UUID> {

    public InMemoryTaskRepository() {
        super(Task::id);   // id extractor: Task::id
    }

    public List<Task> findByStatus(TaskStatus status) {
        return findWhere(t -> t.status() == status);
    }

    public List<Task> findByPriority(Priority priority) {
        return findWhere(t -> t.priority() == priority);
    }

    public List<Task> findByAssignee(String assigneeId) {
        return findWhere(t -> t.assigneeId()
                               .map(a -> a.equals(assigneeId))
                               .orElse(false));
    }

    public List<Task> findOverdue(java.time.LocalDate today) {
        return findWhere(t ->
            t.dueDate()
             .map(d -> d.isBefore(today) && t.status() != TaskStatus.DONE)
             .orElse(false)
        );
    }
}
```

`findByAssignee` and `findOverdue` demonstrate composing Optional logic inside a Predicate — a pattern straight from Module 2.

## Repositories for Project and User

Adding repositories for the other entities is now trivial:

```java
public class InMemoryProjectRepository extends InMemoryRepository<Project, UUID> {
    public InMemoryProjectRepository() { super(Project::getId); }

    public List<Project> findByOwner(String ownerId) {
        return findWhere(p -> p.getOwnerId().equals(ownerId));
    }
}

public class InMemoryUserRepository extends InMemoryRepository<User, UUID> {
    public InMemoryUserRepository() { super(User::id); }

    public Optional<User> findByUsername(String username) {
        return store.values().stream()
                    .filter(u -> u.username().equals(username))
                    .findFirst();
    }
}
```

Both classes are roughly ten lines. Without the generic base class each would be fifty.

## Common mistakes

- **HashMap instead of ConcurrentHashMap** — `HashMap` is not thread-safe. When background threads (introduced in Part 4) start reading and writing the repository simultaneously, a `HashMap` can silently corrupt its internal hash table. Use `ConcurrentHashMap` from day one.
- **Returning a live internal list** — `new ArrayList<>(store.values())` creates a snapshot, but returning `store.values()` directly exposes the map's live key-set view. Use `List.copyOf()` or `.collect(Collectors.toUnmodifiableList())` to prevent callers from mutating the internal state.
- **One massive repository interface** — adding every query method to `Repository<T, ID>` forces every implementation to stub out methods it does not need. Keep the interface minimal (CRUD + `findWhere`) and add domain-specific methods only to the concrete class.
- **Forgetting the id extractor** — without a `Function<T, ID>` the base class cannot know which field is the identifier. Avoid reflection-based magic; an explicit lambda like `Task::id` is clear, fast, and refactoring-safe.

## Recap

- The **Repository pattern** decouples service logic from storage strategy; the service calls an interface, not a concrete class.
- A generic `Repository<T, ID>` interface with `findWhere(Predicate<T>)` covers the vast majority of query needs without bloating the interface.
- `InMemoryRepository<T, ID>` provides all CRUD logic once; concrete repositories extend it and add only domain-specific finders.
- `ConcurrentHashMap` is the correct backing store for an in-memory repository that will be accessed from multiple threads.
- Pagination is a first-class concern — always limit unbounded `findAll` calls.
