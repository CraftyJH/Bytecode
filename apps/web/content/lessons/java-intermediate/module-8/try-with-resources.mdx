Once you start working with files, streams, and connections, you'll meet a recurring chore: every resource you open must be closed, in a `finally` block, even (especially) on exceptions. Get it wrong and you leak file handles, sockets, and database connections — bugs that are invisible until the system runs out of them.

Java 7 introduced **try-with-resources** to make this safe by default.

## The pattern it replaces

Before Java 7, you wrote this:

```java
BufferedReader r = null;
try {
    r = Files.newBufferedReader(p);
    System.out.println(r.readLine());
} finally {
    if (r != null) {
        try { r.close(); } catch (IOException ignored) { }
    }
}
```

Six lines of boilerplate to safely read one line. And if both the body and `close()` throw, the close exception silently swallows the original.

## The new form

```java
try (BufferedReader r = Files.newBufferedReader(p)) {
    System.out.println(r.readLine());
}
```

Resources declared in the parentheses are auto-closed when the try block exits — even on exception. The block reads top-to-bottom: open, use, done. The compiler generates the equivalent finally + close code for you, including the suppressed-exception handling.

To work as a resource, a class must implement **`java.lang.AutoCloseable`** (or its IOException-throwing subtype `java.io.Closeable`). All the standard JDK readers, writers, streams, channels, and connection types do.

<Note>The variable in the parens is implicitly <code>final</code>. You cannot reassign it inside the body — the compiler enforces this so the close() call hits the same object you opened.</Note>

## Multiple resources

You can declare more than one resource, separated by semicolons:

```java
try (
    BufferedReader r = Files.newBufferedReader(in);
    BufferedWriter w = Files.newBufferedWriter(out)
) {
    String line;
    while ((line = r.readLine()) != null) {
        w.write(line.toUpperCase());
        w.newLine();
    }
}
```

Resources are **closed in reverse order** of declaration, so dependent resources tear down correctly. (If `w` somehow depended on `r`, you'd want `w` closed first.)

## Suppressed exceptions

Here's the subtle case the new syntax handles for you. What if the body throws *and* `close()` throws? You can't return both — but throwing only one would lose information. Java's solution: the body's exception wins (the caller sees it), and the close-time exception is attached as a **suppressed** exception, visible in the stack trace and via `Throwable.getSuppressed()`:

```
Exception in thread "main" java.lang.RuntimeException: body failed
    at Main.main(Main.java:10)
    Suppressed: java.io.IOException: close failed
        at Main$Res.close(Main.java:5)
        ...
```

Suppression preserves the diagnostic value of both exceptions without forcing the caller to handle two at once.

## Java 9 enhancement: pre-existing variables

Originally, try-with-resources required you to declare each resource inside the parens. Java 9 relaxed this — if you have a final or effectively-final variable that already implements `AutoCloseable`, you can use it directly:

```java
BufferedReader r = Files.newBufferedReader(p);
// ... maybe pass r to a helper, configure it ...
try (r) {                   // Java 9+: just reference the existing variable
    String line = r.readLine();
    System.out.println(line);
}
```

Useful when a resource is created elsewhere but you want this method to own its lifetime.

## Writing your own AutoCloseable

It's a single method:

```java
class Latch implements AutoCloseable {
    private final String name;
    public Latch(String name) {
        this.name = name;
        System.out.println("acquire " + name);
    }
    @Override
    public void close() {
        System.out.println("release " + name);
    }
}

try (Latch outer = new Latch("outer"); Latch inner = new Latch("inner")) {
    // do work
}
// prints: acquire outer / acquire inner / release inner / release outer
```

Implementing `AutoCloseable` is how you make any "open / use / close" pattern safe for callers — locks, transactions, GUI windows, instrumentation spans, anything.

## Common mistakes

- **Reaching for try/finally instead of try-with-resources** — old habits and old code, but the new form is shorter, safer, and better at exceptions. Use it whenever the resource implements AutoCloseable.
- **Reassigning a resource inside the try** — the compiler refuses; resources are implicitly final. Restructure rather than fight it.
- **Silently swallowing close() exceptions in legacy code** — the old hand-written pattern often did this. Suppressed exceptions in try-with-resources retain the information.
- **Forgetting that order matters for closing** — resources close in reverse declaration order. If A creates B, declare A first, then B; closing B first is what you want.
- **Wrapping a Stream from `Files.list` or `Files.lines` without try-with-resources** — these own OS handles. Skipping the try-with-resources leaks them.

## Recap

try-with-resources is a small piece of syntax that solves a class of bug (resource leaks, lost exceptions) that used to plague every Java codebase. Once you have the muscle memory, you'll never write a manual close again.

- Resources go in the parentheses of `try (...)`. They must implement `AutoCloseable`.
- Multiple resources are allowed, separated by `;`. Closed in reverse order.
- Body exceptions are primary; close exceptions are added as suppressed (visible in stack traces).
- Java 9+ lets you reference an effectively-final existing variable as a resource.
- Implement `AutoCloseable` on your own classes to make any open/use/close pattern compositional.
