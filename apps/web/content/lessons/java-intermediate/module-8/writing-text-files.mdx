Writing text to a file is symmetric to reading it: one method call, no resource management, one important option to know about. This lesson covers `writeString`, `write`, and the open options that change their behavior.

## The two main writes

```java
import java.nio.file.*;
import java.util.*;

Path p = Paths.get("/tmp/notes.txt");

// 1) Write a single String (Java 11+) — overwrites by default
Files.writeString(p, "Hello, world!\nSecond line.\n");

// 2) Write a List<String>, one entry per line, with auto separators
List<String> lines = List.of("alpha", "beta", "gamma");
Files.write(p, lines);
```

Both methods open the file, write the content, and close it. Both default to UTF-8. Both throw `IOException` (a checked exception) on failure.

The choice is just convenience: if you already have a String, use `writeString`. If you already have a `List<String>`, use `write` and let it handle the separators. Either is fine.

<Note>The directory containing the file must already exist. If <code>/tmp/some-dir/notes.txt</code> is the target but <code>/tmp/some-dir</code> doesn't exist, the call throws <code>NoSuchFileException</code>. Use <code>Files.createDirectories</code> to ensure the parent exists first.</Note>

## Default behavior: TRUNCATE then WRITE

By default, `writeString` and `write` follow these rules:

1. If the file does not exist, **create** it.
2. If the file does exist, **truncate** it (set its size to zero).
3. Write the new content from the start.

So calling `writeString(p, "x")` on an existing 1 GB file leaves you with a file containing exactly the byte `x`. There is no warning, no backup. This is usually what you want — but be aware of it before you write production code.

## Append mode

To add to a file instead of overwriting it, pass `StandardOpenOption.APPEND`:

```java
Files.writeString(p, "first call\n");
Files.writeString(p, "second call\n", StandardOpenOption.APPEND);
Files.writeString(p, "third call\n",  StandardOpenOption.APPEND);
// File now contains all three lines, in order.
```

Append mode is what you want for log files, diary-style writes, and any case where each call should add to history rather than replace it.

## Fail-if-exists: CREATE_NEW

Sometimes you want to be sure you don't accidentally overwrite an existing file:

```java
try {
    Files.writeString(p, "fresh content",
        StandardOpenOption.CREATE_NEW, StandardOpenOption.WRITE);
} catch (FileAlreadyExistsException e) {
    System.err.println("Refusing to overwrite: " + p);
}
```

`CREATE_NEW` says "create the file only if it doesn't exist; otherwise throw". This is the safest option for tools that generate output and shouldn't clobber prior runs.

## Standard open options at a glance

| Option | Effect |
| --- | --- |
| `CREATE` | Create the file if it doesn't exist. (Default for write methods.) |
| `CREATE_NEW` | Create the file; fail if it exists. |
| `TRUNCATE_EXISTING` | Wipe the file's contents on open. (Default for write methods.) |
| `APPEND` | Position the writer at the end before writing. |
| `WRITE` | Open for writing. (Implicit for write methods.) |
| `READ` | Open for reading. |
| `DELETE_ON_CLOSE` | Best-effort delete the file when the channel closes. |

You can combine compatible options. `CREATE_NEW + APPEND` is contradictory and will throw at runtime; `CREATE + APPEND` makes sense ("create if missing, append otherwise").

## Atomicity caveats

Neither `writeString` nor `write` guarantees atomic writes. If your program is killed midway, you can end up with a partially written file. For applications where this matters (config files, save games), the standard pattern is:

1. Write the new content to a temporary file.
2. `Files.move(tempPath, finalPath, StandardCopyOption.ATOMIC_MOVE)`.

`Files.move` with `ATOMIC_MOVE` performs a single-step rename on most filesystems. Either the new content is fully there, or the old content is — never an incomplete file.

## Common mistakes

- **Accidental overwrites** — calling `writeString` on a file you didn't realise existed silently destroys it. If you're not sure, check `Files.exists` first or use `CREATE_NEW`.
- **Forgetting to create parent directories** — the file's containing directory must exist. `Files.createDirectories(p.getParent())` is a one-line fix.
- **Mixing append and truncate** — passing both `APPEND` and `TRUNCATE_EXISTING` is contradictory and throws at runtime.
- **Assuming atomic writes** — partial writes can leave a corrupt file if your process dies. For critical writes, use the temp-file + atomic-move pattern.
- **Charset surprises on read** — if you write with one charset and read with another, the result is mojibake. UTF-8 everywhere unless you have a strong reason otherwise.

## Recap

Writing text in modern Java is a single line, with one mental model: by default writes truncate. Anything else — appending, refusing-to-overwrite, atomic writes — is opt-in via `StandardOpenOption`.

- **`Files.writeString(path, content)`** — write a String, defaulting to UTF-8 and TRUNCATE_EXISTING.
- **`Files.write(path, lines)`** — write a `List<String>` with line separators between entries.
- Pass `StandardOpenOption.APPEND` to add to an existing file.
- Pass `StandardOpenOption.CREATE_NEW` to refuse to overwrite.
- Parent directories must exist; use `Files.createDirectories` to ensure them.
