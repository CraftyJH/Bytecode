The legacy `java.io.File` class has been the standard way to talk about files in Java for over twenty years. It works, but it has quirks: methods that return `boolean` instead of throwing on failure, no native support for symbolic links, and an inability to read file metadata atomically. Java 7 introduced the NIO.2 API (`java.nio.file`) to fix all of that.

The new API revolves around two types: **`Path`** describes a location, and **`Files`** is a utility class with a static method for every operation you might want to perform on that location.

## Path — an OS-aware location

A `Path` represents a sequence of name elements separated by the platform's directory separator. It might be absolute (`/etc/hosts`, `C:\Windows\System32`) or relative (`docs/readme.md`).

```java
import java.nio.file.*;

Path p = Paths.get("/tmp/notes/today.md");
// Java 11+: Path.of("/tmp/notes/today.md")

System.out.println(p);              // /tmp/notes/today.md
System.out.println(p.getFileName()); // today.md
System.out.println(p.getParent());   // /tmp/notes
System.out.println(p.getRoot());     // /
System.out.println(p.getNameCount()); // 3
System.out.println(p.getName(0));    // tmp
```

Important properties:
- **Immutable** — every "modifying" method returns a new `Path`.
- **Platform-aware** — on Windows you'd use `\` separators; on Linux/macOS, `/`. The API handles this for you.
- **Just text** — creating a `Path` does not check whether the file exists or open any handle. It's purely a typed wrapper around a string.

<Note>If you have an old <code>File</code>, call <code>file.toPath()</code> to get a Path. Going the other way, <code>path.toFile()</code> hands you back a File. Bridging is cheap and lossless.</Note>

## Combining paths: resolve and relativize

You almost never want to do `new Path(parent + "/" + child)` by hand — the API has built-in helpers that handle separators and absolute paths correctly.

```java
Path base = Paths.get("/tmp");
Path full = base.resolve("notes/today.md");
System.out.println(full);   // /tmp/notes/today.md

// resolve respects absolute arguments
Path other = base.resolve("/etc/hosts");
System.out.println(other);  // /etc/hosts  (absolute argument wins)

// The reverse: from one path, what's the relative way to another?
Path rel = base.relativize(full);
System.out.println(rel);    // notes/today.md
```

`resolve` is the most common Path operation in real code. Use it whenever you assemble a path from a known base and a user-supplied or computed sub-path.

## Files — the static utility class

`Path` describes a location; `Files` does things with it. Some examples you'll use repeatedly in this module:

```java
Files.exists(p);            // true / false
Files.isDirectory(p);       // true if p points to a directory
Files.isRegularFile(p);     // true if p points to an ordinary file
Files.size(p);              // file size in bytes
Files.readString(p);        // entire file as a String (Java 11+)
Files.readAllLines(p);      // List<String> — one entry per line
Files.writeString(p, text); // write/overwrite file content
```

You'll meet many more in the lessons that follow. The pattern is always the same: pick a verb, pass a `Path`, get a result.

## Normalising and comparing paths

Two paths can refer to the same file but look textually different (`/tmp/foo` vs `/tmp/./foo` vs `/tmp/bar/../foo`). Use `normalize()` to clean these up:

```java
Path messy = Paths.get("/tmp/./bar/../foo");
System.out.println(messy.normalize());  // /tmp/foo
```

`normalize()` is purely textual — it does not consult the filesystem. To resolve symbolic links and discover the *canonical* path, use `Files.readSymbolicLink` or `path.toRealPath()` (which does touch the disk and may throw `IOException`).

## Common mistakes

- **Concatenating with `+` instead of `resolve`** — `parent + "/" + child` works only on Unix-like systems and breaks if `child` is already absolute. Always use `resolve`.
- **Assuming a Path means the file exists** — creating a `Path` is just text manipulation. Always check with `Files.exists` before reading or writing if the file's existence matters.
- **Forgetting that Path is immutable** — `p.resolve("x")` does not modify `p`; it returns a new Path. Assign the result to a variable.
- **Using legacy `File` in new code** — apart from interop with old APIs, there's almost no reason to reach for `java.io.File` today. Path + Files is more powerful and less error-prone.

## Recap

The NIO.2 API gives Java a clean, modern way to talk about and manipulate filesystem locations. Once you internalise the Path/Files split, every other file-related lesson in this module reads as obvious applications of the same pattern.

- **`Path`** is an immutable, typed location. Build it with `Paths.get(...)` or `Path.of(...)`.
- Useful Path methods: `getFileName`, `getParent`, `getRoot`, `isAbsolute`, `resolve`, `relativize`, `normalize`.
- **`Files`** is the static utility class — every read, write, list, copy, or delete starts with `Files.something(path)`.
- Path objects don't touch the filesystem on their own. You must call a `Files` method (or `Files.exists`) to interact with disk.
