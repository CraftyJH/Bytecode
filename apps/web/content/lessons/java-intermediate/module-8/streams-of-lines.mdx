You've already met three ways to read a text file: `readString`, `readAllLines`, and `BufferedReader`. They share a problem — the first two load the whole file into memory, the third requires a verbose `while` loop.

`Files.lines(path)` is the modern alternative: it returns a **`Stream<String>`** that produces one line per element, lazily, on demand. Combined with the Stream API, it lets you express common file-processing tasks in a single, readable expression — and it scales to files larger than RAM.

## The basic pattern

```java
import java.nio.file.*;
import java.util.stream.*;

Path log = Paths.get("/tmp/server.log");
try (Stream<String> lines = Files.lines(log)) {
    long errors = lines
        .filter(l -> l.startsWith("ERROR"))
        .count();
    System.out.println("Errors: " + errors);
}
```

The pipeline above reads `server.log` line by line, keeps only lines starting with "ERROR", and counts them. At any moment only one line is in memory — the file could be 5 GB and this code would still work.

The try-with-resources wrap is mandatory: `Files.lines` opens an OS-level file handle, and the returned `Stream` owns it until closed.

<Note><code>Files.lines</code> defaults to UTF-8. Pass a <code>Charset</code> as the second argument to override.</Note>

## Streams are single-pass

Each `Stream` can be consumed exactly once. After a terminal operation (`count`, `collect`, `forEach`, etc.) the stream is finished — calling another terminal op throws `IllegalStateException`.

```java
try (Stream<String> s = Files.lines(p)) {
    long c = s.count();             // ✓ consumes the stream
    long c2 = s.count();            // ✗ IllegalStateException
}
```

If you need two passes, open the file twice — each `Files.lines` returns a fresh stream:

```java
long total;
long errors;
try (Stream<String> s = Files.lines(p)) { total = s.count(); }
try (Stream<String> s = Files.lines(p)) {
    errors = s.filter(l -> l.startsWith("ERROR")).count();
}
```

This re-reads the file, which costs disk time but uses constant memory.

## Common pipelines

The Stream API gives you a vocabulary of building blocks. Here are five patterns you'll reuse constantly.

### 1. Count matching lines

```java
try (Stream<String> s = Files.lines(p)) {
    long n = s.filter(l -> l.contains("WARN")).count();
}
```

### 2. Collect matching lines into a List

```java
import java.util.*;

List<String> errors;
try (Stream<String> s = Files.lines(p)) {
    errors = s.filter(l -> l.startsWith("ERROR")).collect(Collectors.toList());
}
// Or in Java 16+: .toList()
```

### 3. Transform and print

```java
try (Stream<String> s = Files.lines(p)) {
    s.map(String::toUpperCase).forEach(System.out::println);
}
```

### 4. Skip a header, parse the rest

```java
try (Stream<String> s = Files.lines(csv)) {
    s.skip(1)                                       // drop CSV header
     .map(line -> line.split(","))
     .forEach(cols -> System.out.println(cols[0])); // print first column
}
```

### 5. Find the first match (short-circuit)

```java
try (Stream<String> s = Files.lines(p)) {
    Optional<String> first = s.filter(l -> l.contains("FATAL")).findFirst();
    first.ifPresent(System.out::println);
}
```

`findFirst` is lazy: as soon as a match is seen, the stream stops reading the file. On a multi-GB file you might process only the first few KB.

## When NOT to use Files.lines

Streams have overhead. For small files where you'll process every line, the simpler form often reads better:

```java
for (String line : Files.readAllLines(p)) { ... }
```

Reach for `Files.lines` when:

- The file is large (or could be large in the future).
- You want a chain of operations: filter, map, collect, group, sort.
- You only need part of the file (early-exit with `findFirst`, `limit`, `anyMatch`).

For tiny config files or simple line-by-line processing where every line is read, `readAllLines` plus a for-each loop is just as good.

## Close-time errors are real

If the file handle has trouble closing, the close exception is attached as suppressed to whatever exception the body produced. If the body succeeded, the close exception propagates normally. This is the same suppressed-exception machinery you saw in the previous lesson — you don't need to think about it most of the time, but it's there if you need to debug a strange "I/O error after the work succeeded" trace.

## Common mistakes

- **Skipping try-with-resources** — every `Files.lines` call opens a real file handle. Without try-with-resources, the handle leaks until garbage collection (sometimes never).
- **Reusing a stream after a terminal op** — the second op throws. Open a fresh stream for each pass.
- **Reaching for `Files.lines` when `readAllLines` would do** — if you're going to materialise everything anyway, the simpler API is clearer.
- **Forgetting that Files.lines is lazy** — adding `forEach(System.out::println)` is what triggers the actual reading. A pipeline with no terminal operation does nothing.
- **Throwing inside the lambda without thinking about partial state** — if your filter or map throws on line 100 of a 1 million-line file, lines 1–99 are processed and 101+ are not. Make sure that's acceptable for your use case.

## Recap

`Files.lines` is the right tool for any file-processing task that fits the pattern "read a file, do X to each line, produce a result". The Stream API turns those tasks from imperative loops into declarative pipelines.

- **`Files.lines(path)`** returns a `Stream<String>` — lazy, line-by-line, scales to huge files.
- **Always wrap in try-with-resources.** The stream owns a file handle.
- Each stream is single-pass — open the file again to re-read.
- Common patterns: `filter` + `count`, `filter` + `collect`, `map` + `forEach`, `skip` + `map`, `findFirst`.
- Stream operations are lazy until a terminal op (`count`, `collect`, `forEach`, `findFirst`, etc.) triggers consumption.
