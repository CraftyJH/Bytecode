A directory is just a special kind of `Path` that holds other paths. The same `Files` utility class that reads and writes files also creates, lists, and deletes directories. This lesson covers the four operations you'll use most often.

## Creating directories

```java
import java.nio.file.*;

Path single = Paths.get("/tmp/scratch");
Files.createDirectory(single);
// Creates /tmp/scratch — fails if /tmp doesn't exist or scratch already does

Path nested = Paths.get("/tmp/projects/2026/q1");
Files.createDirectories(nested);
// Creates /tmp/projects, /tmp/projects/2026, and /tmp/projects/2026/q1 as needed
```

The plural form (`createDirectories`) is the one you almost always want when programmatically generating output paths — it's a safe no-op if the directory already exists, and it doesn't fail just because an intermediate folder hasn't been made.

`createDirectory` is the strict version. Use it when you specifically want the call to fail if a parent is missing — that can catch typos in your path construction.

<Note>Both methods throw <code>FileAlreadyExistsException</code> if the leaf already exists as a regular file (rather than a directory). This is rare but real — guard against it if you're operating on user-supplied paths.</Note>

## Listing directory contents

`Files.list(dir)` returns a `Stream<Path>` of the directory's immediate children:

```java
import java.util.stream.*;

Path dir = Paths.get("/tmp");
try (Stream<Path> entries = Files.list(dir)) {
    entries
        .filter(Files::isRegularFile)
        .map(p -> p.getFileName().toString())
        .sorted()
        .forEach(System.out::println);
}
```

Two important details:

1. **Always use try-with-resources.** `Files.list` opens an OS-level directory handle. The Stream must be closed to release it. We'll cover try-with-resources properly in the next lesson — for now, copy the pattern.
2. **Order is not guaranteed.** Filesystems can return entries in arbitrary order. If you need alphabetical order, call `.sorted()` in the stream pipeline.

`Files.list` does not descend into subdirectories — it returns only direct children. For recursive traversal, use `Files.walk`:

```java
try (Stream<Path> tree = Files.walk(Paths.get("/tmp/projects"))) {
    tree.filter(Files::isRegularFile).forEach(System.out::println);
}
```

`Files.walk` yields every file and folder under the root, depth-first. By default it follows symbolic links — pass `FileVisitOption.NOFOLLOW_LINKS` if you want to avoid loops.

## Inspecting paths

Common predicates on `Files`:

```java
Files.exists(p);            // true if the file exists
Files.notExists(p);          // !exists, but with stronger semantics under errors
Files.isRegularFile(p);     // true if it's an ordinary file (not a directory)
Files.isDirectory(p);       // true if it's a directory
Files.isReadable(p);        // we have read permission
Files.isWritable(p);        // we have write permission
Files.size(p);              // size in bytes (throws if missing)
```

Note that `exists` and `notExists` are not strict negations: under permission errors or I/O failures, both can return `false`. If you need to distinguish "the file isn't there" from "I can't tell", catch the relevant `IOException` from a stronger call like `Files.size`.

## Deleting

```java
Files.delete(p);          // throws if p doesn't exist
Files.deleteIfExists(p);  // returns boolean — true if deleted, false if missing
```

Both refuse to delete non-empty directories — they throw `DirectoryNotEmptyException`. There is no built-in recursive delete in `Files`, but the pattern is short:

```java
try (Stream<Path> tree = Files.walk(root)) {
    tree.sorted(Comparator.reverseOrder())  // children before parents
        .forEach(p -> {
            try { Files.delete(p); }
            catch (IOException e) { throw new UncheckedIOException(e); }
        });
}
```

Sorting in reverse order ensures children are deleted before their parents — `Files.walk` returns parents before children by default.

## Moving and copying

```java
Files.move(source, target);                              // basic rename/move
Files.move(source, target, StandardCopyOption.REPLACE_EXISTING);
Files.move(source, target, StandardCopyOption.ATOMIC_MOVE);

Files.copy(source, target);
Files.copy(source, target, StandardCopyOption.COPY_ATTRIBUTES);
```

`Files.move` with `ATOMIC_MOVE` is especially useful for the "write to temp file, then rename" pattern that gives you atomic config updates. On most filesystems it boils down to a single rename system call.

## Common mistakes

- **Forgetting try-with-resources around `Files.list`/`Files.walk`** — the returned Stream owns an OS handle that must be released. Leaking handles eventually causes "too many open files" errors.
- **Assuming directory listings are sorted** — they are not. Sort the stream yourself if order matters to you.
- **Calling `Files.delete` on a non-empty directory** — throws `DirectoryNotEmptyException`. Walk and delete depth-first.
- **Mixing `createDirectory` with paths that have missing parents** — fails. Use `createDirectories` unless you specifically want the strict behaviour.

## Recap

Directories use the same `Files` utility class as files — the API is consistent across the whole NIO.2 surface.

- **`Files.createDirectory(p)`** / **`Files.createDirectories(p)`** — single vs. parents-too.
- **`Files.list(dir)`** — `Stream<Path>` of immediate children. Always close it.
- **`Files.walk(dir)`** — `Stream<Path>` for the whole subtree, depth-first.
- **`Files.delete(p)`** / **`Files.deleteIfExists(p)`** — refuses non-empty directories; walk for recursive delete.
- **`Files.move(...)`** / **`Files.copy(...)`** — with optional `StandardCopyOption.ATOMIC_MOVE` and `REPLACE_EXISTING`.
- Predicates like `isDirectory`, `isRegularFile`, and `exists` give you the type and presence checks you need.
