A class declared inside another class is called a **nested class**. Java offers four flavors of them: static nested classes, inner classes, local classes, and anonymous classes. They look similar at first glance but differ in how they capture (or don't capture) the enclosing context. This module walks through each kind and shows when to reach for which.

We start with the simplest variant: the **static nested class**. Despite living inside another class, it does not depend on an instance of that class. Think of it as a top-level class that happens to be namespaced inside its parent for organizational purposes.

## What is a static nested class?

A static nested class is a class declared inside another class with the `static` keyword:

```java
public class Outer {
    static class Nested {
        void hello() {
            System.out.println("hi from Nested");
        }
    }
}
```

The `static` keyword here does **not** mean "only one instance" the way it does on a field. It means "this class is not tied to an instance of `Outer`." You can create as many `Nested` objects as you like, and each one stands alone â€” no hidden reference to an `Outer` lurks inside it.

<Note>The `static` modifier on a nested class has a different meaning than on a field or method. On a field it means "shared by all instances." On a nested class it means "no enclosing instance is captured." It's the same keyword reused for different purposes.</Note>

## Instantiating from outside

From outside `Outer`, you instantiate `Nested` by qualifying the type with the outer class name:

```java
Outer.Nested n = new Outer.Nested();
n.hello();   // hi from Nested
```

The qualifier `Outer.` is only a namespace prefix â€” like writing `java.util.List`. There is no `Outer` object involved. From inside `Outer` itself, you can drop the prefix and write `new Nested()` because the name is already in scope.

## A real-world analogy: nested folders

Picture a folder called `kitchen/` on your computer. Inside it you have a sub-folder called `recipes/`. Files inside `recipes/` are organised under `kitchen/`, but they are independent files â€” they don't require the `kitchen/` folder to be "open" or "running" to exist. A static nested class is the same idea: it lives inside a parent for grouping, but it is otherwise a self-sufficient unit.

## When to use a static nested class

The Java standard library is full of examples. The clearest is `Map.Entry`:

```java
import java.util.Map;
import java.util.HashMap;

Map<String, Integer> ages = new HashMap<>();
ages.put("Alice", 30);
for (Map.Entry<String, Integer> entry : ages.entrySet()) {
    System.out.println(entry.getKey() + " -> " + entry.getValue());
}
```

`Entry` is conceptually tied to `Map` â€” a key/value pair only makes sense in the context of a map â€” but a single `Entry` object doesn't need a `Map` instance to exist. Declaring it as `static` keeps the namespace tidy without imposing a runtime relationship.

Common use cases:

- **Helper data structures** that are an implementation detail of the outer class (linked-list `Node`, tree `Branch`).
- **Builders** for the outer class (`StringBuilder`-style fluent builders are often nested).
- **Simple value/record types** scoped to the outer class.

If your nested type doesn't need to reach into instance state of the outer, use `static`. It's smaller, faster, and easier to reason about.

<Note>If you're on Java 16 or later, a record nested inside another class is implicitly static â€” you cannot remove the staticness. The language designers picked the safer default.</Note>

## A second example: a tree node

```java
public class BinaryTree {
    static class Node {
        int value;
        Node left;
        Node right;

        Node(int value) {
            this.value = value;
        }
    }

    private Node root;

    void add(int value) {
        root = insert(root, value);
    }

    private Node insert(Node node, int value) {
        if (node == null) return new Node(value);
        if (value < node.value) node.left = insert(node.left, value);
        else node.right = insert(node.right, value);
        return node;
    }
}
```

`Node` is a static nested class because a `Node` object on its own â€” without a tree â€” is meaningful. Code outside `BinaryTree` rarely needs to construct a `Node` directly, so nesting it inside the tree class hides the implementation detail nicely.

## Accessing the outer class

A static nested class can read **static** members of the outer class freely (including private ones), but cannot reach instance members without an explicit reference:

```java
public class Outer {
    private static int classConstant = 42;
    private int instanceField = 7;

    static class Nested {
        void demo() {
            System.out.println(classConstant);     // OK â€” static
            // System.out.println(instanceField); // Compile error â€” needs an Outer object
        }
    }
}
```

This is the same rule that applies inside any static method: no implicit `this`, so no instance state.

## Static nested vs top-level: why bother nesting?

A reasonable question: if a static nested class behaves like a top-level class, why nest it at all? Three reasons:

1. **Encapsulation of implementation detail.** A `Node` class is meaningless outside its owning data structure â€” nesting it signals that.
2. **Tidier namespace.** `Map.Entry` is clearly part of `Map`. As a separate top-level class it would just be `Entry`, which is too vague.
3. **Access to private static members.** A static nested class can read the outer class's private static fields and methods, which is sometimes useful for tightly coupled helpers.

If none of these apply, prefer a top-level class â€” fewer levels of nesting are easier to read.

## Common mistakes

- **Forgetting `static` and getting an inner class instead.** Without `static`, every instance of the nested class secretly carries a reference to the enclosing instance. That can leak memory and prevent the outer object from being garbage collected. If your class doesn't need the enclosing instance, mark it `static`.
- **Trying to access instance fields of the outer class.** A static nested class has no implicit `Outer.this`. To read `outer.instanceField` you must accept an `Outer` reference as a parameter. The compiler error message is "non-static variable cannot be referenced from a static context."
- **Writing `new Outer().new Nested()` for a static nested class.** That syntax is reserved for non-static inner classes. For static nested classes, write `new Outer.Nested()` â€” no outer instance required.
- **Confusing the `static` keyword's two meanings.** On a field, `static` means "shared by all instances." On a nested class, it means "no enclosing instance." Different concepts, same word.
- **Over-nesting.** If a class is reused outside its parent, promote it to a top-level class. Deep nesting just to "tidy up" hurts readability when the inner type is also imported elsewhere.

## Recap

Static nested classes are the friendliest member of the nested-class family. They give you namespace organisation and access to the outer class's private statics without the runtime overhead of capturing an enclosing instance.

- A **static nested class** is declared with `static` inside another class
- It does **not** depend on an instance of the outer class
- Instantiate from outside as `new Outer.Nested()`; from inside `Outer`, just `new Nested()`
- Useful for `Entry`-style helpers, tree/list node types, and tightly coupled builders
- It can access static members (including private ones) of the outer class, but not instance members without an explicit reference
- When in doubt, prefer `static` â€” it's the simpler default
