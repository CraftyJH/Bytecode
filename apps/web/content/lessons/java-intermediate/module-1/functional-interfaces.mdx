Every time you write a lambda in Java, you are implementing a **functional interface** — an interface with exactly one abstract method (SAM: Single Abstract Method). The `java.util.function` package ships four archetypal ones that cover the vast majority of use cases. Understanding them unlocks lambdas, streams, and most of modern Java's API surface.

## What makes an interface "functional"

The `@FunctionalInterface` annotation is a compile-time guard that enforces the single-abstract-method rule:

```java
@FunctionalInterface
interface Transformer {
    String transform(String input);  // exactly one abstract method
    // default and static methods are allowed alongside it
}
```

Any interface with exactly one abstract method is a functional interface, annotated or not. The annotation just makes the compiler catch accidental violations:

```java
Transformer shout = input -> input.toUpperCase();
System.out.println(shout.transform("hello"));  // HELLO
```

The lambda `input -> input.toUpperCase()` is shorthand for an anonymous class that implements `Transformer`. The compiler infers the single abstract method and wires the lambda body to it.

## The four archetypes in `java.util.function`

### `Predicate<T>` — test a condition

```java
Predicate<String> startsWithJ = s -> s.startsWith("J");
System.out.println(startsWithJ.test("Java"));   // true
System.out.println(startsWithJ.test("Python")); // false
```

`Predicate<T>` has abstract method `boolean test(T t)`. It answers a yes/no question about its argument. Common uses: filtering collections, validating inputs, conditional logic.

### `Function<T, R>` — transform a value

```java
Function<String, Integer> wordCount = s -> s.split(" ").length;
System.out.println(wordCount.apply("one two three"));  // 3
```

`Function<T, R>` has abstract method `R apply(T t)`. It maps one type to another. Common uses: parsing, formatting, extracting fields from objects.

### `Consumer<T>` — act on a value, return nothing

```java
Consumer<String> logger = msg -> System.out.println("[LOG] " + msg);
logger.accept("Server started");   // [LOG] Server started
logger.accept("Request received"); // [LOG] Request received
```

`Consumer<T>` has abstract method `void accept(T t)`. It is side-effectful by definition — it does something with its argument but produces no result. Common uses: printing, writing to a database, updating state.

### `Supplier<T>` — produce a value, take nothing

```java
Supplier<Double> randomPrice = () -> Math.random() * 100;
System.out.println(randomPrice.get());  // e.g., 73.41...
```

`Supplier<T>` has abstract method `T get()`. It produces a value on demand without input. Common uses: lazy initialization, test data factories, default value providers.

<Note>Memory trick: **P**redicate **t**ests, **F**unction **t**ransforms, **C**onsumer **c**onsumes, **S**upplier **s**upplies. Each word pair starts with the same letter. The method names (`test`, `apply`, `accept`, `get`) are worth memorizing — the compiler won't help you if you call the wrong one.</Note>

## Composing functional interfaces

The `java.util.function` interfaces include `default` methods for composition:

```java
Predicate<String> isLong  = s -> s.length() > 5;
Predicate<String> hasSpace = s -> s.contains(" ");

// Compose with and / or / negate
Predicate<String> longWithSpace = isLong.and(hasSpace);
System.out.println(longWithSpace.test("Hello World")); // true
System.out.println(longWithSpace.test("Hi"));          // false

Function<String, String> trim  = String::trim;
Function<String, String> upper = String::toUpperCase;
Function<String, String> clean = trim.andThen(upper);  // chain
System.out.println(clean.apply("  hello  "));          // HELLO
```

`Predicate.and(other)` returns a new predicate that is true only when both are true. `Function.andThen(after)` pipes the output of the first into the input of the second. These compositions produce new functional interface instances — you can build complex logic from simple pieces.

## Storing and passing lambdas

Because functional interfaces are just interfaces, they can be stored in variables, passed as method arguments, and returned from methods:

```java
static void processAll(String[] items, Consumer<String> action) {
    for (String item : items) {
        action.accept(item);
    }
}

public static void main(String[] args) {
    String[] names = {"Alice", "Bob", "Charlie"};
    processAll(names, name -> System.out.println("Hello, " + name));
}
// Hello, Alice
// Hello, Bob
// Hello, Charlie
```

`processAll` doesn't care how `action` is implemented — it just calls `accept`. Tomorrow you could pass a different lambda (e.g., one that logs to a file) without touching `processAll` at all.

## Common mistakes

- **Calling the wrong method name** — `Predicate` uses `.test()`, `Function` uses `.apply()`, `Consumer` uses `.accept()`, `Supplier` uses `.get()`. These are four distinct names. Calling `.test()` on a `Consumer` is a compile error, not a runtime one, so you'll find out quickly — but it's disorienting at first.
- **Confusing `Function<T,R>` types** — `Function<String, Integer>` takes a `String` and returns an `Integer`. Getting the type parameters backwards (`Function<Integer, String>`) compiles only if the lambda's body accidentally fits the swapped types. Pay attention to the order: input type first, output type second.
- **Using `Function<String, Void>`** instead of `Consumer<String>` — `Void` is a class, not `void`. Using it forces you to `return null` in the lambda body. Always prefer `Consumer` for side-effectful, no-return operations.
- **Forgetting `@FunctionalInterface` makes the compiler your ally** — if you design your own functional interface, add the annotation. It costs nothing and prevents a future teammate from accidentally adding a second abstract method and breaking all existing lambdas.
- **Thinking lambdas are magical** — a lambda is just a concise anonymous class. `s -> s.toUpperCase()` compiles to something that implements the target functional interface's single abstract method. All the rules of interfaces still apply.

## Recap

Functional interfaces are the bridge between Java's object-oriented type system and the functional-style programming enabled by lambdas. The four standard archetypes cover almost every scenario:

- **`Predicate<T>`** — `boolean test(T t)` — test a condition
- **`Function<T,R>`** — `R apply(T t)` — transform a value
- **`Consumer<T>`** — `void accept(T t)` — act on a value, return nothing
- **`Supplier<T>`** — `T get()` — produce a value, take nothing
- Any interface with exactly one abstract method is a functional interface; `@FunctionalInterface` enforces this at compile time
- Lambdas implement functional interfaces — the compiler matches the lambda signature to the single abstract method
- Composition methods (`and`, `or`, `negate`, `andThen`, `compose`) build complex logic from simple pieces
