Java 8 introduced two game-changing additions to interfaces: **`default` methods** (concrete methods with a body that implementing classes inherit) and **`static` methods** (utility functions tied to the interface type itself). Together they let interfaces evolve without breaking existing code and serve as home for closely related helper logic.

## The problem `default` methods solve

Imagine you ship an interface `Collection` in version 1.0. A year later you want to add a `forEach` method. In old Java, adding any method to the interface would break every class that ever implemented it — suddenly they all fail to compile. With `default` methods, you add a body right in the interface:

```java
interface Collection<E> {
    void add(E element);
    int size();

    // Added later — existing classes don't need to change
    default boolean isEmpty() {
        return size() == 0;
    }
}
```

Any existing class that implements `Collection` now automatically inherits `isEmpty()`. Classes that prefer their own implementation can override it.

## Declaring a `default` method

The `default` keyword precedes the return type, and the method has a full body:

```java
interface Greeter {
    String getName();  // abstract — must be implemented

    default String greet() {
        return "Hello, " + getName() + "!";
    }
}

class User implements Greeter {
    private String name;
    User(String name) { this.name = name; }

    public String getName() { return name; }
    // greet() is inherited — no override needed
}

// Usage:
User u = new User("Alice");
System.out.println(u.greet());  // Hello, Alice!
```

`greet()` calls `getName()`, which is abstract. This works because at runtime the actual class always has a `getName()` implementation — the JVM dispatches to it dynamically, exactly as with abstract methods.

<Note>`default` methods in interfaces can call the interface's own abstract methods (as the example above shows) but they cannot access instance fields — interfaces have none. If you need shared state, use an abstract class.</Note>

## Overriding a `default` method

A class can override a `default` method just like any other:

```java
class FormalUser implements Greeter {
    private String name;
    FormalUser(String name) { this.name = name; }

    public String getName() { return name; }

    @Override
    public String greet() {
        return "Good day, " + getName() + ". How do you do?";
    }
}
```

The class's own version wins. If a class inherits conflicting `default` implementations from two interfaces, it *must* override the method to resolve the ambiguity — the compiler will insist.

## Static interface methods

`static` methods in interfaces belong to the interface type, not to any instance. They are called via the interface name, not through an object:

```java
interface MathUtils {
    static int clamp(int value, int min, int max) {
        return Math.max(min, Math.min(max, value));
    }
}

// Call it through the interface name:
int x = MathUtils.clamp(150, 0, 100);  // 100
```

They are not inherited by implementing classes. You cannot call `MathUtils.clamp(...)` on an instance of a class that implements `MathUtils` — only through `MathUtils.clamp(...)` directly.

## Factory methods: a classic static interface method pattern

Static methods are ideal for factory patterns — creating instances of implementations without exposing the class name:

```java
interface Validator {
    boolean validate(String input);

    static Validator nonEmpty() {
        return input -> !input.isEmpty();
    }

    static Validator maxLength(int max) {
        return input -> input.length() <= max;
    }
}

Validator v1 = Validator.nonEmpty();
Validator v2 = Validator.maxLength(10);
System.out.println(v1.validate("hello"));   // true
System.out.println(v2.validate("hello world"));  // false
```

The caller never needs to know which class backs the implementation — the factory method is the only contact point. The Java standard library uses this pattern: `Comparator.naturalOrder()`, `List.of(...)`, `Predicate.not(...)`.

## When to use each

| | `default` method | `static` method |
|---|---|---|
| Called on | An instance (or via interface ref) | The interface type only |
| Inherited? | Yes — by implementing classes | No |
| Best for | Shared convenience behaviour, evolving APIs | Factory methods, utility helpers |
| Can call abstract methods? | Yes | No (no `this` reference) |

## Common mistakes

- **Calling a static interface method through an instance** — `myFormatter.format(...)` where `format` is `static` is a compile error. Use `Formatter.format(...)`. The compiler will not silently allow the instance-based call the way it does for static class methods.
- **Forgetting that `default` methods can be overridden** — a `default` is a fallback, not a final. If your class needs different behaviour, override it freely. The `@Override` annotation works here too and catches typos in the signature.
- **Assuming `default` methods can access fields** — interfaces have no instance fields. A `default` method can only use its parameters, call other interface methods, or call static methods. For stateful helpers, use an abstract class.
- **Diamond problem with conflicting defaults** — if two interfaces both declare `default void doIt()`, a class implementing both *must* override `doIt()`. The compiler enforces this; the override can explicitly delegate to one interface's default with `InterfaceName.super.doIt()`.
- **Overriding a `default` method with weaker semantics** — technically legal, but dangerous. If the `default` documents a contract (e.g., "always wraps output in brackets"), silently breaking that contract in an override causes confusing bugs for anyone using the interface type.

## Recap

Default and static interface methods dramatically increased the power of interfaces in Java 8. They let library designers evolve APIs over time and keep utility logic co-located with the interface that motivates it.

- **`default` methods** provide concrete inherited behaviour — ideal for adding methods to existing interfaces without breaking implementors
- **`static` methods** provide type-level utilities — called as `Interface.method()`, not inherited
- `default` methods can call abstract methods (dynamically dispatched at runtime)
- Conflicting `default` methods from two interfaces must be resolved by an explicit override
- `InterfaceName.super.method()` lets you delegate to a specific interface's default from within the override
