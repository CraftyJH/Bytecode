Between a plain interface (pure contract, zero implementation) and a concrete class (full implementation) lies a useful middle ground: the **abstract class**. It can declare obligations that subclasses must fulfil — just like an interface — but it can also provide concrete methods and fields shared by all subclasses.

The real-world analogy is a template or a partially-filled form. An "Employee" template might already specify how payslips are formatted and how vacation days accumulate, while leaving "calculateBonus()" blank for each department to fill in their own formula.

## Declaring an abstract class

Add the `abstract` modifier to the `class` keyword:

```java
abstract class Shape {
    String color;  // concrete field — all shapes have a color

    Shape(String color) {
        this.color = color;
    }

    // Concrete method — shared by all subclasses
    void printColor() {
        System.out.println("Color: " + color);
    }

    // Abstract method — each subclass must provide its own formula
    abstract double area();
}
```

Two rules follow immediately:
1. You cannot write `new Shape()` — the compiler forbids instantiating abstract classes.
2. Any class that `extends Shape` must implement `area()` — or itself be declared `abstract`.

## Extending an abstract class

```java
class Rectangle extends Shape {
    double width, height;

    Rectangle(String color, double width, double height) {
        super(color);   // call the abstract class's constructor
        this.width = width;
        this.height = height;
    }

    @Override
    public double area() {
        return width * height;
    }
}

class Triangle extends Shape {
    double base, height;

    Triangle(String color, double base, double height) {
        super(color);
        this.base = base;
        this.height = height;
    }

    @Override
    public double area() {
        return 0.5 * base * height;
    }
}
```

Both subclasses inherit `color`, the constructor logic, and `printColor()` for free. They only need to supply the unique piece: the area formula.

<Note>Abstract classes *can* have constructors, even though you can't call them directly via `new`. Subclass constructors call them with `super(...)`, which must be the first line. If an abstract class declares a constructor with parameters, every subclass constructor must explicitly call it — Java will not insert a no-arg call automatically.</Note>

## Polymorphism with abstract classes

Because all subclasses share the `Shape` type, you can write generic code that works for any shape:

```java
public class Main {
    static void printInfo(Shape s) {
        s.printColor();                    // concrete — inherited
        System.out.println("Area: " + s.area());  // dispatches to subclass
    }

    public static void main(String[] args) {
        Shape[] shapes = {
            new Rectangle("blue", 3, 5),
            new Triangle("red", 6, 4)
        };
        for (Shape s : shapes) {
            printInfo(s);
        }
    }
}
// Output:
// Color: blue
// Area: 15.0
// Color: red
// Area: 12.0
```

`printInfo` never knows whether it received a `Rectangle` or a `Triangle`. It only knows the `Shape` API. This is the **Template Method pattern**: the abstract class defines the algorithm skeleton (`printColor` then `area`), and subclasses fill in the variable step.

## When abstract classes shine

Abstract classes are best when:

- You want to share **fields and state** across subclasses (interfaces cannot hold instance fields).
- Subclasses share significant **common behaviour** that shouldn't be duplicated.
- You need to **control construction** — abstract classes can enforce initialization through constructors.

```java
abstract class DatabaseConnection {
    private final String url;

    // Every subclass must supply a URL at construction time
    DatabaseConnection(String url) {
        this.url = url;
        connect();  // run shared startup logic
    }

    abstract void connect();
    abstract void disconnect();

    String getUrl() { return url; }
}
```

No subclass can skip the `url` — it's baked into the constructor chain.

## Common mistakes

- **Trying to instantiate an abstract class** — `new Animal("Rex")` where `Animal` is abstract is a compile error. Always instantiate a concrete subclass and assign it to the abstract type variable if needed.
- **Forgetting `super(...)` in the subclass constructor** — when the abstract class has a parameterized constructor and no no-arg constructor, the compiler requires an explicit `super(...)` call as the very first statement. Omitting it gives "constructor Animal in class Animal cannot be applied to given types."
- **Leaving an abstract method unimplemented in a concrete subclass** — a subclass that doesn't override every abstract method must itself be declared `abstract`. The compiler will point out which methods are missing.
- **Confusing abstract methods with concrete methods** — abstract methods end with a semicolon: `abstract void speak();`. A concrete method has braces: `void speak() { return "Hi"; }`. Accidentally adding a body to an abstract method (or leaving off the body from a concrete one) produces a compile error.
- **Overusing abstract classes when an interface suffices** — if you don't need shared fields or implemented methods, prefer an interface. Using an abstract class consumes the single inheritance slot every Java class has, closing off future options.

## Recap

Abstract classes let you express partial templates: concrete shared behaviour combined with abstract obligations that subclasses must fulfil. They are fundamental to the Template Method pattern and any design where a family of related classes share a common algorithmic skeleton.

- Declared with `abstract class ClassName`
- Cannot be instantiated with `new` — only concrete subclasses can
- May contain any mix of concrete fields, concrete methods, and abstract methods
- Abstract methods — `abstract returnType methodName();` — have no body; subclasses must override them
- `super(...)` in the subclass constructor calls the abstract class's constructor
- Polymorphism through the abstract type works exactly as with interfaces
