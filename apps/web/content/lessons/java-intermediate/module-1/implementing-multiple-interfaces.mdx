A real-world object often belongs to more than one category simultaneously. A smartphone is both a camera and a phone and a music player. A Swiss Army knife is a blade, a screwdriver, and a bottle opener. Java models this with **multiple interface implementation**: a single class can sign as many contracts as it needs to.

This is intentional and powerful. Java deliberately restricts a class to one superclass (to avoid the "diamond problem" of competing inherited implementations) but imposes no limit on the number of interfaces. Interfaces carry no implementation to conflict over — just pure method signatures.

## The syntax

List all the interfaces after a single `implements` keyword, separated by commas:

```java
class Smartphone implements Camera, Phone, MusicPlayer {
    // must implement every method from all three interfaces
}
```

The compiler checks each interface independently. If any abstract method from any interface is missing, the build fails.

## A concrete example

Suppose you're building a document management system. Documents need to be printed and exported:

```java
interface Printable {
    void print();
}

interface Exportable {
    String export();
}

class Invoice implements Printable, Exportable {
    private String client;
    private double amount;

    Invoice(String client, double amount) {
        this.client = client;
        this.amount = amount;
    }

    @Override
    public void print() {
        System.out.println("Invoice for " + client + ": $" + amount);
    }

    @Override
    public String export() {
        return "CSV:" + client + ":" + amount;
    }
}
```

`Invoice` satisfies both contracts. You can now pass an `Invoice` to any method that accepts a `Printable`, any method that accepts an `Exportable`, or both — without duplication.

## Treating the object as different types

Because `Invoice` implements both interfaces, it is assignment-compatible with either interface type:

```java
Invoice inv = new Invoice("ACME", 500.0);

Printable p = inv;    // valid — Invoice IS-A Printable
Exportable e = inv;   // valid — Invoice IS-A Exportable

p.print();            // Invoice for ACME: $500.0
System.out.println(e.export());   // CSV:ACME:500.0
```

Both variables point to the same object in memory. There's no copying — just different "views" of the same `Invoice` through different interface lenses.

<Note>This IS-A relationship is bidirectional: `instanceof` will return `true` for all implemented interfaces. `inv instanceof Printable` is `true`, and so is `inv instanceof Exportable`.</Note>

## Interface segregation: why split the contracts?

You might wonder: why two interfaces instead of one `Document` interface with both methods? The answer is the **Interface Segregation Principle** — clients should not depend on methods they don't use.

Imagine a legacy printer that can print but cannot export. With a single merged interface, the printer's class would be forced to implement `export()` with a stub or throw an exception — messy. With separate interfaces, the printer implements only `Printable`, and the export pathway simply doesn't apply to it:

```java
class LegacyPrinter implements Printable {
    @Override
    public void print() {
        System.out.println("Printing via legacy driver");
    }
    // No export() needed — LegacyPrinter isn't Exportable
}
```

Smaller, focused interfaces compose more flexibly than large, monolithic ones.

## What happens when two interfaces share a method name?

If both `Saveable` and `Loadable` declare `void reset()`, a class implementing both only needs — and can only have — one `reset()` method:

```java
interface Saveable  { void save();  void reset(); }
interface Loadable  { void load();  void reset(); }

class DataStore implements Saveable, Loadable {
    public void save()  { System.out.println("Saved"); }
    public void load()  { System.out.println("Loaded"); }
    public void reset() { System.out.println("Reset"); }  // satisfies both
}
```

A single implementation satisfies both contracts simultaneously because the signature is identical. A true conflict only arises when the same name appears with *different return types* in two interfaces — that is a compile error Java cannot resolve.

## Common mistakes

- **Repeating the `implements` keyword** — `class Foo implements A implements B` is a syntax error. Use a comma-separated list: `class Foo implements A, B`.
- **Forgetting one method from one interface** — if a class implements three interfaces and each has two methods, you owe six implementations. Read compiler errors carefully — they name the exact interface and method that's missing.
- **Thinking each interface gets its own method copy** — implementing `Runnable` and `Callable` doesn't give you two run-like methods. One method body satisfies whichever interface declared it (or both, if signatures match).
- **Confusing interface type with class type** — `Printable p = new Printable()` is illegal. Interfaces cannot be instantiated. You instantiate a class, then assign it to an interface variable.
- **Assuming interface order matters** — `class Foo implements A, B` and `class Foo implements B, A` are equivalent. The order does not affect which methods must be implemented or the object's runtime behaviour.

## Recap

Multiple interface implementation is one of Java's most powerful compositional tools. It lets objects wear many hats without the fragility of multiple inheritance of implementation.

- A class may implement as many interfaces as needed with `class Foo implements A, B, C`
- Every abstract method from every listed interface must be implemented
- An object can be stored in a variable typed as any interface it implements
- Separate, focused interfaces are more reusable than large combined ones (Interface Segregation Principle)
- Identical method signatures across multiple interfaces collapse into one implementation; different return types are a compile error
