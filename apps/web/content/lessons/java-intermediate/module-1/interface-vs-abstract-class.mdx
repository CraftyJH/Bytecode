You now know both tools. The question every Java developer faces is: **which one should I use?** The answer is not arbitrary — there are clear principles that guide the choice, and in practice the two constructs are often combined.

## The core difference

| | Interface | Abstract Class |
|---|---|---|
| Instance fields | No (only `static final` constants) | Yes |
| Constructor | No | Yes |
| Concrete methods | Only `default`/`static` (Java 8+) | Yes, any |
| Multiple "inheritance" | A class can implement many | A class extends exactly one |
| Keyword | `implements` | `extends` |
| Best for | *Capability* contracts across unrelated types | *Shared implementation* within a family |

## When to choose an interface

Prefer an interface when:

- The contract cuts across unrelated types. A `Comparable` duck and a `Comparable` integer have nothing else in common — the interface is the only sensible choice.
- You want a class to be of multiple types simultaneously without forcing it into a hierarchy.
- You are defining an API others will implement and you don't want to lock callers into your class hierarchy.

```java
interface Serializable {
    byte[] serialize();
}

// Totally unrelated classes can all become Serializable:
class UserProfile   implements Serializable { ... }
class GameSaveState implements Serializable { ... }
class InvoicePDF    implements Serializable { ... }
```

None of these classes share a useful superclass. The interface binds them to a common capability without forcing a hierarchy.

## When to choose an abstract class

Prefer an abstract class when:

- Subclasses share **concrete implementation** — fields, methods, or constructor setup that would be copied otherwise.
- You need to **enforce initialization** — constructors with required parameters guarantee that every subclass provides the data the template needs.
- The subclasses form a true **IS-A family** with a meaningful shared identity (not just a shared capability).

```java
abstract class Report {
    protected String title;
    protected String author;

    Report(String title, String author) {
        this.title = title;
        this.author = author;
    }

    // Shared implementation
    void printHeader() {
        System.out.println("=== " + title + " by " + author + " ===");
    }

    // Abstract obligation
    abstract void printBody();

    // Template method: fixed skeleton
    final void print() {
        printHeader();
        printBody();
        System.out.println("=== END ===");
    }
}
```

`printHeader()` would be duplicated in every report type without the abstract class. The `final print()` method locks the order of operations — subclasses can only customise `printBody()`.

<Note>The `final` modifier on a concrete method in an abstract class prevents subclasses from overriding it. This is a key technique in the Template Method pattern: you permit variation only in the designated abstract "hooks."</Note>

## Combining both: the best-of-both-worlds pattern

The most powerful pattern is to declare a public interface (the contract the whole world sees) and then provide an abstract class that implements common behaviour:

```java
// Public contract — anyone can implement it
interface Shape {
    double area();
    double perimeter();
}

// Abstract helper — provides shared logic for shape families
abstract class AbstractShape implements Shape {
    protected String color;

    AbstractShape(String color) { this.color = color; }

    // Concrete shared method
    void printInfo() {
        System.out.println(color + " shape: area=" + area() + ", perim=" + perimeter());
    }
}

// Concrete leaf class — only implements the geometry
class Circle extends AbstractShape {
    private double radius;

    Circle(String color, double radius) {
        super(color);
        this.radius = radius;
    }

    public double area()      { return Math.PI * radius * radius; }
    public double perimeter() { return 2 * Math.PI * radius; }
}
```

`Circle` is a `Shape` (via the interface) and an `AbstractShape` (via the abstract class). External code programs to `Shape`; internal subclasses extend `AbstractShape`. This pattern is used extensively in the Java Collections Framework (`List` → `AbstractList` → `ArrayList`).

## Common mistakes

- **Using an abstract class when an interface would suffice** — if you have no shared implementation and no shared state, an abstract class wastes the single inheritance slot. Prefer the interface; leave the superclass slot open for the future.
- **Using an interface when you need shared state** — interfaces cannot hold instance fields. If you find yourself writing the same field initialization in five `default` methods, you need an abstract class.
- **Forgetting that `extends` comes before `implements`** — `class Duck extends Bird implements Flyable` is correct; `class Duck implements Flyable extends Bird` is a syntax error. Extend first, then implement.
- **Not calling `super(...)` when the abstract class has a mandatory constructor** — if an abstract class has no no-arg constructor, every concrete subclass constructor must explicitly call `super(args)` as its very first line.
- **Overlooking the interface+abstract class combination** — many developers treat this as an advanced trick, but it is the standard pattern in Java's own libraries. Knowing it lets you design APIs that are both flexible (interface) and DRY (abstract class).

## Recap

The choice between interface and abstract class is one of the most important design decisions in Java. Getting it right keeps code flexible, testable, and maintainable.

- **Interface**: maximum flexibility, multiple adoption, no state — ideal for cross-cutting capability contracts
- **Abstract class**: shared implementation and state, enforced initialization — ideal for families of related types
- **Both together**: declare the public contract as an interface; provide shared implementation in an abstract class that implements it
- A class can `extend` one abstract class and `implement` many interfaces simultaneously
- `final` methods in abstract classes lock algorithmic skeletons while leaving designated hooks open to subclasses
