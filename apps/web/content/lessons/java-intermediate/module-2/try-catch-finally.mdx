You know that exceptions propagate up the call stack until something catches them. Now it's time to master the full syntax: multiple catch blocks, the order they must appear in, the `finally` block for cleanup that must always run, and a preview of `try-with-resources` that makes resource management nearly automatic.

Think of a `try` block as a guarded zone. You put risky code inside it. The catch blocks are labelled exits — each one handles a specific kind of trouble. The `finally` block is the hallway outside all exits: no matter which door you leave through, you pass through the hallway.

## Multiple catch blocks

A single `try` block can have several `catch` clauses, each catching a different exception type:

```java
static void parse(String input) {
    try {
        int n = Integer.parseInt(input);
        int result = 100 / n;
        System.out.println("Result: " + result);
    } catch (NumberFormatException e) {
        System.out.println("Not a valid integer: " + input);
    } catch (ArithmeticException e) {
        System.out.println("Cannot divide by zero");
    }
}
```

When an exception is thrown, Java tests each `catch` clause **in order** and runs the first one that matches. Only one catch block ever executes per exception.

## Catch ordering: specific before general

`NumberFormatException` extends `RuntimeException` which extends `Exception`. If you place a `catch (Exception e)` before `catch (NumberFormatException e)`, the specific catch is **unreachable** — Java will refuse to compile it:

```java
// COMPILE ERROR — wrong order
try {
    Integer.parseInt(s);
} catch (Exception e) {           // too broad — swallows everything
    System.out.println("error");
} catch (NumberFormatException e) { // unreachable!
    System.out.println("bad number");
}
```

Always write the **most specific** type first and work toward the most general:

```java
// Correct order
try {
    Integer.parseInt(s);
} catch (NumberFormatException e) {   // specific first
    System.out.println("bad number");
} catch (RuntimeException e) {        // broader second
    System.out.println("runtime error");
}
```

<Note>If you ever see a "exception has already been caught" compile error, the first thing to check is whether a catch clause higher up is a supertype of the one below it.</Note>

## Multi-catch: catching multiple types in one clause

Java 7 introduced the pipe (`|`) syntax to handle several unrelated exceptions with the same code:

```java
try {
    doRiskyOperation();
} catch (IOException | SQLException e) {
    System.out.println("Data layer failed: " + e.getMessage());
    log(e);
}
```

This is cleaner than duplicating the handler body. The variable `e` is implicitly `final` inside a multi-catch clause — you cannot reassign it.

Use multi-catch when two exceptions genuinely deserve identical handling. If they need different responses, keep them separate.

## The `finally` block

`finally` is optional and appears after all catch clauses. Its body **always executes**, whether the try succeeded, a catch ran, or an exception is still propagating:

```java
static void readFile(String path) {
    FileReader reader = null;
    try {
        reader = new FileReader(path);
        // ... read data ...
    } catch (IOException e) {
        System.out.println("Could not read: " + e.getMessage());
    } finally {
        if (reader != null) {
            try { reader.close(); } catch (IOException ignored) {}
        }
        System.out.println("Cleanup done");
    }
}
```

"Cleanup done" prints regardless of whether the file was read successfully. This pattern is essential for releasing resources like file handles, database connections, and network sockets.

<Note>`finally` does NOT run if the JVM is forcibly killed (e.g., `System.exit()`, a hard power-off, or a JVM crash). For normal program flow, including re-thrown exceptions, it always runs.</Note>

## What happens when `finally` itself throws?

If a `finally` block throws an exception, it **replaces** any exception currently in flight. The original exception is lost:

```java
try {
    throw new RuntimeException("original");
} finally {
    throw new RuntimeException("from finally"); // original is suppressed
}
```

This is a common source of subtle bugs. In Java 7+, try-with-resources handles this properly by using "suppressed exceptions" — the primary exception is kept and the secondary one is attached to it. Avoid throwing from `finally` in hand-written code.

## `return` inside `finally`

Similarly, a `return` statement inside `finally` swallows any in-flight exception and also overrides a `return` value from the `try` or `catch` block:

```java
static int dangerous() {
    try {
        throw new RuntimeException("problem");
    } finally {
        return 42;  // exception is silently discarded!
    }
}
```

This is considered very bad style. Never `return` from `finally` unless you have a compelling, well-documented reason.

## try-with-resources (preview)

Managing resources in `finally` is verbose and error-prone. Java 7 introduced **try-with-resources** for any object implementing `AutoCloseable`:

```java
try (BufferedReader reader = new BufferedReader(new FileReader(path))) {
    String line = reader.readLine();
    System.out.println(line);
} catch (IOException e) {
    System.out.println("Failed: " + e.getMessage());
}
// reader.close() is called automatically here
```

The parentheses after `try` hold one or more resource declarations. Each resource is closed in reverse order at the end of the block, even if an exception occurs. You'll see this pattern in depth in Lesson 6.

## A realistic multi-catch example

Here is a method that validates and converts user input, showing multiple concerns handled cleanly:

```java
static int parsePositive(String input) {
    try {
        int value = Integer.parseInt(input.trim());
        if (value <= 0) throw new IllegalArgumentException("Must be positive, got: " + value);
        return value;
    } catch (NumberFormatException e) {
        throw new IllegalArgumentException("Not a number: '" + input + "'", e);
    }
}
```

Notice: the `NumberFormatException` is *wrapped* in an `IllegalArgumentException` with a friendlier message. The original exception is passed as the second constructor argument so its stack trace is preserved — callers can still inspect the root cause.

## Common mistakes

- **Catching a supertype before a subtype** — the compiler catches this for you with a compile error, but it's worth understanding *why*: if `Exception` is caught first, the subtype catch can never be reached because every subtype IS-A `Exception`. Fix by moving the specific type up.
- **Empty `finally` cleanup when null** — `finally` runs even if the `try` block threw before the resource was assigned. Always null-check before calling `close()` in a manual `finally`, or use try-with-resources instead.
- **Swallowing exceptions with `finally` return** — a `return` in `finally` silently discards any exception in flight, turning a crash into a mystery. Reserve `return` for the `try` and `catch` blocks.
- **Multi-catch with related types** — you cannot write `catch (IOException | FileNotFoundException e)` because `FileNotFoundException` is a subtype of `IOException`. The compiler will reject it. Use just the supertype if you want to catch both.

## Recap

The `try-catch-finally` construct gives you fine-grained control over both the handling and the cleanup of exceptional conditions.

- Multiple `catch` blocks allow different responses to different failures — list them **specific first, general last**
- The `|` multi-catch syntax avoids duplication when two types need identical handling
- `finally` is your guaranteed cleanup zone: it runs after the `try` (and any `catch`) completes, no matter what
- Throwing from `finally` or returning from `finally` are both legal but very dangerous — avoid them
- `try-with-resources` replaces manual `finally`-based cleanup for anything that implements `AutoCloseable`
