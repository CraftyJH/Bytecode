Every program encounters unexpected situations: a file is missing, a user types letters where a number was expected, the network drops mid-request. Java's **exception system** is a structured way to signal and handle these situations without tangling error-checking code with normal logic.

Before exceptions existed (think C), functions had to return special error codes. Every caller had to check the return value and pass it up the chain. It was easy to forget one check, and the real logic was buried in a forest of `if (err != 0)` guards. Exceptions separate the happy path from the error path cleanly.

## The exception hierarchy

Java models all exceptional events as objects. The root of the hierarchy is **`Throwable`**, which has two direct children:


**`Error`** signals that the JVM itself is failing. `OutOfMemoryError` means no heap space remains; `StackOverflowError` means infinite (or very deep) recursion has exhausted the call stack. You almost never catch these ŌĆö by the time they fire, there is often nothing you can safely do.

**`Exception`** is what your code works with. It splits further into *checked* and *unchecked* exceptions, which you'll explore fully in Lesson 3.

<Note>You can throw any `Throwable` subclass, but catching `Error` or bare `Throwable` is almost always a mistake. Stick to `Exception` and its subtypes unless you have a very specific reason.</Note>

## How Java signals an exception

When something goes wrong, the JVM (or your code) constructs a `Throwable` object and **throws** it. Normal execution stops at that exact point. Java looks for the nearest matching `catch` block, unwinding the call stack frame by frame until it finds one. If none is found, the thread terminates and Java prints a **stack trace**.

```java
public class Main {
    public static void main(String[] args) {
        divide(10, 0);          // triggers ArithmeticException
        System.out.println("This never runs");
    }

    static int divide(int a, int b) {
        return a / b;           // JVM throws here
    }
}
```

Running this produces:

```
Exception in thread "main" java.lang.ArithmeticException: / by zero
    at Main.divide(Main.java:8)
    at Main.main(Main.java:3)
```

Notice execution stops at `divide` ŌĆö it never reaches the `println`.

## Anatomy of a stack trace

A stack trace is your debugger's best friend. Let's decode the one above:

```
Exception in thread "main" java.lang.ArithmeticException: / by zero
    at Main.divide(Main.java:8)
    at Main.main(Main.java:3)
```

| Part | Meaning |
|---|---|
| `Exception in thread "main"` | Which thread crashed |
| `java.lang.ArithmeticException` | Fully-qualified exception class |
| `/ by zero` | The message stored in the exception object |
| `at Main.divide(Main.java:8)` | Class, method, file, and line where it was thrown |
| `at Main.main(Main.java:3)` | The caller that led to the throw |

Read a stack trace **top to bottom**: the top line is where the problem actually occurred; the lines below show the chain of calls that got there. In deep frameworks the trace can be 40 lines long ŌĆö focus on the lines that mention your own class names.

<Note>In real projects you'll often see "caused by" chains where one exception wraps another. The original root cause is at the very bottom of the trace. Always scroll to the bottom first when diagnosing a new error.</Note>

## Why exceptions beat error codes

Consider two styles:

```java
// Error-code style (painful)
int result = divide(a, b);
if (result == DIVIDE_ERROR) {
    int result2 = doSomethingElse();
    if (result2 == SOMETHING_ERROR) {
        // ...
    }
}
```

```java
// Exception style (clean)
try {
    int result = divide(a, b);
    int result2 = doSomethingElse();
    // more happy-path code...
} catch (ArithmeticException e) {
    handleDivisionError(e);
}
```

Exceptions allow you to group a block of happy-path code together and handle failures in a single place. Critically, if an intermediate method forgets to check a return code, the bug is silent. If an intermediate method fails to catch an exception, it propagates automatically ŌĆö you cannot accidentally ignore it.

## A first look at catching exceptions

The `try` block holds code that might throw. The `catch` block runs only if the matching exception is thrown:

```java
public class Main {
    public static void main(String[] args) {
        try {
            int result = Integer.parseInt("abc");   // throws NumberFormatException
            System.out.println(result);
        } catch (NumberFormatException e) {
            System.out.println("Bad number: " + e.getMessage());
        }
        System.out.println("Program continues");
    }
}
```

Output:
```
Bad number: For input string: "abc"
Program continues
```

The exception was caught, handled, and execution resumed normally. The crucial point: code **after** the try-catch still runs.

## More exception examples

Not all exceptions come from the JVM ŌĆö your own methods can detect invalid situations and throw:

```java
static double squareRoot(double n) {
    if (n < 0) {
        throw new IllegalArgumentException("Cannot take sqrt of " + n);
    }
    return Math.sqrt(n);
}
```

You'll learn the full `throw` syntax in Lesson 4. For now, notice the pattern: check the precondition, describe the problem in the message, stop execution immediately.

Another common source of exceptions is null references:

```java
String name = null;
System.out.println(name.toUpperCase());  // NullPointerException
```

`NullPointerException` (NPE) is the most common exception Java programmers encounter. Java 17+ includes "helpful NPEs" that tell you *which* variable was null:

```
NullPointerException: Cannot invoke "String.toUpperCase()" because "name" is null
```

## Common mistakes

- **Reading the stack trace from the wrong end** ŌĆö the top of the trace is the throw site, but in framework-heavy code the top 20 lines may all be library code. Scan for lines with *your* class name; that's where the real story is. Knowing how to read a trace quickly is one of the highest-value debugging skills you can develop.
- **Confusing `Error` and `Exception`** ŌĆö students sometimes try to catch `StackOverflowError` to "handle" infinite recursion. You can technically catch an `Error`, but by the time one fires the JVM state is too corrupted to do anything useful. The fix is always in your code logic, not the catch block.
- **Assuming the program crashes permanently** ŌĆö catching an exception and handling it properly means the program keeps running. This is the whole point: you can recover from a bad parse, retry a network call, or show the user a friendly message instead of an ugly stack trace.
- **Ignoring the exception message** ŌĆö `e.getMessage()` contains the human-readable explanation. Always include it when logging or displaying errors. An empty catch block that swallows the message makes bugs nearly impossible to diagnose.

## Recap

Exceptions are Java's structured mechanism for signalling and handling runtime errors. They beat error codes because failures propagate automatically and you can keep the happy path readable.

- **`Throwable`** is the hierarchy root; `Error` (JVM-level) and `Exception` (program-level) are its two branches
- When an exception is thrown, execution stops and the JVM searches up the call stack for a matching `catch` block
- A stack trace shows *exactly* where the throw happened and the chain of method calls leading to it
- `try { ... } catch (SomeException e) { ... }` is the fundamental syntax for handling an exception
