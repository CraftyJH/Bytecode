Standard exception types cover many cases, but real applications have domain-specific failure modes that deserve their own names and structure. A `ValidationException` that carries the name of the invalid field is far more useful than a bare `IllegalArgumentException`. Custom exceptions let you encode rich failure information while keeping your API clean.

## When to create a custom exception

Ask three questions:
1. Does this failure mode have a name that is meaningful in your domain? (`InsufficientFundsException`, `DuplicateUserException`)
2. Does the caller need structured data beyond a message string? (field name, error code, rejected value)
3. Does the exception need to sit in a hierarchy specific to your application? (`AppException` → `ServiceException` → `DatabaseException`)

If the answer to any of these is yes, a custom exception is worth the small amount of extra code.

## Extending Exception vs RuntimeException

The choice of parent class determines whether the exception is checked or unchecked:

```java
// Checked — caller must handle or declare
class InsufficientFundsException extends Exception {
    InsufficientFundsException(String message) {
        super(message);
    }
}

// Unchecked — caller is not forced to handle
class ConfigurationException extends RuntimeException {
    ConfigurationException(String message) {
        super(message);
    }
}
```

Use `extends Exception` (checked) when the failure is a foreseeable, recoverable condition that callers should be forced to address — insufficient funds is a perfect example, because a well-written banking app must handle it.

Use `extends RuntimeException` (unchecked) for programming errors, configuration mistakes, or conditions from which the application genuinely cannot recover at runtime.

## Adding fields for extra context

A custom exception shines when it carries structured data. Here's a checked exception that records both the field that failed and why:

```java
class ValidationException extends Exception {
    private final String field;

    ValidationException(String field, String message) {
        super(message);
        this.field = field;
    }

    String getField() { return field; }
}
```

Usage:

```java
static void validateEmail(String email) throws ValidationException {
    if (email == null || !email.contains("@")) {
        throw new ValidationException("email", "Must contain @, got: " + email);
    }
}
```

The catch block can now route the error correctly without parsing the message string:

```java
try {
    validateEmail(input);
} catch (ValidationException e) {
    System.out.println("Field '" + e.getField() + "': " + e.getMessage());
}
```

<Note>Keep extra fields `final` when possible. An exception object should be immutable — its state represents a moment in time when something went wrong.</Note>

## Constructor chaining

Provide multiple constructors so callers can choose how much information to supply. Use `this(...)` to chain them and avoid duplication:

```java
class ApiException extends RuntimeException {
    private final int statusCode;

    ApiException(int statusCode, String message) {
        super(message);
        this.statusCode = statusCode;
    }

    ApiException(int statusCode, String message, Throwable cause) {
        super(message, cause);           // stores message AND cause
        this.statusCode = statusCode;
    }

    int getStatusCode() { return statusCode; }
}
```

Always provide a constructor that accepts a `Throwable cause` — it lets callers wrap lower-level exceptions without losing the original stack trace.

## Building a mini exception hierarchy

In larger projects, you often define a base application exception and extend it for specific cases:

```java
// Base — all application-level exceptions extend this
class AppException extends RuntimeException {
    AppException(String message) { super(message); }
    AppException(String message, Throwable cause) { super(message, cause); }
}

// Specific sub-types
class NotFoundException extends AppException {
    private final String resourceType;
    private final Object id;

    NotFoundException(String resourceType, Object id) {
        super(resourceType + " not found: " + id);
        this.resourceType = resourceType;
        this.id = id;
    }

    String getResourceType() { return resourceType; }
    Object getId() { return id; }
}

class PermissionException extends AppException {
    PermissionException(String action) {
        super("Permission denied: " + action);
    }
}
```

A global error handler can now catch `AppException` to handle all application errors uniformly, while individual catch blocks can still catch `NotFoundException` specifically when needed.

## The `serialVersionUID` note

Custom exception classes that extend `Exception` or `RuntimeException` are `Serializable` (because `Throwable` implements `Serializable`). IDEs will often warn that a `serialVersionUID` field is missing:

```java
class MyException extends Exception {
    private static final long serialVersionUID = 1L;  // suppresses the warning
    // ...
}
```

For most application code you will never actually serialize exceptions. Adding `serialVersionUID = 1L` is fine and silences the warning; the exact value doesn't matter as long as you don't change it after serializing instances to disk.

## A realistic domain-level example

Here is a small user-registration service using a checked validation exception:

```java
class RegistrationException extends Exception {
    private final String field;
    private final Object rejectedValue;

    RegistrationException(String field, Object rejectedValue, String message) {
        super(message);
        this.field = field;
        this.rejectedValue = rejectedValue;
    }

    String getField() { return field; }
    Object getRejectedValue() { return rejectedValue; }
}

class AccountService {
    static void createAccount(String username, int age) throws RegistrationException {
        if (username == null || username.length() < 3) {
            throw new RegistrationException("username", username,
                "Username must be at least 3 characters");
        }
        if (age < 13) {
            throw new RegistrationException("age", age,
                "Must be at least 13 years old");
        }
        System.out.println("Account created: " + username);
    }
}
```

The exception carries `field` and `rejectedValue`, which a UI layer could use to highlight the offending input field directly — impossible with a bare string message.

## Try it

```java
class RangeException extends RuntimeException {
    private final int value;
    private final int min;
    private final int max;

    RangeException(int value, int min, int max) {
        super(value + " is outside [" + min + ", " + max + "]");
        this.value = value;
        this.min = min;
        this.max = max;
    }

    int getValue() { return value; }
}

public class Demo {
    public static void main(String[] args) {
        int[] tests = {-5, 0, 50, 101};
        for (int t : tests) {
            try {
                System.out.println(checkRange(t, 0, 100));
            } catch (RangeException e) {
                System.out.println("Out of range: " + e.getMessage() +
                    " (got " + e.getValue() + ")");
            }
        }
    }

    static int checkRange(int value, int min, int max) {
        if (value < min || value > max) throw new RangeException(value, min, max);
        return value;
    }
}
```

Notice that the catch block can call `e.getValue()` for the structured data, and `e.getMessage()` for the human-readable description.

## Common mistakes

- **Forgetting `super(message)`** — the constructor runs but the message is never stored. `getMessage()` returns null, stack traces print no detail, and debugging becomes significantly harder. Always chain `super(message)`.
- **Mutable fields in exceptions** — making an exception's extra fields `public` and non-final allows callers to modify them after the exception is created, undermining the integrity of the error report. Keep fields `private final`.
- **Over-creating exception types** — not every error needs its own class. If `IllegalArgumentException` with a clear message conveys the failure adequately, a custom class adds ceremony without value. Create custom exceptions when callers need to distinguish them programmatically or when structured data is needed.
- **Omitting the cause constructor** — always provide a `(String message, Throwable cause)` overload so your exception can wrap lower-level failures. Losing the original cause is one of the hardest bugs to diagnose in production.

## Recap

Custom exceptions extend Java's type system into your own domain. They make failure modes first-class concepts, carry structured information, and build exception hierarchies that reflect your application's architecture.

- Extend `Exception` for checked (caller must handle); extend `RuntimeException` for unchecked
- Add `private final` fields for structured data and expose them through getters
- Always call `super(message)` in every constructor to store the message
- Provide a `(message, cause)` constructor overload so wrapping is always possible
- Build hierarchies (`AppException` → specific subtypes) when you need uniform global handling plus specific local handling
- Add `serialVersionUID = 1L` to silence the serialization warning; the exact value doesn't matter for most code
