You've seen exceptions caught; now you'll learn to throw them intentionally. Throwing exceptions is how you enforce *contracts* — the rules your methods expect callers to follow. A method that accepts a file path should refuse null paths loudly rather than silently producing confusing behavior several layers deeper.

## The `throw` statement

`throw` signals an exception from inside any method body. It takes any `Throwable` object:

```java
static void printName(String name) {
    if (name == null) {
        throw new NullPointerException("name must not be null");
    }
    System.out.println("Hello, " + name);
}
```

Execution stops at `throw`. If no catch block in the current stack handles it, the exception propagates up. `throw` takes *one* expression — the exception object — and that expression must be non-null.

<Note>`throw` (no 's') is the action; `throws` (with 's') in the method signature is the declaration. Students commonly confuse the two. Remember: the body *throws*, the signature *declares throws*.</Note>

## Guard clauses: validate early

A **guard clause** is a `throw` (or `return`) at the top of a method that immediately rejects invalid input. It is the most important technique for keeping method bodies clean:

```java
static double divide(double numerator, double denominator) {
    if (denominator == 0.0) {
        throw new ArithmeticException("denominator must not be zero");
    }
    // Happy path — no nesting needed
    return numerator / denominator;
}
```

Compare that to the alternative without a guard:

```java
static double divide(double numerator, double denominator) {
    if (denominator != 0.0) {
        return numerator / denominator;
    } else {
        throw new ArithmeticException("denominator must not be zero");
    }
}
```

Both work, but the guard-clause version keeps the main logic at the top level. With multiple guards, the difference becomes even more dramatic.

## Choosing the right exception type

Java's standard library provides many exception types. Picking the right one makes your code self-documenting:

| Situation | Exception type |
|---|---|
| A method argument violates the contract | `IllegalArgumentException` |
| The object is in a state that doesn't allow this operation | `IllegalStateException` |
| A required argument (or required value) is null | `NullPointerException` |
| An array or list index is out of bounds | `ArrayIndexOutOfBoundsException` / `IndexOutOfBoundsException` |
| An operation is not implemented / not supported | `UnsupportedOperationException` |
| A numeric computation overflowed or underflowed | `ArithmeticException` |

Always prefer a specific type over `RuntimeException` directly — the exception class itself is documentation.

```java
static void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("Age out of realistic range: " + age);
    }
    this.age = age;
}

static void start() {
    if (alreadyRunning) {
        throw new IllegalStateException("Cannot start: already running");
    }
    alreadyRunning = true;
}
```

## Writing good exception messages

An exception message is read by a developer trying to understand what went wrong. Good messages are specific and include the bad value:

```java
// Bad — no context
throw new IllegalArgumentException("invalid input");

// Good — includes the actual value and why it's wrong
throw new IllegalArgumentException(
    "Expected positive integer, got: " + value + " (must be >= 1)");
```

Think of the message as a note to your future self at 2 AM trying to diagnose a production outage.

## Re-throwing exceptions

Sometimes a method catches an exception, does some cleanup or logging, and then re-throws it:

```java
static void saveToDatabase(Record r) throws SQLException {
    try {
        db.insert(r);
    } catch (SQLException e) {
        log.error("Failed to insert record " + r.id, e);
        throw e;   // re-throw the same exception
    }
}
```

`throw e` re-throws the existing exception object, preserving its original stack trace. The caller sees the same `SQLException` that the database driver produced.

## Wrapping exceptions (exception chaining)

Instead of re-throwing the original, you can *wrap* it in a new exception. This is useful for translating low-level exceptions into meaningful domain-level ones:

```java
static User loadUser(int id) {
    try {
        return database.findById(id);
    } catch (SQLException e) {
        // Caller doesn't know about SQL — give them a domain exception
        throw new RuntimeException("Could not load user with id " + id, e);
    }
}
```

The second argument to most exception constructors is the **cause**. The cause is visible in stack traces and accessible via `getCause()`. Never wrap without preserving the cause — you'd lose the original stack trace.

```java
// Bad — cause is lost
throw new RuntimeException("Could not load user");

// Good — cause is preserved
throw new RuntimeException("Could not load user", e);
```

## A realistic validation example

Guard clauses are especially powerful when combined with descriptive messages:

```java
static void transferFunds(Account from, Account to, double amount) {
    if (from == null) throw new NullPointerException("source account must not be null");
    if (to == null)   throw new NullPointerException("destination account must not be null");
    if (amount <= 0)  throw new IllegalArgumentException(
        "Transfer amount must be positive, got: " + amount);
    if (from.getBalance() < amount) throw new IllegalStateException(
        "Insufficient funds: balance " + from.getBalance() + " < requested " + amount);

    from.debit(amount);
    to.credit(amount);
}
```

Each guard addresses a distinct failure mode with a distinct exception type and a message that pinpoints the problem. The actual transfer logic at the bottom is uncluttered and easy to audit.

## Try it

```java
public class Demo {
    public static void main(String[] args) {
        String[] names = {"Alice", "", null, "Bob"};
        for (String name : names) {
            try {
                System.out.println(greet(name));
            } catch (NullPointerException e) {
                System.out.println("NPE: " + e.getMessage());
            } catch (IllegalArgumentException e) {
                System.out.println("IAE: " + e.getMessage());
            }
        }
    }

    static String greet(String name) {
        if (name == null)    throw new NullPointerException("name is null");
        if (name.isBlank())  throw new IllegalArgumentException("name is blank");
        return "Hello, " + name + "!";
    }
}
```

Notice that both guards use the exception type most appropriate to the failure mode, and the loop can handle whichever fires.

## Common mistakes

- **Creating but not throwing** — `new IllegalArgumentException("bad")` without `throw` compiles fine but does nothing. The object is created and immediately garbage-collected. Always write `throw new ...`.
- **Vague exception messages** — "invalid input" or "error occurred" are useless in production logs. Always include the actual value and an explanation of why it's wrong. Future-you will be grateful.
- **Using the wrong exception type** — throwing `RuntimeException` directly when `IllegalArgumentException` exists, or throwing `Exception` when `IOException` would be precise. The type is documentation; choose it deliberately.
- **Losing the cause when wrapping** — `throw new RuntimeException("msg")` discards the original stack trace. Always pass the caught exception as the second argument: `throw new RuntimeException("msg", e)`.
- **Throwing inside `finally`** — as covered in Lesson 2, this swallows any in-flight exception. If you must throw from a cleanup path, wrap it carefully and log the suppressed exception.

## Recap

Throwing exceptions is how you enforce contracts and communicate failures precisely. A method that throws at the right moment, with the right type and a clear message, makes the entire codebase more robust.

- `throw new ExceptionType("message")` is the basic syntax; the object must be non-null
- Guard clauses at the top of a method reject invalid input early, keeping the happy path clean
- Choose the most specific standard exception type available; the type name itself is documentation
- Re-throw with `throw e` to preserve the original stack trace; wrap with `throw new SomeException("msg", e)` to translate exceptions across layers
- Always include the bad value and an explanation in exception messages
