In the previous lesson you learned how to walk a collection with an explicit `Iterator`. Now we tackle the most common *real* reason to drop down from for-each to iterator: **removing elements as you go**. This is one of those topics where naïve code looks reasonable, runs once, and then crashes loudly the next day with a stack trace nobody understands.

Let's understand the crash, fix it the right way, and then meet the modern one-liner that makes it almost unnecessary.

## The naïve attempt — and what goes wrong

Suppose you want to delete every even number from a list. The first thing many learners write is:

```java
List<Integer> nums = new ArrayList<>(List.of(1, 2, 3, 4, 5, 6));
for (int n : nums) {
    if (n % 2 == 0) {
        nums.remove(Integer.valueOf(n));   // BAD
    }
}
```

It compiles. It even *looks* fine. But run it and you get:

```
Exception in thread "main" java.util.ConcurrentModificationException
    at java.base/java.util.ArrayList$Itr.checkForComodification(...)
    at java.base/java.util.ArrayList$Itr.next(...)
    at Main.main(Main.java:5)
```

<Note>The exception name is misleading. There are no threads involved — "concurrent" here means "you modified the collection *while* iterating it", which is logically concurrent even on a single thread.</Note>

## Why the iterator gets angry

Most `java.util` collections (`ArrayList`, `HashSet`, `HashMap`, ...) maintain a hidden field called `modCount` — a counter that bumps every time the *structure* of the collection changes (an element added or removed). When you ask for an iterator, it snapshots the current `modCount`. On every call to `next()` it compares the live count with its snapshot:

```java
// Inside ArrayList.Itr (paraphrased):
final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}
```

If the numbers don't match, *something* changed the collection through a path the iterator didn't know about — so the iterator throws rather than silently return stale data. This is **fail-fast** behavior, and it's a feature, not a bug. Without it you might skip elements, double-visit elements, or read garbage.

## The right way: `Iterator.remove()`

The iterator has one — and only one — sanctioned way to mutate the collection mid-walk: its own `remove()` method. That call is special because the iterator updates its `modCount` snapshot at the same time, so the consistency check still passes on the next `next()`.

```java
List<Integer> nums = new ArrayList<>(List.of(1, 2, 3, 4, 5, 6));
Iterator<Integer> it = nums.iterator();
while (it.hasNext()) {
    int n = it.next();
    if (n % 2 == 0) {
        it.remove();        // safe — removes the element next() just returned
    }
}
System.out.println(nums);   // [1, 3, 5]
```

Three rules to remember about `Iterator.remove()`:

1. It removes the element returned by the **most recent** `next()` call.
2. You may call it **at most once** per `next()`.
3. Calling it before any `next()`, or twice without an intervening `next()`, throws `IllegalStateException`.

That last rule is why a body like `if (it.hasNext()) it.remove();` is *not* valid — `hasNext()` doesn't count.

<Note>Not every iterator implements `remove()`. Iterators returned by immutable collections (e.g. `List.of(...)` or `Collections.unmodifiableList(...)`) throw `UnsupportedOperationException`. Stick with mutable collection types when you intend to remove.</Note>

## The modern shortcut: `Collection.removeIf`

Java 8 added a one-liner that wraps the iterator+remove pattern for the common case "remove everything matching this condition":

```java
List<Integer> nums = new ArrayList<>(List.of(1, 2, 3, 4, 5, 6));
nums.removeIf(n -> n % 2 == 0);
System.out.println(nums);   // [1, 3, 5]
```

It is implemented exactly the way you would write it: walk an iterator, call `remove()` whenever the predicate is true. But it's shorter, and the intent is unmistakable. Reach for `removeIf` whenever your removal rule fits in a lambda.

`removeIf` lives on `Collection`, so it works on `List`, `Set`, `Queue`, and friends — but not on `Map` directly. For maps, use `map.entrySet().removeIf(e -> ...)` or the dedicated `map.values().removeIf(...)`.

## When `removeIf` *isn't* enough

Some scenarios still call for the explicit iterator pattern:

- You need to **inspect more than just the element** — for example, the position, the previous element, or some external state machine.
- You want to **transform some elements and remove others** in the same pass. `removeIf` only deletes; it doesn't update.
- You're iterating something that **isn't a `Collection`** (a custom `Iterable`, a stream of records from a parser, etc.).

In those cases, the manual `while (it.hasNext()) { ... it.remove(); ... }` loop remains the right tool.

## A second example: scrubbing a Set

`HashSet` supports the same iterator-remove pattern. Suppose we have a set of usernames and want to delete every name shorter than four characters:

```java
Set<String> names = new HashSet<>(Set.of("Al", "Bea", "Carol", "Dan", "Eleanor"));
Iterator<String> it = names.iterator();
while (it.hasNext()) {
    String name = it.next();
    if (name.length() < 4) {
        it.remove();
    }
}
System.out.println(names);    // some order of [Carol, Eleanor]
```

Or, more idiomatically:

```java
names.removeIf(name -> name.length() < 4);
```

Both produce the same result. The iterator version is more flexible; the `removeIf` version is more readable.

## A failing example, deliberately

Sometimes the best way to *believe* the rule is to break it:

```java
List<Integer> nums = new ArrayList<>(List.of(1, 2, 3, 4));
for (int n : nums) {
    if (n == 2) nums.remove(Integer.valueOf(n));   // crashes
}
```

Run this and you'll see the dreaded `ConcurrentModificationException` on the second iteration — the iterator's stored `modCount` no longer matches the list's actual `modCount`. The fix is one of the two patterns we've just covered. Memorise the failure mode; it will save you debugging time later.

<Note>Counter-intuitive twist: with an `ArrayList`, removing the *last* element inside a for-each happens to "work" by accident — the loop terminates before the next `hasNext()` check fires. Don't rely on this. The correct rule is simple: never mutate the collection except through the iterator, or with `removeIf`.</Note>

## Common mistakes

- **Calling `list.remove(...)` inside a for-each** — This is the canonical bug. The iterator detects the change and throws `ConcurrentModificationException`. Switch to an explicit `Iterator` plus `it.remove()`, or use `removeIf`.
- **Calling `it.remove()` before any `next()`** — `remove()` only makes sense relative to the most recent `next()`. Calling it on a fresh iterator throws `IllegalStateException`.
- **Calling `it.remove()` twice in a row** — Same reason. After the first `remove()`, you must call `next()` again before another `remove()` is legal.
- **Trying to remove from an immutable collection** — `List.of(...)` and `Collections.unmodifiableList(...)` produce iterators whose `remove()` throws `UnsupportedOperationException`. If you need to remove, build a mutable copy: `new ArrayList<>(immutableList)`.
- **Using a stream's `filter` to mutate the source** — `stream().filter(...)` builds a new sequence; it does not modify the original list. If you intended to delete in place, use `removeIf` instead.
- **Catching `ConcurrentModificationException` to "fix" the bug** — The exception is a symptom, not the disease. Catching it leaves the loop in an undefined state. Fix the mutation pattern instead.

## Recap

Java collections protect you from yourself: they fail fast when something modifies them in a way the iterator can't see. Knowing this lets you write loops that delete exactly what they should, without crashing.

- Mutating a collection during a for-each (or while another iterator is open) throws **`ConcurrentModificationException`**
- Use **`Iterator.remove()`** to delete the most recently returned element safely
- Java 8+ offers **`Collection.removeIf(predicate)`** as a clean one-liner for predicate-based deletion
- `Iterator.remove()` itself throws `IllegalStateException` if called before `next()` or twice in a row
- The fail-fast behavior is a *feature* — silent corruption would be far worse
