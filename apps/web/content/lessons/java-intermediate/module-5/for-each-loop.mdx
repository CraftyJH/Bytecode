Most code that touches a collection has the same shape: visit each element, do something, move on. Java gives you a dedicated, readable way to express exactly that — the **for-each loop**, sometimes called the *enhanced for loop*. It hides the bookkeeping (counter, length, indexing) that a classic `for` requires, and lets the reader focus on the work.

In this lesson you'll learn what powers the for-each loop under the hood, which types you can use it on, and where its convenience starts to bite you back.

## The shape of a for-each loop

The syntax has just three slots: the element type, a name for the current element, and the source you're walking:

```java
for (ElementType name : source) {
    // body — runs once per element
}
```

Read the colon as the word *in*. So `for (String s : names)` is "for each String s in names". On every pass, `name` is rebound to the next element — there is no index variable.

```java
List<String> langs = List.of("Java", "Kotlin", "Scala");
for (String lang : langs) {
    System.out.println("- " + lang);
}
```

That prints three lines. The loop ends when the source has no more elements, with no `i < size` check to forget.

<Note>The variable on the left is a *fresh local variable* per iteration. Reassigning it inside the body has no effect on the underlying collection — it just rebinds your local name.</Note>

## What types qualify?

The compiler accepts a for-each loop in exactly two situations:

1. The right-hand side is an **array** (`int[]`, `String[]`, `Foo[][]`, etc.).
2. The right-hand side implements **`java.lang.Iterable<T>`**.

Almost every collection in `java.util` implements `Iterable`: `ArrayList`, `LinkedList`, `HashSet`, `TreeSet`, `ArrayDeque`, `PriorityQueue`, and so on. That's why for-each "just works" on them.

```java
int[] nums = {1, 2, 3, 4, 5};
int sum = 0;
for (int n : nums) sum += n;          // arrays — first category

Set<String> tags = Set.of("api", "java", "rest");
for (String t : tags) System.out.println(t);  // Iterable — second category
```

## Maps need a view

`Map<K, V>` deliberately does **not** implement `Iterable`. There is no single obvious way to walk a map — keys? values? key/value pairs? — so Java forces you to choose:

```java
Map<String, Integer> ages = new HashMap<>();
ages.put("Alice", 30);
ages.put("Bob", 25);

for (Map.Entry<String, Integer> e : ages.entrySet()) {
    System.out.println(e.getKey() + " is " + e.getValue());
}

for (String name : ages.keySet())  System.out.println(name);
for (int age   : ages.values())    System.out.println(age);
```

Try `for (var x : ages)` and the compiler will tell you `HashMap<String,Integer>` is not iterable.

<Note>`Map.Entry` is a small interface representing one key–value pair. Calling `.getKey()` and `.getValue()` is the most common way to walk a map's contents.</Note>

## How does for-each actually work?

The for-each loop is *syntactic sugar*. The compiler rewrites it into either an indexed loop (for arrays) or a call to `iterator()` (for `Iterable`s). For an `Iterable`, this:

```java
for (String s : names) { use(s); }
```

is roughly equivalent to:

```java
Iterator<String> $it = names.iterator();
while ($it.hasNext()) {
    String s = $it.next();
    use(s);
}
```

You'll meet `Iterator` directly in the next lesson. For now the takeaway is: **for-each is just a readable wrapper around an iterator**. That's why anything `Iterable` works, and why arrays — which aren't `Iterable` — get their own special compiled form.

## Read-only by design

Notice what the desugared loop does *not* expose: the iterator itself, or the index. That has consequences:

- You **cannot remove** an element from the underlying collection during a for-each loop — doing so will throw `ConcurrentModificationException` at runtime (covered in lesson 3).
- You **cannot get the current index** — there is no `i`. If you need it, fall back to a classic `for`.
- You **cannot easily skip ahead or look back** — every element is visited exactly once, in order.

```java
// Classic for — gives you the index
for (int i = 0; i < langs.size(); i++) {
    System.out.println(i + ": " + langs.get(i));
}

// for-each — no index, but cleaner when you don't need one
for (String lang : langs) {
    System.out.println(lang);
}
```

A handy rule of thumb: if you're not using `i` for anything except `list.get(i)`, switch to for-each.

<Note>Calling `list.get(i)` on a `LinkedList` is **O(n)** because linked lists have no random access. A classic indexed loop over a `LinkedList` is therefore O(n^2). The for-each loop uses the iterator and stays O(n) — yet another reason to prefer it when you don't need the index.</Note>

## Auto-unboxing in the loop variable

When the source holds wrapper objects (`Integer`, `Double`, `Boolean`, ...) and you declare the loop variable as a primitive, Java unboxes each element automatically:

```java
List<Integer> scores = List.of(90, 85, 100);
int total = 0;
for (int s : scores) {       // Integer → int per iteration
    total += s;
}
```

If any element were `null`, the auto-unbox would throw `NullPointerException`. Stick with `Integer s` if your list might contain nulls.

## Nested for-each

Loops nest exactly as you'd expect, and the inner loop walks its own source from scratch on every outer pass:

```java
String[][] grid = {
    {"a", "b", "c"},
    {"d", "e", "f"}
};

for (String[] row : grid) {
    for (String cell : row) {
        System.out.print(cell + " ");
    }
    System.out.println();
}
// a b c
// d e f
```

That works because `grid` is a `String[][]` — an array whose elements are themselves arrays. Each `row` is its own iterable target.

## Common mistakes

- **Trying to remove during a for-each loop** — Calling `list.remove(x)` while iterating throws `ConcurrentModificationException`. Use `Iterator.remove()` or `Collection.removeIf(...)` instead. We'll cover both in lesson 3.
- **Reaching for an index that isn't there** — There is no `i` in a for-each loop. If you need the position, either keep a manual counter (`int i = 0; for (...) { ...; i++; }`) or switch to a classic `for`.
- **Iterating a `Map` directly** — `for (var x : myMap)` does not compile. Always pick a view: `entrySet()` for both, `keySet()` for keys only, `values()` for values only.
- **Reassigning the loop variable expecting it to mutate the collection** — Writing `for (String s : names) { s = "X"; }` only changes the local copy. The list is untouched. Use `list.set(i, "X")` in a classic loop, or `replaceAll(...)`.
- **Auto-unbox NullPointerException** — If a `List<Integer>` contains a `null` and you write `for (int x : list)`, the unbox fails at runtime. Declare the variable as `Integer` (or filter out nulls) when nullability is possible.
- **Iterating a `String` directly** — `for (char c : "hello")` does not compile. `String` isn't `Iterable`. Use `"hello".toCharArray()` or `"hello".chars()`.

## Recap

The for-each loop is the default way to read every element of a collection or array in Java. It removes index bookkeeping, works uniformly across `List`, `Set`, `Queue` and arrays, and reads almost like English. Once you know which types qualify and what its limitations are, you'll reach for it 90% of the time.

- For-each works on **arrays** and on anything implementing **`Iterable<T>`**
- It is **read-only** in the sense that you cannot remove during iteration
- There is **no index variable** — fall back to a classic `for` if you need one
- `Map` is **not** `Iterable`; use `entrySet()`, `keySet()`, or `values()`
- Behind the scenes, for-each desugars to an `Iterator`-based loop or an indexed array walk
- Prefer for-each on `LinkedList` to avoid the O(n^2) trap of `list.get(i)`
