You have seen type parameters on classes (`class Box<T>`). Java also lets you place type parameters on individual **methods**, independent of whether the class itself is generic. This makes static utility methods — the kind that live in helper classes — fully type-safe without forcing the entire class to be parameterized.

## Method-level vs class-level type parameters

When a type parameter is on the class, every instance method and field shares it:

```java
class Box<T> {      // T is the class's parameter
    T value;
    T get() { return value; }   // T is in scope here
}
```

When a type parameter is on the method, it applies only to that one method call:

```java
class Utils {                              // not generic
    static <T> T identity(T value) {      // T is this method's parameter
        return value;
    }
}
```

The `<T>` in the method signature comes **after any modifiers** and **before the return type**. That position is how the compiler knows you are introducing a new type parameter rather than referring to an existing type named `T`.

## Type inference

In almost all cases you do not have to specify the type argument when calling a generic method. The compiler infers it from the argument types:

```java
String s  = Utils.identity("hello");   // T inferred as String
Integer n = Utils.identity(42);        // T inferred as Integer
```

In rare cases — usually when the inferred type is ambiguous or you want to be explicit for documentation — you can provide it explicitly:

```java
String s = Utils.<String>identity("hello");  // explicit, rarely needed
```

Prefer implicit inference; explicit invocation is verbose and mainly used when the compiler cannot resolve the type on its own.

<Note>Type inference became significantly smarter in Java 8. Prior to that, complex call chains sometimes required explicit type witnesses. If you ever see code like `Collections.<String>emptyList()`, that was written for an older compiler — modern Java infers it automatically.</Note>

## Static generic utility methods

The most valuable use of generic methods is building stateless utility functions that work across all types. Consider a `swap` utility:

```java
import java.util.List;

public class Util {
    static <T> void swap(List<T> list, int i, int j) {
        T temp = list.get(i);
        list.set(i, list.get(j));
        list.set(j, temp);
    }
}
```

The type parameter `T` exists only to link the list type with the elements being swapped — it carries no other information. The compiler verifies that `list.get(i)` returns a `T` and that `list.set(i, ...)` accepts a `T`. No casts, no `Object` tricks.

## Building useful utilities: contains and zip

```java
import java.util.List;
import java.util.ArrayList;

public class Util {
    // Check membership without casting
    static <T> boolean contains(List<T> list, T target) {
        for (T item : list) {
            if (item.equals(target)) return true;
        }
        return false;
    }

    // Pair up two lists element-by-element
    static <A, B> List<String> zip(List<A> as, List<B> bs) {
        int len = Math.min(as.size(), bs.size());
        List<String> pairs = new ArrayList<>(len);
        for (int i = 0; i < len; i++) {
            pairs.add("(" + as.get(i) + ", " + bs.get(i) + ")");
        }
        return pairs;
    }
}
```

Usage:

```java
List<String> fruits = Arrays.asList("apple", "banana", "cherry");
System.out.println(Util.contains(fruits, "banana"));   // true
System.out.println(Util.contains(fruits, "mango"));    // false

List<Integer> ids    = Arrays.asList(1, 2, 3);
List<String>  labels = Arrays.asList("a", "b", "c");
System.out.println(Util.zip(ids, labels));  // [(1, a), (2, b), (3, c)]
```

The `zip` method uses **two** type parameters `<A, B>`, allowing the two lists to hold different types. The compiler tracks them independently.

## When to use a generic method vs a generic class

The choice comes down to whether the type parameter is needed by the object's **state** or only by one **operation**:

| Situation | Choose |
| --------- | ------ |
| The class stores or remembers values of type `T` across methods | Generic class (`class Foo<T>`) |
| A single utility operation needs type safety but the class keeps no typed state | Generic method (`static <T> ... method(...)`) |
| Several related utility methods all share the same `T` | Possibly a generic class, or a generic interface |

`java.util.Collections` is the canonical example of the generic-method pattern: the class itself is not generic, but methods like `sort`, `shuffle`, `unmodifiableList`, and `frequency` are all generic.

## Another example: a typed toArray helper

Because `T[]` arrays are tricky to create generically, a helper method that wraps the unsafe part in one place is a common pattern:

```java
import java.util.List;

public class Util {
    @SuppressWarnings("unchecked")
    static <T> T[] toArray(List<T> list, T[] dummy) {
        // list.toArray(T[]) already handles the cast internally
        return list.toArray(dummy);
    }
}

// Usage
List<String> words = Arrays.asList("x", "y", "z");
String[] arr = Util.toArray(words, new String[0]);
System.out.println(Arrays.toString(arr));  // [x, y, z]
```

The `dummy` array of length 0 is passed purely to supply the element type at runtime — a common pattern because of type erasure.

```java
import java.util.Arrays;
import java.util.List;
import java.util.ArrayList;

public class Demo {
    static <T> List<T> filter(List<T> list, java.util.function.Predicate<T> pred) {
        List<T> result = new ArrayList<>();
        for (T item : list) {
            if (pred.test(item)) result.add(item);
        }
        return result;
    }

    public static void main(String[] args) {
        List<Integer> nums = Arrays.asList(1, 2, 3, 4, 5, 6);

        List<String> long3 = filter(words, w -> w.length() > 3);
        System.out.println(long3);  // [deer]
    }
}
```

## Common mistakes

- **Missing `<T>` in the signature** — writing `static List<T> wrap(T item)` without `<T>` before the return type is a compile error; the compiler treats `T` as an unrecognized class name.
- **Shadowing the class type parameter** — inside a generic class, writing `static <T> void helper(T v)` introduces a *new*, unrelated `T`. This compiles but creates confusion; choose a different letter for the method-level parameter.
- **Overusing generic methods** — if a method only ever needs `Object`-level operations (like `.equals()` or `.toString()`), a simple `Object` parameter often suffices. Add a type parameter only when the caller needs the return type or parameters to be linked.
- **Explicit type witnesses on obvious calls** — `Utils.<String>identity("hello")` is noisier than `Utils.identity("hello")`. Explicit witnesses are needed only when inference fails, which is rare with Java 8+.

## Recap

Generic methods decouple type safety from class-level design. A plain, non-generic class can contain dozens of generic methods, each with its own type parameter that is inferred at the call site.

- The type parameter list `<T>` appears before the return type in the method signature
- The compiler infers `T` from argument types — explicit witnesses are rarely needed
- Use a generic method when the type is needed only for one operation; use a generic class when the type is part of the object's persistent state
- `java.util.Collections` is the standard-library poster child for the generic-method pattern
