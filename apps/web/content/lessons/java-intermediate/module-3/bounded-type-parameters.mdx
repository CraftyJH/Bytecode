A plain type parameter `<T>` accepts literally any type. That is powerful, but it also means the compiler cannot let you call any methods on a `T` value beyond those defined on `Object`. If you want to sort, compare, or do arithmetic with `T`, you must narrow the range of acceptable types using a **bound**.

## Why bounds are necessary

Consider writing a method that returns the larger of two values. Your first attempt might be:

```java
static <T> T max(T a, T b) {
    return a.compareTo(b) >= 0 ? a : b;  // COMPILE ERROR
}
```

The compiler rejects `a.compareTo(b)` because `T` is unrestricted. It could be a `StringBuilder`, a `BufferedReader`, or any class that does not have `compareTo`. The compiler enforces this: if a method might receive an incompatible type, it flags the problem immediately.

The fix is to declare that `T` must implement `Comparable<T>`:

```java
static <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) >= 0 ? a : b;  // compiles
}
```

Now the compiler knows that whatever `T` turns out to be, it will have a `compareTo(T)` method. The bound effectively restricts the type parameter to the subset of types that satisfy the constraint.

<Note>The keyword `extends` is used for both class bounds and interface bounds. `T extends Comparable<T>` means "`T` is a subtype of `Comparable<T>`" — Comparable is an interface, but `extends` is still the keyword in this position. `implements` is not used in generic bounds.</Note>

## `<T extends SomeClass>` — class bounds

When you bound `T` with a class, callers can only pass that class or its subclasses, and inside the method you have access to all public methods of the bounding class:

```java
import java.util.List;

static <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T n : list) {
        total += n.doubleValue();   // Number.doubleValue() is available
    }
    return total;
}
```

Because `T extends Number`, the compiler knows every element has `doubleValue()`, `intValue()`, `longValue()`, etc. You can pass `List<Integer>`, `List<Double>`, or `List<Long>` — any list whose element type is a `Number` subclass.

```java
System.out.println(sum(Arrays.asList(1, 2, 3)));         // 6.0
System.out.println(sum(Arrays.asList(0.5, 1.5, 2.0)));   // 4.0
System.out.println(sum(Arrays.asList(100L, 200L)));       // 300.0
```

## Multiple bounds with `&`

A type parameter can be constrained by more than one type. All bounds after the first are joined with `&`:

```java
static <T extends Number & Comparable<T>> T clamp(T value, T min, T max) {
    if (value.compareTo(min) < 0) return min;
    if (value.compareTo(max) > 0) return max;
    return value;
}
```

`T` must simultaneously:
- be a subclass of `Number` (gives access to `doubleValue()`, etc.)
- implement `Comparable<T>` (gives access to `compareTo()`)

Callers pass `Integer`, `Double`, or `Long` — all of which satisfy both constraints. Passing a `BigDecimal` (which is also `Number & Comparable`) would work too.

Rules for multiple bounds:
- **At most one class bound** — Java does not support multiple class inheritance.
- **The class bound must come first** — `<T extends Number & Comparable<T>>` is valid; `<T extends Comparable<T> & Number>` is a compile error.
- **Any number of interface bounds** — `<T extends Serializable & Cloneable & Comparable<T>>` is legal.

## Comparable-based algorithms

The canonical use case for bounds is algorithms that require ordering. A fully generic `sort` would need `Comparable`:

```java
import java.util.List;

static <T extends Comparable<T>> void bubbleSort(List<T> list) {
    int n = list.size();
    for (int i = 0; i < n - 1; i++) {
        for (int j = 0; j < n - i - 1; j++) {
            if (list.get(j).compareTo(list.get(j + 1)) > 0) {
                T temp = list.get(j);
                list.set(j, list.get(j + 1));
                list.set(j + 1, temp);
            }
        }
    }
}
```

Usage:

```java
List<Integer> nums = new ArrayList<>(Arrays.asList(5, 1, 4, 2));
bubbleSort(nums);
System.out.println(nums);  // [1, 2, 4, 5]

List<String> words = new ArrayList<>(Arrays.asList("cherry", "apple", "banana"));
bubbleSort(words);
System.out.println(words);  // [apple, banana, cherry]
```

The same method sorts integers and strings because both implement `Comparable<T>`. The comparison logic is entirely inside the type, not scattered through the calling code.

## The Comparable vs Comparator distinction

`Comparable<T>` is the interface a class implements to define its **natural ordering** — the default order that makes sense for that type (`Integer` compares by numeric value, `String` by lexicographic order).

`Comparator<T>` is a **separate strategy object** you pass in when you need a different ordering — sort strings by length, sort employees by salary, sort dates in reverse. Bounded type parameters with `Comparable` cover natural ordering; `Comparator` covers custom ordering and is the subject of a later lesson.

<Note>All Java wrapper types (`Integer`, `Double`, `Long`, `Character`) and `String` implement `Comparable`. User-defined value types (records, data classes) should also implement it when natural ordering makes sense — `Comparable` is how `Collections.sort()` and `TreeSet` work by default.</Note>

## Another example: finding the minimum in a list

```java
import java.util.List;

static <T extends Comparable<T>> T minimum(List<T> list) {
    if (list.isEmpty()) throw new java.util.NoSuchElementException();
    T result = list.get(0);
    for (int i = 1; i < list.size(); i++) {
        if (list.get(i).compareTo(result) < 0) {
            result = list.get(i);
        }
    }
    return result;
}
```

```java
System.out.println(minimum(Arrays.asList(3, 1, 4, 1, 5, 9)));  // 1
System.out.println(minimum(Arrays.asList("mango", "apple", "pear")));  // apple
```

The same fifteen-line method works for any `Comparable` type. Writing this without bounds would require separate versions for each type — exactly the kind of duplication generics eliminate.

## Try it

import java.util.Arrays;
import java.util.List;

public class Demo {
    static <T extends Comparable<T>> boolean isSorted(List<T> list) {
        for (int i = 0; i < list.size() - 1; i++) {
            if (list.get(i).compareTo(list.get(i + 1)) > 0) return false;
        }
        return true;

        System.out.println(isSorted(Arrays.asList("z", "a")));        // false

}

## Common mistakes

- **Forgetting the bound when calling type methods** — writing `<T>` and then trying to call `.compareTo()` or `.doubleValue()` causes a compile error. The compiler needs the bound to know those methods exist on `T`.
- **Putting the interface bound before the class bound** — `<T extends Comparable<T> & Number>` is a compile error. The class bound (`Number`) must come first.
- **Confusing `<T extends Number>` with `<? extends Number>`** — the first declares a named type parameter; the second is a wildcard (covered next lesson). They look similar but serve different purposes.
- **Treating `extends` in bounds as inheritance** — `T extends Number` in a generic bound means "T is assignable to Number", which includes both class extension and interface implementation. It is a subtype relationship, not strictly "this class inherits from that class".

## Recap

Bounds let you unlock the API of a specific type or interface on a generic type parameter. They are the bridge between "works on anything" and "works on anything that supports these operations."

- `<T extends SomeClass>` restricts T to SomeClass and its subclasses/implementors
- Multiple bounds use `&`: `<T extends ClassBound & Interface1 & Interface2>`
- The class bound (at most one) must come first; interface bounds follow
- `Comparable<T>` is the standard bound for any algorithm that requires ordering
- Bounded type parameters make sorting, min/max, and clamping algorithms work across all comparable types without code duplication
