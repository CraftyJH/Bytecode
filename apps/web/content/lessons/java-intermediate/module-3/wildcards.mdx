Type parameters (`<T>`) are named placeholders you introduce yourself. **Wildcards** (`?`) represent an unknown type at a use site — a type someone else chose. They appear as method parameter types and let you write methods that accept many related generic types without sacrificing safety.

## Why wildcards exist: the invariance problem

Suppose `Dog extends Animal`. You might expect `List<Dog>` to be usable wherever `List<Animal>` is required — after all, dogs are animals. But Java generic types are **invariant**:

```java
List<Dog> dogs = new ArrayList<>();
// List<Animal> animals = dogs;  // COMPILE ERROR

List<Animal> animals = dogs;     // hypothetically allowed...
animals.add(new Cat());          // ...this would silently corrupt dogs
Dog first = dogs.get(0);         // ClassCastException — not a Dog!
```

If the assignment were allowed, you could insert a `Cat` through the `animals` reference, then blow up when code elsewhere expects only `Dog` objects. The compiler catches this statically by making generic types invariant.

But sometimes you genuinely want to pass a `List<Dog>` to a method that only reads the list as `Animal` objects. That is exactly what wildcards provide.

## The unbounded wildcard `?`

`List<?>` means "a list of some unknown type." You can iterate it and treat every element as `Object`, but you cannot add anything to it (because you don't know what type the list actually holds):

```java
static void printAll(List<?> list) {
    for (Object item : list) {
        System.out.println(item);
    }
}
```

```java
printAll(Arrays.asList("a", "b", "c"));    // works
printAll(Arrays.asList(1, 2, 3));           // works
printAll(Arrays.asList(true, false));       // works
```

The unbounded wildcard is best for methods that only use `Object`-level operations like `toString()`, `equals()`, or `size()`. It is also common in reflection-heavy code where you truly do not care about the element type.

<Note>You may wonder: why not use `List<Object>` instead of `List<?>`? A `List<Object>` specifically accepts only a `List<Object>` — you could not pass a `List<String>` to it. `List<?>` accepts any list, regardless of element type, because `?` means "I don't know the type but I promise not to add anything unsafe."</Note>

## Upper-bounded wildcard `? extends T` — Producer

`List<? extends Number>` means "a list of some type that is `Number` or a subclass of `Number`." You can **read** elements as `Number`, but you cannot add anything:

```java
static double sum(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) {
        total += n.doubleValue();  // OK — we know each element IS-A Number
    }
    return total;
    // list.add(3);  // COMPILE ERROR — we don't know the exact type
}
```

This method accepts `List<Integer>`, `List<Double>`, `List<Long>`, or `List<BigDecimal>` — any list whose element type is a subtype of `Number`.

```java
System.out.println(sum(Arrays.asList(1, 2, 3)));          // 6.0
System.out.println(sum(Arrays.asList(0.5, 1.5, 2.0)));    // 4.0
```

The list **produces** `Number` values for the method to consume. Hence the mnemonic: **P**roducer **E**xtends.

## Lower-bounded wildcard `? super T` — Consumer

`List<? super Integer>` means "a list of some type that is `Integer` or a supertype of `Integer`." You can **add** `Integer` values to it, but you cannot read elements as anything more specific than `Object`:

```java
static void fill(List<? super Integer> list, int count, int value) {
    for (int i = 0; i < count; i++) {
        list.add(value);  // OK — Integer is always a valid element
    }
}
```

```java
List<Integer> ints    = new ArrayList<>();
List<Number>  numbers = new ArrayList<>();
List<Object>  objects = new ArrayList<>();

fill(ints,    3, 7);
fill(numbers, 2, 9);
fill(objects, 1, 0);

System.out.println(ints);     // [7, 7, 7]
System.out.println(numbers);  // [9, 9]
System.out.println(objects);  // [0]
```

The list **consumes** `Integer` values from the method. Hence: **C**onsumer **S**uper.

<Note>`? super T` is particularly common with `Comparator` and `Consumer` functional interfaces. `Collections.sort(List<T> list, Comparator<? super T> c)` uses a lower-bounded wildcard so you can pass a `Comparator<Object>` to sort a `List<String>`, because any comparison logic that works on Objects also works on Strings.</Note>

## The PECS rule

**Producer Extends, Consumer Super** — abbreviated PECS — is the rule for choosing between the two bounded wildcards:

| The collection... | Use |
| ----------------- | --- |
| produces values you will read | `? extends T` |
| consumes values you will add  | `? super T` |
| both reads and writes         | exact type `T` (no wildcard) |
| neither (just pass around)    | `?` (unbounded) |

A classic PECS example is a `copy` utility:

```java
static <T> void copy(List<? extends T> src, List<? super T> dest) {
    for (T item : src) {
        dest.add(item);
    }
}
```

`src` produces values (`? extends T`), `dest` consumes them (`? super T`). This signature lets you copy a `List<Integer>` into a `List<Number>` or a `List<Object>`.

## `List<Dog>` is NOT a `List<Animal>` — a practical illustration

```java
static void addAnimal(List<Animal> list) {
    list.add(new Cat());   // fine for List<Animal>
}

List<Dog>   dogs = new ArrayList<>();
addAnimal(dogs);   // COMPILE ERROR — List<Dog> is not List<Animal>
```

To write a method that accepts any list of Animal subtypes for **reading only**, use the upper-bounded wildcard:

```java
static void printAnimals(List<? extends Animal> list) {
    for (Animal a : list) {
        System.out.println(a.name());   // reading is fine
    }
    // list.add(new Cat());  // COMPILE ERROR — safe!
}

printAnimals(dogs);     // compiles — List<Dog> is accepted
printAnimals(cats);     // compiles — List<Cat> is accepted
```

The wildcard prevents accidental writes while allowing polymorphic reads.

## Try it

```java
import java.util.Arrays;
import java.util.ArrayList;
import java.util.List;

public class Demo {
    // Copy only elements that satisfy a condition
    static <T> void copyIf(List<? extends T> src,
                            List<? super T>   dest,
                            java.util.function.Predicate<T> pred) {
        for (T item : src) {
            if (pred.test(item)) dest.add(item);
        }
    }

    public static void main(String[] args) {
        List<Integer> source = Arrays.asList(1, 2, 3, 4, 5, 6);
        List<Number>  evens  = new ArrayList<>();

        copyIf(source, evens, n -> n % 2 == 0);
        System.out.println(evens);  // [2, 4, 6]

        List<String> words = Arrays.asList("ant", "bee", "elephant", "ox");
        List<Object> long4 = new ArrayList<>();
        copyIf(words, long4, w -> w.length() >= 4);
        System.out.println(long4);  // [elephant]
    }
}
```

Expected output:
```
[2, 4, 6]
[elephant]
```

## Common mistakes

- **Adding to an upper-bounded list** — `List<? extends Number>` is read-only from the compiler's perspective. Trying to `add` any element (except `null`) is a compile error because the actual type could be `List<Integer>`, `List<Double>`, or anything else.
- **Reading specifically from a lower-bounded list** — `List<? super Integer>` lets you add `Integer`, but `get()` returns `Object`. You must cast if you need a more specific type, which is usually a sign you need a different design.
- **Confusing `<T extends Number>` and `<? extends Number>`** — the first introduces a named type parameter you can reuse across the signature; the second is a wildcard for one specific parameter. Use a named parameter when you need to relate two positions in the signature; use a wildcard when you only need it in one place and don't need to name it.
- **Using wildcards in generic class declarations** — `class Foo<? extends Bar>` is not valid. Wildcards appear in use sites (variable types, method parameters), not in class or method-level type parameter declarations.

## Recap

Wildcards are the mechanism Java provides to handle the invariance of generic types gracefully. They let you write flexible APIs that accept families of related parameterized types without sacrificing the type safety that generics provide.

- `?` — unbounded wildcard; read as `Object`, cannot add
- `? extends T` — upper-bounded; the list produces `T` values for reading
- `? super T` — lower-bounded; the list consumes `T` values for writing
- **PECS**: Producer Extends, Consumer Super
- `List<Dog>` is NOT a `List<Animal>` — generic types are invariant to prevent silent corruption
- `Collections.copy`, `Collections.sort`, and many other library methods use PECS in their signatures
