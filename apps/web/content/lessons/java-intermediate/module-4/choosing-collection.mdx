You've now seen every major collection class in the Java Collections Framework. The last skill is the practical one: **knowing which to reach for**. This lesson provides decision frameworks, a full performance table, patterns for common real-world scenarios, and tools for protecting collections from accidental modification.

## The decision flowchart

When you need a collection, work through these questions in order:

```
1. Do you need key-value pairs?
   YES → Map
         Do you need sorted keys or range queries?
           YES → TreeMap
           NO  → HashMap (default)
                 Do you need insertion order?
                   YES → LinkedHashMap

   NO → continue...

2. Do you need to prevent duplicates automatically?
   YES → Set
         Do you need sorted order?
           YES → TreeSet
           NO  → HashSet (default)
                 Do you need insertion order?
                   YES → LinkedHashSet

   NO → continue...

3. Do you need FIFO/LIFO processing (queue or stack)?
   YES → Queue / Deque
         Large volume, single-threaded?
           YES → ArrayDeque (default)
         Need List + Deque in one?
           YES → LinkedList

   NO → continue...

4. You need an ordered sequence (duplicates OK, index access).
   → List
     Mostly appending / reading by index?
       YES → ArrayList (default)
     Mostly inserting/removing at head or via iterator in middle?
       YES → LinkedList
```

When in doubt, **ArrayList and HashMap are the defaults**. They're correct, fast, and readable. Optimise only when profiling shows a bottleneck.

## Full complexity comparison table

| Operation | ArrayList | LinkedList | HashSet | TreeSet | HashMap | TreeMap |
|-----------|-----------|------------|---------|---------|---------|---------|
| `add(e)` / `put(k,v)` (end/tail) | O(1) amort | O(1) | O(1) avg | O(log n) | O(1) avg | O(log n) |
| `add(i, e)` (middle insert) | O(n) | O(n) seek + O(1) link | — | — | — | — |
| `get(i)` / `get(k)` | O(1) | O(n) | — | — | O(1) avg | O(log n) |
| `contains(o)` / `containsKey(k)` | O(n) | O(n) | O(1) avg | O(log n) | O(1) avg | O(log n) |
| `remove(i/o)` (middle) | O(n) | O(n) seek | O(1) avg | O(log n) | O(1) avg | O(log n) |
| `remove(head/tail)` | O(n)/O(1) | O(1) | — | — | — | — |
| Sorted iteration | No | No | No | Yes | No | Yes |
| Insertion-order iteration | Yes | Yes | No | No | No | No |
| Range queries | No | No | No | Yes | No | Yes |
| Memory overhead | Low | High (2 ptrs/node) | Medium | High (tree nodes) | Medium | High (tree nodes) |

"avg" = amortized average; all hash-based structures are O(1) average, O(n) worst case (or O(log n) since Java 8's tree-bins).

## Common real-world patterns

### Pattern 1: frequency counting (histogram)

```java
import java.util.*;

public class Histogram {
    public static void main(String[] args) {
        String[] votes = {"Alice", "Bob", "Alice", "Carol", "Bob", "Alice"};
        Map<String, Integer> tally = new HashMap<>();
        for (String v : votes) tally.merge(v, 1, Integer::sum);

        // Print in sorted order
        new TreeMap<>(tally).forEach((name, count) ->
            System.out.println(name + ": " + count));
        // Alice: 3
        // Bob: 2
        // Carol: 1
    }
}
```

### Pattern 2: deduplication while preserving order

```java
List<String> raw = Arrays.asList("css", "html", "js", "html", "css", "react");
// LinkedHashSet preserves insertion order AND removes duplicates
Set<String> unique = new LinkedHashSet<>(raw);
System.out.println(unique); // [css, html, js, react]
```

### Pattern 3: grouping (multi-map)

```java
Map<String, List<String>> byDept = new HashMap<>();
String[][] employees = {
    {"Engineering", "Alice"}, {"Marketing", "Bob"},
    {"Engineering", "Carol"}, {"Marketing", "Diana"}
};
for (String[] row : employees) {
    byDept.computeIfAbsent(row[0], k -> new ArrayList<>()).add(row[1]);
}
new TreeMap<>(byDept).forEach((dept, names) -> {
    Collections.sort(names);
    System.out.println(dept + ": " + names);
});
// Engineering: [Alice, Carol]
// Marketing: [Bob, Diana]
```

### Pattern 4: nearest-neighbour / range (sorted map)

```java
TreeMap<Integer, String> errors = new TreeMap<>();
errors.put(400, "Bad Request");
errors.put(401, "Unauthorised");
errors.put(404, "Not Found");
errors.put(500, "Internal Server Error");

// All 4xx errors
System.out.println(errors.subMap(400, 500)); // {400=..., 401=..., 404=...}
```

<Note>When you find yourself writing `Collections.sort(list)` every time you need sorted output, consider whether a `TreeSet` or `TreeMap` would be cleaner — they maintain order automatically, trading O(log n) per insert for never needing a sort.</Note>

## Unmodifiable views

Once a collection is built and handed to other code, you often want to prevent accidental mutation. `Collections` provides read-only wrappers for every interface:

```java
import java.util.*;

public class Immutable {
    public static void main(String[] args) {
        List<String> mutable = new ArrayList<>(Arrays.asList("a", "b", "c"));
        List<String> readOnly = Collections.unmodifiableList(mutable);

        System.out.println(readOnly.get(0));  // a — reads work
        try {
            readOnly.add("d");                // writes throw
        } catch (UnsupportedOperationException e) {
            System.out.println("Blocked: " + e.getClass().getSimpleName());
        }

        // Note: the original is still mutable!
        mutable.add("d");
        System.out.println(readOnly.size()); // 4 — view reflects changes to original
    }
}
```

Analogous methods exist for all collection types: `unmodifiableSet`, `unmodifiableMap`, `unmodifiableSortedMap`, etc.

<Note>For a **truly immutable** collection that doesn't reflect changes to any backing list, use `List.of(...)`, `Set.of(...)`, or `Map.of(...)` (introduced in Java 9). Unlike `unmodifiableList`, these factory methods create collections with no mutable original. They also throw `NullPointerException` on any `null` element, which can be desirable as a contract enforcer.</Note>

## Java 9+ immutable factory methods

```java
// Immutable list — no backing mutable list
List<String> days = List.of("Mon", "Tue", "Wed", "Thu", "Fri");

// Immutable set
Set<String> vowels = Set.of("a", "e", "i", "o", "u");

// Immutable map
Map<String, Integer> http = Map.of(
    "OK", 200,
    "Not Found", 404,
    "Server Error", 500
);

System.out.println(days.get(2));    // Wed
System.out.println(vowels.contains("e")); // true
System.out.println(http.get("OK")); // 200
```

For maps with more than 10 entries, use `Map.ofEntries(Map.entry("k1", v1), Map.entry("k2", v2), ...)`.

## Concurrent collections preview

All standard `ArrayList`, `HashMap`, etc. are **not thread-safe**. Concurrent modification from multiple threads produces silent data corruption. When you need collections shared across threads, use `java.util.concurrent`:

| Standard | Concurrent alternative | Notes |
|----------|------------------------|-------|
| `ArrayList` | `CopyOnWriteArrayList` | Snapshot-on-write; best for rarely-modified, frequently-iterated lists |
| `HashMap` | `ConcurrentHashMap` | Fine-grained locking; high throughput |
| `LinkedList` (queue) | `LinkedBlockingQueue`, `ArrayBlockingQueue` | Thread-safe FIFO with blocking `take()` |
| `TreeMap` | `ConcurrentSkipListMap` | Sorted, thread-safe, O(log n) |

```java
import java.util.concurrent.*;

ConcurrentHashMap<String, Integer> sharedMap = new ConcurrentHashMap<>();
sharedMap.put("requests", 0);
sharedMap.merge("requests", 1, Integer::sum); // thread-safe increment
```

This is just a preview — concurrent programming deserves its own module. The key point: **never share standard collections across threads without synchronisation**.

## Choosing between List.of and Collections.unmodifiableList

| | `List.of(...)` | `Collections.unmodifiableList(list)` |
|---|---|---|
| Allows null elements | No | Yes |
| Reflects changes to original | n/a (no original) | Yes |
| Truly immutable | Yes | No (original can still change) |
| When to use | Known-at-compile-time constants | Defensive copy of a mutable list |

## Try it

```java
import java.util.*;

public class Demo {
    public static void main(String[] args) {
        // 1. Frequency count with merge
        String[] words = "the quick brown fox jumps over the lazy dog the".split(" ");
        Map<String, Integer> freq = new HashMap<>();
        for (String w : words) freq.merge(w, 1, Integer::sum);
        System.out.println("'the' appears: " + freq.get("the")); // 3

        // 2. Sorted unique words
        Set<String> sorted = new TreeSet<>(Arrays.asList(words));
        System.out.println("Unique sorted: " + sorted);

        // 3. Safe API response
        List<String> results = new ArrayList<>(Arrays.asList("Alice", "Bob"));
        List<String> safe = Collections.unmodifiableList(results);
        System.out.println("Safe list: " + safe);

        // 4. Queue draining
        Queue<Integer> q = new LinkedList<>(Arrays.asList(10, 20, 30));
        int sum = 0;
        while (!q.isEmpty()) sum += q.poll();
        System.out.println("Sum: " + sum); // 60
    }
}
```

## Common mistakes

- **Defaulting to `LinkedList` for queues** — `ArrayDeque` is almost always faster for pure queue or stack patterns. Use `LinkedList` only when you need the combined `List`+`Deque` API.
- **Using `List` for membership tests in hot code** — `list.contains(x)` is O(n). If you run this in a loop, switch to a `Set`. The pattern `new HashSet<>(list).contains(x)` inside a loop is still O(n) per call — build the set once, outside the loop.
- **Sharing `Collections.unmodifiableList` and thinking the original is protected** — the wrapper blocks mutations through the read-only reference, but the original list is still mutable. If you want true immutability, use `List.of()` or copy into `new ArrayList<>(original)` before wrapping.
- **Using `List.of()` when the list might grow** — `List.of()` and `Set.of()` are structurally immutable; you cannot add, remove, or set elements. Use them only for constants, not for collections that may be modified.
- **Ignoring thread safety** — passing a `HashMap` or `ArrayList` between threads without synchronisation causes race conditions. Even reads can be unsafe if another thread is concurrently writing. Always use `java.util.concurrent` types or explicit synchronisation.

## Recap

Choosing the right collection is a matter of asking three questions: do you need key-value pairs, do you need uniqueness, and do you need ordering? The answers map directly to a concrete implementation.

- **Default choices**: `ArrayList` for lists, `HashMap` for maps, `HashSet` for sets, `ArrayDeque` for queues
- Upgrade to a **Tree variant** when you need sorted order or range queries (O(log n) trade-off)
- Use **`LinkedHashMap` / `LinkedHashSet`** for insertion-order with hash-speed
- Protect shared collections with **`Collections.unmodifiableX`** or the **`List.of` / `Set.of` / `Map.of`** factory methods
- Never use **legacy classes** (`Vector`, `Hashtable`, `Stack`) in new code
- For **multithreaded** access, use `java.util.concurrent` classes

```java
import java.util.*;

public class Main {
    public static void main(String[] args) {
        // Fast index access
        List<Double> readings = new ArrayList<>();
        for (int i = 1; i <= 5; i++) readings.add(i * 0.5);

        Set<String> tags = new TreeSet<>(
            Arrays.asList("java", "backend", "api", "java", "rest"));
        System.out.println("Unique tags: " + tags);

        Map<String, Integer> freq = new TreeMap<>();
        for (String w : "to be or not to be that is the question".split(" "))
            freq.merge(w, 1, Integer::sum);
        System.out.println("'be' count: " + freq.get("be"));
        System.out.println("Sorted keys: " + freq.keySet());

        // FIFO queue
        Queue<String> requests = new LinkedList<>();
        requests.offer("GET /api/users");
        requests.offer("POST /api/login");
        requests.offer("DELETE /api/session");
        System.out.println("Processing: " + requests.poll());
        System.out.println("Queue size: " + requests.size());
    }
}
```

**What the tests check:**
- `readings.get(3)` returns `2.0` (0.5 × 4)

- `freq.get("be")` is `2`
- `freq.keySet()` is `[be, is, not, or, question, that, the, to]`
- First `poll()` returns `"GET /api/users"`; remaining `size()` is `2`
