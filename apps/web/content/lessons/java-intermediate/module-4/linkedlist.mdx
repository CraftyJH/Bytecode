`LinkedList` implements both `List` and `Deque`. That dual role makes it uniquely powerful — and uniquely misused. In this lesson you'll learn exactly what's inside a linked list, when those O(1) head/tail operations justify its extra memory overhead, and when to prefer `ArrayList` or `ArrayDeque` instead.

## How a doubly-linked list works

Instead of a contiguous array, a `LinkedList` is a chain of **nodes**. Each node holds three things: the element itself, a reference to the **next** node, and a reference to the **previous** node:

```
null ← [prev|"home"|next] ↔ [prev|"about"|next] ↔ [prev|"contact"|next] → null
        ↑                                                  ↑
       head                                               tail
```

The `LinkedList` object itself holds direct references to the head and tail nodes. Because of this:

- **Adding or removing at either end** is O(1): just update the head or tail pointer and fix two neighbour pointers.
- **Accessing by index** is O(n): you must walk the chain from the nearest end to reach position `i`.

Think of it like a physical chain of paperclips. Attaching one to either end is instant; finding the 47th clip requires counting along.

## Core linked-list operations

```java
import java.util.*;

public class LinkedListOps {
    public static void main(String[] args) {
        LinkedList<String> list = new LinkedList<>();

        // End operations — O(1)
        list.addLast("a");    // append to tail
        list.addLast("b");
        list.addLast("c");
        System.out.println(list);           // [a, b, c]

        // Head operations — O(1)
        list.addFirst("z");
        System.out.println(list);           // [z, a, b, c]
        System.out.println(list.removeFirst()); // z — O(1)

        // Index access — O(n) ← avoid in loops!
        System.out.println(list.get(2));    // c
    }
}
```

## The Deque interface — stack and queue in one

`LinkedList` implements `Deque` (Double-Ended Queue), which extends `Queue`. This single interface covers **three** classic data structures:

| Pattern | Methods to use | Analogy |
|---------|---------------|---------|
| **Stack (LIFO)** | `push(e)` / `pop()` / `peek()` | Stack of plates |
| **Queue (FIFO)** | `offer(e)` / `poll()` / `peek()` | Checkout line |
| **Deque** | `addFirst/Last`, `removeFirst/Last`, `peekFirst/Last` | Both ends of a pipe |

```java
import java.util.*;

public class DequeDemo {
    public static void main(String[] args) {
        // As a Stack (LIFO) — used for undo/redo, recursion simulation
        Deque<String> stack = new LinkedList<>();
        stack.push("action1");  // addFirst
        stack.push("action2");
        stack.push("action3");
        System.out.println("Undo: " + stack.pop());   // action3 (LIFO)

        // As a Queue (FIFO) — used for task queues, BFS
        Queue<String> queue = new LinkedList<>();
        queue.offer("task1");
        queue.offer("task2");
        queue.offer("task3");
        System.out.println("Process: " + queue.poll()); // task1 (FIFO)
    }
}
```

<Note>Prefer `ArrayDeque` over `LinkedList` when you only need stack or queue behavior. `ArrayDeque` is backed by a resizable circular array, which has better cache performance (memory-contiguous) and no per-node allocation overhead. Use `LinkedList` when you need both `List` and `Deque` behavior, or when you're inserting/removing in the **middle** via an iterator.</Note>

## Queue interface: the FIFO contract

`Queue` defines two pairs of methods that do the same logical thing but differ in their failure behavior:

| Operation | Throws exception | Returns null / false |
|-----------|-----------------|---------------------|
| Insert | `add(e)` | `offer(e)` |
| Remove head | `remove()` | `poll()` |
| Inspect head | `element()` | `peek()` |

Prefer the null-returning variants (`offer`, `poll`, `peek`) in production code — they let you check the return value rather than wrapping calls in try/catch:

```java
Queue<String> jobs = new LinkedList<>();
jobs.offer("compile");
jobs.offer("test");
jobs.offer("deploy");

String job;
while ((job = jobs.poll()) != null) {
    System.out.println("Running: " + job);
}
// Running: compile
// Running: test
// Running: deploy
```

## Memory overhead: the case against LinkedList for large data

Each node in a `LinkedList` allocates a separate heap object with two reference pointers (16–32 bytes of overhead per node on a typical 64-bit JVM). For an `int`, you also pay boxing cost since Java collections can't hold primitives directly.

An `ArrayList<Integer>` holding 1 million integers uses roughly 4–8 MB (array of references to boxed ints). A `LinkedList<Integer>` for the same data uses roughly 32–48 MB — four to eight times more — due to per-node allocation.

Beyond raw memory, linked nodes are scattered across the heap. CPUs rely on spatial locality (cache lines) to go fast. Walking a linked list generates many cache misses; iterating over an array almost never does. This is why even sequential iteration over a large `LinkedList` is slower in practice than over an `ArrayList`.

## When NOT to use LinkedList

- **Random access by index** — `get(i)` is O(n). A hot loop like `for (int i = 0; i < list.size(); i++) list.get(i)` is O(n²) on a `LinkedList`. Use a for-each iterator instead, or switch to `ArrayList`.
- **You only need stack/queue** — `ArrayDeque` is faster and uses less memory for pure stack or queue patterns.
- **Memory is a concern** — per-node overhead can be 3–8× the memory of the equivalent `ArrayList`.
- **You're sorting frequently** — `Collections.sort()` on a `LinkedList` copies it to an array, sorts, and copies back. It works, but ArrayList sort is meaningfully faster.

## When LinkedList is a good choice

- **Frequent insertions/deletions at the front** — O(1) `addFirst`/`removeFirst` vs O(n) shifts in `ArrayList`.
- **Queue or Deque semantics with moderate-size data** — when cache locality matters less than code clarity and you need the full `List` API alongside `Deque`.
- **You're removing elements in the middle while iterating** — `LinkedList`'s iterator `remove()` is O(1) (just pointer rewiring); `ArrayList`'s is O(n) (element shift).

## Try it

```java
import java.util.*;

public class Demo {
    public static void main(String[] args) {
        // Browser history — new pages pushed to front, "back" pops from front
        Deque<String> history = new LinkedList<>();
        history.push("https://example.com");
        history.push("https://docs.java.com");
        history.push("https://stackoverflow.com");

        System.out.println("Current: " + history.peek());  // stackoverflow
        history.pop();   // navigate back
        System.out.println("After back: " + history.peek()); // docs.java.com

        // Print-job FIFO queue
        Queue<String> printQueue = new LinkedList<>();
        printQueue.offer("invoice.pdf");
        printQueue.offer("report.pdf");
        printQueue.offer("photo.jpg");

        while (!printQueue.isEmpty()) {
            System.out.println("Printing: " + printQueue.poll());
        }
    }
}
```

## Common mistakes

- **Calling `get(i)` inside a for-index loop on a LinkedList** — this turns O(n) iteration into O(n²). Always use a for-each or iterator to traverse a `LinkedList`.
- **Using LinkedList for a pure stack** — `ArrayDeque` is faster for push/pop-only usage. Use `LinkedList` only when you also need `List` methods.
- **Confusing `push`/`pop` with `offer`/`poll`** — `push` and `pop` operate on the *front* (stack/LIFO); `offer` adds to the *tail* and `poll` removes from the *front* (queue/FIFO). Mixing them produces confusing behavior.
- **Calling `remove()` or `element()` on an empty queue** — these throw `NoSuchElementException`. Use `poll()` and `peek()` if the queue may be empty, and check for `null`.
- **Ignoring memory overhead** — linked lists consume far more memory per element than arrays. For collections with millions of elements, the difference matters.

## Recap

`LinkedList` is a doubly-linked list that implements both `List` and `Deque`. Its O(1) head/tail operations make it a natural fit for queues, stacks, and scenarios where you frequently add or remove at the ends. It pays for this flexibility with O(n) index access and higher per-element memory overhead.

- Each node holds **element + prev + next pointers** — no contiguous array
- **O(1)** for add/remove at head or tail; **O(n)** for index access and middle operations
- Implements **`Deque`** — supports stack (`push`/`pop`) and queue (`offer`/`poll`) patterns
- Prefer **`ArrayDeque`** for pure stack/queue; prefer **`ArrayList`** for random access
- Avoid index-based loops on `LinkedList` — they degrade to O(n²)
