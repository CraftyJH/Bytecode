Arrays are fine for a handful of fixed-size values. The moment you need to **grow, shrink, deduplicate, or look things up by key**, you reach for the **Collections Framework** — a family of interfaces and implementations that ships with every version of Java.

In this lesson you'll see the big picture: what interfaces exist, how they relate, when to pick which one, and what time-complexity guarantees matter in practice. The following lessons each zoom into one implementation.

## The interface hierarchy

Almost every collection class you'll ever use traces back to two roots:

```
java.lang.Iterable<T>
  └─ java.util.Collection<T>
       ├─ java.util.List<T>        (ordered, duplicates allowed)
       ├─ java.util.Set<T>         (no duplicates)
       └─ java.util.Queue<T>       (designed for FIFO/priority access)
            └─ java.util.Deque<T>  (double-ended queue)

java.util.Map<K,V>                 (key → value, NOT a Collection)
```

`Iterable` is the true root. It declares a single method, `iterator()`, which is all the enhanced for-each loop (`for (T x : collection)`) needs. `Collection` extends `Iterable` and adds the operations every collection must support: `add`, `remove`, `contains`, `size`, `isEmpty`, `clear`, and bulk variants like `addAll`.

<Note>`Map` is intentionally **not** a `Collection`. A map stores pairs, not individual elements, so bulk-collection semantics like `size()` (which count would you return — keys? values? entries?) don't map cleanly onto a flat element model. It's a parallel hierarchy.</Note>

## The main implementations at a glance

| Interface | Common implementations | Signature property |
|-----------|----------------------|--------------------|
| `List`    | `ArrayList`, `LinkedList` | Ordered, index-addressable |
| `Set`     | `HashSet`, `LinkedHashSet`, `TreeSet` | No duplicates |
| `Queue`   | `LinkedList`, `PriorityQueue`, `ArrayDeque` | Head/tail access |
| `Deque`   | `ArrayDeque`, `LinkedList` | Stack + queue in one |
| `Map`     | `HashMap`, `LinkedHashMap`, `TreeMap` | Key → value lookup |

## When to use which

Picking the right collection up front saves refactoring later. Here is a practical decision guide:

**Do you need to look things up by a key?** → `Map`

**Do you need to store pairs of associated values?** → same answer: `Map`

**Do you care about order and need direct index access?** → `List` (almost always `ArrayList`)

**Do you need to prevent duplicates automatically?** → `Set`

**Do you need to process items in arrival order (FIFO)?** → `Queue`

**Do you need a stack (LIFO) or need to push/pop both ends?** → `Deque` (use `ArrayDeque`)

A secondary question: **do you need sorted order?** If yes, favour `TreeSet` over `HashSet` or `TreeMap` over `HashMap` — but expect O(log n) operations instead of O(1).

## Time-complexity reference

Before you commit to a data structure, it's worth knowing the cost of its core operations.

| Operation | ArrayList | LinkedList | HashSet | TreeSet | HashMap | TreeMap |
|-----------|-----------|------------|---------|---------|---------|---------|
| `add` (end / put) | O(1) amortized | O(1) | O(1) avg | O(log n) | O(1) avg | O(log n) |
| `add` (middle) | O(n) | O(n) index, O(1) with iterator | — | — | — | — |
| `get` / `contains` | O(1) / O(n) | O(n) / O(n) | — / O(1) avg | — / O(log n) | O(1) avg | O(log n) |
| `remove` | O(n) | O(n) index | O(1) avg | O(log n) | O(1) avg | O(log n) |
| Iteration | O(n) | O(n) | O(n) | O(n) sorted | O(n) | O(n) sorted |

"O(1) avg" means *amortized average*. In the worst case (many hash collisions), `HashMap` operations degrade toward O(n), though Java's HashMap has mitigated this with tree-bins since Java 8.

<Note>Don't memorise the table — understand the **shape** of each structure. Hash-based collections trade ordering for O(1) speed. Tree-based collections sacrifice some speed for sorted order. Arrays/ArrayLists trade flexible sizing for O(1) random access. Once you know the shape, the complexity follows naturally.</Note>

## A first look at all three roots

```java
import java.util.*;

public class CollectionsIntro {
    public static void main(String[] args) {
        // List: ordered, duplicates OK
        List<String> cities = new ArrayList<>();
        cities.add("London");
        cities.add("Paris");
        cities.add("London");   // duplicate allowed
        System.out.println(cities);         // [London, Paris, London]
        System.out.println(cities.get(1));  // Paris (index access)

        // Set: no duplicates
        Set<String> uniqueCities = new HashSet<>(cities);
        System.out.println(uniqueCities);   // [London, Paris] (order may vary)

        // Map: key → value
        Map<String, Integer> population = new HashMap<>();
        population.put("London", 9_000_000);
        population.put("Paris",  2_100_000);
        System.out.println(population.get("Paris")); // 2100000
    }
}
```

## Polymorphism through interfaces

One of the most important practices in the Collections Framework is **programming to the interface, not the implementation**:

```java
// Prefer this:
List<String> names = new ArrayList<>();

// Over this:
ArrayList<String> names = new ArrayList<>();
```

By declaring the variable as `List<String>`, you can swap in a `LinkedList` (or a mock in a test) without changing any code that uses `names`. This is the Liskov Substitution Principle in everyday practice.

The same applies to method signatures:

```java
// Accepts any List — flexible
void process(List<String> items) { ... }

// Unnecessarily locked to ArrayList
void process(ArrayList<String> items) { ... }
```

## Try it

```java
import java.util.*;

public class Demo {
    public static void main(String[] args) {
        // Start with a list that has duplicates
        List<Integer> scores = new ArrayList<>(Arrays.asList(88, 95, 72, 95, 88, 100));
        System.out.println("All scores: " + scores);         // [88, 95, 72, 95, 88, 100]

        // Deduplicate with a Set
        Set<Integer> unique = new LinkedHashSet<>(scores);   // LinkedHashSet preserves insertion order
        System.out.println("Unique scores: " + unique);      // [88, 95, 72, 100]

        // Count occurrences with a Map
        Map<Integer, Integer> freq = new HashMap<>();
        for (int s : scores) {
            freq.put(s, freq.getOrDefault(s, 0) + 1);
        }
        System.out.println("Frequency of 95: " + freq.get(95)); // 2
    }
}
```

## Legacy collections to avoid

Java has been around since 1995, and several collection classes predate the modern framework:

| Legacy class | Modern replacement | Why avoid? |
|---|---|---|
| `Vector` | `ArrayList` | Synchronized on every call — expensive when you don't need thread safety |
| `Hashtable` | `HashMap` | Same synchronization problem; doesn't allow `null` keys |
| `Stack` | `ArrayDeque` | Extends `Vector` (inherits its baggage); `Deque` is faster and more principled |
| `Enumeration` | `Iterator` | Older cursor API with fewer capabilities |

You will still see these in legacy codebases, so knowing them is useful — just don't write new code with them.

<Note>If you need a **thread-safe** collection, reach for `java.util.concurrent`: `ConcurrentHashMap`, `CopyOnWriteArrayList`, or `BlockingQueue` implementations. These are designed for concurrency and are far more efficient than the old synchronized wrappers.</Note>

## The Collections utility class

`java.util.Collections` (note the plural) is a helper class full of static methods that operate on any `Collection` or `List`:

```java
import java.util.*;

public class UtilDemo {
    public static void main(String[] args) {
        List<Integer> nums = new ArrayList<>(Arrays.asList(3, 1, 4, 1, 5, 9, 2, 6));

        System.out.println(Collections.min(nums));   // 1
        System.out.println(Collections.max(nums));   // 9
        Collections.sort(nums);
        System.out.println(nums);                    // [1, 1, 2, 3, 4, 5, 6, 9]
        Collections.reverse(nums);
        System.out.println(nums);                    // [9, 6, 5, 4, 3, 2, 1, 1]
        Collections.shuffle(nums);                   // random order each run
    }
}
```

Think of `Collections` as the Swiss Army knife that wraps around any collection the framework defines.

## Common mistakes

- **Using raw types** (`List names = new ArrayList()`) — without the generic type parameter you lose compile-time type checking and get unchecked-cast warnings everywhere. Always supply the type: `List<String> names = new ArrayList<>()`.
- **Reaching for `Vector` or `Hashtable`** — they're slow by default due to blanket synchronization. Use `ArrayList` or `HashMap` in single-threaded code. For concurrent code, use `java.util.concurrent` classes.
- **Declaring the concrete type** (`ArrayList<String> x = new ArrayList<>()`) instead of the interface (`List<String> x`). This locks you into one implementation and makes refactoring harder.
- **Confusing `Collections` (utility class) with `Collection` (interface)** — one has an "s". `Collections.sort()` is the utility method; `Collection` is the interface your list implements.
- **Treating Map as a Collection** — `Map` doesn't implement `Collection`, so you can't pass a `Map` where a `Collection` is expected. Use `.entrySet()`, `.keySet()`, or `.values()` to get a proper view.

## Recap

The Collections Framework gives you a coherent, interface-driven toolbox for every data-structure need. The key insight is that each interface (`List`, `Set`, `Queue`, `Map`) represents a *contract* — a set of behaviours — and each concrete class fulfils that contract with a specific internal structure tuned for different trade-offs.

- **`Iterable → Collection → List/Set/Queue`** form the main hierarchy; `Map` is separate
- **`List`** keeps insertion order and allows duplicates; great for sequences you access by index
- **`Set`** enforces uniqueness; great for membership tests and deduplication
- **`Map`** pairs keys with values; great for lookup tables and counting
- **`Queue`/`Deque`** model processing pipelines and stacks
- Always program to **interfaces**, not concrete types
- Avoid **legacy classes** (`Vector`, `Hashtable`, `Stack`) in new code
