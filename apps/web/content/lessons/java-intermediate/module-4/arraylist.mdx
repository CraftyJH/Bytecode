`ArrayList` is the workhorse of the Java Collections Framework. In most programs, when you need a `List`, `ArrayList` is the right choice. This lesson explains **why**: what it looks like internally, what each operation actually costs, and the pitfalls that catch new Java developers off guard.

## How ArrayList works internally

An `ArrayList` is a **resizable array**. Internally it holds a plain `Object[]` array and an integer `size` that tracks how many logical elements are in the list (which may be less than the array's physical length).

```
ArrayList<String>
  Object[] elementData = ["cat", "dog", "bird", null, null]
  int size = 3
  int capacity = 5   (elementData.length — not directly visible)
```

This is different from `capacity`: `size` is how many elements you've added; `capacity` is how large the backing array currently is. When `size` reaches `capacity`, the list **grows**.

## Capacity doubling

When the backing array is full and you call `add()`, `ArrayList` allocates a new, larger array (roughly 1.5× the current capacity in OpenJDK), copies all existing elements into it, and then appends the new element:

```
Before:  [a, b, c, d, e]   capacity=5, size=5
add("f") → new array of length ~8, copy a–e, append f
After:   [a, b, c, d, e, f, _, _]   capacity=8, size=6
```

Each individual resize is O(n), but it happens so rarely (the list has to fill up first) that if you average the cost across all the `add()` calls that preceded it, each `add` costs O(1) **amortized**. This is the same analysis as a bank account: make small deposits regularly, occasionally make one big withdrawal — average cost per transaction is still small.

<Note>If you **know** in advance how many elements you'll add, construct the list with an initial capacity to avoid any resize at all: `new ArrayList<>(1000)`. This is a micro-optimization — only worth it for very large lists in tight loops.</Note>

## Operation complexity

| Operation | Time complexity | Why |
|-----------|----------------|-----|
| `get(i)` | O(1) | Direct array index |
| `set(i, e)` | O(1) | Direct array write |
| `add(e)` (end) | O(1) amortized | Occasional resize |
| `add(i, e)` (middle) | O(n) | Must shift elements right |
| `remove(i)` (middle) | O(n) | Must shift elements left |
| `remove(Object o)` | O(n) | Linear scan to find it, then shift |
| `contains(o)` / `indexOf(o)` | O(n) | Linear scan |
| `size()` | O(1) | Stored field |

The critical insight: `ArrayList` is **fast at the tail** (add/remove last) and **slow in the middle** (insert/remove forces a shift of all subsequent elements).

## Core operations

```java
import java.util.*;

public class ArrayListOps {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();

        // Adding
        list.add("apple");              // appends at end
        list.add("banana");
        list.add(0, "mango");           // insert at index 0 — shifts others right
        System.out.println(list);       // [mango, apple, banana]

        // Reading
        System.out.println(list.get(1));           // apple
        System.out.println(list.indexOf("banana")); // 2
        System.out.println(list.contains("kiwi"));  // false

        // Updating
        list.set(1, "pear");
        System.out.println(list);                   // [mango, pear, banana]

        // Removing
        list.remove("mango");           // by value — O(n)
        list.remove(0);                 // by index — O(n) due to shift
        System.out.println(list);       // [banana]
    }
}
```

## subList and bulk operations

`subList(from, to)` returns a **live view** of a portion of the list — not a copy. Modifying the sublist modifies the original, and modifying the original after taking a sublist causes a `ConcurrentModificationException` if you touch the sublist:

```java
List<Integer> nums = new ArrayList<>(Arrays.asList(10, 20, 30, 40, 50));
List<Integer> middle = nums.subList(1, 4);  // view of [20, 30, 40]
middle.set(0, 99);
System.out.println(nums);   // [10, 99, 30, 40, 50] — original changed!

// To get an independent copy:
List<Integer> copy = new ArrayList<>(nums.subList(1, 4));
```

<Note>`subList` is a view, not a copy. This is intentional — it avoids allocating a new array. But it means you must be careful: if you need an independent copy, wrap the sublist in `new ArrayList<>()`.</Note>

## The Collections utility class with ArrayList

`java.util.Collections` provides powerful static helpers that work on any `List`:

```java
import java.util.*;

public class UtilMethods {
    public static void main(String[] args) {
        List<Integer> nums = new ArrayList<>(Arrays.asList(3, 7, 1, 9, 4, 1, 7));

        Collections.sort(nums);
        System.out.println("Sorted: " + nums);           // [1, 1, 3, 4, 7, 7, 9]

        Collections.reverse(nums);
        System.out.println("Reversed: " + nums);         // [9, 7, 7, 4, 3, 1, 1]

        System.out.println("Min: " + Collections.min(nums));   // 1
        System.out.println("Max: " + Collections.max(nums));   // 9
        System.out.println("Freq of 7: " + Collections.frequency(nums, 7)); // 2

        Collections.shuffle(nums);   // random order
        System.out.println("Shuffled: " + nums);
    }
}
```

`Collections.sort()` uses a Timsort algorithm — O(n log n) — and is stable (equal elements keep their relative order). For custom ordering, pass a `Comparator`: `Collections.sort(list, Comparator.reverseOrder())`.

## Iterator invalidation

A critical rule: **do not modify an `ArrayList` structurally while iterating over it with an iterator or for-each loop**. Doing so throws `ConcurrentModificationException`:

```java
List<String> words = new ArrayList<>(Arrays.asList("keep", "remove", "keep", "remove"));

// BAD — ConcurrentModificationException
for (String w : words) {
    if (w.equals("remove")) words.remove(w);  // throws!
}

// GOOD — use removeIf (Java 8+)
words.removeIf(w -> w.equals("remove"));
System.out.println(words);  // [keep, keep]

// GOOD — use an explicit Iterator
Iterator<String> it = words.iterator();
while (it.hasNext()) {
    if (it.next().equals("keep")) it.remove();  // safe
}
```

`ArrayList` uses a **modCount** counter. Every structural change (add, remove, clear) increments it. When an iterator sees that the modCount changed since it was created, it throws `ConcurrentModificationException` as a fail-fast safety mechanism.

## Try it

```java
import java.util.*;

public class Demo {
    public static void main(String[] args) {
        List<String> names = new ArrayList<>(Arrays.asList("Charlie", "Alice", "Bob", "Diana", "Alice"));

        // Frequency
        System.out.println("Alice count: " + Collections.frequency(names, "Alice")); // 2

        // Sort alphabetically
        Collections.sort(names);
        System.out.println("Sorted: " + names);  // [Alice, Alice, Bob, Charlie, Diana]

        // Remove all duplicates via LinkedHashSet
        List<String> unique = new ArrayList<>(new LinkedHashSet<>(names));
        System.out.println("Unique: " + unique); // [Alice, Bob, Charlie, Diana]

        // Unmodifiable view
        List<String> locked = Collections.unmodifiableList(unique);
        System.out.println("First: " + locked.get(0)); // Alice
        // locked.add("Eve");  // would throw UnsupportedOperationException
    }
}
```

## Common mistakes

- **Removing by index vs by value** — `remove(int i)` removes the element *at* index `i`; `remove(Object o)` removes the *first element equal to* `o`. With a `List<Integer>`, `remove(2)` removes index 2, but `remove(Integer.valueOf(2))` removes the value 2. If in doubt, be explicit and box the value.
- **Removing inside a for-each loop** — this always throws `ConcurrentModificationException`. Use `removeIf()` or an explicit `Iterator.remove()`.
- **Forgetting subList is a live view** — modifications through the sublist change the original. If you want an independent slice, copy it: `new ArrayList<>(list.subList(a, b))`.
- **Pre-allocating with `new ArrayList<>(n)`** — confusingly, passing an integer to the constructor sets the *initial capacity*, not the *size*. The list is still empty; `get(0)` would throw. Pre-sizing avoids resizes but doesn't pre-fill.
- **Using ArrayList when frequent middle-insertions are needed** — if your code mostly inserts/removes near the beginning or middle, reconsider whether a different structure (a `LinkedList` or a different algorithm) is more appropriate.

## Recap

`ArrayList` is the right default `List` in the vast majority of situations because random access (get/set) and end-append are both O(1). Its internal capacity-doubling mechanism ensures you rarely pay the resize cost in practice.

- Backed by a **resizable array**; capacity doubles (roughly) when full
- **O(1)** for `get`, `set`, and end-`add`; **O(n)** for insertions and removals anywhere except the end
- Use `Collections.sort/reverse/shuffle/frequency/min/max` for list-level operations
- Never structurally modify while iterating — use `removeIf()` or `Iterator.remove()`
- `subList()` returns a **view**, not a copy
