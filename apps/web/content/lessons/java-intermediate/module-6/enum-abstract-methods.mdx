In the previous lesson we gave an enum *data*: each constant carried its own value in `private final` fields. That works beautifully when the data alone is enough to drive behaviour — `cents()` just returns a stored number. But sometimes the work each constant has to do is genuinely different: addition is not subtraction, and a JSON serialiser is not a YAML serialiser. For these cases Java lets each enum constant carry its own implementation of a method through **abstract methods with per-constant bodies**.

This is the closest thing Java's enums have to the classical Strategy pattern, and it is one of the most elegant features in the language once you see it in action.

## The pattern

Declare the method as `abstract` on the enum itself, then provide a body inside each constant's `{ ... }` block:

```java
enum Op {
    ADD { public int apply(int a, int b) { return a + b; } },
    SUB { public int apply(int a, int b) { return a - b; } },
    MUL { public int apply(int a, int b) { return a * b; } };

    public abstract int apply(int a, int b);
}
```

Three rules govern this construct:

1. The method is declared `abstract` once on the enum.
2. **Every** constant must supply a body. Forgetting one constant is a compile error.
3. Each body lives inside the curly braces immediately after the constant name, and is separated from the next constant by a comma — not a semicolon.

The semicolon after the last constant body is the same one we met in lesson 2 — the boundary between the constant list and the rest of the enum body.

## What is really happening: anonymous subclasses

Under the hood, the compiler treats each per-constant body as an *anonymous subclass* of the enum. Conceptually, this:

```java
enum Op {
    ADD { public int apply(int a, int b) { return a + b; } };
    public abstract int apply(int a, int b);
}
```

…is equivalent to writing one Op subclass per constant:

```java
abstract class Op {
    public abstract int apply(int a, int b);
}

class OpAdd extends Op {                          // synthesized as Op$1
    public int apply(int a, int b) { return a + b; }
}

// …and ADD is the single instance of OpAdd.
```

You will see the synthetic class files (`Op$1.class`, `Op$2.class`, ...) in your compiled output. You cannot reference those types by name, and you do not need to — the public face is still just `Op`.

<Note>Because the constants are subclass instances, calling `getClass()` on them returns the synthetic subclass: `Op.ADD.getClass()` is *not* `Op.class`. If you need the declaring enum type, use `getDeclaringClass()` instead.</Note>

## A second example: a tiny logger

Imagine three log levels, each formatting messages a little differently:

```java
enum LogLevel {
    INFO {
        public String format(String msg) { return "[INFO] " + msg; }
    },
    WARN {
        public String format(String msg) { return "[WARN] *** " + msg + " ***"; }
    },
    ERROR {
        public String format(String msg) {
            return "[ERROR] !!! " + msg.toUpperCase() + " !!!";
        }
    };

    public abstract String format(String msg);
}

System.out.println(LogLevel.INFO.format("started"));     // [INFO] started
System.out.println(LogLevel.WARN.format("low disk"));    // [WARN] *** low disk ***
System.out.println(LogLevel.ERROR.format("crashed"));    // [ERROR] !!! CRASHED !!!
```

If a developer adds `DEBUG` tomorrow, the compiler immediately complains until they add a `format` body. The IDE points at the constant, not at some far-away `switch`.

## Field-based dispatch vs. abstract methods

There is more than one way to model per-constant behaviour. Compare these two designs of the same operator enum:

### Field-based dispatch

```java
enum Op {
    ADD("+"), SUB("-"), MUL("*");

    private final String symbol;
    Op(String symbol) { this.symbol = symbol; }

    public int apply(int a, int b) {
        switch (this) {
            case ADD: return a + b;
            case SUB: return a - b;
            case MUL: return a * b;
            default:  throw new AssertionError("Unhandled: " + this);
        }
    }
}
```

### Abstract method dispatch

```java
enum Op {
    ADD("+") { public int apply(int a, int b) { return a + b; } },
    SUB("-") { public int apply(int a, int b) { return a - b; } },
    MUL("*") { public int apply(int a, int b) { return a * b; } };

    private final String symbol;
    Op(String symbol) { this.symbol = symbol; }

    public abstract int apply(int a, int b);
}
```

Both compile, both produce the same output. The differences appear when the codebase grows.

| Concern                | `switch` on `this`                                      | Abstract method bodies                                       |
| ---------------------- | ------------------------------------------------------- | ------------------------------------------------------------ |
| Adding a constant      | Easy to forget the new `case`; falls through to default | Compiler error until you add the body                        |
| Reading the logic      | One method shows everything in one place                | Logic lives next to each constant — local reasoning          |
| Trivial operations     | Compact and fine                                        | A bit more boilerplate                                       |
| Non-trivial operations | `switch` arm grows long, and `default` hides bugs       | Each body is self-contained; can use helpers freely          |

The conventional rule of thumb: prefer a `switch` (or stored-function field) when the per-constant logic is one short line; prefer abstract methods when the logic is several lines, when there are multiple methods that each constant must implement, or when correctness on growth matters more than terse code.

<Note>If you do go the `switch` route, always include a `default` that throws `AssertionError`. It will fire loudly the first time someone adds a constant without updating the switch.</Note>

## Combining fields and abstract methods

The two techniques compose freely. Constants can carry data *and* override methods at the same time:

```java
enum Op {
    ADD("+") { public int apply(int a, int b) { return a + b; } },
    SUB("-") { public int apply(int a, int b) { return a - b; } },
    MUL("*") { public int apply(int a, int b) { return a * b; } };

    private final String symbol;
    Op(String symbol) { this.symbol = symbol; }

    public String symbol() { return symbol; }
    public abstract int apply(int a, int b);
}

System.out.println(Op.MUL.symbol() + " => " + Op.MUL.apply(3, 4));   // * => 12
```

The constructor argument is supplied *before* the body, in the form `CONSTANT(args) { body }`.

## Beyond binary operators: a real-world analogy

Think of each enum constant with a per-constant body as a worker on an assembly line. The line tells every worker "do your job," but each worker performs a different specialty: the welder welds, the painter paints, the inspector inspects. The orchestrator only cares that everyone has a `doYourJob()` method. From the outside it looks uniform; on the inside, each worker's body is different. That is exactly what an abstract enum method buys you in code.

## Strategy pattern in miniature

In classical OOP the Strategy pattern uses a separate class per strategy plus a factory or registry to choose between them. With enum abstract methods, the strategies *are* the constants: there is no registry to maintain, no naming dance, and the closed set is enforced by the type system. The downside — and it is real — is that you cannot add a new strategy without modifying the enum. Use this pattern when the set of strategies is genuinely closed and stable.

## Common mistakes

- **Forgetting a body for a new constant.** Adding `DIV` without a `{ public int apply(...) ... }` block produces a compile error. The error message is your friend; read it carefully.
- **Putting the body in the wrong place.** The body belongs immediately after the constant name, before the comma. Writing it after the constructor arguments works (`MUL("*") { ... }`), but writing it before the arguments does not.
- **Using `switch` on `this` without a `default` that throws.** A silent fall-through is the source of most "we shipped a constant without handling it" bugs. Always throw `AssertionError` from `default`.
- **Calling `getClass()` on a per-constant body and being surprised.** Because the body is an anonymous subclass, `getClass()` returns the synthetic `Op$1`. Use `getDeclaringClass()` when you want the enum type itself.
- **Reaching for abstract methods when one shared implementation would do.** If the logic is identical for every constant, a regular concrete method is simpler and more honest.

## Recap

Per-constant abstract methods turn an enum into a tiny, type-safe family of strategies.

- Declare the method `abstract` on the enum and **every** constant must override it.
- Each constant body is syntactic sugar for an anonymous subclass of the enum.
- The compiler forces completeness: forgetting a body is a compile error, not a runtime surprise.
- Constants can combine constructor arguments and method bodies — `MUL("*") { ... }`.
- Prefer this style when per-constant logic is non-trivial or when correctness on growth matters; a `switch` with a throwing `default` is fine for one-line behaviours.
