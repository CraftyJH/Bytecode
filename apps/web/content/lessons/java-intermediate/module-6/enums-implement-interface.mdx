So far in this module, every method on our enums has been declared inside the enum itself. That makes the enums useful, but isolated: they cannot fit into a function that already accepts some other type. The fix is the same one we use everywhere else in object-oriented Java — declare an interface, and have the enum **implement** it. Suddenly the enum becomes a citizen of any API that speaks that interface.

This lesson covers the syntax, the two main patterns for satisfying the contract (centrally vs. per-constant), and a few of the standard interfaces that enums implement implicitly so you can rely on them out of the box.

## The syntax is exactly what you would guess

```java
interface Greetable {
    String greet();
}

enum Lang implements Greetable {
    ENGLISH { public String greet() { return "Hello"; } },
    SPANISH { public String greet() { return "Hola"; } },
    FRENCH  { public String greet() { return "Bonjour"; } };
}
```

`enum Lang implements Greetable` is the same `implements` keyword you use on a regular class. Each constant supplies a `greet()` body, just like in the previous lesson on abstract methods. The interface declares no body for `greet()`, so the enum body must — either via per-constant overrides as shown, or via a single concrete method shared by all constants.

<Note>An enum cannot `extends` another class. It implicitly extends `java.lang.Enum`, and Java has no multiple inheritance for classes. Interfaces are the only extension axis for enums — but you can implement as many as you need.</Note>

## Two ways to satisfy the contract

Once you have `enum Lang implements Greetable`, you must provide a body for `greet()`. There are two clean ways to do that.

### Pattern A — per-constant bodies

Each constant overrides the method individually. Use this when the behavior really does differ per constant:

```java
enum Lang implements Greetable {
    ENGLISH { public String greet() { return "Hello"; } },
    SPANISH { public String greet() { return "Hola"; } },
    FRENCH  { public String greet() { return "Bonjour"; } };
}
```

This is identical to the abstract-method pattern from lesson 3 — except the abstract method comes from an external interface instead of being declared inside the enum.

### Pattern B — single concrete implementation

If every constant would do the same thing, write one regular method:

```java
enum Lang implements Greetable {
    ENGLISH("Hello"), SPANISH("Hola"), FRENCH("Bonjour");

    private final String word;
    Lang(String word) { this.word = word; }

    @Override
    public String greet() {
        return word;
    }
}
```

The data lives in a `private final` field (lesson 2 territory), and a single `greet()` returns it. There is no duplication and the language list is easy to extend.

The right choice is dictated by whether the behavior is uniform. If `greet()` boils down to "return the stored word," Pattern B wins. If `greet()` does something genuinely different per constant — different formatting, different side effects — Pattern A wins.

## Why interfaces matter for enums

The reason to bother is **interoperability**. Once `Lang implements Greetable`, every API that accepts a `Greetable` accepts a `Lang` constant, and the API does not need to know anything about enums:

```java
static void say(Greetable g) {
    System.out.println(g.greet());
}

say(Lang.FRENCH);                   // Bonjour

List<Greetable> all = new ArrayList<>(Arrays.asList(Lang.values()));
all.forEach(g -> System.out.println(g.greet()));
```

The `say` function knows nothing about `Lang`. Tomorrow you can add a `Robot implements Greetable` class with a totally different greeting strategy, and the same function works without changes. This is the same "code to an interface" payoff that any other implementing class provides.

## Built-in interfaces every enum already implements

Even without writing `implements` yourself, every enum already satisfies two interfaces from the standard library:

- **`Comparable<E>`** — through `java.lang.Enum`, every constant compares to others of the same type by `ordinal()`. That means you can sort, use them as keys in `TreeMap`, or call `Collections.min`/`max` for free.
- **`Serializable`** — enums serialize cleanly without any extra effort, and deserialization preserves the singleton invariant.

```java
List<Lang> langs = new ArrayList<>(Arrays.asList(Lang.SPANISH, Lang.ENGLISH, Lang.FRENCH));
Collections.sort(langs);                                   // uses natural order
System.out.println(langs);                                 // [ENGLISH, SPANISH, FRENCH]
```

<Note>The natural order is **declaration order**, not alphabetical. If you want a different ordering, supply a `Comparator` to `sort` or `TreeMap`. Avoid relying on natural order for business meaning, for the same reason you avoid relying on `ordinal()`.</Note>

## A more substantial example: a validator pipeline

Suppose your application validates user input. Define a tiny interface and an enum of named rules:

```java
interface Validator {
    boolean isValid(String input);
}

enum Rule implements Validator {
    NON_EMPTY {
        public boolean isValid(String input) {
            return input != null && !input.isEmpty();
        }
    },
    EMAIL {
        public boolean isValid(String input) {
            return input != null && input.contains("@");
        }
    };
}

static boolean checkAll(String input, List<Validator> rules) {
    for (Validator r : rules) {
        if (!r.isValid(input)) return false;
    }
    return true;
}
```

The pipeline does not need to know about enums — it just consumes `Validator`s. Tomorrow you can add a non-enum `RegexRule implements Validator` and pass it to the same `checkAll`.

## Implementing multiple interfaces

Enums can implement as many interfaces as needed, separated by commas — exactly like a class:

```java
interface HasCode { int code(); }
interface Greetable { String greet(); }

enum Lang implements Greetable, HasCode {
    ENGLISH("Hello", 1),
    SPANISH("Hola",  2),
    FRENCH ("Bonjour", 3);

    private final String word;
    private final int code;
    Lang(String word, int code) {
        this.word = word;
        this.code = code;
    }

    public String greet() { return word; }
    public int code()     { return code; }
}
```

`Lang.SPANISH` is now a `Greetable`, a `HasCode`, *and* a `Comparable<Lang>`. Each role is independent and can be exercised in different parts of the system.

## Refactoring drill: from per-constant bodies to a shared implementation

Look at the starter code. The three greetings are just three different strings — there is no real per-constant *logic*. A common refactor is to lift the data into a field and replace the bodies with a single concrete method:

```java
enum Lang implements Greetable {
    ENGLISH("Hello"),
    SPANISH("Hola"),
    FRENCH ("Bonjour");

    private final String word;
    Lang(String word) { this.word = word; }

    @Override
    public String greet() { return word; }
}
```

Identical behavior, less code, and adding a new language is a single line. This is a small but real example of the rule of thumb from lesson 3: prefer per-constant bodies when the *logic* differs, prefer a shared method when only the *data* differs.

## A real-world analogy: USB devices

A USB port is an interface — a strict, published contract about pins and protocols. A keyboard, a mouse, and a flash drive all *implement* that interface in their own way, but the laptop only knows about USB. An enum implementing an interface is the same idea: the enum constants are like a fixed catalogue of devices, all of which speak the standard protocol so the rest of the program can treat them uniformly.

## Common mistakes

- **Trying to `extends` another class.** Enums implicitly extend `java.lang.Enum` and cannot extend anything else. Use an interface to share behavior across enums and other classes.
- **Forgetting `public` on the interface methods.** Interface methods are `public` by definition. The compiler rejects any implementation that is less accessible. Always write `public` explicitly on the override.
- **Using per-constant bodies when the behavior is uniform.** It compiles, but it duplicates code. If every constant returns its stored field, write a single concrete method instead.
- **Relying on the natural ordering for business meaning.** Enums implement `Comparable` via `ordinal()`. If you reorder constants, your sorted output changes. Provide an explicit `Comparator` for any ordering with real meaning.
- **Assuming you cannot mix abstract and concrete methods.** You absolutely can. Implement the interface centrally for shared work and add per-constant bodies for the parts that genuinely differ.

## Recap

Implementing an interface promotes an enum from a self-contained catalogue into a first-class participant in your application's APIs.

- Use `enum X implements I { ... }`, optionally with a comma-separated list of interfaces.
- Provide method bodies either **per constant** (when the logic differs) or **once on the enum** (when only the data differs).
- Every enum already implements `Comparable<E>` and `Serializable` for free.
- Implementing an interface lets enum constants flow through code that knows nothing about enums — pure decoupling.
- An enum cannot `extends` another class, but interfaces give it all the polymorphism it needs.
