Every class in Java silently inherits from `Object` — the universal parent — and therefore inherits three methods you'll override over and over again: `toString()`, `equals()`, and `hashCode()`. The defaults are usually wrong for your data, and getting them right is the difference between a class that "just works" and one that surprises you.

## `toString` — the printable form

By default, `Object.toString()` returns something like `Point@1b6d3586` — the class name and a memory hash. Useless for humans. Override it to produce a readable representation:

```java
class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override
    public String toString() {
        return "(" + x + ", " + y + ")";
    }
}

System.out.println(new Point(3, 4));   // (3, 4)
```

`println` and string concatenation both call `toString()` automatically. The moment you override it, your objects start printing usefully everywhere — debugger tooltips, log statements, error messages.

A good `toString` includes enough information to identify the object in a debug log, but isn't so verbose it overwhelms the line.

## `equals` — value equality

The default `Object.equals()` is just `==` — it returns `true` only when both references point to the same object. For most data classes that's the wrong question. You usually want **value equality**: two Points are equal when they have the same x and y, even if they're different objects.

```java
class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;          // optional fast-path
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }
}

new Point(1, 2).equals(new Point(1, 2));   // true
```

The signature must take `Object`, not `Point`. If you write `equals(Point p)`, you've **overloaded** the inherited method instead of overriding it — and frameworks that call `equals(Object)` won't go through your version. Always annotate with `@Override` so the compiler catches this.

The structure of a correct `equals`:

1. Quick reference check (`this == o`) — optional optimization.
2. Type check via `instanceof`.
3. Cast to your type.
4. Compare the fields that matter.

## `hashCode` — and the contract that ties them together

Hash-based containers (`HashMap`, `HashSet`) work by bucketing objects according to their `hashCode`. Two objects that are `.equals()` *must* return the same `hashCode`, or those containers will lose track of them.

```java
class Point {
    int x, y;
    // ...constructor, equals from above...

    @Override
    public int hashCode() {
        return java.util.Objects.hash(x, y);
    }
}
```

`Objects.hash(...)` combines the given values into a reasonable hash. You can write hash code by hand (`return 31 * x + y;`), but `Objects.hash` is shorter and uses the same fields you'd use in `equals` — making it easy to keep them in sync.

The contract:

- If `a.equals(b)`, then `a.hashCode() == b.hashCode()` — **required**
- If `a.hashCode() == b.hashCode()`, the objects might *or might not* be equal — **collisions are allowed**

Get this wrong and you'll have bugs that look like memory corruption: you put a key in a HashMap, then `get` returns null even though the equal key is right there.

<Note>A very common error: override `equals` but forget `hashCode`. Everything seems fine until you add the object to a `HashSet` or use it as a `HashMap` key — then it silently fails to find it. Always override both.</Note>

## When to override the Big Three

Override all three together for any class whose objects you want to compare by value, store in a `Set`, or use as `Map` keys. For purely behavioral classes (a `Logger`, a `RequestHandler`), the defaults are usually fine — there's no meaningful "equal Logger" concept.

Modern Java has `record` types that auto-generate all three for you. We'll cover those later — for now, learn to write them by hand, because you'll see the pattern in every Java codebase.

## Try it

```java
public class Demo {
    public static void main(String[] args) {
        Point a = new Point(2, 3);
        Point b = new Point(2, 3);
        System.out.println(a);              // (2, 3)
        System.out.println(a.equals(b));    // true
        System.out.println(a.hashCode() == b.hashCode());   // true
    }
}

class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override public String toString() { return "(" + x + ", " + y + ")"; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }

    @Override public int hashCode() { return java.util.Objects.hash(x, y); }
}
```

## Common mistakes

- **`equals(Point p)` instead of `equals(Object o)`** → that's an overload, not an override. The framework's `equals(Object)` calls won't reach it. `@Override` will catch this mistake immediately — use it on every intended override.
- **Overriding `equals` but not `hashCode`** → your objects appear equal but vanish into the wrong bucket of any HashMap or HashSet. The rule is simple: if you touch `equals`, touch `hashCode` in the same commit.
- **Comparing strings with `==` inside `equals`** → for object fields like Strings, use `.equals()`. Only primitives (`int`, `double`, `boolean`, etc.) use `==` for value comparison.
- **Including transient or derived state in equals** → only compare the fields that define the identity of the value. A cached computation, a logging counter, or a timestamp-of-last-access aren't part of "what makes this object equal to that one".

## Recap

The "Big Three" methods are what make Java objects play nicely with the rest of the ecosystem. Libraries, frameworks, and the JDK itself all call `equals`, `hashCode`, and `toString` — often implicitly. Getting them right is part of making your class a first-class citizen of the Java world.

- Every class inherits `toString`, `equals`, and `hashCode` from `Object`
- Override `toString` for readable output in logs, debuggers, and println
- Override `equals(Object)` for value equality — guard with `instanceof` and cast inside
- Override `hashCode` consistently with `equals` — equal objects must hash to the same value
- Always annotate overrides with `@Override` so the compiler catches signature mistakes
