So far every field and method you've written has been per-object: each Dog has its own `name`, each Counter its own `value`. Sometimes you want the opposite — one piece of data or one method that belongs to the **class**, not to any particular instance. That's what `static` is for.

A helpful mental model: imagine a company's HR database. Each employee record holds that person's name, salary, and title — those are instance fields, one per employee. But there's also a single counter of total employees, and a company-wide method for computing tax brackets. Those are static — they belong to the company (class), not to any individual.

## Static fields

A static field exists once, no matter how many objects you create:

```java
class User {
    static int count = 0;     // one shared int, on the class
    String name;              // one per User

    User(String name) {
        this.name = name;
        count++;              // increment the shared counter
    }
}
```

Every `new User(...)` runs the constructor, which increments the same shared `count`. To read it from outside, address the class directly:

```java
new User("Ada");
new User("Linus");
new User("Grace");

System.out.println(User.count);   // 3
```

You'll occasionally see `someUser.count` — Java allows it, but it's misleading. Always access static fields through the class name (`User.count`) so it's obvious the field is shared.

## Static methods

A static method belongs to the class. It has no `this`, so it cannot read or write instance fields directly:

```java
class MathUtils {
    static int square(int n) {
        return n * n;
    }
}

System.out.println(MathUtils.square(7));   // 49
```

You've already been using static methods all along — `Math.max(a, b)`, `Integer.parseInt("42")`, and your own `main` method are all static.

The right rule of thumb: if the method's result depends only on its parameters, make it static. If it needs the state of a specific object, make it an instance method.

## Static and instance can coexist

A class often has a mix:

```java
class User {
    static int count = 0;          // class-wide counter
    private String name;           // per-user

    User(String name) {
        this.name = name;
        count++;
    }

    public String getName() {       // instance — reads this.name
        return name;
    }

    public static int howMany() {  // static — reads the class field
        return count;
    }
}

User u1 = new User("Ada");
User u2 = new User("Grace");
System.out.println(u1.getName());     // Ada
System.out.println(User.howMany());   // 2
```

Inside `howMany`, you cannot write `name` — there's no `this`. But you *can* write `count`, because static methods can freely access static fields.

## Constants — `static final`

A constant is a value that never changes. Combine `static` (one copy) with `final` (no reassignment):

```java
class Circle {
    static final double PI = 3.14159265;

    double radius;
    Circle(double r) { this.radius = r; }

    double area() {
        return PI * radius * radius;
    }
}
```

Convention: name `static final` constants in `UPPER_SNAKE_CASE`. The compiler will refuse any code that tries to reassign `Circle.PI`.

<Note>Java's built-in `Math.PI` is itself a `static final double`. Whenever you see `Math.PI`, `Integer.MAX_VALUE`, or `Double.NaN` in code — those are all `static final` constants following exactly this pattern.</Note>

## Common mistakes

- **Reading an instance field from a static method** → "non-static field cannot be referenced from a static context". The static method has no object to read from — it simply doesn't know which object's field you want.
- **Accessing static through an instance reference** → `someUser.count` compiles but reads as if it were a per-object field. Always go through the class name: `User.count`. This is the clearest signal to every reader that the field is shared.
- **Making everything static** → if you find yourself passing the same object repeatedly to static methods, those should probably be instance methods on that class. Static is not "simpler" — it just removes access to object state.
- **Forgetting `final` on constants** → `static int PI = 3.14` is technically reassignable. Use `static final` (and uppercase the name) so the compiler enforces immutability.

## Recap

Static members are the bridge between procedural and object-oriented code. Utility methods (`Math.abs`, `Arrays.sort`), class-wide state (a running counter), and named constants are all natural uses. Instance members are for everything tied to a specific object's identity.

- `static` makes a field or method belong to the class, not to any one instance
- Access static members through the class name (`User.count`, `Math.max(...)`)
- Static methods have no `this` — they can't read instance fields without an explicit object
- Use static for utilities and class-wide state; use instance for behavior tied to a specific object
- `static final UPPER_SNAKE_CASE` is the standard for constants
