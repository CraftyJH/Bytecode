You've been writing fields as if anyone could touch them directly: `dog.age = -5;` is a perfectly legal assignment. That's a problem when those fields need to follow rules — a balance shouldn't go negative, an email needs an `@`, an age can't be a string. **Encapsulation** is the practice of hiding the fields behind methods so the class can enforce its own rules.

Think of a vending machine. You can't reach inside and grab a snack directly — you interact through a defined interface (insert coin, press button). The machine's internal mechanics are hidden. If the machine needs to track inventory or reject insufficient payment, it does so on its own terms. That's encapsulation: a well-defined public interface, hidden internals.

## The `private` modifier

Add `private` before a field, and only the class itself can read or write it:

```java
class Account {
    private int balance = 0;

    public void deposit(int amount) {
        if (amount > 0) {
            balance += amount;
        }
    }

    public int getBalance() {
        return balance;
    }
}
```

From outside `Account`:

```java
Account a = new Account();
a.deposit(100);              // OK — public method
System.out.println(a.getBalance());  // 100

a.balance = 999;             // compile error — balance is private
```

The compiler refuses to let outside code touch `balance` directly. Anyone who wants to change the balance has to go through `deposit`, which contains the validation.

## Public, private, and the default

Java has four access levels. The two you'll use 95% of the time:

| Modifier   | Visible to                              |
| ---------- | --------------------------------------- |
| `private`  | Only the class itself                   |
| `public`   | Everyone                                |
| (none)     | Code in the same package                |
| `protected` | Same package + subclasses (covered with inheritance) |

Default to `private` for fields, `public` for methods you intend to be the class's API. Make private anything that is an implementation detail.

## Getters and setters

When outside code needs to read or write a private field, expose a method:

```java
class Person {
    private String name;
    private int age;

    public String getName()      { return name; }
    public void   setName(String n) { this.name = n; }

    public int  getAge()         { return age; }
    public void setAge(int a) {
        if (a >= 0) this.age = a;     // reject negative ages
    }
}
```

These are called **getters** (`getX()`) and **setters** (`setX(...)`). The naming convention is universal — IDE tools, JSON libraries, and frameworks all rely on it.

A setter is the right place to put validation. If `setAge(-5)` silently rejects the value, no caller can corrupt the object.

## When NOT to add a setter

A getter doesn't have to come with a setter. If a value should never change after construction, leave it without one:

```java
class Person {
    private final String id;       // assigned once, in the constructor
    private String name;

    Person(String id, String name) {
        this.id = id;
        this.name = name;
    }

    public String getId()    { return id; }
    public String getName()  { return name; }
    public void setName(String n) { this.name = n; }
    // no setId() — id is fixed for the lifetime of the Person
}
```

This is **immutability** for a single field. `final` makes the compiler refuse a second assignment, and the lack of a setter prevents anyone from trying.

<Note>In large codebases, blindly generating getters and setters for every field defeats the purpose of encapsulation. Ask: "does the caller legitimately need to change this?" If not, don't add a setter.</Note>

## Common mistakes

- **Public fields with no validation** → if `balance` is public, any caller can write any value, and your invariants are unenforceable. Make fields private by default and add methods only when outside access is genuinely needed.
- **Exposing every field with a getter and setter** → if a field shouldn't change after construction, leave the setter off. Getters and setters aren't free — each one is part of your class's public contract, and changing it later is a breaking change.
- **Getter that returns a mutable internal object** → if `getItems()` returns the actual `List` field, callers can call `.clear()` on it and bypass your encapsulation. Return a copy or a read-only view.
- **Validation in the wrong place** → if every caller has to remember to check `amount > 0`, eventually someone forgets. Put the check inside the setter or mutator method where it's enforced automatically.

## Recap

Encapsulation is the OOP principle that keeps classes manageable as they grow. A well-encapsulated class is a self-contained unit: you can change its internals (how it stores data, what algorithm it uses) without breaking anything outside, because the outside only interacts through the public methods.

- `private` restricts a field or method to the class that declares it
- `public` exposes a method or field to all code
- Default to private fields + public methods — let the class be the gatekeeper of its own state
- A getter exposes a read; a setter exposes a write (often with validation)
- Omit setters for fields that should be immutable after construction; consider `final` to enforce that
