Inheritance gives subclasses their own behavior. **Polymorphism** — literally "many forms" — is what happens when you write code that doesn't care *which* subclass it's working with: one variable type, multiple actual classes, each behaving as itself.

The payoff is code that stays the same while the world around it grows. You write a loop that processes `Animal` objects. Next month someone adds a `Parrot` class. The loop doesn't change — it still calls `speak()`, and the Parrot speaks in its own way. That's polymorphism at work.

## A parent reference can hold any subclass

If `Dog` extends `Animal`, a variable of type `Animal` can point to a `Dog`:

```java
Animal a = new Dog();    // legal — a Dog is an Animal
a.speak();               // dispatches to Dog.speak()
```

The variable `a` has the **declared type** `Animal` but the **runtime type** `Dog`. When you call `a.speak()`, Java looks up the method on the runtime type and runs Dog's override. This is **dynamic dispatch**.

The declared type matters too — it tells the compiler which methods you're allowed to call. If `bark()` is only declared on `Dog` (not `Animal`), then `a.bark()` won't compile, even when `a` actually holds a Dog.

## Why this matters: one loop, many types

The real value of polymorphism shows up when you have a collection of mixed types:

```java
Animal[] zoo = { new Dog("Rex"), new Cat("Whiskers"), new Cow("Bessie") };

for (Animal a : zoo) {
    a.speak();            // dispatches to whichever subclass each element is
}
```

The loop has no idea what's in the array — it just calls `speak()`. Java picks the right override per element. Adding a new subclass (`Sheep`) doesn't require touching this loop at all.

## Static dispatch (overloading) vs dynamic dispatch (overriding)

It's worth contrasting:

- **Overloading** (multiple methods with the same name in the *same* class, different parameters) is resolved at **compile time** based on the argument types.
- **Overriding** (a subclass replaces an inherited method's body) is resolved at **runtime** based on the actual object's type.

```java
class Printer {
    void print(int n)    { System.out.println("int: " + n); }
    void print(String s) { System.out.println("str: " + s); }
}
// overloading: which print runs is decided by the compiler
```

```java
Animal a = new Dog();
a.speak();
// overriding: which speak runs is decided at runtime
```

## Casting back down

Sometimes you genuinely need to call a subclass-only method on a parent-typed variable. You can downcast — but only if the runtime type really is that subclass:

```java
Animal a = new Dog();
if (a instanceof Dog) {
    Dog d = (Dog) a;
    d.bark();              // safe — we just checked
}
```

`(Dog) a` is a cast: a runtime check that throws `ClassCastException` if `a` isn't actually a Dog. Always guard with `instanceof` first. Better: design your method hierarchies so you don't need to cast in the first place.

## Abstract — methods that *must* be overridden

Sometimes the parent class has no useful default behavior — every subclass should provide its own. Mark those methods `abstract`:

```java
abstract class Shape {
    abstract double area();         // no body — every subclass must define it
}

class Circle extends Shape {
    double radius;
    Circle(double r) { this.radius = r; }
    @Override
    double area() {
        return Math.PI * radius * radius;
    }
}
```

An `abstract` class can't be instantiated directly (`new Shape()` is a compile error). It's a contract: any concrete subclass must implement the abstract methods, or the compiler refuses.

<Note>Abstract classes and interfaces (covered later) are how Java enforces contracts at compile time. If you want to guarantee that every Shape has an `area()` method, making it abstract is far more reliable than documentation or convention.</Note>

## Common mistakes

- **Calling a subclass-only method through a parent reference** → the compiler only sees what's declared on the parent type. Either change the variable's type or downcast with an `instanceof` check first. If you're doing this frequently, it's a sign the method should be on the parent.
- **Casting without checking** → `(Dog) someAnimal` throws ClassCastException at runtime if the object isn't actually a Dog. The `instanceof` guard is not optional — it's the safety net that makes casts safe.
- **Confusing overloading with overriding** → overloading varies parameters in the same class; overriding replaces an inherited method in a subclass. They're resolved at different times and behave differently. A common pitfall: `equals(MyClass other)` overloads instead of overriding `equals(Object o)`.
- **Reaching for casts often** → if you find yourself casting all over the place, the design is fighting you. Lift the method up to the parent (or an abstract method) so polymorphism does the work.

## Recap

Polymorphism is the mechanism that lets Java code be open for extension but closed for modification. A loop that processes `Shape` objects works correctly today with Circle and Square, and will work correctly tomorrow when Triangle is added — without a single line of the loop changing.

- A parent-typed variable can hold any subclass instance
- Method calls are dispatched on the runtime type, not the declared type
- The declared type controls which methods are *visible* to the compiler
- `abstract` methods force subclasses to provide their own implementation
- Polymorphism + a parent-typed collection = one loop, any number of behaviours
