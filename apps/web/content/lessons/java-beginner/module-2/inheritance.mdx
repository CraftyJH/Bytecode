When two classes share most of their data and behaviour, you don't have to copy code between them. **Inheritance** lets you declare one class as an extension of another — a `Dog` is an `Animal`, a `Manager` is an `Employee` — and reuse everything the parent already does.

Think of it like specialization in biology. An Animal defines the general traits: every animal has a name, can make a sound, needs food. A Dog is an Animal with all those general traits, plus specific ones: it barks, fetches, and wags its tail. Inheritance in code mirrors this relationship: the subclass gets the parent's fields and methods automatically, then adds its own.

## `extends` — declaring a subclass

Use `extends` to say "this class is a kind of that class":

```java
class Animal {
    String name;
    Animal(String name) {
        this.name = name;
    }
    void speak() {
        System.out.println(name + " makes a sound");
    }
}

class Dog extends Animal {
    Dog(String name) {
        super(name);
    }
}
```

Now `Dog` automatically has the `name` field and the `speak()` method:

```java
Dog rex = new Dog("Rex");
rex.speak();   // Rex makes a sound
```

You wrote no `name` field and no `speak()` method on `Dog`, but the inherited ones work as if you did.

## `super(...)` — calling the parent constructor

A subclass constructor must initialise the parent before it does its own work. The `super(...)` call invokes a parent constructor with whatever arguments it needs:

```java
class Dog extends Animal {
    int age;
    Dog(String name, int age) {
        super(name);          // calls Animal(String) with name
        this.age = age;       // then initialises Dog's own field
    }
}
```

`super(...)` must be the first statement in the constructor. If you don't write one, Java inserts a hidden `super()` call to the parent's no-arg constructor — which is a compile error if the parent doesn't have one.

<Note>The relationship is asymmetric. A Dog is an Animal, but an Animal is not necessarily a Dog. You can write `Animal a = new Dog("Rex");` (a Dog assigned to an Animal-typed variable). You cannot write `Dog d = new Animal("X");` — not every Animal is a Dog.</Note>

## Overriding methods

When the inherited behaviour isn't what you want, **override** it: write a method in the subclass with the same signature as the parent's, and add `@Override` so the compiler can verify it really is replacing something:

```java
class Dog extends Animal {
    Dog(String name) { super(name); }

    @Override
    void speak() {
        System.out.println(name + " says woof!");
    }
}
```

Now:

```java
Animal a = new Animal("Generic");
Dog d = new Dog("Rex");
a.speak();   // Generic makes a sound
d.speak();   // Rex says woof!
```

The `@Override` annotation is optional but strongly recommended. If you mistype the signature (`speek` instead of `speak`, or wrong parameter types), you'll silently create a *new* method instead of overriding — and `@Override` makes the compiler catch that.

## Calling the parent's version

Inside an override, you can still reach the parent's behaviour with `super.methodName(...)`:

```java
class Dog extends Animal {
    Dog(String name) { super(name); }

    @Override
    void speak() {
        super.speak();                 // first do the generic Animal sound
        System.out.println("...woof!"); // then add the Dog flavour
    }
}
```

This is useful when you want to extend the parent's behaviour rather than replace it entirely.

## What's inherited (and what isn't)

A subclass inherits everything from its parent **except**:

- `private` fields and methods — those stay locked inside the parent class
- Constructors — every class declares its own, and chains to the parent via `super(...)`

`public`, `protected`, and package-private members all come along automatically.

<Note>If you need a subclass to access a parent's field directly, declare the field `protected` instead of `private`. `protected` is visible inside the class itself and in all subclasses, making it a middle ground between private and public.</Note>

## Try it

```java
public class Demo {
    public static void main(String[] args) {
        Cat c = new Cat("Whiskers");
        c.speak();   // Whiskers says meow
    }
}

class Animal {
    String name;
    Animal(String name) { this.name = name; }
    void speak() { System.out.println(name); }
}

class Cat extends Animal {
    Cat(String name) { super(name); }
    @Override
    void speak() { System.out.println(name + " says meow"); }
}
```

## Common mistakes

- **Forgetting `super(...)` when the parent has no no-arg constructor** → Java inserts an implicit `super()`. If the parent doesn't have one, you must call `super(args)` explicitly on the first line of the child constructor.
- **Spelling override wrong** → without `@Override`, a typo creates a *new* method silently. The original inherited version keeps running unchanged. Always annotate intended overrides.
- **Trying to inherit `private` members** → private is private even from subclasses. Declare the field `protected` if a subclass legitimately needs direct access, or expose a getter/setter.
- **Inheritance for code reuse alone** → if a Truck isn't really an Engine, don't extend Engine just to share methods. Use composition (a Truck *has* an Engine field) instead. Inheritance should model a genuine "is a" relationship.

## Recap

Inheritance is powerful when used for genuine "is a" relationships. It eliminates duplication and provides a natural place to add shared behaviour. But it creates tight coupling — a change to the parent affects every subclass — so use it judiciously.

- `class B extends A` makes B a subclass of A, inheriting non-private members
- `super(args)` calls a parent constructor; must be the first statement of a child constructor
- Override a method by writing the same signature in the subclass; annotate with `@Override`
- `super.method(...)` calls the parent's version from within an override
- Use inheritance only when there's a real "is a" relationship, not just for code sharing
