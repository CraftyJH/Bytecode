Strings are the workhorse of any program that deals with text — user names, messages, file paths, configuration values. In Java, `String` is a **class**, not a primitive type. That distinction has a practical consequence you'll hit quickly: you cannot compare strings with `==`. Understanding how strings work saves you from one of the most common Java bugs.

## String basics

A String literal is enclosed in double quotes. String objects come with a rich set of built-in methods:

```java
public class StringBasics {
    public static void main(String[] args) {
        String name = "Java Programming";

        System.out.println(name.length());           // 16
        System.out.println(name.toUpperCase());      // JAVA PROGRAMMING
        System.out.println(name.toLowerCase());      // java programming
        System.out.println(name.charAt(5));          // P
        System.out.println(name.substring(5));       // Programming
        System.out.println(name.substring(5, 8));    // Pro
        System.out.println(name.contains("Java"));  // true
        System.out.println(name.replace("Java", "Python")); // Python Programming
    }
}
```

Strings are **immutable** — every method returns a new String rather than modifying the original. `name.toUpperCase()` does not change `name`; it produces a fresh String object.

## == vs .equals()

This is the single most common Java string bug. Use `.equals()` to compare content:

```java
String a = new String("hello");
String b = new String("hello");

System.out.println(a == b);        // false — different objects
System.out.println(a.equals(b));   // true  — same content
```

When you write a string literal like `"hello"`, Java may reuse an existing object from a pool, so `==` can sometimes return `true` by coincidence. Never rely on this — always use `.equals()` for string comparison.

To avoid NullPointerException when one side might be null, put the known-safe string on the left: `"expected".equals(userInput)`.

## String concatenation and StringBuilder

The `+` operator concatenates strings. For simple cases it's fine. When you're building a string inside a loop, use `StringBuilder` to avoid creating many temporary objects:

```java
// Fine for a few concatenations
String greeting = "Hello, " + "World" + "!";

// Better for loop-based building
StringBuilder sb = new StringBuilder();
for (int i = 1; i <= 5; i++) {
    sb.append("item").append(i).append("\n");
}
String result = sb.toString();
System.out.print(result);
```

## String.format()

`String.format()` builds strings using placeholders — cleaner than chaining `+` for complex output:

```java
String name = "Alice";
int score = 95;
double gpa = 3.87;

String report = String.format("Student: %-10s | Score: %3d | GPA: %.2f", name, score, gpa);
System.out.println(report);
// Student: Alice      | Score:  95 | GPA: 3.87
```

Common format specifiers: `%s` for strings, `%d` for integers, `%f` or `%.2f` for floating-point numbers.

## Common mistakes

- **Using `==` to compare strings** → almost always gives wrong results outside the string pool. Use `.equals()` every time.
- **Expecting string methods to mutate** → `s.toUpperCase()` does not change `s`. Capture the return value: `s = s.toUpperCase();`
- **Off-by-one in `substring`** → `substring(start, end)` includes the character at `start` but excludes the one at `end`.
- **Concatenating in a tight loop** → each `+` creates a new object. Use `StringBuilder.append()` when building strings iteratively.

## Recap

- `String` is an immutable class; every method returns a new `String`
- Key methods: `length()`, `charAt()`, `substring()`, `contains()`, `toUpperCase()`, `toLowerCase()`, `replace()`, `trim()`
- Always compare string content with `.equals()`, never with `==`
- Use `StringBuilder` for efficient loop-based string construction
- `String.format()` produces readable formatted output using `%s`, `%d`, `%.2f` placeholders
